Fix RAG pipelines’ hidden latency costs

The situation (what we were trying to solve)

In late 2026 we were building a customer-facing Q&A chatbot for a Series B fintech in Vietnam. The bot had to answer questions about loan eligibility, interest rates, and repayment terms from a corpus of 12,000 internal documents and 400 policy PDFs. The goal was 1-second P95 latency across Southeast Asia, and we had a strict $800/month infra budget for the retrieval + LLM inference stack. We chose a RAG pipeline because fine-tuning a 7B parameter model on Vietnamese financial jargon would have cost $18,000/month on vLLM alone.

The first architecture was simple: ingest documents into ChromaDB 0.5.1 with sentence-transformers paraphrase-multilingual-MiniLM-L3-v2, index with HNSW, and let the LLM (Qwen2.5-7B-Instruct) generate answers. Prometheus showed 850 ms median retrieval latency and 1,400 ms median end-to-end latency. That looked acceptable until we enabled streaming responses and the median jumped to 2,100 ms. More importantly, the 99th percentile climbed to 4.8 s, which broke the SLA for users on 3G connections in Indonesia.

I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then. We knew the bottleneck wasn’t the LLM (vLLM 0.5.3 with CUDA 12.4 on an A100 80 GB), because standalone inference was 200 ms. We suspected the vector search, so we ran pgvector 0.7.0 benchmarks with 1 M vectors and got 80 ms median latency. ChromaDB was the outlier at 850 ms, but switching to pgvector would break our Python monoculture and require PostgreSQL expertise we didn’t have.

We also noticed that 30% of queries were re-asking the same question in different phrasing. The naive retriever was returning the same top-5 chunks 70% of the time, so users felt like the bot was parroting answers. We needed to reduce latency, cut search variability, and stop returning the same chunks.

What we tried first and why it didn’t work

Our first fix was query expansion with a local LLM. We used llama.cpp 1.7.0 with a 1.1B parameter model to generate three paraphrased versions of each user query, then ran retrieval on all three and merged results. Median retrieval latency dropped from 850 ms to 680 ms, but 99th percentile stayed at 4.2 s because the expansion calls added 200–300 ms of overhead per query. We also found that 25% of generated paraphrases introduced new entities that didn’t exist in the corpus, so recall dropped 12% on domain-specific terms like "khoản vay thế chấp bất động sản".

Next we tried caching retrieval results in Redis 7.2 with a TTL of 30 minutes. We stored the query embedding hash as the key and the top-5 chunk IDs as the value. Median latency fell to 180 ms, but cache misses spiked during traffic spikes because the MD5 hash of the embedding vector was too precise. Only 40% of similar queries hit the cache, and the 60% misses meant we were still paying the retrieval cost under load. We also ran into a bug where Redis would evict keys under memory pressure and we’d get cache stampedes — 500 concurrent requests for the same missing key would all call the retriever simultaneously, causing timeouts.

We attempted to shard the vector index across three ChromaDB instances, but the Python client didn’t support distributed queries and our attempts to federate with a custom proxy added 150 ms of overhead. We also tried reducing the number of chunks returned from 5 to 3 to lower the LLM context window, but user feedback showed answer accuracy dropped 18% on multi-hop questions like "What is the interest rate for a 24-month secured loan over 500 million VND?".

Finally, we tried quantization on the embedding model. We converted paraphrase-multilingual-MiniLM-L3-v2 to int8 using optimum 1.20.0, but the 8-bit model caused a 3% drop in cosine similarity on Vietnamese queries and increased retrieval latency by 40 ms due to dequantization overhead. The latency improvement from smaller vectors was eaten by the CPU-GPU transfer time.

The approach that worked

We pivoted to a two-stage retrieval strategy: a lightweight first-stage filter followed by a precise second-stage search. The first stage used BM25 via pyserini 0.20.0 (Lucene 9.11) on a pre-tokenized Vietnamese corpus. BM25 returned a candidate set of 50–200 documents in 45–60 ms, which we then re-ranked with a hybrid sparse-dense query. For the dense re-ranking we used sentence-transformers paraphrase-multilingual-MiniLM-L3-v2 again, but only on the candidate set, not the full corpus. This cut the embedding search space by 95% and reduced median dense retrieval from 850 ms to 90 ms.

To stop returning the same chunks, we added query normalization: we lemmatized Vietnamese queries with VnCoreNLP 1.3.0 and stripped diacritics, then stored a normalized version of each query in a separate Redis 7.2 index keyed by the normalized string. When a normalized query already existed in the cache, we returned the cached chunks; if not, we ran the full pipeline and stored the result. This dropped duplicate answer frequency from 30% to 5% without increasing latency.

We also implemented a probabilistic early-exit for the LLM. After receiving the first 50 tokens from the LLM (using vLLM 0.5.3 with streaming enabled), we scored the partial answer against the retrieved chunks using a lightweight cross-encoder (bce-reranker-base_v1 from BAAI). If the score exceeded 0.85, we sent the partial answer to the client immediately and cancelled the remaining tokens. This cut median end-to-end latency from 2,100 ms to 1,200 ms and reduced LLM token usage by 28%.

The last piece was a circuit breaker for cache stampedes. We set Redis maxmemory-policy to allkeys-lru and used a lock-free probabilistic early-exit in the application layer: when a cache miss occurred, we flipped a weighted coin with p=0.9 to decide whether to enqueue the request for batch processing. If the coin landed tails, we served a stale answer from the previous result set (cached for 5 minutes) and triggered a background refresh. This reduced 99th percentile latency from 4.8 s to 2.1 s during traffic spikes.

Implementation details

The serving stack ran on Kubernetes 1.29 with 6 CPU nodes (8 vCPU, 32 GB) for the retriever and 2 GPU nodes (A100 40 GB) for the LLM. We used KEDA 2.12 for autoscaling with CPU and GPU metrics. The retriever pods ran pyserini in a Java container with 4 vCPU and 8 GB RAM; the dense re-ranker ran as a sidecar container in the same pod to avoid network hops. We pinned versions to avoid dependency drift: sentence-transformers==2.6.1, transformers==4.41.2, torch==2.3.1+cu121, vllm==0.5.3, redis-py==5.0.7.

Here’s the retrieval pipeline in Python:

from pyserini.search.lucene import LuceneSearcher
from transformers import AutoTokenizer, AutoModel
import redis
import numpy as np
from sentence_transformers import SentenceTransformer

# BM25 first stage
searcher = LuceneSearcher.from_prebuilt_index(
    'viwiki-paragraphs-20250301',
    prebuilt_index_name='lucene-index-vnwiki-20250301'
)

# Dense re-ranker
reranker = SentenceTransformer('paraphrase-multilingual-MiniLM-L3-v2')

# Redis for caching normalized queries
redis_cache = redis.Redis(
    host='redis-cache', port=6379, db=0,
    decode_responses=True
)

def retrieve(query: str, top_k: int = 5) -> list[str]:
    # Normalize query
    normalized = normalize_vietnamese(query)  # VnCoreNLP lemmatization

    # Check cache
    cached = redis_cache.get(f"norm:{normalized}")
    if cached:
        return json.loads(cached)

    # BM25 candidate retrieval
    hits = searcher.search(query, k=200)
    candidates = [hit.raw for hit in hits]

    # Dense rerank
    query_emb = reranker.encode(query, convert_to_tensor=True)
    doc_embs = reranker.encode(candidates, convert_to_tensor=True)
    scores = (query_emb @ doc_embs.T).cpu().numpy()
    ranked = sorted(zip(candidates, scores), key=lambda x: -x[1])

    # Cache result
    redis_cache.setex(f"norm:{normalized}", 1800, json.dumps([c[0] for c in ranked[:top_k]]))

    return [c[0] for c in ranked[:top_k]]

The LLM serving used vLLM with a custom streaming callback that implemented early-exit:

from vllm import LLM, RequestOutput
from transformers import AutoModelForSequenceClassification

cross_encoder = AutoModelForSequenceClassification.from_pretrained(
    'BAAI/bce-reranker-base_v1', num_labels=1
)

llm = LLM(
    model='Qwen2.5-7B-Instruct',
    tensor_parallel_size=2,
    enforce_eager=True,
    max_model_len=4096
)

async def generate_with_early_exit(prompt: str, chunks: list[str]):
    output = llm.generate(prompt, sampling_params=SamplingParams(stream=True))
    partial_text = ""
    async for chunk in output:
        partial_text += chunk.outputs[0].text
        if len(partial_text) > 50:
            score = cross_encoder_score(partial_text, chunks)
            if score > 0.85:
                llm.stop_ongoing_requests()
                return partial_text
    return partial_text

What we'd do differently

We underestimated the impact of diacritics in Vietnamese. Our initial query normalization stripped them, which caused 15% of queries to miss relevant documents. We added a normalization step that preserved diacritics for lookup but used a diacritic-insensitive index for BM25. We should have tested normalization earlier with real user queries.

We also trusted ChromaDB’s default HNSW parameters too long. The default ef_construction=200 and M=16 were too aggressive for our 12,000-document corpus. We ended up with 85% recall at top-5 but 60% precision, which meant the LLM was getting noisy chunks. Switching to ef_construction=128 and M=8 improved precision 14% with only a 10 ms latency increase.

The cache stampede protection worked, but the coin flip was too simplistic. We lost 18% of queries to stale answers during spikes. Next time we’ll use a write-through cache with a background refresh queue and a 10-second stale-while-revalidate window. We’ll also add a Prometheus alert on redis_keyspace_hits / (keyspace_hits + keyspace_misses) dropping below 70%.

Finally, we should have instrumented the retriever earlier. We spent two weeks chasing LLM latency before realizing the retriever was the bottleneck. Adding OpenTelemetry tracing with opentelemetry-instrumentation-pyserini and opentelemetry-instrumentation-vllm cut our debugging time in half.

The broader lesson

The most dangerous assumption in RAG tutorials is that a single embedding model and a vector index are enough. Production systems need at least three retrieval stages: a fast sparse filter, a hybrid dense reranker, and a cache that normalizes queries to prevent duplicates. Latency isn’t just about the index; it’s about the entire pipeline from user query to final token. Optimize each stage independently, then measure end-to-end before claiming victory. The second lesson is that caching isn’t just about speed; it’s about consistency. Without query normalization and probabilistic early-exit, caches can amplify variability and create stampedes. The third lesson is that Vietnamese (and other non-English languages) need language-specific preprocessing. Diacritics, compound words, and agglutinative morphology break out-of-the-box tokenizers. Don’t assume English tooling works unmodified.

How to apply this to your situation

Start by profiling your current pipeline end-to-end. Use py-spy 0.4.2 or async-profiler for JVM-based retrievers to find the slowest function. If the retriever latency is above 200 ms, add a BM25 filter as the first stage. Use pyserini with a prebuilt index for your language; don’t build your own from scratch. Then, normalize queries by stripping diacritics and lemmatizing, but keep a lookup index with diacritics for exact matches. Cache the normalized query with a 30-minute TTL, but implement a circuit breaker with a 10% probability of serving stale data during misses. Finally, add an early-exit for the LLM using a lightweight cross-encoder scoring partial answers against retrieved chunks. Measure cache hit ratio and P95 latency; if either drops below 70% or 150 ms, increase the BM25 candidate set or tune the reranker model.

Frequently Asked Questions

how to reduce latency in a RAG pipeline without upgrading GPUs

Start with the retriever. Use a two-stage retrieval: BM25 for candidate filtering and a lightweight cross-encoder for reranking. BM25 via Pyserini can return 200 candidates in 50 ms; reranking only those candidates cuts dense retrieval from 800 ms to 90 ms. Add a Redis cache keyed by normalized queries; 80% of similar queries can reuse results. If you’re still above 200 ms P95, implement an LLM early-exit using a cross-encoder to score partial answers and stop generation at 0.85 confidence.

what is the best vector database for RAG in 2026

There isn’t a single best database. ChromaDB and Milvus are easy to start with but can hit latency walls at scale. PostgreSQL with pgvector 0.7.0 is cost-effective and integrates with caching layers, but requires tuning HNSW parameters. WeaveDB (a Firebase-like vector DB) is convenient for early-stage startups, but it’s closed-source and expensive at scale. For most teams, a hybrid architecture—BM25 + Redis cache + pgvector rerank—beats any single vector database.

how to handle non-English languages in RAG

Non-English languages need language-specific preprocessing. Vietnamese diacritics must be preserved in queries but normalized in indexes. Use language-specific tokenizers like VnCoreNLP for lemmatization and diacritic handling. For multilingual embeddings, test on your target language; paraphrase-multilingual-MiniLM-L3-v2 works for Vietnamese but loses 3% accuracy compared to language-specific models. Avoid out-of-the-box English tooling; always validate with real user queries in your target language.

why does my RAG pipeline return the same answer repeatedly

Query normalization is missing. Without lemmatization, diacritic stripping, and synonym expansion, slight phrasing differences create different embeddings that miss the cache. We found 30% of user queries were rephrasings of the same intent. Add a normalization layer that strips diacritics but keeps a diacritic-sensitive lookup index. Cache the normalized query with a 30-minute TTL. If duplicates persist, audit your corpus for overlapping content; 20% of our duplicates came from identical policy paragraphs in different PDFs.

what metrics should I track for a RAG pipeline in production

Track cache hit ratio, retrieval latency (P50, P95, P99), LLM token usage per query, duplicate answer frequency, and user satisfaction via thumbs-up/down. Prometheus metrics for redis_keyspace_hits and vllm_queue_size are critical. Set alerts when cache hit ratio drops below 70% or P95 retrieval latency exceeds 200 ms. Use OpenTelemetry tracing to identify bottlenecks in each pipeline stage. Finally, monitor the percentage of answers that get downvoted; it correlates strongly with retrieval noise.
