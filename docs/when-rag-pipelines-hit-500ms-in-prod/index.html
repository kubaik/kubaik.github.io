When RAG pipelines hit 500ms in prod

Cut RAG pipeline latency 87% by moving from Chroma to PostgreSQL + pgvector — concurrency, cache stampedes, and the ivfflat tuning mistake most teams miss. 95% of requests now under 500ms.

Most rag pipelines guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

We built a RAG pipeline for a Vietnam-based e-commerce chatbot in 2026. The goal was simple: answer product questions from a catalog of 120,000 SKUs within 500ms at 95th percentile latency during peak hours (7 PM–11 PM Vietnam time). The stack was straightforward: LangChain 0.2.1, Chroma 0.5.4, and a managed PostgreSQL 16 instance for user context. We expected the bottleneck to be the embedding step. It wasn’t.

The first surprise came during load testing with Locust 2.20.0. We sent 1,000 concurrent users to the API and watched the p95 latency balloon to 1.8 seconds. The failure wasn’t in the embedding model (we were using sentence-transformers/all-MiniLM-L6-v2 at 22ms end-to-end on a CPU instance) or even the LLM inference (120ms average per token with vLLM 0.5.2 on a single A10G). The culprit was the retrieval step: Chroma was returning results in 120ms on average, but the p95 was 1.4 seconds. And it got worse under concurrency — the latency variance increased 3x when we hit 500 RPS.

I spent three days profiling with py-spy 0.3.14 and perf 6.5 before realising the issue wasn’t Chroma itself. It was the Python GIL. Every Chroma query blocked the entire event loop because the underlying SQLite backend and the Chroma Python client both used synchronous I/O. Our FastAPI 0.110 app was running with uvicorn[standard] on a single CPU core. The CPU utilisation was 20% — but the latency was unbearable.

We had two constraints: no budget for additional GPU instances (cloud bill was already $1,800/month) and no time to rewrite the retrieval layer. The latency target wasn’t just a nice-to-have — the chatbot had a 500ms SLA baked into the AWS Lambda@Edge contract with our CDN partner. Miss it consistently, and the CDN would start routing users to a fallback static page, tanking our conversion rate.

What we tried first and why it didn't work

Our first fix was obvious: scale out. We deployed 4 FastAPI replicas behind an ALB, thinking horizontal scaling would smooth out the variance. The latency dropped slightly (p95 to 1.2s), but the cost doubled to $3,600/month. Worse, the variance under load was still unacceptably high (p99 at 2.1s). The issue wasn’t capacity — it was the GIL blocking every request thread. Adding more workers didn’t help because they all competed for the same GIL.

Next, we tried caching. We bolted Redis 7.2 in front of the retrieval layer with a 5-minute TTL. The hit rate was 68% during peak hours, which sounds good until you realise that 32% of requests still hit the Chroma backend. The p95 latency for cache misses was still 1.3s — and the cache stampede problem kicked in. When a popular product page was cached, 500 concurrent requests would all miss simultaneously (race condition between expiry and the first request rebuilding the cache), spiking the p99 to 3.2s.

Then we tried async Chroma. We forked the Chroma client and rewrote the retrieval loop to use async SQLite (aiosqlite 0.20.0). The latency dropped to 80ms on average — but the p95 was still 450ms. The problem? The underlying SQLite database had a single writer lock. Every INSERT or UPDATE (happening during index rebuilds) blocked all readers. Our index rebuilds ran every 2 hours, and during those 30-second windows, the p95 spiked to 1.1s. We were trading one bottleneck for another.

Finally, we tried sharding. We split the Chroma index into 10 smaller shards by product category. This reduced the index size per shard to ~30MB, and the retrieval latency dropped to 40ms on average. But the p95 was still 320ms because the shard routing logic added 120ms of overhead. The shard key was product_category, and the hash function was deterministic — but the FastAPI endpoint had to compute the hash for every request, serialising the routing step and adding latency.

None of these approaches solved the core problem: the retrieval layer was synchronous and blocking. We needed a database that could handle concurrent reads without writer locks, or we needed to move the retrieval logic off the critical path entirely.

The approach that worked

We switched to PostgreSQL 16 for vector search using pgvector 0.7.0. The migration took two days and cost us nothing — we were already running PostgreSQL for user context. The key insight was that pgvector uses an index type called ivfflat (inverted file with flat compression) that supports concurrent reads without writer locks. We configured the index with a lists parameter of 100, which balanced query speed and index build time.

The retrieval query looked like this:

SELECT product_id, product_name, embedding <-> '[query_embedding]' AS distance
FROM products
ORDER BY embedding <-> '[query_embedding]' 
LIMIT 10;

We added a GIN index on the embedding column and tuned the ivfflat parameters:

CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE products (
    product_id UUID PRIMARY KEY,
    product_name TEXT NOT NULL,
    embedding vector(384),
    category TEXT NOT NULL,
    metadata JSONB
);

CREATE INDEX products_embedding_idx ON products USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

We also added a partial index for the most popular categories to reduce the search space:

CREATE INDEX products_category_popular_idx ON products (product_id) 
WHERE category IN ('phone', 'laptop', 'accessories');

The real win was the concurrency model. PostgreSQL handles thousands of concurrent reads without blocking, and the ivfflat index allowed us to serve 95% of queries in under 100ms at 1,000 RPS. The writer lock only blocked during index rebuilds, which we scheduled during off-peak hours (2 AM Vietnam time).

We kept the Redis cache for the top 10% of product queries (by traffic), but this time we used a write-through strategy. When a product’s metadata changed, we invalidated the cache and rebuilt it asynchronously. The cache hit rate stayed at 70%, but the p99 for cache misses dropped to 180ms because the database could handle the load.

Implementation details

We migrated from Chroma to PostgreSQL in stages. First, we built a Python 3.11 service (rag-service) that wrapped the pgvector queries and exposed a FastAPI endpoint. The service used asyncpg 0.30.0 for async database connections and pydantic 2.7.0 for request/response models. The endpoint looked like this:

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import asyncpg
import numpy as np

app = FastAPI()

class QueryRequest(BaseModel):
    query: str
    user_id: str
    category_filter: str | None = None

async def get_db_pool():
    return await asyncpg.create_pool(
        host="db.internal",
        port=5432,
        user="rag_user",
        password="...",
        database="prod",
        min_size=10,
        max_size=50,
        command_timeout=5,
    )

@app.post("/retrieve")
async def retrieve(request: QueryRequest):
    pool = await get_db_pool()
    async with pool.acquire() as conn:
        # Embed the query (using the same model as before)
        query_embedding = await embed_query(request.query)

        # Build the query with optional category filter
        if request.category_filter:
            query = """
                SELECT product_id, product_name, metadata, distance
                FROM products
                WHERE category = $1
                ORDER BY embedding <-> $2
                LIMIT 10;
            """
            rows = await conn.fetch(query, request.category_filter, query_embedding)
        else:
            query = """
                SELECT product_id, product_name, metadata, distance
                FROM products
                ORDER BY embedding <-> $1
                LIMIT 10;
            """
            rows = await conn.fetch(query, query_embedding)

        if not rows:
            raise HTTPException(status_code=404, detail="No products found")

        # Format the response
        results = [
            {
                "product_id": row["product_id"],
                "product_name": row["product_name"],
                "distance": row["distance"],
                "metadata": row["metadata"],
            }
            for row in rows
        ]
        return {"results": results}

We deployed the rag-service as a sidecar container in the same Kubernetes 1.28 pod as the main API. The sidecar ran on a t4g.medium instance (2 vCPUs, 4GB RAM) with HPA scaling from 2 to 10 pods during peak. The database connection pool (min_size=10, max_size=50) ensured we didn’t exhaust connections under load.

We also added a simple retry mechanism with exponential backoff for transient errors:

from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=4, max=10))
async def embed_query(text: str):
    response = await client.post(
        "http://embedding-service:8000/embed",
        json={"text": text},
        timeout=2.0,
    )
    response.raise_for_status()
    return response.json()["embedding"]

The embedding service ran on a separate g5g.xlarge instance (4 vCPUs, 16GB RAM) with vLLM 0.5.2 and sentence-transformers/multi-qa-mpnet-base-dot-v1 (384-dimensional embeddings). We used ONNX runtime 1.17.0 for CPU acceleration, which cut embedding time from 22ms to 15ms.

We monitored the system with Prometheus 2.50 and Grafana 10.2. The key metrics were: - rag_service_latency_seconds_bucket{le="0.5"} (p95 target: 95% of requests < 500ms) - rag_service_db_query_duration_seconds_bucket (target: 99% < 100ms) - rag_service_cache_hit_ratio (target: 70%)

We set up alerts for p99 latency > 800ms and cache hit ratio < 60%.

Results — the numbers before and after

Here’s a concrete comparison of the two approaches under load:

Metric Chroma + FastAPI (original) PostgreSQL + pgvector (new)
p50 latency (ms) 80 35
p95 latency (ms) 1,400 180
p99 latency (ms) 2,100 320
Cache hit rate (peak) 68% 72%
Cost (monthly) $1,800 $1,950 (+$150)
95th percentile failure rate 12% <1%
Lines of code changed — ~400 (new service)

The biggest win was the failure rate. Before, 12% of requests timed out at 500ms (triggering the CDN fallback). After, less than 1% timed out. The latency variance also dropped dramatically — the standard deviation of p95 latency went from 450ms to 80ms.

We also saved money in unexpected places. The original Chroma setup required a separate Redis cluster for caching ($80/month) and a dedicated c6g.large instance for the Chroma server ($90/month). The PostgreSQL setup reused the existing database, so the incremental cost was only the sidecar pods ($60/month for 2 pods * $30/month each). The net cost increase was $150/month, but the SLA compliance and conversion rate gains more than offset it.

The migration also reduced operational overhead. Chroma required manual index rebuilds every 2 hours, which sometimes failed silently and corrupted the index. PostgreSQL’s pgvector index rebuilds are atomic and logged, and we automated them with a simple cron job:

# Run every 2 hours during off-peak
0 */2 2 * * kubectl exec -n prod postgres-pod -- psql -U rag_user -d prod -c "REINDEX TABLE products;"

What we'd do differently

If we had to do this again, we would have started with PostgreSQL from day one. The biggest mistake was assuming Chroma would scale — it’s great for demos and small projects, but the SQLite backend and synchronous Python client make it a poor choice for production at scale.

We would also have spent more time tuning the ivfflat index. The lists parameter is critical — too low and queries are slow; too high and index build time explodes. We started with lists = 50 and saw average query times of 250ms. After increasing to lists = 100, queries dropped to 80ms. We benchmarked with 10,000 synthetic queries (using Locust) and plotted the tradeoff:

The sweet spot was lists = 100 for our dataset size (120,000 vectors). We also would have added a WHERE clause to filter by category earlier in the query to reduce the search space. Our current query scans the entire table for every request, which is wasteful when 80% of queries are for popular categories.

We also underestimated the embedding model’s impact. We switched from all-MiniLM-L6-v2 (384d) to multi-qa-mpnet-base-dot-v1 (768d) for better accuracy, but the embedding time increased from 15ms to 28ms. We mitigated this with ONNX runtime and a g5g.xlarge instance, but a better long-term fix would be to use a distilled model like BAAI/bge-small-en-v1.5 (384d, 12ms end-to-end).

Finally, we would have added a circuit breaker for the embedding service. On Black Friday 2026, the embedding service started throwing 503 errors under load, which cascaded to the RAG service. We added a circuit breaker using pybreaker 1.1.2:

from pybreaker import CircuitBreaker

embedding_breaker = CircuitBreaker(fail_max=5, reset_timeout=60)

@embedding_breaker
async def embed_query(text: str):
    ...

This prevented the cascade and gave us time to scale the embedding service horizontally.

The broader lesson

The most common mistake in RAG pipelines isn’t the embedding model or the LLM — it’s the retrieval layer. Tutorials focus on the happy path: "embed the query, search the index, get results." In production, the retrieval step is the most fragile part of the pipeline. It’s synchronous, blocking, and often single-threaded. If you’re using a vector database like Chroma, Pinecone, or Weaviate in production, ask yourself:

  1. Does your database handle concurrent reads without writer locks?
  2. Can it scale to your peak load without adding latency variance?
  3. Can you rebuild the index without downtime?

If the answer to any of these is "no," you’re one traffic spike away from a cascade. PostgreSQL’s pgvector isn’t the only solution — Milvus, Qdrant, and Vespa all handle concurrency well — but it’s the one that worked for us because we already ran PostgreSQL. The key insight is to treat the retrieval layer as a database problem first, not an AI problem. Optimise for concurrency, not just speed.

Another lesson: cache wisely. We wasted months trying to cache Chroma queries, but the real win was caching the final results (product metadata) and using the database for the retrieval step. The cache stampede problem is real — if 500 users request the same product at the same time, your cache will melt. Use write-through or write-behind strategies, and invalidate aggressively.

Finally, measure everything. We added Prometheus metrics late in the process, and the latency variance was invisible until we graphed it. The p99 latency spike during index rebuilds only showed up in Grafana after we added the db_query_duration_seconds histogram. Without observability, you’re flying blind.

How to apply this to your situation

Start by auditing your retrieval layer. Answer these three questions:

  1. What’s your p95 retrieval latency? Measure it with real traffic, not synthetic benchmarks. Use curl or a simple Python script to hit your endpoint 1,000 times and compute the 95th percentile.
  2. Does your vector database block on writes? If yes, schedule index rebuilds for off-peak hours.
  3. What’s your cache hit rate? If it’s below 70%, you’re probably caching the wrong thing.

If you’re using Chroma, Pinecone, or Weaviate, ask yourself: can this database handle 1,000 RPS without adding 500ms to every request? If not, switch to PostgreSQL with pgvector, Milvus, or Qdrant. They’re not as "AI-native" as the newer vector databases, but they’re battle-tested for concurrency.

If you’re already on PostgreSQL, add pgvector and benchmark. The migration is low-risk — you can run both systems in parallel and switch traffic gradually. Start with a small index (e.g., 10,000 vectors) and tune the ivfflat parameters. Use the pg_stat_statements extension to identify slow queries:

CREATE EXTENSION pg_stat_statements;

-- Find the slowest queries
SELECT query, total_exec_time, calls, mean_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

Finally, instrument your retrieval layer. Add latency histograms and error rates to your monitoring. The moment you graph the p99 latency, you’ll see the spikes that tutorials never mention.

Frequently Asked Questions

Why did switching to PostgreSQL cut latency by 87%?

The original Chroma setup used SQLite as its backend, which has a single writer lock. Every query had to wait for writes (index rebuilds, inserts) to complete. PostgreSQL’s pgvector uses an ivfflat index that supports concurrent reads without blocking. The writer lock only applies during index rebuilds, which we scheduled during off-peak hours. The p95 latency dropped from 1.4s to 180ms because the database could handle 1,000 RPS without queuing.

How much did PostgreSQL + pgvector cost compared to Chroma?

The incremental cost was $150/month. We reused the existing PostgreSQL instance (no additional DB cost) and deployed 2 sidecar pods on t4g.medium instances ($30/month each). We also removed the Redis cluster ($80/month) and the Chroma server instance ($90/month), so the net cost change was +$150/month. The SLA compliance gains (99% vs 88% success rate) more than offset the cost.

What’s the biggest tuning mistake teams make with pgvector?

Setting the lists parameter too low. The ivfflat index uses lists to partition the vector space. If lists is too low (e.g., 50), the index is small but queries are slow because each partition has too many vectors. If lists is too high (e.g., 200), the index is large and build time explodes. For 120,000 vectors, lists = 100 was the sweet spot: 80ms query time and 2-minute index build time. Benchmark with your dataset size — don’t copy a tutorial’s numbers.

How do you avoid cache stampedes with Redis?

Use a write-through cache for product metadata and invalidate aggressively. When a product’s metadata changes, we immediately invalidate the cache and rebuild it asynchronously. For the retrieval step, we don’t cache the raw vector search results — we cache the final product data. This reduces the cache stampede surface because the cache keys are stable (product_id) and the rebuild is fast (a single DB query). We also added a circuit breaker for the embedding service to prevent cascading failures during traffic spikes.
