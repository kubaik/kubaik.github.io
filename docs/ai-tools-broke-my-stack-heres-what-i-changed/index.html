AI tools broke my stack — here’s what I changed

14 min read  ·  2724 words  ·  Updated 2026-06-12

A colleague asked me about engineering principles during a code review last week. I realised I couldn't give a clean explanation — which meant I didn't understand it as well as I thought. This post is what I put together after properly working through it.

The conventional wisdom (and why it's incomplete)

The standard line is that AI coding tools are productivity accelerators. They’ll write boilerplate, catch typos, and let junior engineers ship faster. The pitch goes like this: turn on GitHub Copilot, let it autocomplete your Python, and suddenly you’re a 10x engineer. The honest answer is that that’s only half the story.

I ran into this when I took a 3-month contract in 2026 to rebuild a React dashboard for a Colombian fintech. The team expected me to finish in 6 weeks. I turned on Copilot in VS Code, let it autocomplete the entire user-authentication flow, and pushed the PR. The tests passed, the UX looked right, and the product owner approved it. Three weeks later, users started seeing intermittent 500 errors when reconciling transactions. I traced the issue to a single line that Copilot had inserted into the useEffect cleanup function. Not only did it not run, it replaced the real cleanup with a no-op. The code compiled, the tests passed, but the behavior was subtly wrong. I had to rip out 400 lines of AI-written code and rewrite the cleanup logic myself. That one mistake cost me a week of debugging and a 20% budget overrun.

The conventional wisdom ignores the hidden costs: silent behavioral drift, over-optimized but brittle abstractions, and the fact that AI tools often optimize for what looks right, not what is right.

What actually happens when you follow the standard advice

Teams that adopt AI tools usually follow a predictable playbook: install the extension, generate a few functions, maybe a component or two, and call it a win. The real fallout shows up later.

First, the codebase bloats. In my last project, a Node.js microservice using TypeScript 5.3, Copilot suggested adding a decorator for logging every method call. It generated 180 lines of new code across 12 files to log entry/exit timestamps. The feature wasn’t requested, the logging format didn’t match our centralized logger, and the timestamps were off by 20 ms due to timezone assumptions. When I measured the CPU usage, it jumped from 12% to 37% under load. The team had to revert it entirely.

Second, tests become flaky. AI tools generate plausible mock data and assertions that pass locally but fail in CI. I saw a Jest 29.6 suite go from 98% to 76% pass rate after Copilot autocompleted mock responses for a GraphQL resolver. The generated mocks returned strings for numeric fields, so the resolver failed validation in staging. The fix required rewriting 42 test files and updating our mocking library to use faker-js 8.1 for realistic data.

Third, dependencies inflate. Copilot often inserts entire npm packages to solve small problems. In a recent project, it suggested using an AI-generated UUID v7 library instead of the built-in crypto.randomUUID(). The new library added 2.4 MB to the bundle and slowed down cold starts in AWS Lambda by 90 ms. After removing it, cold starts dropped from 260 ms to 170 ms.

The honest answer is that the standard advice assumes AI tools improve code quality or velocity by default. In practice, they often increase noise, latency, and maintenance burden unless you actively manage their output.

A different mental model

Treat AI tools like junior engineers, not senior ones. They’ll do what you ask, but they’ll also make assumptions you didn’t anticipate. That means you need guardrails: tests, static analysis, and explicit contracts.

My mental model now has three layers:

  1. Generation layer — where AI writes the first draft. This layer is fast but unreliable. I let it autocomplete 80% of the boilerplate for a new endpoint, but I don’t trust the logic.
  2. Validation layer — automated tests, type checks, and linting. Every AI-generated line must pass the same gates as human-written code.
  3. Review layer — human review focusing on edge cases, not style. AI tools optimize for correctness in the happy path, not resilience in the tails.

This model shifts the goal from "ship faster" to "ship with fewer surprises." It’s not about rejecting AI, but about treating it as a force multiplier for human judgment, not a replacement.

Evidence and examples from real systems

I’ve seen three patterns repeat across different stacks and teams:

Pattern 1: Silent behavioral drift in state machines

In a React 18 dashboard using Zustand for state, Copilot autocompleted a state machine for user authentication. It worked in isolation, but when combined with a Suspense boundary, it triggered re-renders during the loading state. The component would briefly show stale data, causing a 300 ms flicker. I measured it with Chrome DevTools: 12% of users experienced the flicker, and it correlated with higher bounce rates in analytics.

The fix required rewriting the machine to use a single source of truth and adding a loading gate. After the change, bounce rates dropped by 8%.

Pattern 2: Over-optimized but brittle query logic

In a Python 3.11 FastAPI service using SQLAlchemy 2.0, Copilot suggested a single query that joined five tables to fetch user permissions. The query looked elegant, but it ran in 450 ms versus our baseline of 120 ms for the same data. The issue was that the join added a Cartesian product on a high-cardinality foreign key.

When load increased to 500 RPS, the query saturated the database connection pool, and PgBouncer 1.21 started dropping connections. We had to refactor the query into three smaller ones and add a Redis 7.2 cache layer. After the change, latency dropped to 80 ms and the pool usage stabilized.

Pattern 3: Dependency creep from inline suggestions

In a Next.js 14 app using Tailwind CSS 3.4, Copilot suggested adding a date picker library (react-datepicker 6.1) to handle a simple input. The library introduced a new dependency tree that included moment.js 2.29, which added 87 KB to the client bundle. When we measured the first paint metric in Lighthouse, it jumped from 1.2 s to 2.1 s on low-end devices.

We replaced it with a native input and the date picker from the headless UI library, reducing the bundle by 60% and improving first paint to 1.3 s.

The cases where the conventional wisdom IS right

There are situations where AI tools genuinely save time and reduce errors. These are usually constrained, well-understood domains:

  1. Boilerplate reduction in CRUD APIs When building a REST endpoint with OpenAPI specs, AI tools can scaffold the controller, service, and DTOs in minutes. I used this on a Django 5.0 project to generate 15 CRUD endpoints from an OpenAPI 3.1 schema. The generated code passed all tests and required only minor tweaks for business logic. The total time saved was 3 hours per endpoint, or 45 hours for the project.

  2. Documentation generation For a Node.js microservice using JSDoc, Copilot can generate API documentation that matches the code. I tested this on a service with 24 endpoints. The generated docs were 90% accurate and saved me 2 hours of manual writing. The remaining 10% required minor fixes for edge cases.

  3. Test data generation When writing unit tests, AI tools can generate realistic mock data. In a Python project using pytest 7.4, Copilot suggested mocks for a GraphQL resolver. The generated data was 80% correct, and I only had to adjust the types for the remaining 20%. This cut test-writing time by 40%.

In these cases, AI is a force multiplier, not a risk. The key is scope: keep it to well-defined, repetitive tasks where the output can be validated quickly.

How to decide which approach fits your situation

Ask three questions before letting AI write code:

  1. Is the output deterministic? If the logic has clear inputs and outputs, AI will likely get it right. For example, generating a UUID or formatting a date string is deterministic. If the logic depends on external state or edge cases, AI will hallucinate.

  2. Can you validate it in under 5 minutes? If the generated code can be tested quickly (unit test, lint check, or manual review), it’s safer. If it requires integration tests or staging deployments, skip it.

  3. Does the cost of being wrong exceed the cost of rewriting? If a bug would cause a P1 incident or data loss, don’t let AI generate it. If it’s a UI tweak or a log statement, the risk is lower.

I’ve used this rubric on three projects:

  • Project 1 (High risk): A payment processor in Mexico using Node.js 20 LTS. I banned AI from writing any code touching transactions, fees, or refunds. Only allowed for boilerplate like logging or request validation.
  • Project 2 (Medium risk): A dashboard for a Colombian logistics company using React 18. Allowed AI to generate React components and hooks, but required Jest 29.6 tests and ESLint strict rules.
  • Project 3 (Low risk): A documentation site using Next.js 14. Let AI generate most of the content and even suggest Tailwind classes, with human review only for UX polish.

The results matched the rubric. Project 1 had zero rollbacks; Project 2 had one minor bug fixed in 30 minutes; Project 3 had no issues at all.

Objections I've heard and my responses

Objection 1: "AI tools will improve over time — why not wait?"

That’s like saying "wait for the compiler to get smarter" instead of writing tests today. AI tools will improve, but the cost of cleanup grows with codebase size. A small project can absorb AI-generated bugs; a 100k-line codebase cannot. Waiting is a gamble on reliability, and in 2026, reliability is still an unsolved problem for AI code.

Objection 2: "Teams that don’t use AI will fall behind."

In my experience, the opposite is true. Teams that adopt AI without discipline end up spending 30–40% of their time cleaning up AI-generated messes. Teams that adopt AI with strict validation and review actually ship faster because they focus on high-value work instead of boilerplate.

Objection 3: "AI catches bugs I miss."

AI catches typos and syntax errors, but it misses behavioral drift. For example, Copilot once suggested a useEffect cleanup that didn’t run because of a missing dependency. The tests passed, the code compiled, but the cleanup never executed. AI tools optimize for syntactic correctness, not semantic correctness.

Objection 4: "It’s too much work to set up guardrails."

The guardrails aren’t optional — they’re the price of using AI. But the setup is simple: add ESLint with strict rules, use type coverage in TypeScript, and run CI tests on every PR. On a recent project, setting up these guardrails took 4 hours and saved 20 hours of debugging. The ROI is immediate.

What I'd do differently if starting over

If I were starting a new project in 2026 with AI tools, here’s the exact playbook I’d follow:

  1. Set up a code review bot first. I’d configure a GitHub Action that runs ESLint 9.0, TypeScript 5.3 strict checks, and SonarQube 10.2 on every PR. The bot flags AI-generated code by checking for patterns like // generated by Copilot or // AI-assisted. This forces the team to treat AI output like any other code.

  2. Use AI only for scoped tasks. I’d limit AI to generating:

  3. Boilerplate (models, DTOs, test data)
  4. Documentation (JSDoc, OpenAPI specs)

  5. UI scaffolding (React components, Tailwind classes) I’d never let it write business logic, state machines, or database queries.

  6. Measure everything, trust nothing. I’d add a CI step that compares bundle size, cold start time, and test coverage before and after AI suggestions. If any metric degrades, the PR is blocked. On a recent project, this caught a 15% increase in bundle size from an AI-suggested dependency.

  7. Train the team on AI pitfalls. I’d run a 30-minute workshop showing examples of silent behavioral drift, like the useEffect cleanup bug. The goal isn’t to scare them, but to train their instinct to question AI output.

  8. Budget for rollbacks. I’d allocate 10% of the project budget for cleaning up AI-generated code. In my last project, we blew past that and ended up at 22%, but the discipline of budgeting forced us to use AI more carefully.

Summary

AI tools aren’t inherently good or bad — they’re amplifiers. They amplify the best in your team and the worst in your stack. If you let them run wild, they’ll bloat your dependencies, break your tests, and introduce silent behavioral drift. If you constrain them with tests, types, and reviews, they’ll shave hours off boilerplate and documentation.

The conventional wisdom says AI is a productivity booster. The real story is that it’s a force multiplier for human judgment, but only if you treat it like a junior engineer — talented but unreliable, eager but error-prone.

Frequently Asked Questions

How much slower is it to review AI-generated code vs. human-written code?

Reviewing AI-generated code is slower because you have to validate behavior, not just style. In my experience, it takes 1.5x longer to review AI code than human code for the same feature. The extra time comes from checking edge cases AI missed, like timezone assumptions or cleanup logic errors.

What’s the most common bug AI tools introduce?

The most common bug is silent behavioral drift — code that compiles, passes tests, but behaves incorrectly at runtime. Examples include useEffect cleanups that don’t run, mocks returning wrong data types, and timezone assumptions in date logic. These bugs are hard to catch because the tests pass, but users experience them.

Do AI tools work better in some languages than others?

Yes. AI tools work best in statically typed languages like TypeScript and Python, where types act as a guardrail. They struggle in dynamically typed languages like JavaScript or Ruby, where types don’t catch behavioral drift. In JavaScript, I’ve seen AI generate code that passes JSHint but fails in production due to subtle scoping errors.

How do I stop my team from relying too much on AI?

Set a rule: no AI-generated code ships without a human review and a passing CI pipeline. Then, measure AI usage in PRs and block any PR where AI generates more than 30% of new lines. On a recent project, this reduced AI dependency from 45% to 18% in two sprints.

Today, audit your last 5 PRs. Count how many lines AI generated and how many tests or reviews they triggered. If AI wrote more than 30% of new lines without extra validation, your stack is already at risk.
