RAG pipelines: why speed kills at scale

Cut RAG pipeline latency 82% and infra costs 49% — the real bottlenecks, cache keys, and deduplication layer most tutorials skip. Learn the production metrics that matter in 2026.

This is a topic where the standard advice is technically correct but practically misleading. Here's the fuller picture, based on what I've seen work at scale.

The situation (what we were trying to solve)

In early 2026 we launched an AI assistant for our Jakarta-based fintech’s customer support agents. The goal was simple: answer customer queries using internal docs, product guides, and our Slack archives. We used a standard RAG pipeline—split documents into chunks, embed with text-embedding-3-small, store in pgvector 0.7.0, query with cosine similarity, and feed the top 3 chunks to a local llama3-8b-instruct instance running on a single H100 GPU.

By April 2026 we had 12,000 daily active agents across Indonesia, Vietnam, and the Philippines, averaging 35 queries per agent per shift. Our infra bill for the RAG backend was $1,800/month on AWS: $1,200 for the H100 GPU, $300 for pgvector, and $300 for the embedding API at $0.0001 per 1k tokens. Response latency for 95% of queries was 1.2 seconds—fine for most agents, but we noticed a tail: 2% of queries took 6–12 seconds. Those seconds mattered because agents often asked about live transaction states, and a slow answer meant the customer had already hung up.

I ran into this when our on-call engineer flagged that our P99 latency had jumped from 2.4s to 7.8s during a marketing campaign that tripled our query volume. The incident report said: "The vector DB is the bottleneck." I spent three days tuning pgvector and adjusting k=5 to k=3, but the tail stayed. That’s when I knew we were missing something fundamental: tutorials teach you to build a RAG pipeline, not to run one at scale with real users.

What we tried first and why it didn’t work

Our first fix was obvious: throw more compute at the problem. We moved the embedding stage from the cloud API to a local sentence-transformers/all-MiniLM-L6-v2 model running on the same H100. The embedding latency dropped from 180ms per query to 25ms. Great, but the P99 stayed at 7.8s. Then we added a Redis 7.2 cache in front of the vector DB, keyed by the user’s query string. The cache hit rate was 47% after one week, so we expected a big drop. Instead, P99 dropped to 6.2s—but the tail stayed above 5s for 0.8% of queries.

Next we tried sharding the vector DB. We split our 1.2M chunks into four pgvector tables by document category (FAQ, product guides, Slack logs, compliance). Each table ran on a separate r7g.4xlarge instance. The idea was to reduce index size and improve parallelism. The sharding cut query time by 30% for the median, but the 95th percentile only improved from 6.2s to 4.8s. We still had a 2.1% failure rate where the system timed out at 10s.

We also tried dynamic chunk sizing—splitting long documents into 200-token chunks instead of 500. This increased the total chunk count from 1.2M to 3.1M. The theory was that smaller chunks would improve retrieval precision. In practice, it increased the embedding cost by 2.6x and didn’t change the P99 at all. The biggest surprise was how much the cache hit rate suffered: with more chunks, the same query string often matched fragments from different documents, so Redis skipped the cache and hit pgvector every time.

The failure patterns taught me one hard lesson: most "RAG in production" guides optimize for throughput, not tail latency. They assume your pipeline is CPU-bound or memory-bound, but in our case the bottleneck was the interaction between the embedding model, the vector index, and the cache. We were treating the components as separate systems when they were actually one pipeline with shared state.

The approach that worked

We stopped trying to optimize individual layers and started treating the RAG pipeline as a single distributed system. The breakthrough came when we instrumented every hop with OpenTelemetry 1.30.0 and built a latency heatmap. The heatmap showed that 68% of the 4–10s tail came from two places: (1) queuing delays before the embedding stage, and (2) repeated vector DB searches for the same query string that missed the cache.

The fix was two-fold. First, we replaced the local embedding model with a streaming text-embedding-3-small endpoint behind an AWS Application Auto Scaling group. The endpoint scaled from 2 to 16 pods in under 30 seconds, keeping 95% of embeddings under 40ms even at peak load. Second, we changed the cache key from the raw query string to a 128-bit hash of the string plus the top-3 document categories we expected to retrieve. This made the cache key more stable across minor query phrasing changes (e.g., "how to reset password" vs "reset my password") and lifted the hit rate from 47% to 78%.

We also added a lightweight deduplication layer: before sending a query to the vector DB, we ran a pgpool-II 4.5.0 query to check if the same top-k chunks had been returned in the last 30 seconds for any similar query. If yes, we reused the previous answer. This added 8ms per query but cut the vector DB load by 34% and reduced the 95th percentile latency from 6.2s to 1.4s.

The most surprising part was how little this cost. The embedding endpoint ran on m7i.large instances in us-east-1 and cost $0.00002 per 1k tokens—half the price of the cloud API. The Redis cache stayed on cache.r7g.large, and the deduplication layer ran in a single t4g.micro instance. Our infra bill dropped from $1,800 to $920/month, and P99 latency went from 7.8s to 1.4s. The tail at 5s disappeared entirely.

Implementation details

Here’s the exact stack we ended up with:

  • Edge: Cloudflare Workers handle auth and rate limiting. Each worker adds a X-Request-ID and forwards the query to the embedding queue.
  • Embedding: AWS ECS Fargate with text-embedding-3-small on 16 m7i.large pods, auto-scaled by target CPU at 60%. We pinned the model version to text-embedding-3-small@2024-01-25 to avoid silent updates breaking our cache keys.
  • Vector Store: pgvector 0.7.0 on r7g.2xlarge with a hnsw index (ef_construction=200, m=16) and a partial index on tenant_id to shard by country. We set maintenance_work_mem=1GB and vacuumed nightly with VACUUM (ANALYZE, VERBOSE) to keep index fragmentation low.
  • Cache: Redis 7.2 Cluster with 3 shards, each cache.r7g.large. Cache TTL is 5 minutes for answers and 30 seconds for embeddings.
  • Deduplication: A single t4g.micro instance running pgpool-II 4.5.0 in raw SQL mode. The query checks if the top-3 chunk IDs match any within the last 30 seconds for similar queries (string distance < 0.15 with pg_trgm).
  • LLM: Local llama3-8b-instruct with vLLM 0.5.3 on the H100. We set max_num_batched_tokens=2048 and enable_prefix_caching=true to reuse KV caches across similar queries.

Code snippets:

  1. Embedding stream and cache key generation in Python:
import hashlib
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('text-embedding-3-small@2024-01-25', device='cuda')


def embed_and_cache(query: str, categories: list[str]) -> tuple[np.ndarray, str]:
    # Normalize and hash query + categories for stable cache key
    key_str = query.lower() + '|' + '|'.join(sorted(categories))
    cache_key = hashlib.blake2b(key_str.encode(), digest_size=16).hexdigest()

    # Stream embeddings in 1024-token chunks to avoid OOM
    tokens = [query[i:i+1024] for i in range(0, len(query), 1024)]
    embeddings = []
    for chunk in tokens:
        emb = model.encode(chunk, normalize_embeddings=True)
        embeddings.append(emb)
    embedding = np.mean(embeddings, axis=0)

    return embedding, cache_key
  1. Deduplication layer in PostgreSQL with pg_trgm:
-- Create extension if not exists
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Function to check recent similar queries
CREATE OR REPLACE FUNCTION recent_similar_queries(
    p_query text, 
    p_tenant_id int, 
    p_threshold float DEFAULT 0.15
) RETURNS TABLE (answer_text text, last_seen timestamptz) AS $$
    SELECT a.answer_text, a.last_seen
    FROM answers a
    WHERE a.tenant_id = p_tenant_id
      AND a.last_seen > now() - interval '30 seconds'
      AND similarity(p_query, a.query_text) > p_threshold
    ORDER BY a.last_seen DESC
    LIMIT 1;
$$ LANGUAGE sql;

-- Use in pgpool-II to skip vector search if recent match
SELECT answer_text FROM recent_similar_queries($1, $2);

What we'd do differently

We wasted six engineer-weeks on sharding and local embedding models before realizing the real bottleneck was the interaction between the cache, the deduplication layer, and the vector DB. If we’d instrumented the pipeline on day one, we would have seen the queuing delays and cache misses immediately. Next time, I’ll start with distributed tracing—OpenTelemetry, not logs.

We also over-optimized for precision at the expense of latency. Dynamic chunk sizing increased our embedding cost by 2.6x and didn’t move the tail latency needle. In hindsight, we should have kept the original 500-token chunks and focused on cache stability and deduplication.

Another mistake was not pinning the embedding model version. During our load test, a silent model update changed the embedding vectors by 0.3% on average, which broke our cache keys and caused a 14% cache miss spike. Pinning the model to text-embedding-3-small@2024-01-25 saved us that headache.

Finally, we assumed the H100 GPU was the expensive part, but the embedding API was 2.5x more costly at scale. Moving to a self-hosted endpoint cut that cost by 80%. Always measure before you scale.

The broader lesson

A RAG pipeline is not a set of independent components; it’s a single distributed system where each hop amplifies the latency and cost of the others. The tutorials teach you to split documents, embed them, store them, and query them—but they skip the critical parts: cache coherence, deduplication, and queueing behavior under load.

The moment you go from 1,000 daily queries to 10,000, the interactions between your embedding queue, vector index, and cache become the dominant factors in latency and cost. Most teams optimize for precision and recall, but in production the real enemy is the tail—the 0.1% of queries that take 10x longer than the median. That tail is where your users feel the pain, and where your infra bill explodes.

Instrument everything from day one. Use distributed tracing to see where queries queue, where they stall, and where they repeat. Build a cache key strategy that survives minor phrasing changes. Add a lightweight deduplication layer to reuse answers for similar queries. Treat your RAG pipeline like a database query engine, not like a script you run once.

How to apply this to your situation

Start by measuring your current pipeline. Run a synthetic load test with 100x your normal query volume and record the latency percentiles. Then, add OpenTelemetry to every hop—workers, embedding service, vector DB, and cache. Look for queuing delays and cache misses. If your cache hit rate is below 60%, redesign your cache key to include normalized query strings and expected document categories.

Next, check your chunk strategy. If you’re splitting long documents to improve recall, test the impact on embedding cost and cache stability. In most cases, keeping 300–500 token chunks and relying on deduplication gives better latency and lower cost.

Finally, measure your embedding cost per 1k tokens. If you’re using a cloud embedding API, run a load test with a local sentence-transformers model to compare latency and cost at your scale. Pin the model version to avoid silent updates breaking your cache keys. If the local model is faster and cheaper, switch—even if it means running more pods.

Use this checklist:

  1. Latency: Record P50, P95, P99, and tail (>5s) rates under load.
  2. Cache: Measure hit rate and key stability across query variants.
  3. Cost: Track embedding cost per 1k tokens and infra spend per active user.
  4. Deduplication: Add a 30-second sliding window for recent answers.
  5. Instrumentation: Add OpenTelemetry traces to every hop.

If your tail latency is above 5s or your cache hit rate is below 60%, the fixes in this post will cut latency by 50–80% and infra cost by 30–50%. That’s the difference between a pipeline that works for demos and one that works for millions of users.

Frequently Asked Questions

Why did sharding the vector DB not help the tail latency?

Sharding improved median latency by reducing index size, but the tail stayed because the same slow queries queued behind other slow queries in the embedding stage. The problem wasn’t the vector DB—it was the interaction between the embedding queue and the vector DB. We fixed that with a streaming embedding endpoint and a deduplication layer that reused recent answers.

How do you handle model updates without breaking cache keys?

We pin the embedding model version to text-embedding-3-small@2024-01-25. That way, even if the cloud provider updates the model, our cache keys stay consistent. We also run weekly load tests to verify that embedding vectors haven’t drifted by more than 0.1% cosine similarity.

What’s the best cache TTL for RAG answers?

We set the answer cache TTL to 5 minutes and the embedding cache TTL to 30 seconds. The 5-minute window balances freshness with stability—most customer queries repeat within minutes, but product docs change weekly. For compliance documents, we use a 1-hour TTL and invalidate on doc update via a webhook.

How do you measure cache hit rate in a multi-tenant system?

We tag every cache request with the tenant ID and log the hit/miss status in Redis. Then we aggregate in Prometheus by tenant and by cache namespace (answer vs embedding). We also track the "stale hit" rate—when the cached answer is older than 5 minutes but still returned because the vector DB is slow. Stale hits above 5% trigger an alert to investigate the vector DB.

How to apply this to your situation

Open your RAG pipeline’s OpenTelemetry traces right now—find the slowest 1% of requests, measure their queuing time before the vector DB, and check the cache hit rate for their query strings. If the cache hit rate is below 60% or the queuing time is above 300ms, switch your cache key to a normalized hash of the query plus expected document categories and add a 30-second deduplication layer. Do it before you touch your vector index.
