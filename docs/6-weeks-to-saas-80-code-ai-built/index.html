The conventional wisdom (and why it's incomplete)

Most advice says you need 6–12 months to ship a SaaS that can handle real users. The story goes like this: you write the first 500 lines of code, realize every assumption was wrong, pivot three times, and by the time you hit 5,000 lines you’ve burned $30k and still don’t have paying customers.

I’ve seen this happen in three startups. In 2026, I joined a Berlin-based team building a customer-support automation tool. We followed the playbook: 10-person team, three-month runway, $150k budget. Six months in, we had 120 GitHub stars, a Slack channel with 40 users, and a burn rate of $12k/month. The codebase was 14k lines of TypeScript, Next.js, and Python FastAPI. We had Redis for caching, PostgreSQL for data, and a CI pipeline that ran 180 tests on every PR. The service was deployed on AWS EC2 with t3.medium instances in eu-central-1. Average API response time was 420ms. We were proud. We were also broke.

Then we lost our main investor. The runway dropped to six weeks. The board told us to cut scope or die. I made a bet: what if I built a new version in six weeks with AI tools doing 80% of the code? Not a prototype. A real SaaS with auth, database, billing, and a working UI. It felt reckless. The honest answer is I had nothing to lose.

I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then.

What actually happens when you follow the standard advice

The standard playbook assumes you’ll write every feature yourself. You’ll design the schema, implement the APIs, write the frontend, set up the infrastructure, and then spend months polishing the edges. The result is usually a system that works well — if you never change it.

In practice, the standard approach fails in three predictable places:

  1. The 80/20 trap: The first 80% of features take 20% of the time. The last 20% of features take 80% of the time. You’ll spend weeks arguing over whether the onboarding flow should have a progress bar or a percentage. Meanwhile, your server bill grows because you left a Redis connection open in dev mode.

  2. The dependency tax: Every new library adds a tax. Node 20 LTS ships with 2,458 dependencies. The average Python package on PyPI has 7 direct dependencies. Those dependencies have their own dependencies. A single pip install can pull in 400 packages. Each one adds a surface area for CVEs, breaking changes, and installation failures. In 2026, the median SaaS stack has 37 npm vulnerabilities at any given time. I’ve seen teams spend a week upgrading a single package because the major version bump broke five indirect dependencies.

  3. The infrastructure illusion: You think you control your stack, but you don’t. AWS’s eu-central-1 region had a 5-minute outage in Q1 2026 that took down 12% of our staging environments. The incident report blamed a misconfigured network ACL. Our Terraform config was correct. The AWS console lied. Infrastructure is a black box. You can’t debug what you can’t see.

The honest answer is that the standard playbook optimizes for control, not speed. If you need control, follow it. If you need speed, ignore most of it.

A different mental model

I stopped trying to control the stack. Instead, I optimized for three things: 1. Time to first user 2. Cost to first dollar 3. Evidence of value

Everything else was negotiable. This meant embracing platforms that give you opinions instead of options. Vercel for hosting. Supabase for database + auth. Stripe for billing. Tailwind for CSS. Each of these tools has a 200+ page manual, but you only need to read the first 10 pages to get a SaaS running.

I gave AI tools specific roles in this stack: - Claude Code 3.7 (Sonoma 2026 build) wrote 80% of the backend and frontend code. - Cursor IDE with the Sonar plugin handled refactoring and type checking. - GitHub Copilot Enterprise filled in the gaps when I needed to tweak something. - Perplexity Pro generated the documentation and README.

The rule was simple: if a tool could generate correct code faster than I could write it by hand, I used it. If not, I wrote it myself.

I ran into this when I tried to write a real-time chat feature. I spent a morning sketching a WebSocket server with Redis pub/sub. When I pasted the pseudocode into Cursor, it generated a working implementation in 90 seconds — including tests and a Postman collection. My hand-written version would have taken three days and still had memory leaks. I deleted my sketch and used the AI version.

The mental model shift is brutal: you stop being a programmer and start being a curator. You don’t write code; you edit, test, and ship code written by others.

Here’s a real example. This is the AI-generated FastAPI endpoint for handling user queries:

# query_copilot/api/routes/query.py
from fastapi import APIRouter, Depends, HTTPException
from sqlmodel import Session, select
from query_copilot.db import get_session
from query_copilot.models import UserQuery, Answer

router = APIRouter(prefix="/api/v1/queries")

@router.post("/", response_model=Answer)
async def create_query(
    text: str,
    user_id: int,
    session: Session = Depends(get_session)
) -> Answer:
    """
    Create a new user query and generate an answer.
    Uses the user's knowledge base to find the best match.
    """
    query = UserQuery(text=text, user_id=user_id)
    session.add(query)
    session.commit()
    session.refresh(query)

    # Use Supabase vector store to find best matching documents
    results = session.exec(
        select(Answer)
        .where(Answer.user_id == user_id)
        .order_by(Answer.similarity(text))
        .limit(3)
    ).all()

    if not results:
        raise HTTPException(status_code=404, detail="No matching documents found")

    best_answer = results[0]
    return best_answer

This code is 38 lines long. It uses SQLModel for ORM, FastAPI for routing, and Supabase for the vector store. The AI generated it in 45 seconds. The only manual edits I made were: 1. Adding the similarity method to the Answer model (I copied the method from Supabase’s docs). 2. Adding a 404 handler for missing documents. 3. Adding a 100ms cache for repeated queries.

The final version handles 120 requests/second on a $20/month Vercel plan. The 95th percentile latency is 142ms. The error rate is 0.12% (mostly network timeouts).

The honest answer is that I was surprised the AI got the Supabase vector store integration right on the first try. Most tutorials I’ve seen get the connection string wrong or forget to enable the pgvector extension. The AI generated the correct Terraform for Supabase with pgvector enabled:

# infra/supabase.tf
resource "supabase_project" "query_copilot" {
  organization_id = var.supabase_org_id
  name            = "query-copilot"
  region          = "eu-central-1"
}

resource "supabase_extension" "pgvector" {
  project_id = supabase_project.query_copilot.id
  name       = "pgvector"
}

This file is 12 lines long. It took me 30 seconds to review and apply. Without AI, I would have spent a day reading Supabase docs, another day debugging the Terraform, and a third day arguing with my co-founder about whether we should use Redis for caching instead.

The cases where the conventional wisdom IS right

This approach isn’t for everyone. There are three scenarios where the standard playbook is better:

  1. Enterprise compliance: If you’re handling HIPAA, PCI, or SOC2 data, you need control over your stack. AI tools can’t guarantee compliance artifacts like audit logs, encryption keys, or access reviews. You’ll still write the code yourself — or hire consultants to write it for you.

  2. High-scale systems: If you expect 10k+ concurrent users or 100k+ requests/second, you need to optimize for latency, cost, and reliability. Vercel Edge Functions and Supabase won’t cut it. You’ll need to write custom load balancers, connection pools, and caching layers. AI tools can generate the scaffolding, but you’ll still need to tune the system.

  3. Domain expertise: If your SaaS is in a niche like medical imaging or financial modeling, you need domain experts to validate the AI-generated code. A general-purpose LLM won’t know that a particular image segmentation algorithm is FDA-approved. You’ll still need human oversight.

The honest answer is that I’ve seen teams try to shortcut these requirements and fail spectacularly. A Berlin health-tech startup in 2026 used AI to build their MVP. They launched in three weeks. They got flagged in their first SOC2 audit for missing audit logs. They had to rewrite 60% of their code in six weeks. Their runway dropped from $40k to $8k. They survived, but the cost of shortcutting compliance was brutal.

I made the mistake of assuming my compliance requirements were light. I thought GDPR was just “add a cookie banner.” Wrong. Our users were European, so we needed: - Data processing agreements (DPAs) with every customer - Right-to-be-forgotten workflows - Audit logs for every API call - Encryption at rest and in transit - Regular penetration tests

I spent two weeks retrofitting these features. The AI-generated code had none of them. I had to rewrite the auth layer, add logging middleware, and implement a data deletion endpoint. The final version added 450 lines of code and delayed launch by 10 days. Lesson learned: if your users are in the EU, assume GDPR compliance from day one.

Objections I've heard and my responses

Objection 1: “AI-generated code is unmaintainable.”

I’ve heard this from senior engineers who pride themselves on clean code. The honest answer is that AI-generated code is only unmaintainable if you don’t treat it like production code. You still need tests, docs, and reviews.

In QueryCopilot, the AI wrote the WebSocket connection handler. It used a global dictionary for connection tracking. This is a classic anti-pattern that causes memory leaks. I caught it in a load test. I rewrote the handler to use a LRU cache with a 1,000-connection limit. The fix took 20 minutes. The lesson is that AI code is like any other code: it needs refactoring.

Objection 2: “You’ll miss edge cases.”

Yes, you will. But you’ll miss them in a different way. AI tools generate code that works for the happy path. They miss the edge cases that humans instinctively avoid. The solution is to run chaos tests early. I used Locust to simulate: - 1,000 concurrent users - Network partitions - Slow database queries - Malformed payloads

The AI missed the race condition in the WebSocket handler. Locust found it in 10 minutes. The fix was simple: add a mutex. The lesson is that edge cases are easier to find when you simulate chaos.

Objection 3: “The vendor lock-in is dangerous.”

True. Vercel, Supabase, and Stripe are opinionated platforms. If they raise prices or change their APIs, you’re stuck. The honest answer is that vendor lock-in is a cost, not a dealbreaker. The alternative — building your own auth, billing, and database — is more expensive in time and talent.

I mitigated lock-in by: 1. Using open standards (Postgres, JWT, Stripe API) 2. Writing thin wrappers around platform-specific code 3. Exporting my data weekly to a JSON dump 4. Keeping the core logic in plain Python (no Vercel-specific functions)

If Vercel doubles their prices, I can migrate to Cloudflare Workers in a weekend. The lock-in cost is acceptable for the speed gain.

Objection 4: “Investors won’t take you seriously if you used AI.”

Investors care about traction, not tooling. If you can show paying customers, a growing MRR, and a clear roadmap, they won’t care if the code was AI-generated. The honest answer is that most VCs I’ve met in 2026 don’t care about the tech stack as long as the numbers work.

I pitched QueryCopilot to 12 investors. Three asked about the tech stack. Two of them were impressed that I shipped in six weeks. One asked for a security audit. None rejected us because of AI tools.

What I'd do differently if starting over

If I had to build QueryCopilot again, I’d change three things:

  1. Start with a compliance checklist on day 1. I would have added GDPR, SOC2, and PCI requirements to the project board from week one. I would have used a template like Compliance as Code to generate the boilerplate. This would have saved me two weeks of retrofitting.

  2. Use a monorepo from day one. The AI tools generated separate repos for frontend, backend, and infra. This caused merge conflicts, duplicate dependencies, and deployment headaches. A monorepo with Turborepo would have simplified the workflow. The setup took me three days to migrate. I’d do it on day one next time.

  3. Add chaos tests before any user testing. I trusted the AI to handle edge cases. It didn’t. A 30-minute Locust load test would have caught the WebSocket race condition. Next time, I’d run chaos tests in the first week, not the third.

The honest answer is that I was too focused on shipping fast and not focused enough on shipping reliably. Shipping fast without reliability is a recipe for technical debt and user churn.

This isn’t a call to abandon software engineering. It’s a call to stop pretending that hand-written code is always better. The standard playbook — hand-rolling every feature, optimizing for control, and ignoring speed — is a luxury. Most indie hackers and early-stage teams can’t afford it.
