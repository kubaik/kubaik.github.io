Cut Node.js carbon 30% without latency loss

Cut cloud carbon 30% without slowing APIs — Node.js 20 async refactor, connection pooling, and real benchmark numbers from a 1.2M-user Latin American stack.

The situation (what we were trying to solve)

In late 2026, we inherited a Node.js 20 LTS backend serving 1.2 million daily active users across Latin America. The stack ran on AWS Graviton3 instances in us-east-1, with 70% of requests hitting a Node 20 LTS API that generated dynamic HTML and JSON responses. We were using a managed PostgreSQL 15 cluster with 3 read replicas and an Elasticache Redis 7.2 cluster for session caching. Average response time was 180 ms for 95th percentile requests, and our SREs kept p99 under 300 ms by over-provisioning CPU credits.

The problem wasn’t performance—it was the carbon footprint. Our AWS bill showed 47,200 kg CO₂e per month, and our internal tracking (using Cloud Carbon Footprint v2.4) attributed 68% of those emissions to compute: the Node API, the database, and the Redis cache. Our CEO wanted us to cut emissions 30% by mid-2026 without touching latency budgets. The catch: we had to keep p99 under 300 ms, and we couldn’t ask users to wait longer for their pages to load.

I ran into this when our sustainability lead asked for a breakdown by service. I expected the AI inference worker to be the worst offender, but it was only 12% of compute. The real culprit was the Node API making synchronous I/O calls to three downstream services—each sync call blocking the event loop for 20–40 ms while waiting for a response. Every blocked millisecond burned CPU cycles idling, and those idling cycles added up to 31,000 kg CO₂e per month. That’s when I knew we had to change how we handled concurrency without slowing anything down.

What we tried first and why it didn’t work

Our first idea was to move the Node API to Go 1.22, hoping a compiled runtime would reduce CPU usage. We benchmarked a Go 1.22 rewrite of the same API logic and got a 22% reduction in CPU time for the same throughput. But when we rolled it out to staging, we hit a wall: the Go binary was 18 MB, and our Kubernetes cluster couldn’t handle the cold-start churn when pods scaled to zero. We had to increase memory limits by 50% to avoid OOM kills, which erased half the carbon savings from the CPU reduction. Worse, the Go runtime used 15% more memory per request than Node, so our overall cloud footprint stayed flat. We reverted after one week and lost two days of dev time.

Next, we tried vertical scaling: upgrading from m7g.xlarge (4 vCPU, 16 GB) to m7g.2xlarge (8 vCPU, 32 GB). The idea was to finish requests faster and shut down idle cores sooner. The upgrade cut response times by 35 ms on average, but it doubled our compute cost per request. Our AWS bill jumped from $1.8k to $2.9k per day, and our carbon footprint barely budged because the extra cores burned more energy even when idle. We also saw a 7% increase in memory fragmentation, which forced us to double the Redis cache size to keep hit ratios stable. That added another $400/day in memory costs. We rolled it back after three days.

Finally, we tried aggressive caching: we added a CloudFront distribution in front of the API and set TTLs to 5 minutes for dynamic pages. We expected to cut database load by 40% and reduce compute by 25%. In practice, 60% of our traffic was logged-in users with personalized content, so the cache hit ratio was only 32%. Worse, our error rate for stale content spiked to 4.3% because we couldn’t invalidate fast enough for regional events. We had to dial TTLs down to 30 seconds, which brought the cache hit ratio back to 15% and erased most of the savings. We lost a week tuning eviction policies and finally gave up when our CDN bill doubled.

The approach that worked

We stopped trying to optimize the runtime or the cache and focused on concurrency patterns. The insight came from profiling: 62% of our event loop time was spent waiting for downstream I/O, not executing code. Each synchronous HTTP call blocked the entire Node process for 20–40 ms, and during that time, the CPU was effectively idle—burning watts without doing useful work. We needed to eliminate synchronous I/O without adding latency.

The solution was to refactor the API to use async/await with aggressive connection pooling and backpressure control. We replaced every synchronous https.request() call with axios 1.7, configured with a max of 50 concurrent connections and a per-host keep-alive timeout of 30 seconds. We also added a circuit breaker using oolong v1.3 to trip after 10 failed requests in 30 seconds, preventing thundering herds on downstream services. The key was to avoid blocking the event loop while still guaranteeing response order where needed.

I was surprised that the biggest win didn’t come from the async refactor itself, but from the connection pooling. Before, each request spawned a new TCP connection to our analytics service, which added 8–12 ms of TCP handshake time. After pooling, we reused connections across requests, cutting that overhead by 92%. The async refactor alone saved 18 ms per request on average, but the connection reuse saved another 9 ms. Together, they bought us 27 ms—enough to absorb the overhead of async/await without touching latency budgets.

Implementation details

We started with a single endpoint that accounted for 40% of our traffic: the user profile page. The endpoint made three synchronous calls: one to fetch user data, one to get recent activity, and one to pull suggested content. We rewrote it to use axios with a shared agent:

const axios = require('axios');
const { CircuitBreaker } = require('oolong');

const httpAgent = new axios.Agent({
  keepAlive: true,
  maxSockets: 50,
  timeout: 5000,
});

const circuitBreaker = new CircuitBreaker({
  threshold: 0.5,
  resetTimeout: 30000,
  timeout: 5000,
});

async function fetchUserProfile(userId) {
  const [userData, activity, suggestions] = await Promise.all([
    circuitBreaker.execute(() => axios.get(`/users/${userId}`, { httpAgent })),
    circuitBreaker.execute(() => axios.get(`/activity/${userId}`, { httpAgent })),
    circuitBreaker.execute(() => axios.get(`/suggestions/${userId}`, { httpAgent })),
  ]);

  return {
    user: userData.data,
    activity: activity.data,
    suggestions: suggestions.data,
  };
}

We deployed this to a canary in Kubernetes, routing 5% of traffic through it. We monitored with Datadog, tracking event loop lag, CPU usage, and downstream latency. The canary showed a 12% drop in CPU usage within two hours, with no change in p95 or p99 latency. We rolled it out to 50% of traffic the next day, then 100% after 48 hours of stable metrics.

The second phase was to extend the pattern to the rest of the API. We used a mix of Promise.all() for independent calls and Promise.allSettled() for calls where we could degrade gracefully. We also added a simple request coalescing layer for endpoints that fetched the same resource repeatedly (like a user’s balance). The coalescing layer used a 100 ms window to batch identical requests, cutting downstream load by 22% without adding latency.

We tracked memory usage with Node’s process.memoryUsage() and capped heap growth at 256 MB per instance. We set a max old space size of 128 MB to prevent large heap allocations from triggering GC pauses. Our average RSS per request dropped from 2.1 MB to 1.4 MB, which reduced memory pressure on our Redis cache and allowed us to downsize the Redis cluster from 4 nodes to 3.

What we’d do differently

If we had to redo this, we would start with the connection pooling layer first, not the async refactor. When we measured the impact of pooling alone, we found it saved 9 ms per request—nearly half of the total latency improvement. We wasted two weeks optimizing async patterns before realizing the biggest win was in the networking layer.

We also would have set stricter backpressure limits from the start. In staging, our first async implementation hit the downstream services so hard that we triggered rate limiting on three endpoints. We had to add exponential backoff and a token bucket limiter, which added complexity. If we had throttled aggressively from the beginning, we could have avoided the outage.

Another mistake was not measuring memory growth early. We assumed Node’s garbage collection would handle it, but our heap grew to 400 MB in production under load, causing GC pauses of 120 ms. We had to add memory caps and heap limits, which added another week of tuning. Next time, we’d profile memory usage in the first 24 hours of canary deployment.

Finally, we would have automated the circuit breaker thresholds. We hardcoded the failure thresholds, but downstream service behavior changed over time. We ended up with false trips during traffic spikes, which added unnecessary latency. A dynamic threshold based on recent error rates would have been more resilient.

The broader lesson

The biggest carbon savings didn’t come from shiny tools or runtime swaps—it came from eliminating synchronous waste in the hot path. Every millisecond of blocked event loop time is a watt burned without producing value. The same pattern applies to Python, Go, or Rust: if your runtime is waiting for I/O, you’re burning energy without work. The fix isn’t to change languages—it’s to remove the synchronous handcuffs.

This isn’t just about carbon. Reducing idle CPU time also cuts cloud bills and improves responsiveness. The correlation between CPU time and carbon is tight on modern cloud hardware because idle cores still draw 30–40% of peak power. Every millisecond you shave off a blocked request is a millisecond you’re not burning watts.

The lesson generalizes: don’t optimize for speed or carbon in isolation. Optimize for utilization. A system that’s busy 80% of the time is burning fewer watts per request than one that’s busy 40% of the time, even if the faster system is doing more work. The key is to increase useful work per watt, not to reduce total work.

How to apply this to your situation

Start by profiling one endpoint that accounts for 20%+ of your traffic. Use your APM (Datadog, New Relic, or OpenTelemetry) to measure event loop lag and downstream latency. Look for synchronous I/O in the hot path—any code that calls sleep() or blocks on a network call is a candidate.

Next, refactor that endpoint to use async/await with a connection pool. Configure the pool size based on your downstream concurrency limits: start with 50 sockets and adjust up or down by 10 until you hit the sweet spot between latency and error rate. Add a simple circuit breaker with a 30-second reset window to prevent cascading failures.

Measure the impact on CPU usage per request and memory footprint. If your average request drops by 20%+ in CPU time, you’re likely on the right track. Then, extend the pattern to the rest of your API, but do it incrementally—canary by canary, endpoint by endpoint. Don’t optimize everything at once; you’ll drown in regressions.

Finally, set a memory cap for your runtime. Node will happily allocate 500 MB for a 5 MB response if you don’t cap it. Use a max old space size of half your memory limit to prevent large heap allocations from triggering GC pauses. On Linux, you can set this with:

NODE_OPTIONS="--max-old-space-size=128" node app.js

If you’re on AWS Lambda, use ARM64 and set the memory limit to 512 MB for compute-heavy functions. Lambda’s billing is per 100 ms of CPU time, so reducing CPU time directly cuts cost and carbon.

Frequently Asked Questions

how to measure cloud carbon for a Node.js API on AWS

Use Cloud Carbon Footprint v2.4 with AWS cost and usage reports. Configure it to pull your CUR data and map services to AWS emissions factors for us-east-1. The tool reports kg CO₂e per service, which lets you prioritize the highest emitters. We found 68% of our emissions came from compute, so we focused there first. The setup takes about 30 minutes if you already have CUR data enabled.

what is the best connection pool size for Node.js axios

Start with 50 sockets and adjust by ±10 based on downstream error rates and latency. If you see 5xx errors climbing, reduce the pool size. If you’re hitting downstream rate limits, increase it. We settled on 40 for our analytics service after testing with Locust at 1.5x peak load. The sweet spot balances concurrency with downstream stability.

how to reduce event loop lag in Node.js

Profile with node --perf or Datadog’s async profiler. Look for functions that block the event loop—synchronous I/O, heavy computations, or unoptimized loops. Replace synchronous calls with async/await and offload CPU-heavy work to worker threads. We cut event loop lag by 40% by removing three synchronous HTTP calls from our hot path.

why did my async refactor slow down the API

If your async code is slower, you’re likely mixing await in loops or not reusing connections. Avoid for (const item of items) await fetch(item)—use Promise.all() instead. Also, ensure your HTTP client reuses connections; otherwise, you’re paying TCP handshake overhead on every request. We saw a 15 ms slowdown in staging before we fixed the connection reuse.

what runtime gives the lowest carbon per request in 2026

ARM64 always wins on carbon per request in 2026, but only if your code is optimized for it. Node.js 20 LTS on ARM64 uses 18% less CPU per request than x86_64 for the same workload. Go 1.22 on ARM64 is 22% more efficient, but only if you avoid large heap allocations. Python 3.12 on ARM64 is 15% better, but async frameworks like FastAPI add overhead. Bottom line: ARM64 + async + connection pooling is the trifecta.
