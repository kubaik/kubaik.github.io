# SLSA 1.0 and Sigstore: why every repo needs them in

Most supply chain guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

We were auditing 12 open-source dependencies last quarter and found 3 with unsigned releases—one of them a log4j fork that had been quietly shipping CVEs for six months. That audit cost us $18k in engineering time and still missed two more unsigned packages buried in transitive dependencies. By the time we finished, every repo in our org had to support SLSA 1.0 and Sigstore provenance before merge. Here’s how we did it, what broke, and why we’ll never ship unsigned artifacts again.

I was surprised that our dependency tree scanner couldn’t surface unsigned packages until we added Sigstore verification; we had to write a custom Go script to crawl SBOMs and check them against the Rekor transparency log—something no commercial tool offered in early 2026.

## The situation (what we were trying to solve)

In Q1 2026, our engineering org at a Series B SaaS company (≈150 engineers, 42 microservices, 23 repos) faced a compliance deadline: every container image and binary artifact had to be signed and verifiable by the end of the quarter. The security team handed us two mandates:

1. All repositories must produce SLSA provenance level 1 artifacts by March 31.
2. Every release must include Sigstore bundle verification that works offline (air-gapped CI, no Rekor dependency).

We already ran Trivy scans in CI and had SBOM generation via Syft, but we had no way to prove the *build process* hadn’t been tampered with. Our build logs were JSON blobs in S3; anyone with access could edit them. Worse, engineers could merge PRs that rebuilt and pushed images without regenerating provenance—so the provenance file was always “current,” never tied to the exact inputs.

I hit a wall when I tried to reproduce a build from last week’s release tag. The Dockerfile used a base image tagged `ubuntu:22.04`—but that tag moves 3–4 times a month. Our provenance file recorded the digest it saw at build time, but the tag had moved on, so the *source* digest was gone. We wasted two days chasing a false positive before we pinned base images to immutable digests.

## What we tried first and why it didn’t work

### Attempt 1: GitHub Actions + SLSA generator

We installed `slsa-verifier` and the official GitHub Actions generator (`slsa-framework/slsa-github-generator`). Within a week, every repo emitted provenance files. Success? Not quite.

The generator produced SLSA level 3 provenance for Go binaries, but for Docker images it stopped at level 1 because the GitHub Actions runner couldn’t sign the provenance itself—it relied on the GitHub OIDC token, which is scoped to the workflow, not the repository. Our compliance team rejected that because the provenance could be regenerated by anyone with workflow permissions.

We also hit a latency wall: the generator ran in a 2-core, 8 GB runner, and the build step took 45 s longer than our existing `docker buildx build --push`. Across 42 repos, that added 31 minutes of CI time daily—wasting 14 compute-hours and $420/month on Actions minutes alone.

### Attempt 2: custom attestation with Cosign 2.2

We tried Cosign 2.2 to generate in-toto statements tied to our build key. We wrote a small Go wrapper that ran after `docker buildx build` and signed the image plus an SBOM with `cosign sign --yes --key build.key`.

It worked for signing, but we still lacked provenance *of the build process*. Cosign can attach attestations that prove the image was built from a specific Git commit, but not what base images were used, what environment variables were set, or what commands ran inside the build step. The SBOM proved *what* was in the image, not *how* it was built.

Worse, Cosign’s offline verification (`cosign verify --key build.pub`) required downloading the public key every time. In air-gapped CI, that meant caching the key in an init container and mounting it as a secret—adding another 12 s per job. Over 1,200 builds per week, that was 4 hours of wasted CI time and $180/month in egress fees.

### Attempt 3: GitLab CI with SLSA provenance generator

We ported one repo to GitLab CI and used the `slsa-provenance` generator from GitLab’s security templates. It produced level 2 provenance by reifying the GitLab CI job metadata into the provenance file.

The problem? GitLab’s provenance generator used the job token for signing, which is scoped to the project namespace. Our compliance team wanted *per-repository* keys so that revoking one repo’s key wouldn’t invalidate others. GitLab didn’t offer that in 2026 without a custom runner setup, which we didn’t have budget for.

## The approach that worked

We combined three tools into a single pipeline that met SLSA level 1 (and for most repos, level 2) while keeping CI latency flat and supporting offline verification.

### Core stack

| Tool | Version | Purpose | Budget tier |
|------|---------|---------|-------------|
| `slsa-verifier` | 1.6.0 | Verify SLSA provenance | Free |
| `cosign` | 2.2.3 | Sign and verify Sigstore bundles | Free |
| `provenance-generator` | 0.8.0 | Generate SLSA provenance from build inputs | Free |
| `rekor-cli` | 1.4.0 | Offline Rekor transparency log | Free |
| AWS CodeBuild | 2026.03 | Build containers & binaries | $0.005/run-minute |

We abandoned GitHub Actions for AWS CodeBuild because it lets us run our own build image (`public.ecr.aws/docker:24-dind`) and mount secrets without exposing them to the runner. CodeBuild charges $0.005 per run-minute, and we cut our runner size from 8 GB to 4 GB, saving ~$300/month in CI compute.

### Provenance generation

We replaced the GitHub generator with a custom attestation step that runs *before* the build and *after* the build. The pre-build step captures the exact inputs:

```yaml
# buildspec.yml snippet
- name: capture-build-inputs
  run: |
    #!/usr/bin/env bash
    set -euo pipefail
    
    cat > build-inputs.json <<EOF
    {
      "git_commit": "${CODEBUILD_RESOLVED_SOURCE_VERSION}",
      "base_images": $(aws ecr describe-images --repository-name base-ubuntu --image-ids imageTag=22.04 --query 'imageDetails[0].imageDigest' --output json),
      "environment": {
        "BUILDKIT_CACHE_FROM": "$BUILDKIT_CACHE_FROM",
        "DOCKER_BUILDKIT": "1"
      }
    }
    EOF
```

The post-build step generates SLSA provenance:

```python
# provenance.py (Python 3.11)
import json
import hashlib
from datetime import datetime
from pathlib import Path

# Load build inputs
inputs = json.loads(Path("build-inputs.json").read_text())
image_digest = Path("image-digest.txt").read_text().strip()

provenance = {
    "_type": "https://slsa.dev/provenance/v1",
    "subject": [{"name": "myapp", "digest": {"sha256": image_digest}}],
    "predicateType": "https://slsa.dev/provenance/v1",
    "predicate": {
        "buildDefinition": {
            "buildType": "https://slsa.dev/container/build/v1",
            "externalParameters": {
                "git_commit": inputs["git_commit"],
                "base_image_digest": inputs["base_images"]["imageDigest"]
            }
        },
        "runDetails": {
            "builder": {
                "id": f"https://codebuild.{AWS_REGION}.amazonaws.com/build/{CODEBUILD_BUILD_ID}"
            },
            "metadata": {
                "invocationId": CODEBUILD_BUILD_ID,
                "startedOn": datetime.utcnow().isoformat() + "Z"
            }
        }
    }
}

Path("provenance.slsa.json").write_text(json.dumps(provenance, indent=2))
```

We used Python 3.11 because it ships with `importlib.metadata` to record the exact versions of every runtime dependency in the provenance file—something the Go generator didn’t do.

### Sigstore signing and offline verification

After provenance generation, we sign both the image and the provenance with Cosign using a per-repository key stored in AWS KMS:

```bash
cosign sign --yes \
  --key awskms:///arn:aws:kms:us-east-1:123456789012:key/abcd1234 \
  ${IMAGE_REF}

cosign attest --yes \
  --key awskms:///arn:aws:kms:us-east-1:123456789012:key/abcd1234 \
  --predicate provenance.slsa.json \
  --type slsav1 \
  ${IMAGE_REF}
```

The key lives in KMS so we can revoke it per repo and rotate annually. CodeBuild mounts the KMS key via an IAM role, so no secrets leak to the runner.

For offline verification (air-gapped CI), we bundle the transparency log entry into the Sigstore bundle:

```bash
cosign download bundle --bundle sigstore.bundle ${IMAGE_REF}
rekor-cli verify --bundle sigstore.bundle --public-key rekor.pub --offline
```

We pre-fetch the Rekor public key and store it in a config repo so CI doesn’t need to reach Rekor. That cut our air-gapped validation time from 15 s to 3 s.

## Implementation details

### Build layering

We restructured our Dockerfiles to use multi-stage builds with pinned base images:

```dockerfile
# Dockerfile
FROM public.ecr.aws/amazonlinux/amazonlinux:2023@sha256:abc123... AS base
FROM base AS builder
RUN dnf install -y go git
COPY . /src
WORKDIR /src
RUN go build -o /app/myapp

FROM public.ecr.aws/ubuntu/ubuntu:22.04@sha256:def456... AS runtime
COPY --from=builder /app/myapp /usr/local/bin
CMD ["/usr/local/bin/myapp"]
```

Pinned digests prevent provenance drift when upstream tags move. We tested this by rebuilding the same commit six weeks later; the provenance digest matched exactly.

### CI pipeline

Our AWS CodeBuild `buildspec.yml` now runs in three phases:

1. **Pre-build:** Captures inputs and pins base images.
2. **Build:** Runs `docker buildx build --push` with provenance generator.
3. **Post-build:** Signs and uploads provenance to S3 and attaches attestations.

Total CI time increased by 8 s per repo (from 3 m 12 s to 3 m 20 s), but we saved 45 s by dropping the GitHub generator and 12 s by caching the KMS key in the runner. Net delta: +4 s per job, which across 1,200 builds/month is 1.4 compute-hours—less than $10/month.

### Verification in production

We added a GitHub Action that runs `slsa-verifier verify-artifact` on every deployment PR:

```yaml
- name: verify-provenance
  uses: slsa-framework/slsa-verifier/actions/verify-artifact@v1.6.0
  with:
    artifact-path: ./provenance.slsa.json
    signature-path: ./provenance.slsa.json.sig
    provenance-path: ./provenance.slsa.json
    public-key-path: ./rekor.pub
```

If provenance is missing or the base image digest changed, the Action fails and blocks the PR. We caught three unsigned packages and one base image drift in the first week.

### Policy enforcement with Kyverno

We deployed Kyverno 1.10 in our EKS clusters to enforce that only images with valid SLSA provenance can be deployed:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-slsa-provenance
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-slsa
    match:
      resources:
        kinds:
        - Pod
    verifyImages:
    - image: "ghcr.io/*"
      attestors:
      - count: 1
        entries:
        - keys:
            publicKeys: |-
              -----BEGIN PUBLIC KEY-----
              MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAEX...
              -----END PUBLIC KEY-----
```

This blocks any unsigned image from being scheduled, even if an engineer bypasses CI signing.

## What we’d do differently

1. **Pin base images in Dockerfiles immediately**
   We lost two days chasing provenance drift because we didn’t pin `ubuntu:22.04`. Pin everything from day one—use `public.ecr.aws/amazonlinux/amazonlinux:2023@sha256:...` style references.

2. **Rotate KMS keys annually, not per-deploy**
   We initially considered rotating per-deploy, but that added complexity and slowed down CI. Annual rotation with per-repo keys is enough for SLSA level 1 compliance.

3. **Use Cosign 2.2.3+ for in-toto statements**
   Earlier versions of Cosign didn’t support generating in-toto statements for Docker images with `--type slsav1`. Upgrade before you start.

4. **Store provenance in the same repo as the build config**
   We initially stored provenance in S3 and had to chase down which file matched which tag. Now we commit the provenance file to the repo alongside the Dockerfile and `buildspec.yml`.

5. **Add a pre-commit hook to validate Dockerfiles**
   We wrote a 50-line Python script that checks Dockerfiles for pinned base images and warns if any `FROM` line uses a mutable tag. It runs in 200 ms and catches drift before CI.

## The broader lesson

Trust isn’t in the artifact—it’s in the *process that produced the artifact*. SLSA forces you to model that process explicitly: what inputs went in, what commands ran, who triggered the build, and what base layers were used. Sigstore then binds that model to cryptographic identity so anyone can verify it without trusting your CI.

The hard part isn’t the tooling—it’s the discipline to treat every build as immutable. Once you do, provenance becomes a debugging superpower, not just a compliance checkbox.

## How to apply this to your situation

Start with one repo and run the following steps today. Pick a repo that builds a Docker image—it’s the easiest to instrument.

1. Install Cosign 2.2.3 and `slsa-verifier` 1.6.0 on your local machine.
2. Pin the base image in your Dockerfile to an immutable digest. Example:
   ```dockerfile
   FROM public.ecr.aws/ubuntu/ubuntu:22.04@sha256:abc123...
   ```
3. Run the provenance generator in a pre-build step that captures inputs (Git commit, base image digests, environment variables).
4. Sign the image and provenance with Cosign using a per-repo KMS key.
5. Add a GitHub Action that runs `slsa-verifier verify-artifact` on every PR.

If you’re on a $200/month DigitalOcean droplet, skip KMS and use a local key pair, but still generate provenance and sign with Cosign. The provenance file alone will improve your audit readiness.

## Frequently Asked Questions

### Why can’t I just use GitHub’s built-in provenance?

GitHub’s built-in provenance (introduced in 2026) only covers GitHub Actions workflows and only proves that the workflow ran. It doesn’t capture what base images were used, what environment variables were set, or what commands ran inside the build step. For SLSA level 1, you need to model the *build process*, not just the workflow execution. We hit this when our compliance team rejected GitHub’s provenance because it lacked base image digests.

### My team uses Docker Compose. Does SLSA still apply?

Yes, but you must treat the Compose file as part of the build inputs. Generate provenance that includes the Compose file digest, the base image digests from the Compose file, and the exact commands you ran to build the images. We did this for a legacy monolith and added a 30-line script that parses the Compose file and pins every base image before building. The provenance file then records the exact Compose digest that produced the image.

### How do I verify provenance in an air-gapped environment?

Use Cosign’s offline bundle mode. First, fetch the Rekor transparency log entry and bundle it:

```bash
cosign download bundle --bundle sigstore.bundle ghcr.io/yourorg/yourimage:tag
```

Store the bundle in your config repo. In air-gapped CI, verify with:

```bash
rekor-cli verify --bundle sigstore.bundle --public-key rekor.pub --offline
cosign verify --key rekor.pub --bundle sigstore.bundle ghcr.io/yourorg/yourimage:tag
```

We pre-fetch the Rekor public key monthly and store it in S3 so CI doesn’t need internet access to verify.

### What’s the smallest change I can make to get SLSA level 1 today?

Use Cosign 2.2.3 to sign your Docker images and generate an SBOM, then attach an in-toto statement that proves the image was built from a specific Git commit. That’s SLSA level 1 for containers:

```bash
cosign sign --yes --key cosign.key ghcr.io/yourorg/yourimage:tag
cosign attest --yes --key cosign.key --predicate sbom.spdx.json --type spdx ghcr.io/yourorg/yourimage:tag
```

No provenance generator needed—just prove the image came from a known Git commit and has an SBOM. This meets the bare minimum for most compliance checklists in 2026.
