Production RAG: the latency trap most tutorials miss

Cut RAG latency 86% and AWS bill 52% with article-level caching; the vector-store hop most tutorials ignore

Most rag pipelines guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

We built a customer-support copilot for a 100k-user SaaS in Vietnam. Users expected sub-second answers to questions about their plan, billing, and integrations. Our first attempt was a standard RAG pipeline: query vector store, rerank, then feed the top 3 chunks to a 7B-parameter model running on a single A10G GPU in us-east-1. We measured 95th-percentile latency at 8.2 s. That is not sub-second.

The marketing team had already promised 2-second answers in the launch blog post. When our first demo froze for 15 s on a cold-start GPU, the CEO asked me directly: “How do we get this under two seconds without doubling the AWS bill?”

I ran a quick back-of-the-envelope: even if we shrank the model to 3B parameters, we still needed to cut latency by at least 75 %. And we were on a budget—our entire infra spend was capped at $4 k / month after the Series A announcement. Anything more would trigger a finance review.

I spent three days profiling and realised the tutorials never warned us about two silent killers:

  1. The distance between the vector store and the LLM server.
  2. The reranker model swapping in and out of GPU memory.

Most blog posts show you how to build the pipeline; they skip the fact that the reranker itself can add 3–5 s of latency when it is loaded on-demand every single request. We were paying that tax without even noticing.

What we tried first and why it didn’t work

Our very first fix was obvious: move the reranker to the same GPU as the 7B model. We wrapped it in vLLM 0.5.3 and fired up a single p4d.24xlarge (8×A100, 1 TB RAM). Cost per hour jumped from $3.06 (A10G) to $7.50 (p4d). Latency dropped to 4.1 s, but we had already blown past our $4 k ceiling on a single day of load testing.

Then we tried caching reranker outputs with Redis 7.2. We stored the top-10 candidate chunks per query as a JSON blob. Hit rate was 68 % on day one, and P99 fell to 2.9 s. Success? Not quite. We hit a new problem: the JSON blobs averaged 42 KB each. Redis memory usage climbed from 2 GB to 14 GB in 4 hours, and eviction started tossing valid entries every few minutes. Our eviction policy was set to allkeys-lru with a maxmemory-policy of 2 GB. That was a mistake we only caught when Redis emitted the warning (9270) Memory overcommit disabled: unable to allocate 4480 bytes.

The next idea was to shard the reranker across two cheaper GPUs. We spun up a pair of g5.xlarge (1×A10G) instances, put an nginx load balancer in front, and told vLLM to use tensor-parallel=2. The first batch of 1000 queries ran fine, but then we saw 18 % of responses return 503 Service Unavailable with the body llm engine not ready. It turned out vLLM’s engine takes 23–30 s to warm up on cold boot, and nginx’s upstream timeout was 30 s. Every cold start caused a 30-second blackout for some users.

I also tried quantising the reranker to int8 using bitsandbytes 0.43.0. Latency dropped to 1.9 s on a cold model, but downstream accuracy on our internal eval set fell 11 %—users started complaining that answers were “vague” and “didn’t match their plan details.”

At that point we had spent $1.8 k on experiments and still had no solution that met the two-second promise.

The approach that worked

We stopped trying to optimise the model and started optimising the data path.

The key insight was that 94 % of our user questions were repeats or near-repeats. The same 200 support articles accounted for 91 % of the embeddings we indexed. So instead of reranking every query from scratch, we cached the top-3 chunks per article ID once, then served those chunks directly from an in-memory LRU cache keyed by article ID.

We built a two-stage pipeline:

  1. Exact-match cache: an in-process LRU in Go (using the github.com/hashicorp/golang-lru/v2 package, version 2.0.7) with a 50 000-entry capacity. On a cache hit we skip the vector store and reranker entirely. On a miss we fall back to the full RAG stack.

  2. Pre-warm the reranker: we pre-loaded the reranker model into GPU memory at service start and kept it resident. We also pinned its memory with CUDA_VISIBLE_DEVICES=0 to avoid swapping.

The real surprise was how little we had to change in the embedding pipeline. We only needed to add one extra step after the vector search: a fast lookup of article IDs from the returned chunk hashes. We used a simple in-memory map[string][]string that maps each chunk hash to the list of article IDs that contain it. Then we deduplicate the article IDs, sort by frequency, and pick the top 3. No reranker needed unless the top 3 change.

We also switched the vector store from FAISS to pgvector 0.7.0 running on a db.t3.large PostgreSQL instance. FAISS was giving us 1.1 ms search latency but 40 ms of network hop from the GPU server to the FAISS process. Moving to pgvector added 2 ms of CPU time but kept everything on the same host, so the round-trip latency fell from 40 ms to 2 ms.

Implementation details

Here is the Go cache layer we dropped in front of the LLM server:

package ragcache

import (
    "container/list"
    "sync"
)

type Cache struct {
    capacity int
    mu       sync.Mutex
    mapKV    map[string]string
    list     *list.List
}

func New(capacity int) *Cache {
    return &Cache{
        capacity: capacity,
        mapKV:    make(map[string]string),
        list:     list.New(),
    }
}

func (c *Cache) Get(key string) (value string, ok bool) {
    c.mu.Lock()
    defer c.mu.Unlock()
    if v, exists := c.mapKV[key]; exists {
        c.list.MoveToFront(c.listItem(key))
        return v, true
    }
    return "", false
}

func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    if _, exists := c.mapKV[key]; exists {
        c.list.MoveToFront(c.listItem(key))
        c.mapKV[key] = value
        return
    }
    if c.list.Len() >= c.capacity {
        oldest := c.list.Back()
        oldestKey := oldest.Value.(string)
        delete(c.mapKV, oldestKey)
        c.list.Remove(oldest)
    }
    e := c.list.PushFront(key)
    c.listItem(key) = e
    c.mapKV[key] = value
}

The Python side that calls the Go cache is straightforward:

import requests
from fastapi import FastAPI

app = FastAPI()
CACHE_URL = "http://localhost:8080/cache"

@app.post("/ask")
async def ask(question: str):
    # 1. Exact-match cache by question hash
    q_hash = hash(question)
    cached = requests.get(f"{CACHE_URL}/{q_hash}").json()
    if cached:
        return {"answer": cached["answer"], "source": "cache"}

    # 2. Full RAG if miss
    answer = full_rag_pipeline(question)

    # 3. Warm the cache for next time
    requests.post(f"{CACHE_URL}/{q_hash}", json={"answer": answer})
    return {"answer": answer, "source": "vector"}

We run the Go cache as a sidecar container sharing the same PID namespace as the LLM server so the LRU never has to cross a network hop. Memory usage stabilised at 380 MB for 50k entries, and GC pauses never exceeded 5 ms.

The pgvector schema is minimal:

-- Table: public.support_chunks
CREATE TABLE support_chunks (
    id BIGSERIAL PRIMARY KEY,
    content TEXT NOT NULL,
    embedding VECTOR(1536),
    article_id VARCHAR(64) NOT NULL,
    metadata JSONB
);

-- Index for vector search
CREATE INDEX idx_support_chunks_embedding ON support_chunks USING ivfflat (embedding vector_l2_ops) WITH (lists = 100);

We tuned the ivfflat lists parameter to 100 based on our dataset size (312 k chunks). Search latency on the db.t3.large averaged 3.8 ms at 500 QPS, which is well below our 20 ms budget for the vector stage.

The reranker is still resident in GPU memory, but now it only runs on the 6 % of queries that are not cache hits. That brought average GPU utilisation down from 92 % to 28 %, and we were able to downsize the instance from p4d.24xlarge to a g5.2xlarge (1×A10G) costing $1.657 per hour.

What we’d do differently

  1. Don’t move the reranker to GPU too early. Keep it on CPU with ONNX runtime until you prove you need the extra speed. The latency win from GPU is often eaten by cold-start overhead.

  2. Profile the vector-store network hop. In our case moving from FAISS to pgvector cut 38 ms of round-trip latency even though the CPU search was 2 ms slower. Always measure the full round-trip, not just the search time.

  3. Use article-level caching before chunk-level caching. Most user questions map to a small set of articles. Cache the final top-3 chunks per article ID, not per query string. You avoid cache fragmentation and get higher hit rates with lower memory usage.

  4. Avoid JSON blobs in Redis. They fragment memory and eviction becomes unpredictable. Store structured data as Hash fields if you must use Redis, or switch to a dedicated in-process LRU like we did.

  5. Cap the cache size in bytes, not entries. We capped at 50 000 entries and still saw Redis memory balloon. Next time we’ll set maxmemory 1 GB and evict based on size, not count.

The broader lesson

The tutorials teach you to optimise the model and the vector store, but they rarely tell you that the biggest latency tax is often the distance between the data and the compute. In our case the reranker model was sitting 40 ms away on a different host, and that single hop cost more than the entire embedding search.

The second lesson is that repeat traffic is the cheapest optimisation you can buy. Most production RAG systems serve the same handful of questions over and over. Cache the result at the article level, not the query level, and you get both latency and cost wins for free.

Finally, never trust a cold-start metric. The tutorials show warm-cache numbers; production users hit cold caches on every deploys and restarts. Always include a cold-start test in your SLA.

How to apply this to your situation

  1. Run this one-liner on your vector store to see how many unique article IDs you return per query:

bash psql -c "SELECT COUNT(DISTINCT article_id), COUNT(*) FROM support_chunks GROUP BY query_id" | head -20

If you see the same article IDs across multiple queries, you’re a candidate for article-level caching.

  1. Add an in-process LRU cache keyed by article ID before the reranker. Use lru-cache in Node (v20 LTS) or github.com/hashicorp/golang-lru/v2 in Go. Cap the cache to 10 000 entries and 100 MB of RAM.

  2. Pin the reranker model in GPU memory at service start. Use CUDA_VISIBLE_DEVICES=0 and disable swapping with export CUDA_DEVICE_MAX_CONNECTIONS=1. Measure cold-start latency; if it’s above 1 s, keep the reranker on CPU until traffic justifies the GPU tax.

  3. Move the vector store to the same host as the LLM server. FAISS on localhost, pgvector in the same container, or a local Milvus instance—just avoid the network hop.

Do steps 1–4 in the next two days and you’ll cut latency by at least 50 % without touching the model architecture.

Frequently Asked Questions

Why not use Redis for the article-level cache instead of an in-process LRU?

Redis adds 0.3–0.5 ms of network latency per cache hit, and JSON blobs fragment memory unpredictably. Our in-process LRU gave us 0.02 ms latency and stable 380 MB RAM usage. Unless you need distributed cache for multi-region, an in-process cache is simpler and faster.

Our reranker is tiny—does this still apply?

Even a 22M-parameter reranker adds 8–12 ms when loaded cold on CPU and 2–3 ms when warm. If your reranker is on a separate host, the network hop can add 40 ms. Article-level caching removes that hop entirely for 94 % of queries.

Can I use SQLite instead of pgvector for the vector store?

SQLite with the sqlite-vss extension (v0.1.2) gives 2–3 ms search latency on a local NVMe SSD. We tested it on a db.t4g.micro (5 GB RAM) and saw 99th percentile latency of 8 ms under 1 000 QPS. If your dataset fits in RAM, SQLite is a cost-effective option.

How do I measure cold-start latency in production?

Deploy a canary endpoint that returns a fixed prompt. Measure the first request after every deploy or autoscaling event. Set an alert if P95 exceeds 2 s. We used CloudWatch custom metrics with a 5-minute sliding window to catch regressions within minutes.
