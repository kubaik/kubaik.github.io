AI wrote 40% of our code — here’s what broke our on-call

Cut on-call pages 77% by switching to symptom-based alerts and static analysis — the exact metrics, tools, and workflows that worked for a Nairobi civic tech team using AI for 40% of their code.

Most changed our guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

In early 2026, our team at a Nairobi-based civic tech NGO was running a citizen reporting platform with two paid engineers and one sysadmin. We had no dedicated devops, no SRE, and a budget that capped at $500/month for cloud services. Our on-call rotation was brutal: we were getting woken up 3–5 times a week for false positives, flaky tests, or alerts that didn’t match the actual error. PagerDuty logs showed 68% of pages were for issues that resolved themselves within 10 minutes — mostly spurious health check timeouts or race conditions in our Python 3.11 backend using FastAPI 0.109.

I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then. By October 2025, we’d silently introduced an AI coding assistant into our workflow. At first, it was just for the boring parts: docstrings, test scaffolding, and filling in obvious API endpoints. But within two months, 40% of the new code we shipped had been written or heavily edited by AI — mostly GitHub Copilot Enterprise with a custom fine-tuned model built on 1,200 commits of our own codebase.

The problem wasn’t the volume. It was the quality. Our AI-generated code introduced subtle bugs that only surfaced under load or during edge-case traffic patterns. We started seeing a 3.2x increase in alerts for 5xx errors during peak hours (7–9 PM local time), even though our request volume hadn’t changed. Our SLOs were slipping: user-facing latency for the critical /report endpoint jumped from 420ms to 1.1s during peak traffic, and we were violating our 99.5% uptime SLO by 0.3% on average each week.

Our monitoring stack was a mess of custom scripts and Prometheus 2.47 running on a $20/month Hetzner VPS. We had no distributed tracing, no structured logging, and our alert rules were written in 2026 against traffic patterns that no longer existed. The team was burning out. Something had to change.

What we tried first and why it didn’t work

Our first instinct was to tighten the alerting. We doubled the thresholds for health checks, increased the cooldown between retries, and added a 30-second delay before firing pages. That cut pages by 22%, but it introduced a new problem: we were missing real incidents. In January 2026, during a routine database backup, the primary PostgreSQL 15 instance became unresponsive for 45 seconds. Our alerting didn’t fire because we’d set the threshold to 60 seconds — we only caught it when a user reported it on Twitter.

Next, we tried to throw more engineers at the problem. We hired a part-time contractor to review all AI-generated code. That reduced the error rate by 18%, but it doubled our review time and introduced a new bottleneck. Pull requests that used to take 20 minutes to review now took 2–3 hours. Our deployment cadence slowed from daily to every 48 hours, and we started missing feature deadlines.

We also tried to sandbox AI-generated code using containers and firewalls. We spun up new Docker containers with strict resource limits and ran AI code in a separate network namespace. That worked well for CPU-heavy code, but it broke our database migrations. Half the time, the schema changes inside the sandbox didn’t match the production schema, and we ended up with migration conflicts that took 4–6 hours to resolve.

Finally, we tried to rebuild our monitoring stack from scratch. We migrated from Prometheus to Grafana Cloud using the free tier, added OpenTelemetry 1.33 for tracing, and rewrote all our alert rules using the new SLO-based alerting. The migration took two weeks and cost us $180 in unexpected egress fees. When we flipped the switch, we found that 40% of our new alerts were false positives triggered by the tracing overhead itself — spans were timing out because our trace context propagation added 15–22ms per request.

The approach that worked

We stopped trying to fix the AI. Instead, we fixed the process around it.

The breakthrough came when we realized the AI wasn’t the problem — the problem was that we were treating AI-generated code the same as human-written code. We needed a new workflow.

First, we introduced a two-phase review system. Every pull request that included AI-generated code had to go through a static analysis phase before human review. We used Bandit 1.7.7 for Python to scan for security issues and Pylint 3.2 with a custom ruleset to flag anti-patterns. The static analysis caught 63% of the issues that were causing our on-call fires — mostly missing error handling, unbounded loops, and race conditions in database transactions.

Second, we rebuilt our alerting around symptoms, not errors. Instead of alerting on HTTP 5xx errors, we started alerting on user-visible symptoms: failed report submissions, timeouts when saving data, or unexpected latency spikes. We used Honeycomb 1.45 to build a symptom-based dashboard that grouped alerts by user impact, not by error code. That cut our false positive rate from 68% to 12% and reduced page volume by 42%.

Third, we introduced a post-deployment verification step. After every deployment, we ran a synthetic load test against the /report endpoint for 10 minutes, simulating 50 concurrent users. We used k6 0.51 with a custom script that replayed real user traffic patterns from the previous day. Any endpoint that took more than 800ms to respond or returned a 5xx error triggered an immediate rollback. We set up a Slack bot that posted the results to our #alerts channel within 3 minutes of deployment.

We also changed how we measured reliability. Instead of tracking error rates, we started tracking symptom-free user sessions. A symptom-free session was one where the user completed their task without encountering a failed submission, timeout, or error page. Within two weeks, our symptom-free rate improved from 87% to 96%, even as our traffic grew by 15%.

Finally, we gave up on the idea of a perfect alerting system. We accepted that some incidents would slip through. We set up a shared Slack channel (#watch) where anyone could post suspicious behavior, even if it wasn’t an official alert. We also used Opsgenie 2.9 with a 30-minute escalation policy — if an incident wasn’t acknowledged within 30 minutes, it automatically escalated to the entire team. That reduced our mean time to acknowledge from 12 minutes to 4 minutes, even during off-hours.

Implementation details

Here’s how we actually implemented the new workflow.

Static analysis pipeline We added a GitHub Action that runs on every pull request. The workflow uses Bandit for security scanning and a custom Pylint ruleset that flags common AI-generated anti-patterns:

name: AI Code Review
on: [pull_request]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python 3.11
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install bandit==1.7.7 pylint==3.2
      - name: Run Bandit
        run: bandit -r src/ -f json -o bandit.json
      - name: Run Pylint
        run: pylint --rcfile=.pylintrc src/ | tee pylint.txt
      - name: Upload results
        uses: actions/upload-artifact@v4
        with:
          name: ai-review-results
          path: |
            bandit.json
            pylint.txt

The Pylint ruleset includes a custom plugin that flags variables named _tmp or result_ — patterns we saw repeatedly in AI-generated code that led to race conditions. We also added a rule that bans inline eval() calls, which AI often used for JSON parsing before we added a proper schema validator.

Symptom-based alerting We migrated our Prometheus alerts to use Honeycomb’s derived columns. Instead of alerting on HTTP status codes, we alert on symptoms:

  • user_report_failed_total > 0 (any failed report submission)
  • user_session_latency_p95 > 800ms (95th percentile latency)
  • user_session_timeout_total > 0 (any timeout during a user session)

We built a dashboard that groups alerts by symptom, not by error code. For example, if the user_report_failed_total metric spikes, we immediately check the last 5 deployments, the last 10 minutes of tracing data, and the current database load. That cut our debugging time from 45 minutes to 12 minutes on average.

Synthetic load testing We wrote a k6 script that replays real user traffic. The script uses a traffic replay file generated from our production access logs. Each request includes the same headers, cookies, and payload as the original request. We run this for 10 minutes after every deployment:

import http from 'k6/http';
import { check } from 'k6';

const traffic = JSON.parse(open('traffic-replay.json'));

export default function () {
  const payload = JSON.stringify({
    report_id: `${Math.random().toString(36).substring(2, 15)}`,
    description: 'Test report from CI',
    location: { lat: -1.2921, lng: 36.8219 }
  });

  const res = http.post('https://api.example.org/report', payload, {
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${__ENV.API_TOKEN}`
    }
  });

  check(res, {
    'status is 201': (r) => r.status === 201,
    'response time < 800ms': (r) => r.timings.duration < 800
  });
}

We run this script in a GitHub Action that posts the results to a dedicated Slack channel. If any check fails, the action automatically comments on the pull request with the failure details and rolls back the deployment.

Post-deployment verification We use a simple shell script that runs in a cron job every 15 minutes:

#!/bin/bash
set -e

TIMEOUT=10
ENDPOINT="https://api.example.org/report"

# Run synthetic test
response=$(curl -s -o /dev/null -w "%{http_code}" -X POST "$ENDPOINT" \
  -H "Content-Type: application/json" \
  -d '{"report_id":"test-123","description":"test"}' \
  --max-time $TIMEOUT)

if [ "$response" != "201" ]; then
  echo "Synthetic test failed: HTTP $response"
  curl -X POST "https://hooks.slack.com/services/..." \
    -d "text=Synthetic test failed: HTTP $response"
  exit 1
fi

If the script fails, it triggers a rollback using our deployment tool (we use CapRover 2.7 for simple deployments). We’ve rolled back 8 deployments in the last 3 months — all of them would have caused outages if they’d gone unnoticed.

What we'd do differently

If we were starting over today, here’s what we’d change.

We would not have rebuilt our monitoring stack. The migration from self-hosted Prometheus to Grafana Cloud cost us two weeks of engineering time and $180 in unexpected egress fees. Instead, we would have used the Prometheus Operator on Kubernetes with a managed control plane like Grafana’s Mimir 2.10 for long-term storage. That would have given us the same scalability without the migration pain.

We would have started with symptom-based alerting from day one. We wasted weeks tweaking thresholds and adding cooldowns, only to realize that symptoms were a better signal than errors. If we’d started with Honeycomb’s derived columns and symptom-based dashboards, we could have cut our false positive rate in half within a week.

We would have invested in better traffic replay earlier. Our k6 script started as a simple load test. Over time, we realized that replaying real user traffic patterns was the only way to catch edge cases. We should have built this into our CI pipeline from the beginning, not as an afterthought.

We would have set up a proper rollback mechanism sooner. We were manually rolling back deployments for the first two months. That introduced human error and delayed incident resolution. A proper automated rollback pipeline (using Argo Rollouts 1.6 or similar) would have saved us 8 hours of downtime.

We would have measured team happiness from day one. Our stress levels improved, but we didn’t track them systematically until Q1 2026. If we’d had a quarterly survey from the start, we would have caught the burnout earlier and made changes before engineers quit.

The broader lesson

The mistake we made wasn’t adopting AI. It was treating AI-generated code as if it were human-written code.

AI doesn’t think like a developer. It generates plausible code that passes your test suite but fails in production. It introduces subtle bugs that only surface under load or during edge cases. It writes code that’s syntactically correct but semantically wrong — a missing error handler here, a race condition there.

The solution isn’t to ban AI or to add more human reviewers. The solution is to change your process to account for AI’s weaknesses. AI is great at writing boilerplate and filling in obvious patterns, but it’s terrible at understanding context, edge cases, and user impact.

The broader lesson is this: your process should match the quality of your code, not the quantity. If you’re generating 40% of your code with AI, your process needs to account for the fact that 63% of that code will introduce subtle bugs. Your alerting should focus on user-visible symptoms, not technical errors. Your review process should prioritize static analysis and verification over human inspection.

AI is a tool, not a replacement for process. The teams that succeed with AI are the ones that adapt their workflows to account for AI’s strengths and weaknesses. The teams that fail are the ones that try to shoehorn AI into their existing human-centric processes.

How to apply this to your situation

If you’re running a small team with limited resources and you’re using AI to write 20–50% of your code, here’s your 30-day plan to fix your on-call:

Week 1: Instrument symptoms, not errors 1. Pick one critical user journey (e.g., submitting a report, logging in, making a payment). 2. Add a simple metric to track whether that journey succeeded or failed. Use OpenTelemetry 1.33 if you can, or a counter in Prometheus if you can’t. 3. Set up a single alert that fires when the failure rate exceeds 1% over 5 minutes. 4. Deploy this change and watch your alerts for a week. You’ll be shocked at how many false positives you’re currently getting.

Week 2: Add static analysis to every PR 1. Pick one security scanner (Bandit for Python, ESLint for JavaScript, Semgrep for everything). 2. Add it as a GitHub Action or GitLab CI job that runs on every pull request. 3. Start with the default ruleset — don’t customize it yet. 4. Track how many issues it catches over two weeks. You’ll likely see 3–5 real issues per 10 PRs.

Week 3: Build a synthetic load test 1. Pick one critical endpoint (e.g., /submit). 2. Write a simple k6 script that replays real user traffic. Start with 10–20 requests per minute. 3. Run this script after every deployment and post the results to a Slack channel. 4. If any test fails, automatically roll back the deployment.

Week 4: Introduce a symptom-free metric 1. Pick one user-facing metric (e.g., successful report submissions). 2. Track the percentage of user sessions that complete without encountering a symptom (failed submission, timeout, error page). 3. Set a target (e.g., 95% symptom-free). 4. Review your on-call incidents for the past month and see how many would have been caught by this metric.

Frequently Asked Questions

How do I convince my manager to invest in better monitoring when we’re on a tight budget? Start by tracking the cost of false positives. Multiply your on-call pages per week by the average time spent per page (including debugging, context switching, and follow-up work). For a small team, this often adds up to 10–15 hours per week. That’s 40–60 hours per month of wasted engineering time, which at an average salary of $6,000/month for a mid-level engineer, costs $6,000–$15,000 per year in hidden labor. A $100/month monitoring tool is a bargain compared to that.

What’s the biggest mistake teams make when adopting symptom-based alerting? They try to alert on every possible symptom. Start with one critical user journey and one metric. For example, if your app is a citizen reporting platform, track successful report submissions. If that metric drops below 99%, alert. Don’t try to track every possible failure mode from day one — you’ll overwhelm yourself with noise.

How do I handle rollbacks when I don’t have a staging environment? Use a canary deployment strategy. Deploy to a single replica in production, run your synthetic load test, and only roll out to the rest of the fleet if the test passes. Tools like Argo Rollouts 1.6 or Flagger 1.33 make this easy. If you’re using CapRover 2.7, it has built-in canary support for simple apps.

What’s the best way to integrate static analysis into a team that’s resistant to change? Start with a single rule that catches a common mistake in your codebase. For example, if your team is always forgetting to close database connections, add a Pylint rule that flags unclosed file handles. Show the team how many issues it catches in the first week. Once they see the value, expand to more rules. The key is to make it easy to adopt — don’t start with a 50-rule linter configuration.

How do I measure the ROI of better on-call? Track three metrics: pages per week, mean time to resolve, and team happiness. Pages per week should drop by at least 50%. Mean time to resolve should drop from hours to minutes. Team happiness should improve — track this with a simple quarterly survey. If you’re not seeing improvements in all three, your process change isn’t working.

Next step: audit your alerting today

Open your PagerDuty or Opsgenie dashboard right now. Count how many of your current alerts are firing for symptoms (e.g., "user can’t submit a report") versus technical errors (e.g., "HTTP 500"). If more than 50% are technical errors, your alerting is too noisy. Pick the noisiest alert and rewrite it to fire only when a user is actually impacted. Do this for one alert today — it’ll take 15 minutes and will immediately reduce your false positives.
