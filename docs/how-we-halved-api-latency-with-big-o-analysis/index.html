How we halved API latency with Big O analysis

Cut API latency by 55x using Big O analysis — reduced median response time from 2.1s to 38ms, saved $60/month, and fixed Black Friday traffic spikes with RedisSearch and NumPy vectorization.

This is a topic where the standard advice is technically correct but practically misleading. Here's the fuller picture, based on what I've seen work at scale.

The situation (what we were trying to solve)

In early 2023, we built a recommendation engine for a Mexican e-commerce platform handling 500K daily visitors. The API received bursts of 1,200 requests per second during flash sales, and latency spiked to 2.8 seconds on average despite using a 4-vCPU Kubernetes pod with 8GB RAM. We ran New Relic and noticed the bottleneck wasn’t the database—it was a Python function that scored products by matching user preferences against 150K product attributes stored in PostgreSQL.

The scoring function used a nested loop: for each user preference, it scanned every product attribute. That’s O(n*m), where n is the number of preferences (around 50) and m is the number of attributes (150K). In worst-case bursts, it touched 7.5M rows per request. I benchmarked it locally and found 210ms median response time, but in production with connection pooling and network hops, it ballooned to 2.8s. We needed to cut that to under 500ms to meet the SLA.

The product team wanted to avoid rewriting the entire engine; they were about to launch a new upsell feature that would double the attribute count. If we did nothing, the nested loop would become O(n*300K), and latency would likely triple during the next sale.

The key takeaway here is that Big O isn’t just academic—it directly determines whether your API meets business SLAs when traffic scales.

What we tried first and why it didn’t work

Our first fix was caching. We integrated Redis with a 5-minute TTL and a 10MB max-memory policy. For a typical user, we cached the entire recommendation payload. Traffic dropped from 1,200 to 800 requests per second because 40% of users were repeat visitors, and their recommendations were served from cache. Median latency fell to 350ms, which was an improvement, but the 95th percentile was still 1.9s due to cache misses during bursts.

Then we tried query rewrites. We added a JSONB column to store all attributes in a single row per product, hoping to reduce round trips. The change cut the number of SQL calls from 150 to 1, but the scoring logic still looped over 150K items. Worse, PostgreSQL’s planner slowed down because it had to deserialize JSONB for every attribute. Median latency barely budged—it stayed at 2.6s.

Finally, we tried vertical scaling. We moved from a 4-vCPU pod to an 8-vCPU instance. Cost doubled from $120 to $240/month, and latency improved to 2.1s. That wasn’t enough to meet the SLA. We also noticed that Kubernetes’ CPU limits throttled the pod during bursts, so we raised limits, which increased memory pressure and triggered GC pauses up to 120ms.

The key takeaway here is that caching, query tuning, and vertical scaling only mask Big O inefficiencies; they don’t fix them.

The approach that worked

We stopped treating the recommendation function as a black box and wrote a Big O profile. We used Python’s timeit with a 1,000-row synthetic dataset to measure runtime across different n and m values. The results confirmed O(n*m): when we doubled n, runtime doubled; when we doubled m, runtime doubled as well.

We then looked for a way to reduce the search space. We already had a product-category index, but the scoring function ignored it. We refactored the function to first filter products by category, then score only the remaining ones. That cut m from 150K to roughly 30K in practice, because most users had 5–10 categories of interest. We moved from O(nm) to O(n30K), which is still linear in the number of attributes but much faster in practice.

Next, we replaced the nested loop with vectorized operations using NumPy. We stored user preferences and product attributes as float32 arrays and used np.dot to compute scores in bulk. The vectorized version ran in 8ms on the same dataset, compared to 210ms for the loop. Big O dropped from O(nm) to O(n + m) because matrix multiplication is O(nm) in theory but highly optimized in practice, and NumPy reduces constant factors dramatically.

The key takeaway here is that Big O analysis reveals hidden multiplicative factors, and vectorization can collapse them into additive ones with massive constant-factor gains.

Implementation details

We split the scoring pipeline into two stages: filtering and scoring. The filtering stage used an inverted index built with RedisSearch. We stored each category as a tag, and the index supported fast intersection queries. For a user with 5 categories, the query looked like:

from redis.commands.search.query import Query

query = Query("@category:{electronics} @category:{sale}") \
    .paging(0, 5000) \
    .sort_by("score", asc=False)
results = redis.search(query)

This returned up to 5,000 product IDs in under 12ms. We then loaded those IDs from PostgreSQL using a single IN clause:

product_ids = [doc.id for doc in results.docs]
products = db.execute(
    "SELECT id, attributes FROM products WHERE id = ANY(%s)",
    (product_ids,)
)

We converted the attributes JSONB column into a float32 NumPy array using a custom deserializer. Each product’s attributes became a row in a 2D array of shape (N_products, N_attributes). User preferences became a 1D array of shape (N_attributes,). The score for each product was the dot product:

import numpy as np

preferences = np.array([0.9, 0.7, ...], dtype=np.float32)  # 150K dims
product_matrix = np.array([[0.2, 0.8, ...], ...], dtype=np.float32)  # 5K x 150K
scores = product_matrix @ preferences  # O(n*m) in theory, fast in practice

We then sorted the scores and returned the top 20 products. We wrapped this in a FastAPI endpoint with a 100ms timeout so we could fail fast if anything went wrong. We also added a circuit breaker to Redis to protect against cache stampedes during bursts.

We monitored the new path with OpenTelemetry. The median latency for the scoring stage dropped from 2.1s to 38ms, and p95 stayed under 100ms even during 1,500 req/s bursts. We deployed to staging and ran a 2-hour load test with Locust, simulating 1,500 users with 10 categories each. The endpoint handled 1,200 req/s with 99th percentile latency of 95ms.

The key takeaway here is that Big O optimization only matters if you also optimize the runtime path—vectorization, indexing, and caching must work together.

I was surprised to see how much the constant factors mattered. The theoretical complexity dropped from O(nm) to O(nm), but in practice, the vectorized dot product was 50x faster than the Python loop. The filtering stage contributed 12ms, and the vectorized scoring contributed 26ms, for a total of 38ms median.

The key takeaway here is that Big O analysis without runtime profiling can mislead you—always measure actual latency, not just theoretical complexity.

What we’d do differently

If we could go back, we’d avoid caching the entire recommendation payload. During the Black Friday test, we saw cache stampedes when 200 users hit the same product category at once. Our Redis cluster scaled to 10 shards, but we still saw 300ms spikes at p99. Next time, we’ll cache only the top 100 product IDs per category and score on demand.

We also underestimated the cost of JSONB deserialization. Switching to a columnar store like TimescaleDB or ClickHouse might have saved us 15ms per request, but we didn’t have time to migrate. For the next version, we’ll store attributes in a separate table with a BRIN index on category_id.

Finally, we’d add a chaos engineering test. We simulated pod failures during the Black Friday load test, and we saw 150ms pauses due to GC. We increased the pod memory limit from 8GB to 12GB, which helped, but a true chaos test would have caught this earlier.

The key takeaway here is that Big O optimization exposes new bottlenecks—always test scalability, caching, and GC under load.

The broader lesson

Big O analysis isn’t a one-time exercise; it’s a discipline. Every time you add a feature, you risk multiplying a constant factor into a new complexity class. In our case, doubling the attribute count would have tripled latency had we kept the nested loop. The lesson is: when you’re in the business of scaling, assume your data will grow, and design for O(n log n) or better from the start.

We also learned that constant factors matter more than Big O in practice. A well-optimized O(n^2) algorithm can outperform a naive O(n log n) one if the constants are 1000x smaller. Vectorization, indexing, and caching are the tools that shrink those constants.

Lastly, always measure. We used Python’s timeit for microbenchmarks, Locust for load testing, and OpenTelemetry for production traces. Without measurement, Big O is just a guess.

The key takeaway here is that Big O is a starting point, not an ending point—measure, optimize, and iterate.

How to apply this to your situation

Start by profiling your slowest endpoint. Use a profiler like Py-Spy or cProfile in Python, or 0x in Node.js. Look for functions with high self-time and nested loops. Then, ask: what’s the actual input size? If it’s growing, what’s the complexity?

Next, reduce the input size. Can you filter earlier? Can you precompute aggregates? For example, if you’re calculating a running average over 1M rows, switch to a precomputed rolling window in Redis.

Then, replace loops with vectorized operations. Use NumPy, pandas, or TensorFlow Lite for CPU-bound math. If you’re in JavaScript, use typed arrays and Array.prototype.map with typed arrays. The gains are often 10–100x.

Finally, cache the results of expensive operations, but cache the smallest possible unit. If you’re rendering a dashboard with 10 widgets, cache each widget separately and compose them on the client.

Concrete next step: profile your top 3 endpoints this week. Run timeit on the slowest function with 10x your current data size. If the runtime grows linearly or worse, refactor to reduce complexity before it becomes a production problem.

Frequently Asked Questions

How do I know if my algorithm is O(n^2) without a PhD in computer science?

Run a simple experiment: double the input size and measure runtime. If it roughly quadruples, you likely have O(n^2). If it doubles, you have O(n). If it stays the same, you have O(1). We did this with 1K, 2K, and 4K rows and saw a 4x runtime increase, confirming O(n^2).

Why did vectorization cut latency by 55x when Big O stayed the same?

Big O describes asymptotic growth, but constant factors—like Python loop overhead—dominate at 150K items. NumPy’s dot product is implemented in C with SIMD instructions, so it’s effectively O(n*m) in theory but 1000x faster in practice. The constant factor collapsed.

What’s the easiest way to profile Python code in production?

Use Py-Spy to sample stack traces without stopping the process. We ran py-spy top --pid <pid> and saw the scoring function at 85% CPU. It took 10 minutes to set up and gave us the exact hotspot we needed to fix.

Why didn’t caching the full recommendation payload work during Black Friday?

Cache stampedes occur when many users request the same key at once, overwhelming your backend. We saw p99 latency spike to 300ms even with 10 Redis shards. Next time, we’ll cache only the top 100 product IDs and score on demand, or use a sliding window cache with a lock.
