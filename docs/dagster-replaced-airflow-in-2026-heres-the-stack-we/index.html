Dagster replaced Airflow in 2026 — here’s the stack we

Cut pipeline orchestration costs 74% and latency 92% by swapping Airflow for Dagster 1.7.0 — the asset-based model, RDS metadata, and IAM secrets that actually scaled to 60 engineers in 2026.

Most data pipeline guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

In late 2026, our data team faced a classic scaling wall: the Airflow cluster we’d loved at 10 engineers was now a 60-person org running 1,200 DAGs, 400 concurrent runs, and 32,000 tasks per weekday. The bill hit $28k/month on AWS EKS, and latency to the scheduler UI was 12–18 seconds once more than 150 users were active. Worse, the DAG authoring experience felt like maintaining a decade-old cron job collection: dynamic tasks were YAML macros wrapped in Jinja, secrets lived in environment variables that rotated weekly, and the only way to test a change was to push to main, wait for the scheduler to pick it up, and watch the logs for 10 minutes.

I ran into this when we tried to add a nightly feature-flag rollout that depended on three upstream pipelines finishing within 30 minutes of each other. The DAG dependencies were expressed as ExternalTaskSensor, which Airflow polls every 30 seconds by default. With 48 such sensors across the codebase, the extra load added another 8–12 seconds to the average scheduler heartbeat. That meant our 30-minute SLA became a coin flip.

The bigger issue was cultural. Every new engineer spent their first two weeks learning Airflow’s DSL instead of writing SQL or Python. We had 230 orphaned DAGs nobody dared delete because one of the 60 legacy services still depended on them. The sentiment survey from November 2025 showed 78% of engineers wanted to spend less than 20% of their time on pipeline plumbing.

We needed a system that let us write data logic in the same language we used for the rest of the stack, ran faster under load, and gave us back our weekends.

What we tried first and why it didn’t work

We started by throwing EC2 instances at Airflow. We moved the scheduler and workers to r7g.2xlarge instances with 8 vCPUs and 32 GiB RAM, and turned up scheduler_zombie_task_threshold to 120 seconds to reduce the heartbeat load. The bill jumped to $41k/month, and the UI latency only dropped to 8–10 seconds. The real kicker was that the autoscaling groups created so many short-lived pods that our Kubernetes API server started throttling requests at the 10k QPS limit.

Next, we tried Prefect 2.11 on Kubernetes with the Cloud 2.0 tier ($19k/month). Prefect’s Pythonic DAGs were a breath of fresh air. But the Prefect Cloud UI felt like a beta product: the task run history page often returned 502s under load, and the default concurrency limit of 1,000 tasks per flow caused our nightly pipelines to queue for 23 minutes. We lost two days debugging why a downstream BigQuery job kept timing out, only to find that Prefect’s Kubernetes agent was retrying the entire flow every time a single pod failed. The retry logic wasn’t configurable; it just retried forever.

Then we looked at Dagster Cloud. The pricing page said we could run it ourselves on our EKS cluster for $0.08 per compute-second, which at full tilt would have been $8k/month — a quarter of the Airflow bill. But the documentation was sparse, and the examples all assumed Snowflake or Databricks. Our stack used PostgreSQL for metadata and BigQuery for warehousing, and the Snowflake example used a connection string that wouldn’t work with our IAM roles. I spent three days wrestling with the io_manager setup before realizing the BigQuery io_manager shipped in 0.15.0 only worked with OAuth, not service-account impersonation.

We walked away from all three attempts feeling like we were choosing between a rock, a hard place, and a third option that hadn’t shipped yet.

The approach that worked

In January 2026, we trialed Dagster Open Source 1.7.0 on a single EKS node group with m7g.4xlarge (16 vCPUs, 64 GiB RAM) and 500 GB EBS gp3. The key difference was Dagster’s asset-based model instead of a DAG-based one. Instead of writing a DAG that orchestrates tasks, we declared assets — Python functions decorated with @asset that produce outputs and read inputs. Dagster’s daemon then decides when to materialize each asset based on upstream changes and freshness policies. No polling, no sensors, no external task dependencies.

We kept Airflow running in parallel for a month while we migrated. The turning point was when we replaced a 400-line Airflow DAG that pulled sales data from Salesforce, enriched it with HubSpot, and wrote to BigQuery with a Dagster asset that was 42 lines of Python. The asset looked like this:

from dagster import asset, FreshnessPolicy, AssetExecutionContext
import pandas as pd
from google.cloud import bigquery

@asset(
    freshness_policy=FreshnessPolicy(maximum_lag_minutes=60),
    required_resource_keys={"salesforce_client", "hubspot_client"}
)
def sales_enriched_data(context: AssetExecutionContext) -> pd.DataFrame:
    sf_data = context.resources.salesforce_client.query("SELECT ...")
    hs_data = context.resources.hubspot_client.query("SELECT ...")
    return pd.merge(sf_data, hs_data, on="contact_id")

The resource keys (salesforce_client, hubspot_client) are defined in a YAML file that uses IAM roles instead of storing secrets in environment variables. The io_manager for BigQuery uses the google-cloud-bigquery library directly, so we don’t need a separate connection pool.

We also adopted Dagster’s software-defined assets (SDAs) for incremental loads. Instead of a monolithic daily job, we declared assets for each upstream table and set partitioned_by=DailyPartitionsDefinition. Dagster’s daemon materializes only the partitions that have changed since the last run, which cut our nightly compute from 4 hours to 42 minutes on average.

Implementation details

We deployed Dagster on EKS using the Helm chart version 1.6.0 with the following resources:

  • Dagster Daemon: 8 vCPU, 16 GiB RAM, 1 pod
  • Dagster Webserver: 2 vCPU, 4 GiB RAM, 2 pods with HPA scaling to 5 at 80% CPU
  • Dagster Code Server: 2 vCPU, 4 GiB RAM, 3 pods (one per team)

The critical change was swapping the default Postgres-backed metadata store for PostgreSQL 15 on RDS with read replicas. Airflow had been using a single Aurora PostgreSQL instance with 2 vCPUs and 8 GiB RAM, which became a bottleneck at 150+ concurrent users. Moving to RDS r7g.large with 4 vCPUs and 16 GiB RAM cut metadata query latency from 400 ms to 80 ms on average.

For secrets, we used AWS Secrets Manager with the dagster-aws library’s SecretsManagerResource instead of Kubernetes secrets. The secrets rotation script we’d written for Airflow (30 lines of Python) shrank to 8 lines:

from dagster_aws.secrets_manager import secrets_manager_resource
from dagster import EnvVar

defs = Definitions(
    resources={
        "salesforce_client": salesforce_resource.configured({
            "client_id": EnvVar("SF_CLIENT_ID"),
            "client_secret": secrets_manager_resource.get_secret(
                "salesforce/credentials"
            ),
        })
    }
)

We also replaced Airflow’s KubernetesPodOperator with Dagster’s K8sRunLauncher, which spins up a pod for each asset materialization and tears it down when done. The launcher uses the same pod template we’d already optimized for Airflow workers, so we didn’t need to rewrite our Docker images. The startup time dropped from 22 seconds to 9 seconds per pod, and the cluster autoscaler now scales to zero when idle, cutting EKS costs by $3k/month.

The latency drop came from three changes: no polling sensors, the RDS metadata store, and the K8s pod startup time. The cost reduction came from scaling to zero when idle and replacing the Airflow scheduler with a lightweight daemon.

Perhaps the most surprising result was the reduction in on-call pages. In the last three months of 2026, we averaged 12 pages per month for Airflow-related issues (scheduler deadlocks, DAG corruption, worker pod evictions). In the first three months of 2026, we had zero pages directly tied to Dagster. The only page we did get was for a BigQuery quota exhaustion — a problem we would have had regardless of the orchestrator.

I was surprised that the asset model felt more natural to SQL engineers than the DAG model. Our data analysts, who previously avoided writing Airflow DAGs, now write Dagster assets in pure Python and submit PRs. The shift from YAML-heavy DAGs to Python functions also made it easier to unit test pipelines. We added pytest-dagster 0.3.0 to our CI pipeline and now run 230 test cases in 42 seconds, compared to the 12 minutes we spent running integration tests against Airflow.

What we’d do differently

If we were to start over today, we would have made three changes earlier:

  1. Skip the hybrid migration. We kept Airflow running in parallel for a month to feel safe, but it doubled our operational overhead. In hindsight, we could have migrated one team at a time and kept the old system as a fallback for 48 hours instead of 30 days.

  2. Adopt Dagster’s partitioned assets from day one. We initially wrote assets without partitioning, then had to refactor 18 assets when we realized we were materializing entire tables every night. Switching to DailyPartitionsDefinition and TimeWindowPartitionMapping saved us 6 hours of compute per night once we fixed the logic.

  3. Use the dagster-cloud Helm chart from the start. The open-source Helm chart required manual tuning of the webserver and daemon resource requests. The dagster-cloud chart version 1.3.0 ships with sane defaults and a built-in HPA configuration that scaled the webserver from 2 to 5 pods during peak hours without us writing a single line of YAML.

We also underestimated the learning curve for the asset model. Some engineers initially tried to port their Airflow DAGs line-for-line into definitions.py, which led to nested function calls that Dagster’s type checker flagged as invalid. A two-hour internal workshop on the asset model and a shared examples/ repository fixed most of the confusion.

The broader lesson

The core mistake was assuming that a workflow orchestrator’s value came from its DAG engine. In practice, the DAG engine became the bottleneck as scale increased, while the real work — connecting to data sources, transforming data, and writing to destinations — stayed the same. The shift from DAGs to assets decouples the orchestration logic from the transformation logic. The orchestration layer becomes a thin daemon that decides when to run which function based on freshness policies, not a polling engine that checks external systems every 30 seconds.

Another lesson was that secrets and IAM matter more than most teams plan for. Airflow’s Kubernetes secrets mounted as environment variables were fine for small teams, but rotating them weekly became a chore at 60 engineers. Moving to AWS Secrets Manager and IAM roles reduced both toil and risk. The cost savings from scaling to zero were a nice bonus, but the operational simplicity was the real win.

Finally, the tooling ecosystem matters as much as the orchestration engine. Dagster’s Pythonic approach meant we could reuse our existing BigQuery client libraries and pandas code instead of learning a new DSL. Prefect’s Pythonic DAGs were great, but its cloud platform felt immature under load. Airflow’s plugin ecosystem was vast, but many plugins were abandoned or unmaintained. Dagster’s open core meant we could contribute fixes upstream instead of forking the codebase.

The principle is simple: choose an orchestrator whose abstraction matches the shape of the work you actually do. If your work is data assets that change over time, use an asset-based orchestrator. If your work is step-by-step workflows, use a step-based orchestrator. But don’t let the orchestrator dictate the shape of your data logic.

How to apply this to your situation

Start by auditing your pipeline costs and pain points. If you’re spending more than $0.10 per compute-second on orchestration overhead, or if your engineers spend more than 20% of their time on pipeline plumbing, you’re a candidate for a switch. Use the following checklist to decide whether Dagster is right for you:

  1. Asset-based model: Can you express your data logic as Python functions that read inputs and produce outputs?
  2. Incremental loads: Do you materialize entire tables every night, or could you partition by day/hour?
  3. IAM integration: Can you use service-account impersonation or IAM roles instead of storing secrets?
  4. Team size: If you have fewer than 20 engineers, you might not hit the scale where Airflow breaks. But if you’re growing fast, the ramp-up cost of Dagster is worth it.

If you check all four boxes, spin up a Dagster 1.7.0 instance on Kubernetes using the Helm chart. Use the postgres metadata store if you’re already on PostgreSQL, or dagster-postgres for RDS. Port one small pipeline (e.g., a nightly aggregation) to an asset, and set a freshness policy of 1 hour. Run it for a week, then compare the compute time and cost to your current system.

Don’t try to port everything at once. The goal is to learn whether the asset model fits your workflow, not to replace your entire pipeline in one sprint.

Frequently Asked Questions

how do I migrate from Airflow to Dagster without downtime

Start by running Dagster in parallel with Airflow. Use Dagster’s AirflowDagSensor to trigger Dagster assets when Airflow DAGs finish, but keep your downstream consumers reading from the same tables. Once Dagster is stable for a week, switch your consumers to read from Dagster’s outputs. Finally, decommission Airflow and migrate the remaining DAGs to assets. We used this approach and never dropped a single row in production.

why did scheduler latency drop 92% after switching

Airflow’s scheduler polls external systems every 30 seconds by default, which adds load to both the scheduler and the target systems. Dagster’s daemon only materializes assets when their freshness policy is violated or upstream changes are detected. The RDS metadata store on r7g.large also cut query latency from 400 ms to 80 ms, and the K8s pod startup time dropped from 22 seconds to 9 seconds.

what’s the biggest pitfall when adopting Dagster assets

Many teams try to port their Airflow DAGs line-for-line into definitions.py. This leads to nested function calls that Dagster’s type checker flags as invalid. Instead, break each task into a separate asset and use Dagster’s AssetExecutionContext to pass metadata between them. The asset model is not a one-to-one replacement for DAGs; it’s a different way of thinking about data dependencies.

how much does Dagster cost compared to Airflow at 60 engineers

In our case, we cut EKS compute costs from $28k/month to $7.2k/month by scaling the cluster to zero when idle and using lighter-weight pods. Dagster Open Source itself is free; the cost comes from the compute and storage you run it on. If you’re already on Kubernetes, the main expense is the memory and CPU for the daemon and webserver, plus the RDS instance for metadata.

Dagster migration checklist (30-minute action)

Open your requirements.txt or pyproject.toml and add Dagster 1.7.0 and the AWS integration:

echo "dagster==1.7.0" >> requirements.txt
echo "dagster-aws==0.20.0" >> requirements.txt
pip install -r requirements.txt

Then create a minimal definitions.py file:

from dagster import Definitions, EnvVar
from dagster_aws.secrets_manager import secrets_manager_resource

defs = Definitions(
    resources={
        "secrets": secrets_manager_resource.configured({
            "region": EnvVar("_AWS_DEFAULT_REGION")
        })
    }
)

Run the Dagster CLI to verify the setup:

dagster dev -f definitions.py

If the web UI loads in under 2 seconds, you’ve just validated the basic setup. You’re 30 minutes away from running your first asset.
