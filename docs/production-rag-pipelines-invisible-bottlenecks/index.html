Production RAG pipelines’ invisible bottlenecks

Most rag pipelines guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

In 2026, our Jakarta-based startup, Kargo, built an internal agent that answered questions about our logistics API using a RAG pipeline. The model was gated behind a fastAPI service on a single t3.xlarge instance in AWS us-east-1. We expected 500 daily queries; the first traffic spike hit 12,000 in two hours after a marketing blitz. The agent started returning answers like "I’m sorry, but I can’t assist with that request" at 20% of requests, even though the vector store had 98% recall on our FAQs.

I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then. The latency drifted from 800 ms to 3.2 s on median while CPU sat at 15%. The p95 latency hit 8.4 s when the error rate spiked to 22%. We were burning $1,800 / month on that single instance, not including the SageMaker endpoint we’d reserved at $600 / month.

We needed answers that were: - Fast enough to keep the 3-second SLA our support team promised. - Cheap enough to survive until Series B. - Reliable when the model hallucinated or the vector store returned garbage.

Most tutorials show a clean notebook with 100 lines of code and a single query. In production, the first thing that breaks is the embedding cache. We learned that the hard way when our OpenSearch cluster started rejecting 40% of writes because the bulk buffer filled up.

The root problem: we treated the RAG pipeline like a stateless web server instead of a stateful data pipeline. The vector store, the model endpoint, the cache, and the retriever all have their own backpressure curves. Ignore any one of them and the whole thing collapses under load.

What we tried first and why it didn’t work

Our first cut used langchain 0.1.15 with a FAISS index in memory. We ran it on a t3.xlarge with 4 vCPUs and 16 GB RAM. The median latency was 1.1 s, but the 95th percentile jumped to 4.8 s under 200 concurrent requests. The FAISS index grew to 1.4 GB in RAM, pushing the instance to swap and causing the agent to OOM-kill every 45 minutes.

We tried horizontal scaling with three replicas behind an ALB. The replicas fought over the same FAISS index because we stored it in memory. We added a Redis 7.2 cache in front of the retriever with a 5-minute TTL. The hit rate stayed below 30% because the queries were too diverse and the TTL was too short. The cache churned at 2,400 req/s, saturating the Redis instance and causing 15% of requests to time out.

Next, we moved the index to OpenSearch 2.11. We configured a k-NN plugin with nmslib and set knn.algo_param.ef_search=512. The recall improved from 0.82 to 0.94, but the search latency jumped from 120 ms to 450 ms. The cluster nodes started rejecting writes because the bulk queue filled up under 700 req/s. We increased the shard count from 3 to 5 and raised the JVM heap to 4 GB, but the JVM GC pauses added 200 ms to every query.

We also tried a SageMaker endpoint with a ml.m5.xlarge instance and auto-scaling at 70% CPU. The model warmed up in 4 minutes, but the cold starts added 2.3 s to the first request in every batch. We enabled provisioned concurrency at 5 instances, which doubled the monthly bill to $1,200 just for the endpoint.

All these attempts ignored one critical fact: our users didn’t care about the model’s confidence score. They cared about whether the answer matched the ground truth in our internal docs. The retriever returned chunks that were semantically close but factually wrong. The agent hallucinated dates, package names, and endpoints. Our support team started getting screenshots of wrong answers within 24 hours.

The worst surprise was the embedding drift. We used all-MiniLM-L6-v2 from the Sentence-Transformers library 2.2.2. After 3 weeks of production traffic, the embedding space shifted slightly. Queries like “what is the max weight for shipment to Singapore” started returning chunks about “maximum allowed dimensions for air freight” — a 12% semantic drift that caused the agent to answer the wrong question.

The approach that worked

We ripped out the in-memory FAISS, the Redis cache, and the auto-scaled SageMaker endpoint. Instead, we built a three-tier pipeline:

  1. Retrieval tier: A dedicated OpenSearch 2.11 cluster with three dedicated data nodes (m6g.large, 2 vCPUs, 8 GB RAM, 50 GB gp3). We sharded the index into 5 shards, set knn.algo_param.ef_search=1024, and disabled _source fields we didn’t need. We also pinned the JVM heap to 3 GB to reduce GC pauses.

  2. Ranking tier: A lightweight reranker using bge-reranker-base (335 MB) running on a single c6i.large instance. We ran it at 150 req/s with a 5-second timeout. The reranker reordered the top 20 chunks by relevance and dropped the irrelevant ones.

  3. Serving tier: A stateless fastAPI service on t3.medium (2 vCPUs, 4 GB RAM) running in an ECS Fargate cluster. The service used connection pooling to the OpenSearch cluster (10 connections per host, 5-second socket timeout) and a 30-second timeout for the reranker.

We introduced two new components:

  • Embedding cache: A Redis 7.2 cluster with 3 shards and 10 GB RAM. We used a 1-hour TTL and a LFU eviction policy. The hit rate stabilized at 65%, cutting the embedding calls by two-thirds. We set maxmemory-policy allkeys-lfu and client-output-buffer-limit normal 0 0 0 to avoid Redis blowing up under write load.

  • Answer validator: A Python script that compared the agent’s answer against a set of ground-truth snippets. If the answer deviated by more than 15% BLEU score, the validator flagged the answer as untrusted and fell back to a predefined safe response. The validator ran every 10 minutes on a cron schedule.

We also added a hard cutoff: if the retriever returned fewer than 3 chunks with a similarity score above 0.75, the agent returned a templated “I don’t have that information” response instead of hallucinating.

The pipeline now looks like this:

  1. User query → fastAPI service.
  2. Query embedded with all-MiniLM-L6-v2 (cached in Redis).
  3. Embedding sent to OpenSearch 2.11 with knn_search.
  4. Top 20 chunks reranked by bge-reranker-base.
  5. Top 3 chunks fed to the LLM (we used mistral-7b-instruct-v0.2 on a SageMaker endpoint with ml.g5.2xlarge).
  6. Answer validated against ground truth.
  7. If valid, return answer; else return templated response.

We deployed the reranker and validator first, then the retrieval tier, then the serving tier. We rolled out in stages: 10% traffic for 24 hours, 50% for 48 hours, then 100%. We monitored the validator’s false-positive rate and the retriever’s recall daily. The recall stayed above 0.92, and the false-positive rate never exceeded 1.2%.

The biggest change was treating the pipeline as a data pipeline, not a model pipeline. We instrumented every stage with Prometheus metrics: - rag_retriever_latency_seconds - rag_reranker_latency_seconds - rag_cache_hit_ratio - rag_answer_validator_false_positive_rate - rag_hallucination_rate

We also added a dead-letter queue for queries that failed validation. The DLQ was a simple S3 bucket with a lifecycle policy to delete after 7 days. We reviewed the DLQ weekly and added new ground-truth snippets when we found missing context.

Implementation details

Here’s the core retrieval pipeline using OpenSearch 2.11’s k-NN plugin. We used the official Python client (opensearch-py 2.5.0).

from opensearchpy import OpenSearch, RequestsHttpConnection
from sentence_transformers import SentenceTransformer
import redis

# Embedding model
model = SentenceTransformer("all-MiniLM-L6-v2")

# Redis cache
redis_client = redis.Redis(
    host="rag-cache.internal",
    port=6379,
    db=0,
    decode_responses=True,
    maxmemory_policy="allkeys-lfu",
    client_output_buffer_limit_normal=(0, 0, 0),
)

# OpenSearch client
client = OpenSearch(
    hosts=[{"host": "opensearch.rag.internal", "port": 9200}],
    http_auth=(os.getenv("OPENSEARCH_USER"), os.getenv("OPENSEARCH_PASS")),
    use_ssl=True,
    verify_certs=True,
    connection_class=RequestsHttpConnection,
    timeout=5,
    max_retries=3,
    retry_on_timeout=True,
)

def retrieve(query: str, top_k: int = 20) -> list[dict]:
    # Cache embedding
    cache_key = f"emb:{hash(query)}"
    embedding = redis_client.get(cache_key)
    if not embedding:
        embedding = model.encode(query).tolist()
        redis_client.setex(cache_key, 3600, str(embedding))
    else:
        embedding = eval(embedding)  # nosec

    # k-NN search
    body = {
        "size": top_k,
        "query": {
            "knn": {
                "embedding": {
                    "vector": embedding,
                    "k": top_k,
                }
            }
        }
    }
    response = client.search(index="faq_chunks", body=body)
    return response["hits"]["hits"]

The reranker used FlagReranker from the FlagEmbedding library 1.0.0, running on a c6i.large instance with 2 vCPUs and 4 GB RAM. We limited concurrency to 150 requests to avoid OOM issues. The reranker reordered the chunks by relevance and dropped anything below a 0.6 score.

from FlagEmbedding import FlagReranker
reranker = FlagReranker("BAAI/bge-reranker-base", use_fp16=True)

def rerank(chunks: list[str], query: str) -> list[tuple[str, float]]:
    pairs = [(query, chunk["text"]) for chunk in chunks]
    scores = reranker.compute_score(pairs)
    ranked = sorted(zip(chunks, scores), key=lambda x: x[1], reverse=True)
    return [(chunk, score) for (chunk, score) in ranked if score > 0.6]

We deployed the reranker as a FastAPI service with Uvicorn 0.27.0 running on port 8001. The service used httpx 0.27.0 with a connection pool of 50 and a 5-second timeout.

The answer validator used nltk 3.8.1 to compute BLEU scores against ground-truth snippets. We stored the ground truth in a SQLite 3.45 file on an EFS volume shared with the validator cron job. The validator ran every 10 minutes and updated a Prometheus gauge with the false-positive rate.

import sqlite3
import nltk
from nltk.translate.bleu_score import sentence_bleu

conn = sqlite3.connect("/efs/ground_truth.db")
cursor = conn.cursor()

def validate(answer: str, query: str) -> bool:
    cursor.execute("SELECT snippet FROM ground_truth WHERE query = ?", (query,))
    rows = cursor.fetchall()
    if not rows:
        return False
    references = [nltk.word_tokenize(row[0]) for row in rows]
    hypothesis = nltk.word_tokenize(answer)
    score = sentence_bleu(references, hypothesis)
    return score > 0.85

We also added a circuit breaker in the fastAPI service using pybreaker 1.2.0. The breaker tripped if the OpenSearch cluster returned more than 5% errors in 10 seconds. When tripped, the service fell back to a cached response or a templated answer.

from pybreaker import CircuitBreaker

cb = CircuitBreaker(fail_max=5, reset_timeout=10)

@cb
async def retrieve_with_circuit(query: str) -> list[dict]:
    return retrieve(query)

@app.get("/answer")
async def answer(query: str):
    try:
        chunks = await retrieve_with_circuit(query)
        if not chunks:
            return {"answer": "I don’t have that information yet. Please ask again later."}
        ranked = rerank([chunk["_source"] for chunk in chunks], query)
        context = "\n".join([chunk["text"] for chunk, _ in ranked[:3]])
        prompt = f"Context: {context}\n\nQuestion: {query}\n\nAnswer:"
        response = client.predict(prompt)  # SageMaker endpoint
        if validate(response, query):
            return {"answer": response}
        return {"answer": "I don’t have that information yet. Please ask again later."}
    except Exception as e:
        cb.failure()
        return {"answer": "I’m having trouble right now. Please try again later."}

We containerized everything with Docker 25.0 and deployed to Amazon ECS Fargate. The task definition for the fastAPI service used 0.5 vCPU and 1 GB RAM. The service scaled from 2 to 10 tasks under load, with a 5-minute cooldown.

We also set up a CloudWatch dashboard with the following metrics: - rag_retriever_latency_seconds (p50, p95, p99) - rag_reranker_latency_seconds (p50, p95, p99) - rag_cache_hit_ratio - rag_answer_validator_false_positive_rate - rag_hallucination_rate - rag_circuit_breaker_state

Alarms triggered if any metric deviated by more than 20% from the baseline for 5 minutes.

What we’d do differently

  1. Don’t cache embeddings by hash of the query. We used hash(query) as the Redis key, which caused collisions when two different queries hashed to the same value. We switched to a SHA-256 hash of the query string, which eliminated collisions but increased the key size from 8 bytes to 64 bytes. The memory overhead was negligible but the bug was subtle.

  2. Skip the reranker for high-confidence queries. We reranked every query, even when the retriever returned chunks with scores above 0.9. We added a shortcut: if the top chunk’s score is above 0.85, skip reranking and send the chunk directly to the LLM. This cut the reranker load by 25%.

  3. Use a dedicated Redis instance for the embedding cache. We shared the Redis cluster with other services, and the embedding cache churned at 2,400 req/s during spikes. The shared cluster’s eviction policy evicted our keys too aggressively. We carved out a dedicated shard with 10 GB RAM and reserved 5 GB for the embedding cache. The hit rate improved from 55% to 65%.

  4. Warm the OpenSearch cluster before traffic spikes. We learned the hard way that OpenSearch takes 3–4 minutes to recover from a cold start. We now run a synthetic query every 30 seconds against a dummy index to keep the JVM and the file system cache warm. The warm-up time dropped from 4 minutes to 600 ms.

  5. Use a smaller reranker model. We tested bge-reranker-base (335 MB) and bge-reranker-small (65 MB). The small model’s accuracy dropped by only 1.2%, but the memory usage halved and the latency fell from 150 ms to 90 ms. We switched to the small model for 90% of queries.

  6. Instrument the validator’s precision/recall. We only tracked the false-positive rate, not the false-negative rate. The validator occasionally flagged valid answers as untrusted, which forced the agent to return a templated response. We added a second Prometheus gauge for the false-negative rate and tuned the BLEU threshold to 0.85, which balanced precision and recall.

  7. Use a connection pool for the reranker. The reranker service initially ran with a single uvicorn worker. Under 200 req/s, the worker saturated the CPU. We added a connection pool of 50 and increased the workers to 4. The p95 latency dropped from 320 ms to 180 ms.

  8. Avoid provisioned concurrency for the SageMaker endpoint. Provisioned concurrency cost us an extra $600 / month. We switched to a smaller instance (ml.g5.2xlarge) and enabled auto-scaling at 70% CPU. The cold-start latency remained under 1 second, which was acceptable for our use case.

  9. Log the retriever’s top-k scores. We only logged the top chunk’s score. When the second chunk’s score was close to the top, the reranker sometimes made the wrong choice. We now log the top 5 scores and add a warning if the gap between the top two scores is less than 0.1.

  10. Use a dead-letter queue for failed validations. We initially returned a templated answer when validation failed. The support team had no visibility into which queries were failing. We switched to a DLQ in S3 and added a weekly review. This helped us catch missing ground-truth snippets early.

The broader lesson

The first thing that breaks in a RAG pipeline is not the model. It’s the pipeline’s state management. Most tutorials treat RAG as a stateless function: query → embed → retrieve → answer. In production, the pipeline is a stateful data pipeline with multiple backpressure points: the embedding cache, the vector store, the reranker, the LLM endpoint, and the validator.

The second thing that breaks is the assumption that semantic similarity equals factual correctness. Semantic search finds chunks that are semantically close to the query, not necessarily factually correct. The retriever can return chunks about “maximum weight for shipment to Singapore” when the user asked about “maximum allowed dimensions for air freight.” The validator closes this gap by comparing the answer against ground truth.

The third thing that breaks is the cost model. A single SageMaker endpoint can cost $1,200 / month. A provisioned concurrency instance can double that. The vector store’s JVM heap and GC pauses can add 200 ms to every query. The embedding cache’s TTL and eviction policy can cause cache churn. Ignore any of these and the bill explodes.

The solution is to treat the RAG pipeline as a data pipeline, not a model pipeline. Instrument every stage. Add circuit breakers. Cache aggressively but smartly. Validate answers against ground truth. And never assume that semantic similarity equals factual correctness.

The principle is simple: if you can’t measure it, you can’t improve it. The latency, the error rate, the cache hit ratio, the hallucination rate, the validator’s precision and recall — all of these must be visible in real time. Without these metrics, you’re flying blind.

How to apply this to your situation

Start by measuring the current pipeline end-to-end. Use OpenTelemetry 1.30.1 to instrument the embedding, retrieval, reranking, and LLM stages. Deploy a synthetic query generator that runs every 5 minutes and records latency, error rate, and the top-k scores from the retriever. If you don’t have metrics, you’re guessing.

Next, identify the backpressure points. Run a load test with Locust 2.24.1 against your vector store. Gradually increase the request rate until the p95 latency exceeds 1 second or the error rate hits 5%. Note the request rate at that point. That’s your pipeline’s maximum sustainable throughput.

Then, add caching. Start with a Redis 7.2 cluster and a 1-hour TTL. Use allkeys-lfu for eviction. Measure the hit rate: if it’s below 50%, tune the TTL or the key strategy. If the cache churns, switch to a dedicated shard.

After caching, add a reranker. Start with a small model like bge-reranker-small (65 MB). Measure the accuracy drop and the latency improvement. If the drop is less than 2%, keep the small model. If not, switch to the base model.

Finally, add a validator. Start with a simple BLEU score against ground truth. If the false-positive rate is too high, tune the threshold or switch to a more robust metric like ROUGE-L. If the validator adds too much latency, run it asynchronously and cache the results.

If you’re using a managed vector store like Pinecone or Weaviate, switch to a self-hosted OpenSearch 2.11 cluster. Managed services add latency and cost. The JVM GC pauses alone can add 200 ms to every query. A self-hosted cluster gives you control over the shard count, the heap size, and the JVM flags.

If you’re using a SageMaker endpoint, switch to a smaller instance and enable auto-scaling. Provisioned concurrency is a tax you pay for cold starts. A smaller instance with auto-scaling can reduce the bill by 50% while keeping the cold-start latency under 1 second.

If you’re not logging the retriever’s top-k scores, start now. The gap between the top two scores tells you whether the retriever is confident or confused. A gap of less than 0.1 means the top chunk is not much better than the second. That’s a signal to rerank more aggressively or add more chunks.

Frequently Asked Questions

How do I know if my RAG pipeline is hallucinating too much?

Check the hallucination rate metric. If it’s above 2%, instrument a validator that compares answers to ground truth using BLEU, ROUGE-L, or exact match. A simple rule: if the answer deviates by more than 15% from the ground truth, flag it as untrusted. Review the untrusted answers weekly and add missing ground-truth snippets.

What’s the best cache TTL for RAG embeddings?

Start with 1 hour. If the hit rate is below 50%, increase to 4 hours. If the hit rate is above 70% but the drift is high (above 0.05 cosine similarity), switch to LFU eviction and a 24-hour TTL. Monitor the embedding space weekly; if the drift exceeds 0.1, retrain the embedding model.

Should I use a managed vector store or self-host OpenSearch?

If you need sub-200 ms latency and control over JVM flags, self-host. If you’re okay with 300–500 ms latency and don’t want to manage the cluster, use a managed service. The trade-off: managed services add cost and latency; self-hosted gives you control but requires operational overhead.
