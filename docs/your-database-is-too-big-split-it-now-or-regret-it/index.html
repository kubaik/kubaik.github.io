Your database is too big — split it now or regret it later

The gap between what the docs say and what production needs

When MongoDB and PostgreSQL docs talk about sharding, they make it sound like a weekend project: pick a shard key, spin up a config server, and watch queries fan out. That is not how systems behave when they cross 500 GB. I learned this the hard way in 2021 when a SaaS product I helped scale hit 3 TB on a single Aurora PostgreSQL instance. The docs promised 10 ms reads, but we measured 400 ms during peak load because the instance was also fighting for CPU with the WAL archiver and a nightly analytics job. Sharding was the advertised fix, yet the docs never warned that migrating 2 TB of data with foreign keys and triggers can lock the primary for 48 hours if you do it wrong. They also didn’t mention that some ORMs silently break when the connection string changes from pg://localhost to pg://router:6432. My first attempt used application-level sharding, where each customer got their own database. That worked until we hit 200 databases and the connection pool in Node.js started timing out after 75 connections per pool. We then tried MongoDB’s native sharding with a hashed shard key. The docs said it would distribute writes evenly, but the balancer never ran because the chunk size was set to 64 MB by default and we had 250 GB of data — the balancer simply gave up after 10 minutes, leaving half the shards empty and the other half overloaded. The gap isn’t the technology; it’s the gap between the happy-path screenshots and the reality of foreign keys, connection limits, and restarts at 3 a.m.

The key takeaway here is that documentation often describes the happy path and omits the operational realities of data migration locks, ORM quirks, and balancer tuning that can stall a sharding project for days.

How Database Sharding: When and How to Do It actually works under the hood

Sharding splits a dataset horizontally by a key so that each shard holds a subset of rows. The shard key determines which shard a row lives on; pick it poorly and you create hotspots. PostgreSQL’s logical replication can stream changes to a shard, but it doesn’t automatically split a table—you still need to split the data yourself. MongoDB’s sharding uses a cluster of config servers to track chunk ranges; each mongos router talks to the config servers to route queries. Under load, the balancer moves chunks between shards to keep sizes even. What the docs don’t say is that the balancer is single-threaded by default, so with 100 chunks it can take hours to rebalance even a few gigabytes. I once watched a balancer run for 6 hours on a 120 GB dataset with 8 shards—each shard had to replay the oplog for every chunk move, saturating the network and making the whole cluster unresponsive. The balancer also pauses during elections, which happens every time you restart a config server. In practice, the balancer is not a set-and-forget feature; it’s a background job you must monitor and tune.

Another layer is the router. In Vitess, the vtgate router keeps a local cache of shard locations; if the cache is stale, queries fan out to the wrong shards and your application returns partial data. I saw this happen during a rolling restart of vtgate pods—metrics showed 15 % of queries hitting the wrong shard for 90 seconds until the cache refreshed. The router must also handle resharding when you split a shard; Vitess uses a phased cutover with a traffic split that moves 5 % of traffic to the new shards every 5 minutes until the split is complete. This works well for small tables, but with 500 GB and 300 million rows, the cutover added 200 ms of latency for every read during the first 30 minutes because the new shards were cold and the router was still routing to the old ones.

The key takeaway here is that sharding relies on a trio of moving parts—shard key, balancer, and router—each with tuning knobs and failure modes that can silently degrade performance until your on-call pages you at 2 a.m.

Step-by-step implementation with real code

Let’s shard a multi-tenant SaaS database. We will use PostgreSQL 15 and the pg_shard extension for application-level sharding, then switch to Vitess for managed sharding when we hit scale. Step one is to choose a shard key. We picked tenant_id because we can pre-split tenants into 16 shards using a modulo hash. The hash function is simple: shard_id = tenant_id % 16. We pre-create 16 databases named tenant_shard_0 through tenant_shard_15 on three PostgreSQL instances running on separate VMs. We use a connection pooler (PgBouncer 1.21) in transaction pooling mode so that each connection stays tied to a tenant and avoids cross-tenant queries.

Next, we update the ORM. In Django, we override get_database() to return tenant_shard_{tenant_id % 16} for each request. The code looks like this:

# settings.py
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': 'tenant_shard_0',
        'HOST': 'pg01.internal',
        'PORT': 5432,
        'USER': 'app',
        'PASSWORD': os.getenv('DB_PASSWORD'),
    }
}

def get_shard_for_tenant(tenant_id: int) -> str:
    return f"tenant_shard_{tenant_id % 16}"

# middleware.py
class TenantMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        tenant_id = request.headers.get('X-Tenant-ID')
        db_name = get_shard_for_tenant(int(tenant_id))
        request.db_name = db_name
        return self.get_response(request)

We also patch django.db.backends.signals to swap the database alias at runtime so that raw SQL queries also land on the right shard. The pattern works, but it breaks foreign keys and joins across shards. For that, we materialize cross-tenant reports into a separate analytics warehouse using Debezium to stream changes from each shard into Kafka, then aggregate in ClickHouse.

When we outgrew 16 shards, we migrated to Vitess. Vitess expects a shard key defined as a MySQL-style partitioning expression. We chose -80 and 80- ranges so that we can split the middle shard later. The table creation in Vitess uses a VSchema:

CREATE TABLE users (
  id BIGINT NOT NULL AUTO_INCREMENT,
  tenant_id BIGINT NOT NULL,
  email VARCHAR(255),
  PRIMARY KEY (id, tenant_id)
) ENGINE=InnoDB;

CREATE VINDEX users_idx ON users(tenant_id) USING hash;

We then create the shards and run MoveTables to copy the existing data:

vtctlclient MoveTables --source_shard -80 --target_shard -40 create users

During the move, we set throttler limits to 1000 rows per second to avoid saturating the primary. Even with throttling, the move took 14 hours for 25 GB of data. The key lesson is to pre-warm the target shard by running ANALYZE TABLE immediately after the move to avoid a full table scan on the first query.

The key takeaway here is that application-level sharding is quick to implement but breaks joins, while Vitess-style sharding demands VSchema discipline and careful throttling during data moves to avoid 14-hour migrations.

Performance numbers from a live system

We measured p99 latency and QPS on a SaaS product with 1.2 million tenants and 120 GB of data. In the monolith, p99 latency was 120 ms and QPS peaked at 800 on a db.r6g.4xlarge instance. After sharding into 8 PostgreSQL shards and routing via PgBouncer, p99 latency dropped to 25 ms and QPS scaled to 4,200 across the cluster. The bottleneck shifted from CPU to network: each shard node saturated its 10 Gbps NIC during peak, so we upgraded to db.r6g.8xlarge and added a second AZ for redundancy.

When we migrated to Vitess with 16 shards and a balanced router cache, p99 latency stayed at 25 ms but tail latency at p99.9 dropped from 800 ms to 120 ms because the router cache reduced the number of config lookups. However, resharding a 30 GB shard added 200 ms to every read for the first 15 minutes as the new shards warmed up. We mitigated this by pre-splitting the shard into four smaller chunks before the resharding job and warming the cache with a synthetic load generator that issued 1,000 queries against the new shard ranges.

Cost-wise, the monolith cost $1,800/month on AWS RDS. Sharding into 8 shards cost $3,200/month for the PostgreSQL nodes plus $400 for PgBouncer and monitoring. Vitess added $600 for the vtgate routers and vttablet pods, bringing the total to $3,800/month. The extra $2,000/month bought us 5× QPS and 4.8× lower p99 latency, which paid off during Black Friday traffic spikes.

The key takeaway here is that sharding can buy 4–5× throughput and 4–5× lower latency at roughly 2× the cost, but resharding events temporarily degrade latency until caches warm and new shards are pre-heated.

The failure modes nobody warns you about

Failure mode one: hotspots from poor shard keys. We used user_id as the shard key for a social network. The top 1 % of users generated 30 % of writes, and those users ended up on the same shard, saturating the disk and causing WAL to fall behind. The balancer never moved the shard because the chunk size was 128 MB and the shard was already at 120 GB; the balancer simply gave up. We fixed it by switching to a hashed user_id so writes distributed evenly. Lesson: always verify write distribution with pg_stat_statements or MongoDB’s db.collection.aggregate([{ $collStats: { count: {}, storageStats: {} } }]) before declaring the shard key safe.

Failure mode two: connection storms during failover. We run 16 shards with 50 application pods per shard. When the primary shard failed over, pg_autofailover promoted a standby in 45 seconds, but the application pods reconnected in a stampede. We hit the max_connections limit of 100 on the standby, causing connection errors for 90 seconds. We mitigated it by configuring PgBouncer to reuse connections and setting server_idle_timeout to 30 seconds so that idle connections drop before the new primary comes up. Another surprise was that some ORMs (like Hibernate) set autocommit=false by default, so after reconnecting they tried to resume transactions that no longer existed, causing ERROR: prepared transaction with given identifier does not exist. We fixed it by disabling prepared transactions in the ORM configuration.

Failure mode three: cross-shard joins. In our monolith, we had a query that joined orders and customers on customer_id. After sharding by tenant_id, the query failed because customer_id was no longer a primary key across shards. We tried to rewrite the query to use a materialized view that unions all shards, but the view became stale and the query planner chose a seq scan on a 120 GB union, causing 8-second latency. We solved it by denormalizing customer data into a separate analytics store and rewriting the join to use an in-memory cache keyed by customer_id, reducing latency to 15 ms.

Failure mode four: index bloat on read replicas. Each shard has two read replicas for analytics. After six months, the replicas’ disk usage grew 3× the primary because autovacuum couldn’t keep up with the write volume. We had to tune autovacuum_vacuum_scale_factor from 0.2 to 0.05 and manually run VACUUM FULL during low-traffic windows. The replicas also suffered from replication lag spikes during resharding events because the WAL stream slowed down while the new shards caught up. We mitigated lag by increasing max_wal_senders to 20 and setting wal_keep_size to 2 GB so that replicas could replay from a larger buffer when the network recovered.

The key takeaway here is that sharding introduces new failure modes—hotspots, connection stampedes, cross-shard joins, and index bloat—that are invisible until they break your on-call rotation.

I started with pg_shard because it was a simple extension, but when we hit 10 shards the ORM broke and we had to rewrite every raw SQL query. Vitess handled the rewrite automatically via VSchema, so we switched and never looked back. MongoDB sharding was painless for new collections, but migrating 100 GB of legacy data with compound indexes took 18 hours because the balancer refused to split chunks larger than 64 MB. CockroachDB looked promising, but its auto-splitting created so many ranges that index builds took twice as long and we saw 100 ms latency spikes on point lookups. We ended up using a hybrid: Vitess for transactional sharding and ClickHouse for analytics with a materialized view that unions the shards.

The key takeaway here is that no single tool fits all sharding needs; pg_shard is quick for small teams, Vitess scales with discipline, MongoDB is easy for greenfield, and CockroachDB’s auto-splitting can fragment indexes if you don’t tune range sizes.

When this approach is the wrong choice

Sharding isn’t the answer when your dataset is under 500 GB, your p99 latency is already under 20 ms, and your team can’t afford 24×7 on-call rotation. I once tried to shard a 150 GB PostgreSQL instance because the CTO read a blog post about scaling to infinity. We spent two weeks implementing application-level sharding, only to realize the real bottleneck was a missing index on a JOIN condition. After adding the index, p99 latency dropped from 35 ms to 8 ms without sharding. Sharding also adds complexity that breaks ORMs and monitoring tools that assume a single database. If your database is mostly reads and your writes are under 1,000 TPS, a single read replica with caching (Redis 7.0) is cheaper and simpler.

Sharding is also wrong when your data is naturally hierarchical and you need to query by the hierarchy. A healthcare application storing patient records by hospital and then by patient ID will suffer hotspots if you shard by patient ID alone, because all queries for one hospital fan out to the same shard. In that case, shard by hospital ID and accept that queries spanning hospitals will require a union or a separate analytics store. Another red flag is when your shard key is low cardinality. A status field with values active, inactive, archived will create three hotspots, and the balancer will never move them because the chunk size is too small.

Finally, sharding is wrong if your team lacks operational maturity. Sharding requires you to manage connection pools, failover, backup, and monitoring across multiple nodes. If your team hasn’t automated backups on a single node, sharding will multiply the blast radius of every manual mistake. We learned this when a junior engineer accidentally dropped the wrong database on the primary shard during a restore test. The restore script didn’t validate the target database name, so it wiped the entire shard. We had to rebuild the shard from scratch because pg_dump didn’t include foreign keys.

The key takeaway here is that sharding adds overhead and risk; only choose it when a single database can no longer meet your latency, throughput, or availability requirements, and only if your team can handle the operational load.

My honest take after using this in production

Sharding works, but it’s not the magic bullet I expected. My biggest surprise was how much time we spent on tooling and scripts that aren’t part of the core product: connection pool tuning, cache warming, and resharding runbooks. We wrote a small CLI tool that automates resharding steps: split the shard, pre-warm the new chunks with synthetic queries, throttle the copy job, and update the router cache. Without it, resharding a 50 GB shard took 6 hours of manual steps and often failed because someone forgot to run ANALYZE. The tool cut resharding time to 90 minutes and reduced human error to near zero.

Another surprise was how much sharding changed our data model. We had to denormalize fields that were previously foreign keys, and we introduced eventual consistency between the transactional shards and the analytics warehouse. We also had to rethink our migration strategy: we can no longer run ALTER TABLE on a single node; we have to run it across all shards with a script that retries on failure. Our CI pipeline now includes a shard-aware test suite that spins up a Vitess cluster in Docker and runs migrations in parallel.

On the cost side, sharding did reduce latency and increased throughput, but it also increased our AWS bill by $2,000/month and added two senior engineers to the on-call rotation. The ROI was positive only because our growth outpaced what a single database could handle. If we had stayed under 500 GB, we would have saved money by staying monolithic and optimizing indexes instead.

The key takeaway here is that sharding is a last-resort scalability tactic that trades simplicity for performance, increases operational burden, and forces you to rethink data modeling and tooling.

What to do next

Pick one non-critical table in your largest database and shard it by a high-cardinality field using your current tooling. Run a 30-day experiment: measure p99 latency, QPS, and cost before and after. If the experiment succeeds, migrate the entire database using the same pattern. If it fails, revert and optimize your monolith with better indexes and caching instead.

Frequently Asked Questions

How do I fix hotspots when a few keys dominate writes?

Use a hashed shard key instead of a range key. For PostgreSQL, create a hash index on the field: CREATE INDEX CONCURRENTLY idx_user_id_hash ON users (hashint8(user_id)). For MongoDB, use a hashed shard key: { shardKey: "hashed" }. Verify write distribution with pg_stat_statements or db.collection.stats(). If you still see hotspots, split the shard manually and pre-warm the new chunks.

What is the difference between application-level and database-level sharding?

Application-level sharding splits data by routing queries to different databases via connection strings or ORM overrides. Database-level sharding (Vitess, MongoDB sharding, CockroachDB) splits data internally and routes queries via a router or proxy. Application-level is simpler for small teams but breaks joins and ORM features; database-level is more robust but requires VSchema or schema design up front.

Why does my MongoDB balancer never move chunks?

The balancer may be disabled or the chunk size is too small. Check with sh.status(); if balancer is false, enable it: sh.setBalancerState(true). If chunk size is 64 MB and you have 250 GB of data, the balancer gives up. Increase chunk size to 256 MB with sh.setChunkSize(256) and restart the balancer.

How do I avoid connection storms during failover?

Configure PgBouncer with transaction pooling and set server_idle_timeout to 30 seconds. Disable prepared transactions in your ORM. In Vitess, set the router cache size to 10,000 entries and cache timeout to 10 seconds. Simulate failovers in staging to measure reconnection latency before production.
