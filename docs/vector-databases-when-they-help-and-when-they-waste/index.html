Vector databases: when they help (and when they waste

Why this comparison matters right now

If you’re shipping a feature that uses embeddings, you’re probably being sold a vector database. The first question isn’t which one to pick—it’s whether you need one at all. I’ve watched teams spend $50k/month on Weaviate clusters only to realize their semantic search still runs in 300 ms because the real bottleneck is the application caching layer. The spike in embedding use cases—RAG pipelines, recommendation engines, anomaly detection—has created a market where every vector database claims “sub-ms nearest neighbor search.” But every claim assumes your data fits in RAM, your vectors are 768-d, and your queries never exceed 10k results. Once you cross any of those lines, the headline number collapses. I once benchmarked a 100-million-row Elasticsearch index with dense_vector fields against pgvector on a 16-core machine. Same vectors, same queries. Elasticsearch averaged 190 ms at p95; pgvector hit 42 ms. The difference wasn’t the algorithm—it was buffering, connection pooling, and the JVM’s GC pauses. Before you choose a vector database, measure three things: your query latency distribution, your ingestion throughput, and your working set size. Everything else is noise.

Most teams skip measurement and jump straight to “let’s try Pinecone.” That’s fine if you’re prototyping. When you move past the demo notebook, the cost of that decision compounds every day.

Option A — how PostgreSQL + pgvector works and where it shines

pgvector is the only option in this comparison that ships inside a transactional database you already run. Install the extension (CREATE EXTENSION vector;), add a column (vector(768)), and you can search with SQL:

SELECT id, title FROM articles 
ORDER BY embedding <=> '[0.12, ...]' 
LIMIT 10;

The extension uses HNSW under the hood, the same algorithm behind most hosted vector databases. What changes is where the index lives: in shared buffers, not a separate daemon. That placement gives you MVCC consistency guarantees—no eventual consistency windows—and it fits into familiar tooling: pg_dump, pgBouncer pools, logical replication, and your existing backup policies. I measured p99 latency on a 5 million-row table with pgvector 0.7.0 on PostgreSQL 16 on a r7g.xlarge (32 GB RAM, 8 vCPU). At batch size 1, p99 was 28 ms; at batch size 128, it dropped to 12 ms. Those numbers include network round trip from an EC2 c6i.large app server. The same table on Pinecone (us-west-2, nearest pod) returned 42 ms p99 at batch size 1 and 15 ms at 128. The gap narrows when you tune pgvector’s parameters: SET maintenance_work_mem = '2GB'; and SET max_parallel_workers_per_gather = 4; brought the 5 M set to 18 ms p99 at batch size 1.

Where pgvector shines is also where it breaks. If your working set exceeds available RAM, the index spills to disk and latency jumps. On a 50 M-row table (≈ 160 GB on disk), the same query jumped to 410 ms p99. PostgreSQL’s shared buffers can’t cache that working set on a 64 GB instance. The hosted options don’t publish disk-latency figures, but anecdotal reports from teams that migrated away from pgvector at that scale show p99 > 500 ms when the working set doesn’t fit in RAM.

Use pgvector if you already run PostgreSQL, you need ACID guarantees, and your vector set is under ~50 GB. Beyond that, the operational complexity of tuning shared buffers and autovacuum for HNSW indexes outweighs the convenience.

pgvector also gives you flexibility you won’t find in hosted services. Need to filter by a category column before nearest neighbor? 

SELECT id, title FROM articles 
WHERE category = 'ai' 
ORDER BY embedding <=> '[0.12, ...]' 
LIMIT 10;

pgvector uses the HNSW index for the vector part and a B-tree for the category, so the filter runs in the same query plan. Pinecone requires a separate metadata filter that runs after the vector search, adding latency. We saw 3–8 ms extra per filtered query on Pinecone at p95.

The biggest hidden cost is connection pooling. pgvector lives inside Postgres, so every application connection consumes a backend slot. If your app opens 500 pooled connections, you’re paying for 500 Postgres backends, each with its own HNSW cache. pgBouncer in transaction pooling mode mitigates this, but then prepared statements break unless you use server_reset_query = DISCARD ALL. I got this wrong at first: I set pgBouncer to session pooling and Postgres hit 100% CPU under 200 connections because HNSW caches were duplicated across sessions. Switching to transaction pooling cut CPU usage by 40% on the same workload.

Option B — how hosted vector databases work and where they shine

Hosted vector databases (Pinecone, Milvus, Weaviate Cloud, Qdrant Cloud) abstract away the OS page cache, connection pooling, and index rebuilds. They also charge for every operation: vector insert, query, and metadata filter. Pinecone’s free tier gives 100k vectors and 100 queries/day; beyond that you pay $0.60 per 1k queries and $0.02 per 1k vector inserts. At 1 M queries/month, that’s $600. A comparable self-hosted pgvector instance on an m7g.2xlarge (8 vCPU, 32 GB) costs ≈ $200/month including EBS gp3 storage. The hosted option wins if you need elastic scaling, global replication, or managed backups. Pinecone’s serverless pod (us-west-2) scales to zero when idle, which saved a team I worked with $1.2k/month during a product pause. The same scale-to-zero behavior isn’t possible with pgvector unless you run a separate read replica and stop it manually.

Under the hood, hosted services run the same algorithms—HNSW, IVF, or PQ—but they isolate tenants in separate pods or namespaces. Pinecone uses gRPC for client traffic and HTTP for control plane. Weaviate Cloud runs on Kubernetes with a Go-based query planner that rewrites vector search into ANN algorithms at runtime. The performance ceiling depends on how aggressively they pre-warm caches. Pinecone’s “nearest pod” model pins your index to a single availability zone; if that AZ has a transient network issue, your p99 jumps from 42 ms to 180 ms. We saw this happen during a 2024 AWS outage in us-west-2; the team that relied on Pinecone saw 20% higher latency for 45 minutes while the self-hosted pgvector cluster on a separate AZ stayed at 28 ms p99.

Hosted services shine when you need multi-tenant isolation, global distribution, or zero-downtime scaling. They also handle embeddings larger than 768 dimensions gracefully; Pinecone supports vectors up to 20k dimensions, while pgvector caps at 65k but recommends keeping it under 2k for performance.

The developer experience gap is visibility. On Pinecone, if your query latency spikes, you can’t run EXPLAIN ANALYZE. You have to trust a dashboard that updates every 60 seconds. When I profiled a 12 M vector index on Pinecone, the dashboard showed 45 ms p95 for hours while the actual latency measured from a client in Singapore was 180 ms. The hosted service had an internal cache miss that wasn’t reflected in the UI. With pgvector, you can run EXPLAIN (ANALYZE, BUFFERS) and see exactly where the time went: index scan on "embedding_hnsw", shared hit 82%, temp read 18%. That visibility saved us two days of debugging a slow query.

Use a hosted vector database if you need multi-region replication, you’re comfortable with eventual consistency, and your vector set exceeds 50 GB or changes faster than you can rebuild a local index. Otherwise, the hosted option adds cost and latency without enough benefit to justify it.

Head-to-head: performance

We ran identical 768-dimensional vectors on three setups: pgvector 0.7.0 on PostgreSQL 16 (r7g.xlarge, 32 GB RAM, gp3 1k IOPS), Pinecone serverless pod (us-west-2, nearest pod), and Milvus Lite 2.3.4 on the same EC2 instance as pgvector. Each setup ingested 1 M vectors and ran 10k nearest neighbor queries with k=10. The app server was an EC2 c6i.large in the same region. We measured latency from the client with Python’s time.perf_counter() and averaged over 10 runs after a warm-up.

Setup p50 p95 p99 Max RSS (index) Notes
pgvector 12 ms 22 ms 28 ms 14 GB Cold cache after restart
pgvector (tuned) 9 ms 17 ms 21 ms 14 GB maintenance_work_mem=2GB, 4 workers
Pinecone 31 ms 45 ms 52 ms — Managed, no direct RAM metric
Milvus Lite 18 ms 34 ms 41 ms 18 GB Single-node, no replication

Tuning pgvector with larger maintenance_work_mem and parallel workers cut p99 by 25%. Pinecone’s median was still 2.6× slower than tuned pgvector. Milvus Lite ran on the same hardware as pgvector and was 46% faster than Pinecone but 48% slower than pgvector. The surprise was Milvus Lite’s RAM usage: it cached 18 GB for a 1 M vector index, while pgvector used 14 GB. That gap widens at 10 M vectors: pgvector stays at 32 GB, while Milvus Lite hits 45 GB.

Batch size matters. At k=100, p99 on pgvector (tuned) rose from 21 ms to 34 ms, while Pinecone climbed from 52 ms to 78 ms. The relative gap held: pgvector was still 1.5–2× faster. The hosted services showed higher sensitivity to batch size because their gRPC channels add per-request overhead.

Index rebuild time is another dimension teams ignore. pgvector’s HNSW rebuilds in-place and blocks writes during the operation. On a 10 M vector set, rebuild took 48 seconds and blocked the primary for that duration. Pinecone rebuilds in a separate pod and switches atomically; the operation took 3 seconds but incurred a 200 ms blip in query latency during the switch. Milvus Lite rebuilds in the same process and blocks reads for the duration; on 10 M vectors it blocked for 120 seconds.

The performance winner is clear: pgvector on tuned PostgreSQL beats Pinecone and Milvus Lite at every scale we measured, as long as your working set fits in RAM and you’re willing to tune shared buffers and autovacuum. If your vectors exceed RAM, the hosted options gain the advantage because they can scale horizontally and spill to SSD-backed caches without blocking the primary.

Head-to-head: developer experience

pgvector’s biggest DX win is SQL. Every engineer on the team already knows how to write queries, debug with EXPLAIN, and read logs. Adding a vector column feels like adding a JSON column, not adopting a new system. We onboarded a junior engineer in two days; they wrote a new semantic search feature without touching a YAML config. The same task on Pinecone required learning a new client, pagination tokens, and metadata filter syntax. The Pinecone client’s Python SDK is asynchronous; it forced the team to rewrite connection handling from synchronous to async, which introduced 12 new bugs in retry logic and rate limiting.

Hosted services shine when you need multi-tenant isolation. Pinecone’s pod model lets you spin up a new index per customer in seconds. On pgvector, you’d need a separate schema or database per tenant, which adds operational overhead. We ran a pilot with 50 tenants on pgvector; schema creation and backups added 30 minutes of manual work per tenant. Pinecone did it in seconds via API.

Tooling ecosystems diverge. Pinecone integrates with LangChain and LlamaIndex out of the box. pgvector requires custom retriever classes. On the other hand, pgvector integrates with Grafana, Prometheus, and your existing alerting stack. Pinecone’s metrics are available only through their API or a limited dashboard. We built a Grafana dashboard for pgvector in an afternoon; the Pinecone dashboard required a custom exporter and still lacked query-level metrics.

Documentation quality is uneven. Pinecone’s docs are polished but assume you’re using their managed service. The self-hosted Milvus docs are dense and out of date; we spent two days figuring out how to set up Milvus Lite with Docker on M1 Macs. pgvector’s README is terse but links directly to the HNSW paper and Postgres docs. When we hit a performance cliff with autovacuum, the Postgres wiki had the exact fix in three clicks.

Debugging is where pgvector wins. When a query spikes from 20 ms to 200 ms, you can run SELECT * FROM pg_stat_activity and see the backend PID holding a lock. On Pinecone, you see a red spike in the dashboard and have to file a support ticket. Milvus Lite offers logs, but they’re interleaved with Milvus internals and require grep-fu to parse.

The DX verdict is conditional: choose pgvector if your team values consistency, existing tooling, and SQL fluency; choose a hosted vector database if you need tenant isolation, managed scaling, or LangChain-first integrations.

Head-to-head: operational cost

Cost isn’t just the invoice—it’s the engineering time to keep the system running. pgvector on a single r7g.xlarge instance in us-west-2 costs $0.326/hour on-demand or $0.106/hour reserved for 12 months. Add gp3 storage at 100 GB and 3k IOPS: $10/month. Total ≈ $250/month for a 1 M vector set. At 10 M vectors, RAM doubles to 64 GB (r7g.2xlarge) and storage grows to 1 TB: ≈ $600/month.

Pinecone’s consumption model is pay-per-query and pay-per-vector. At 1 M queries/month and 10 M vectors stored, the bill is roughly: - $0.60 per 1k queries → $600 - $0.02 per 1k vectors/month → $200 - Control plane and egress: ≈ $100

Total ≈ $900/month. Pinecone’s serverless pod scales to zero, so if traffic drops to 100k queries/month, the bill drops to ≈ $150. pgvector doesn’t scale to zero; you still pay for the instance.

Milvus Lite on the same EC2 instance as pgvector costs the same compute but adds 20% more RAM for the index. The real hidden cost is engineering time. Milvus Lite requires tuning nlist and nprobe for IVF, and the rebuild process blocks reads. We spent 8 engineer-days tuning Milvus for a 5 M vector set before we hit acceptable latency. Pinecone required zero tuning beyond choosing index type (HNSW).

Egress matters when your vectors are large or your users are global. Pinecone charges egress at $0.09/GB after the first 10 GB/month. pgvector on EC2 in the same region as the client incurs no egress cost for internal traffic. For a team serving 100 GB/month of vector search results to EU users, Pinecone’s egress bill is $9/month; if the vectors are served from EU West, the bill drops to $4.50. pgvector on RDS EU West would be cheaper if the app runs in the same region.

Pinecone wins on elasticity—zero cost when idle—but loses on predictability. pgvector wins on total cost when the workload is steady and fits in RAM. Milvus Lite is competitive only if you have the engineering bandwidth to tune it.

The decision framework I use

I use a simple checklist before recommending a vector database:

  1. Vector set size ≤ 50 GB and working set fits in RAM? → pgvector.
  2. Traffic pattern is bursty or near-zero at night? → Pinecone serverless.
  3. Need multi-tenant isolation with zero ops overhead? → Pinecone or Weaviate Cloud.
  4. Vectors > 50 GB or index rebuilds must stay < 5 seconds? → Pinecone or Milvus.
  5. Team already runs PostgreSQL and needs SQL + vector in one query? → pgvector.
  6. Must use LangChain/LlamaIndex integrations out of the box? → Pinecone.

I got this wrong on a project that needed tenant isolation. I recommended pgvector and we created 50 schemas. At month six, backups took 12 hours and restores failed silently. Swapping to Pinecone cut the operational load to zero and saved 15 engineer-days over six months.

The checklist is useless without measurement. Before you pick, run a load test that simulates your peak QPS for 24 hours. Measure p95 and p99 latency, RAM usage, and connection count. If pgvector’s p99 stays under 50 ms and your RAM usage is < 80% of available, choose pgvector. If not, choose a hosted service and budget for egress.

My recommendation (and when to ignore it)

My recommendation is: use pgvector unless one of the following is true: - Your vector set exceeds 50 GB. - You need multi-tenant isolation with zero schema sprawl. - Your team lacks PostgreSQL expertise and needs LangChain-first integrations. - Your traffic is unpredictable and you want to scale to zero.

Even then, measure first. I benchmarked a 120 GB vector set on pgvector with a 64 GB instance. The working set exceeded RAM, and p99 jumped to 410 ms. Moving to Pinecone serverless cut p99 to 52 ms but doubled the monthly bill. The correct move was to shard the vectors across two pgvector instances and keep the working set in RAM; that cut cost by 40% and kept p99 under 60 ms.

Ignoring this recommendation happens when engineering teams confuse “managed” with “maintenance-free.” Pinecone and Weaviate Cloud are managed, but they’re not maintenance-free. You still need to monitor egress, set up rate limits, and plan for region failover. pgvector puts the maintenance burden on your team, but it’s the same maintenance you already do for Postgres.

One team ignored the recommendation and built their RAG pipeline on Pinecone. When they scaled to 100 M vectors, Pinecone’s p99 jumped to 300 ms. They migrated to pgvector on a 256 GB instance and cut p99 to 80 ms while cutting costs by 60%. The migration took six engineer-days of schema redesign and connection pooling tuning. They measured first, then migrated.

Final verdict

If you’re building a new feature that needs vector search and your vector set is under 50 GB, start with pgvector in PostgreSQL. You’ll get SQL fluency, sub-50 ms p99 latency on a tuned instance, and operational overhead you already manage. Budget for RAM—aim for 2–3× your vector set size—and tune maintenance_work_mem and max_parallel_workers_per_gather. Use pgBouncer in transaction pooling to avoid HNSW cache duplication across sessions.

If your vector set exceeds 50 GB, or you need multi-tenant isolation without schema sprawl, or your traffic is unpredictable and you want to scale to zero, choose a hosted vector database. Pinecone is the most mature option; Weaviate Cloud offers better filtering and multi-modal support. Budget for egress and expect 2–3× the cost of self-hosted pgvector at steady state.

Measure before you migrate. Run a 24-hour load test with your peak QPS and vector dimensions. Compare p95/p99 latencies, RAM usage, and total cost. If pgvector’s p99 stays under 50 ms and your RAM usage is < 80%, commit to it. If not, budget for the hosted option and accept the egress cost. The worst outcome is choosing a hosted vector database because “it’s easier,” then discovering your RAM-bound working set makes every query a disk seek.

Start today: clone your production vectors into a pgvector instance, run pgbench with your real queries, and compare the latency histogram to your current system. If pgvector is within 20% of your SLA, ship it. If not, budget for Pinecone and negotiate a PoC budget with your finance team before you commit.
