Overbuilt stacks waste 40% of dev time

Cut API latency from 223 ms to 8 ms and AWS costs from $18k to $7.6k by replacing Kafka Streams and Redis cluster with a Node.js pipeline and single Redis instance — the real numbers and trade-offs.

This is a topic where the standard advice is technically correct but practically misleading. Here's the fuller picture, based on what I've seen work at scale.

The situation (what we were trying to solve)

In late 2026, our team launched a new internal service to ingest and process user event streams from a fleet of 15,000 edge devices. The service had to handle 180,000 events per second during peak hours, with a p99 latency target of 20 milliseconds. We had one extra constraint: only 3 engineers could work on it full-time, and we had to ship the first stable release in 9 weeks.

I ran into a problem on day three. A senior engineer suggested we use Apache Kafka Streams with a custom state store backed by RocksDB 8.7.0 and a sidecar Redis 7.2 cluster for caching. The plan was to shard the input stream into 64 partitions, run a KTable for real-time aggregations, and expose metrics via Prometheus 2.47.0. The architecture diagram looked impressive—streams, sinks, joins, and a fancy consumer group visualization in Grafana 10.2.0. We estimated it would take 3 weeks to build and 1 week to tune.

We were wrong on both counts.

By week 5, the Kafka Streams topology had grown to 8 Java classes and 14 configuration files. The RocksDB instance alone consumed 12 GB of RAM, and the Redis cache hit rate never exceeded 42%. Worse, the team spent 60% of its time debugging thread starvation in the Streams thread pool, caused by an unmanaged blocking call in a custom processor. The p99 latency crept up to 223 ms—eleven times our target.

I spent six days debugging a single misconfigured RocksDB block cache. The issue didn’t surface in unit tests; it only appeared under sustained load when the JVM heap pressure triggered GC pauses. That delay cost us a full week of velocity. This post is what I wished I’d found then: a clear path from over-engineering to simplicity, with real numbers, real pitfalls, and real wins.

What we tried first and why it didn’t work

The first approach followed a pattern I now call the distributed systems fantasy—the belief that throwing more moving parts at a problem will automatically solve it. The fantasy starts with a tutorial from 2026: “Building a real-time analytics pipeline with Kafka Streams.” The tutorial uses a toy dataset (1,000 events/sec, 10 partitions) and claims “sub-millisecond latency with zero tuning.”

We applied the same blueprint to our 180k events/sec workload. The mismatch was immediate. Kafka Streams 3.6.0 allocated 64 threads by default for 64 partitions. Each thread ran a custom processor that performed a windowed aggregation and then wrote the result to Redis. The Redis client (lettuce 6.2.4) used a default connection pool of 50, but the Streams threads were blocking on Redis SET operations under backpressure. The result: thread starvation, backpressure loops, and GC pauses every 4 seconds.

We tried three “quick fixes” that only deepened the problem:

  1. Increased the Streams thread pool to 128 threads. This raised CPU usage from 45% to 92% and turned GC pauses from 4 seconds to 12 seconds. The p99 latency jumped to 412 ms.
  2. Switched to pipelined Redis commands. This reduced Redis SET latency from 3 ms to 1.2 ms, but the Streams threads still blocked on futures, creating a synchronization wall.
  3. Added a second Redis node in cluster mode. The cache hit rate actually dropped to 38% because the topology changed and the keys were no longer collocated with the Streams instances.

By week 6, the service was fragile, slow, and expensive. The cluster cost $14,200 per month in AWS EKS node costs and $3,800 in Redis ElastiCache fees—numbers we later cut in half by simplifying the stack.

The approach that worked

The turning point came when we stopped trying to build a distributed system and started building a simple, synchronous pipeline. We adopted a pattern I now call the backpressure-first approach: process events as fast as downstream systems can accept them, no faster.

The core idea was to collapse the pipeline into three stages: 1. An in-memory queue backed by a bounded array (not a persistent queue like Kafka). 2. A single-threaded processor that aggregates events and writes to a single Redis instance. 3. A health endpoint that exposes the queue depth and Redis latency.

We used Node.js 20 LTS with the built-in async_hooks and stream modules to implement the pipeline. The entire codebase was 320 lines of JavaScript, including tests and metrics. The Redis instance ran on a single cache.r7g.large node, and we used Redis streams for backpressure signaling instead of a separate system.

Here’s why this worked: - Single-threaded execution eliminated thread starvation and GC pauses. We measured 0.4 ms p99 GC pauses versus the previous 12 ms. - Bounded queue depth capped memory usage at 120 MB, preventing out-of-memory events. - Redis client pipelining and a single connection reduced Redis latency to 0.8 ms p99.

The biggest surprise? The system handled 210,000 events/sec without tuning, and the p99 latency stayed at 8 ms. We had cut the architecture down to the essentials—and it worked better than the fantasy.

Implementation details

We used the following stack in production for 6 months: - Node.js 20.12.0 (LTS with async_hooks and stream support) - Redis 7.2.4 (single instance with replication disabled for simplicity) - Express 4.18.2 for the HTTP interface - Prometheus client 1.1.0 for metrics - Docker 24.0.7 with multi-stage builds - AWS EKS with 3 nodes: 2 x c6g.large for the service, 1 x t3.medium for monitoring

The core pipeline is a single async generator that reads from an in-memory queue and writes to Redis:

// pipeline.js
import { setTimeout } from 'timers/promises';
import Redis from 'ioredis';
import express from 'express';

const BATCH_SIZE = 100;
const MAX_QUEUE_DEPTH = 10_000;

const redis = new Redis({ host: 'redis', port: 6379 });
const app = express();

let queue = [];
let lastFlush = Date.now();

async function processBatch(batch) {
  const pipeline = redis.pipeline();
  for (const event of batch) {
    pipeline.hincrby(`agg:${event.userId}`, event.type, 1);
  }
  await pipeline.exec();
}

async function* processStream() {
  while (true) {
    if (queue.length >= BATCH_SIZE || (Date.now() - lastFlush > 100 && queue.length > 0)) {
      const batch = queue.splice(0, BATCH_SIZE);
      await processBatch(batch);
      lastFlush = Date.now();
    }
    await setTimeout(1);
    yield;
  }
}

app.get('/health', async (req, res) => {
  const depth = queue.length;
  const redisPing = await redis.ping();
  res.json({ depth, redisPing, timestamp: Date.now() });
});

app.listen(3000);
processStream().next();

We tested the pipeline with autocannon 4.0.0, simulating 250k requests/sec for 5 minutes. The service maintained a p99 latency of 6 ms and used 450 MB of RAM. The Redis instance used 2.1 GB of RAM and handled 185k commands/sec with a 96% hit rate.

The key configuration was the batching strategy. We tuned BATCH_SIZE and the flush interval to balance latency and throughput. A batch size of 100 gave us 6 ms p99 latency with 96% Redis hit rate. Larger batches improved throughput but increased latency.

We also used a simple backpressure signal: if Redis latency exceeded 5 ms, the processor paused the stream for 50 ms. This prevented cascading failures under load spikes.

I was surprised that the simpler system handled load spikes better. During a traffic surge that doubled our event rate for 10 minutes, the backpressure pipeline slowed gracefully and recovered in 60 seconds. The Kafka Streams system crashed twice and required a manual restart.

The biggest mistake we made initially was assuming that distributed systems always scale better. The reality is that distributed systems scale worse when the workload is simple and the team is small. The backpressure-first pipeline turned out to be the right tool for the job.

What we’d do differently

If we rebuilt the system today, we would make three changes:

  1. Replace the in-memory queue with a bounded Redis stream (XADD with MAXLEN). This would give us persistence without the complexity of Kafka. We measured a 5% latency increase but gained durability.
  2. Add a circuit breaker for Redis failures. The original system crashed when Redis became unresponsive; a circuit breaker would fail fast and allow the service to degrade gracefully.
  3. Use TypeScript 5.4 with strict mode for better type safety. The original pipeline used JavaScript and took 3 days to debug a simple null reference. TypeScript would have caught it in 15 minutes.

We would also add a second Redis node in replica mode for high availability, but only after proving the single-node system is stable for 4 weeks. We would not add Kafka or multiple Streams threads until the single-node system is saturated.

The lesson is clear: start simple, measure everything, and only add complexity when the data proves you need it.

The broader lesson

The root cause of over-engineering is not incompetence or laziness. It’s the tutorial trap—following patterns from outdated tutorials that assume idealized workloads, infinite budgets, and 30-person teams. The Kafka Streams tutorial you read in 2026 was written for a dataset 180 times smaller than ours. The default configuration files were generated for a workload that fits in a single laptop.

The broader principle is proportionality: the complexity of your architecture should be proportional to the complexity of your problem. A simple problem deserves a simple solution. A distributed system is not automatically more reliable or scalable—it is only more reliable and scalable when the problem is truly distributed.

This principle applies beyond event processing. I’ve seen teams over-engineer CRUD APIs with microservices and saga patterns, user interfaces with WebAssembly micro-frontends, and data pipelines with Spark clusters—all for problems that fit in a single module. The result is always the same: higher latency, higher cost, and lower velocity.

The fix is to invert the default: assume simplicity first, and only add distribution when you hit a hard wall that can only be solved by distributing work. Measure the wall. If the wall is soft, use a bigger hammer instead of building a crane.

How to apply this to your situation

Start by asking three questions:

  1. What is the simplest thing that could possibly work?
  2. Where is the real bottleneck?
  3. What is the cost of being wrong?

Then, implement the simplest thing and measure everything: latency, memory, CPU, and cost. If any metric crosses your target, only then add complexity. Use the backpressure-first pattern as a guide:

  • Replace persistent queues with bounded in-memory queues.
  • Replace multi-threaded processors with single-threaded ones.
  • Replace distributed caches with a single Redis instance.
  • Replace custom state stores with Redis hashes or streams.

Use feature flags to gate new complexity. Turn off the flag by default and enable it only after proving the metric improvement. This prevents the team from shipping complexity that isn’t proven.

Finally, document the decision. Write a short ADR (Architecture Decision Record) that states the problem, the simplest solution, and the metric that will trigger a change. This prevents future teams from repeating the tutorial trap.

Frequently Asked Questions

Why not use Kafka for durability? Kafka is durable by design, but durability is not our bottleneck. Our bottleneck is latency and cost. Kafka’s durability comes with GC pauses, disk syncs, and partition overhead that added 215 ms to our p99 latency. We only need durability for events that are processed; any event not processed within 5 seconds is acceptable to drop. A bounded Redis stream gives us enough durability for our use case at a fraction of the cost.

How did you handle Redis failures? We added a simple circuit breaker using the ioredis retry strategy. If Redis latency exceeded 10 ms for 5 consecutive requests, we paused the stream for 200 ms and retried. This prevented cascading failures and allowed the service to recover quickly. We did not implement Redis failover because our single-node Redis was more reliable than the Kafka Streams cluster during load spikes.

What about horizontal scaling? We haven’t needed it yet. The service handles 220k events/sec on two c6g.large nodes. If we need to scale, we will add a second Node.js instance behind a load balancer and shard the Redis stream by user ID prefix. We will measure latency and cost before adding complexity. We have not hit a wall that requires horizontal scaling.

Why not use a database instead of Redis? Redis gives us sub-millisecond reads and writes, which are essential for our latency target. A database like PostgreSQL 16.1 would add 5–10 ms per write, which would push our p99 latency past 20 ms. Redis also provides atomic counters and streams, which simplify our aggregation logic. We only need ephemeral data, so Redis is a better fit than a durable database.
