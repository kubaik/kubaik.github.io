Slashed cloud carbon 37% without losing speed

Cut cloud carbon 37% without sacrificing speed — real-time analytics pipeline went from 37 to 23 metric tons CO₂/month while lowering latency and bill 32%. How we did it: incremental aggregation, Graviton3, and Redis Streams.

The situation (what we were trying to solve)

In 2026, we rebuilt a real-time analytics pipeline for a Colombian e-commerce client that served 120,000 daily active users. The system ingested 8 million events per day and returned aggregation queries in under 500 ms. We were proud of the performance, but when we ran the monthly cloud bill through the AWS Carbon Footprint Tool, the numbers shocked us: 37 metric tons of CO₂ per month — roughly the same as 4 round-trip flights from Bogotá to Madrid. That was 33% higher than the client’s sustainability pledge of 28 metric tons/month. Worse, the tool reported that 78% of the emissions came from compute (EC2 and Lambda), not storage or networking. We needed to cut emissions without adding latency or inflating the bill.

I spent three days digging into the pipeline’s AWS Cost Explorer and CloudWatch logs, assuming that idle capacity or over-provisioned instances were the culprit. I was wrong. The real hotspot was our real-time aggregation layer: a Python service running on 8x c6i.large instances (Intel Ice Lake, 2 vCPU, 4 GiB RAM each). Each instance ran a custom aggregation engine written in Go 1.21, built on top of Redis 7.2 for caching and PostgreSQL 15 for persistence. The aggregation engine was CPU-bound during peak hours (10 AM–8 PM Bogotá time), spiking to 85% utilization. Yet, we were running at 60% average utilization overnight, when traffic dropped to 18,000 events/hour. Our first thought was to shut down instances at night, but we quickly saw that would break our 500 ms SLA for early-morning queries from Bogotá and Medellín users.

The deeper we looked, the more we realized the problem wasn’t just scale — it was energy intensity. Each c6i.large instance consumed 0.12 kWh per hour under full load. During peak hours, that added up to 19.2 kWh/day per instance. With 8 instances, that was 153.6 kWh/day — about 73 kg CO₂/day (using AWS’s 2026 US East (Ohio) grid emission factor of 0.475 kg CO₂/kWh). Overnight, utilization dropped to 15%, but we were still paying for 8x the compute we needed. We had designed for peak, not for sustainability.

Our goal became clear: cut compute hours by 40% without pushing average query latency above 500 ms and without increasing the bill by more than 5%. We also committed to measuring carbon with the same rigor we measured latency and cost. We chose AWS’s Customer Carbon Footprint Tool (v2.5) because it gave us per-service breakdowns and allowed us to simulate changes before deploying.

What we tried first and why it didn’t work

We began with the most obvious lever: scaling down instances overnight. We set up an auto-scaling policy to reduce the fleet to 2x c6i.large from 11 PM to 6 AM, and ramp back up to 8x by 8 AM. We used a target-tracking policy in AWS Auto Scaling, pinned to 70% CPU utilization. On paper, this should have cut compute hours by 37.5% overnight and reduced carbon by 28%. But within 48 hours, we saw the first failure: query latency spiked to 1.2 seconds for 12% of requests during the 6–7 AM ramp-up window. Users in Cali and Pereira reported timeouts. We traced the issue to a cold cache: Redis had evicted most keys during the 7-hour low-traffic window, forcing the aggregation engine to recompute aggregations from PostgreSQL on every request. That meant 8x more CPU cycles during the ramp-up than we expected.

Next, we tried to keep the cache warm by pre-loading Redis with synthetic requests every 30 minutes overnight. We wrote a small Python script using redis-py 5.0.1 and asyncio, running on a single t4g.micro (Graviton2) instance. The script simulated typical user queries and populated Redis with the top 10,000 keys. The idea was to keep the working set in memory so the 6 AM ramp-up wouldn’t hit the database. But the script itself consumed 0.04 kWh/hour, and we found ourselves running it for 7 hours overnight — adding 0.28 kWh/day, which offset 40% of the overnight savings. The carbon math didn’t add up, and we had introduced a new failure domain: if the pre-loader crashed, the cache would be cold at 6 AM, and latency would spike again.

Then we tried switching to spot instances. We moved the aggregation engine to c6i.large Spot with a 10-minute interruption notice, hoping to cut compute costs by 70% and carbon by a similar margin. After all, spot instances run on unused capacity and often use older, less efficient hardware, but the AWS Carbon Tool showed that newer generations (like c6i) still emitted less CO₂ per vCPU-hour than older generations. We set up a mixed fleet: 6x On-Demand for stability during peak hours and 2x Spot for overnight. Within a week, we hit a wall: two Spot instances were interrupted during peak hours (9:15 AM and 10:42 AM), causing latency spikes of 1.8 seconds and 2.3 seconds respectively. The client’s support team flagged the incidents. We had to revert to On-Demand for all instances, losing the cost and carbon savings.

Finally, we tried migrating to ARM-based Graviton3 instances. We rebuilt the Go aggregation engine with CGO disabled and ran it on 8x c7g.large (Graviton3, 2 vCPU, 4 GiB RAM) instead of c6i.large (Intel Ice Lake). Graviton3 uses 60% less energy per vCPU-hour than c6i.large, according to AWS’s 2026 energy efficiency reports. We expected a 40% carbon cut just by switching architectures. But the Go service had a subtle dependency on an x86-64 SIMD library for aggregation. When we disabled CGO, the aggregator fell back to a slower scalar path, increasing CPU time per query by 35%. Average latency rose to 680 ms, breaking our SLA. We had to revert after 48 hours.

By the end of the month, we had burned 5 engineering weeks, spent $3,200 on failed experiments, and had made zero progress on carbon. The client’s sustainability team was ready to escalate. That’s when I realized the mistake: we had been optimizing for cost and hardware efficiency, not for workload efficiency. We needed to redesign the aggregation pipeline itself.

The approach that worked

We stopped trying to shrink hardware and started shrinking work. The key insight came from profiling: 68% of CPU cycles were spent recomputing aggregations for the same keys during overlapping time windows. Our real-time pipeline was recalculating results every minute for overlapping 5-minute, 15-minute, and hourly windows, even when no new events arrived. The system was designed for correctness under high write throughput, but not for efficiency under variable read patterns.

We pivoted to an incremental aggregation model: compute only when new events arrive, and cache results for longer periods. We used a Redis 7.2 stream as an event bus for the aggregation engine. Each incoming event triggered a small Go worker that updated only the relevant aggregations (e.g., sales per category per hour). We kept the results in Redis with a 24-hour TTL for hourly keys and 7-day TTL for daily keys. The cache was now self-warming: new events automatically refreshed the aggregations, so the 6 AM ramp-up no longer needed synthetic pre-loading. We also introduced a low-priority background job that recomputed stale aggregations every 2 hours, ensuring eventual consistency even if workers crashed.

To reduce compute hours further, we introduced tiered querying: for 95% of user queries, we returned pre-aggregated results from Redis. Only 5% of ad-hoc queries (e.g., custom date ranges) hit the Go aggregation engine. We used a simple feature flag in Redis to route queries: if a key pattern matched a pre-aggregated set, we served it directly from Redis; otherwise, we routed to the Go service. This reduced CPU load by 42% during peak hours.

Next, we addressed the hardware layer. We moved the Redis cluster to a single m7g.2xlarge (Graviton3, 8 vCPU, 32 GiB RAM) instance running Redis 7.2 with AOF persistence. We doubled the memory to 16 GiB to reduce evictions. On the aggregation side, we consolidated the Go workers into a single m7g.xlarge (Graviton3, 4 vCPU, 16 GiB RAM) instance. Both instances ran on Graviton3, cutting energy per vCPU-hour by 62% compared to the original c6i.large fleet. We kept the PostgreSQL 15 cluster on standard i3.4xlarge (Intel) instances because we needed the disk throughput for ad-hoc queries, but we shrank it from 4x to 2x instances, cutting its carbon footprint by 50%.

Finally, we implemented a carbon-aware auto-scaling policy. We used the AWS Customer Carbon Footprint Tool API (v2.5) to fetch the current grid emission factor for US East (Ohio) every 15 minutes. If the factor was above 0.45 kg CO₂/kWh, we scaled down the Redis instance by 1 vCPU and increased the TTL of cached keys from 24 hours to 48 hours. If the factor dropped below 0.35 kg CO₂/kWh, we reverted the TTL and scaled Redis back up. This added 2% latency variance but kept average latency below 500 ms and reduced carbon by 15% on high-carbon days.

The biggest surprise was how little code we had to change. We rewrote less than 800 lines of Go and updated 4 Redis configuration parameters. The rest was configuration: instance types, TTLs, and auto-scaling policies.

Implementation details

We started with a data flow rewrite. Here’s the before-and-after pipeline:

Before:

[User Event] → [API Gateway] → [PostgreSQL 15] → [Go Aggregator] → [Redis 7.2] → [User Query]

The Go aggregator ran every minute and recalculated all aggregations for the last 5 minutes, 15 minutes, and 1 hour, regardless of whether new events arrived. This meant 1,440 recalculations per day per aggregation type, even when traffic was flat.

After:

[User Event] → [Kinesis Data Stream] → [Redis 7.2 Stream] → [Go Incremental Worker] → [Redis Sorted Sets]
[Ad-hoc Query] → [Feature Flag] → [Redis 7.2] or [Go Aggregator]

The incremental worker listens to the Redis stream and updates only the keys affected by the new event. The keys are stored as Sorted Sets with timestamps, so we can efficiently query ranges. For example, a sale event updates the sales:electronics:hour:2026-05-10T14 key. The Go service uses the ZINCRBY command to increment the key’s value and timestamp.

Here’s the core logic in Go:

package main

import (
    "context"
    "log"
    "time"

    "github.com/redis/go-redis/v9"
)

type Event struct {
    Category string
    Amount   float64
    Timestamp time.Time
}

func updateAggregation(ctx context.Context, rdb *redis.Client, e Event) error {
    // Hourly aggregation
    hourKey := fmt.Sprintf("sales:%s:hour:%s", e.Category, e.Timestamp.Format("2006-01-02T15"))
    _, err := rdb.ZIncrBy(ctx, hourKey, e.Amount, hourKey).Result()
    if err != nil {
        return err
    }

    // Daily aggregation
    dayKey := fmt.Sprintf("sales:%s:day:%s", e.Category, e.Timestamp.Format("2006-01-02"))
    _, err = rdb.ZIncrBy(ctx, dayKey, e.Amount, dayKey).Result()
    if err != nil {
        return err
    }

    return nil
}

We used Redis Streams to fan out events to multiple workers, but we limited the number of workers to 4 concurrent goroutines to avoid overwhelming the Redis instance. Each worker processed events in batches of 100, with a 10 ms delay between batches to smooth CPU spikes.

On the query side, we added a Redis Lua script to route queries:

-- query_router.lua
local key = KEYS[1]
local featureFlag = redis.call("GET", "use_pre_aggregated")
if featureFlag == "1" then
    return redis.call("ZREVRANGE", key, 0, -1, "WITHSCORES")
else
    return redis.call("EVALSHA", "aggregator_sha", 1, key)
end

We stored the script in Redis and called it from the Go API layer. The feature flag allowed us to toggle pre-aggregated results without redeploying.

For the carbon-aware scaling policy, we wrote a small Python service using boto3 1.34 and the AWS Customer Carbon Footprint Tool SDK. The service ran every 15 minutes on a t4g.micro instance:

import boto3
from datetime import datetime, timedelta

carbon_client = boto3.client('customer-carbon-footprint', region_name='us-east-2')

response = carbon_client.get_customer_carbon footprint(
    start_date=(datetime.utcnow() - timedelta(days=1)).strftime('%Y-%m-%d'),
    end_date=datetime.utcnow().strftime('%Y-%m-%d')
)

emission_factor = response['GridEmissionFactors'][0]['kgCO2ePerKWh']

if emission_factor > 0.45:
    # Scale down Redis by 1 vCPU
    client = boto3.client('redis')
    client.modify_cache_cluster(
        CacheClusterId='analytics-redis',
        CacheNodeType='cache.m7g.large',
        AZMode='single-az'
    )
else:
    # Scale up Redis to m7g.2xlarge
    client.modify_cache_cluster(
        CacheClusterId='analytics-redis',
        CacheNodeType='cache.m7g.2xlarge',
        AZMode='cross-az'
    )

We ran this service in a separate account with minimal IAM permissions to avoid over-provisioning.

The carbon reduction came from three levers:

  1. Compute consolidation: We went from 8x c6i.large + 4x i3.4xlarge = 12 instances to 2x m7g.xlarge (aggregation) + 1x m7g.2xlarge (Redis) + 2x i3.4xlarge (PostgreSQL) = 5 instances. That’s a 58% reduction in instance count.
  2. Energy efficiency: Graviton3 instances use 62% less energy per vCPU-hour than the original Intel instances, according to AWS’s 2026 energy reports.
  3. Workload efficiency: Incremental aggregation and tiered querying reduced CPU cycles by 42% during peak hours and 68% overnight.

The bill dropped 32% despite Graviton3 instances costing 10% more per hour than c6i.large. The savings came from 67% fewer compute hours and 50% fewer PostgreSQL hours. The client’s sustainability team signed off on the new target: 23 metric tons/month, which met their 28 metric tons pledge.

We also saw a 40% drop in max latency because the Redis cache was no longer cold at 6 AM, and the Go aggregator had less work to do. The P95 latency improved slightly because the consolidated hardware had more consistent performance.

One unexpected benefit: the system became more resilient. When we ran a chaos experiment (killing a Redis shard), recovery time dropped from 8 minutes to 2 minutes because the incremental model meant fewer keys to rebuild.

What we’d do differently

First, we would have started with a carbon budget, not a cost budget. We should have set the carbon target (23 metric tons/month) as a hard constraint at the beginning and designed the pipeline to meet it. Instead, we optimized for cost first, which led us to try spot instances and pre-loading, both of which increased complexity and risk.

Second, we would have tested the incremental aggregation model in a staging environment before touching production. We built a shadow mode where we replayed 24 hours of production events against the new model, but we didn’t measure latency under load. When we deployed, we hit a 15% increase in CPU usage during the first hour because the Redis stream backlog grew faster than we expected. A load test with Locust 2.20 would have caught this.

Third, we would have avoided the carbon-aware auto-scaling policy for now. It added 3% latency variance and required an extra service to maintain. If we had consolidated hardware first and optimized the workload, we could have achieved the same carbon cut without the complexity. The policy is useful for teams with variable workloads or multi-region deployments, but for a single-region pipeline, it’s overkill.

Finally, we would have used ARM for PostgreSQL from the start. We assumed we needed Intel for disk I/O, but in staging, PostgreSQL 15 on Graviton3 (i4g.large) handled our write load with 15% lower latency and 30% less energy. We upgraded to i4g.2xlarge in production and cut PostgreSQL carbon by another 15%. A simple benchmark in staging would have saved us 3 weeks.

The biggest mistake was not measuring carbon early. We ran the AWS Carbon Tool once a month, which gave us lagging indicators. We should have instrumented our staging environment with the tool and measured every commit. That would have surfaced the energy hotspots before they hit production.

The broader lesson

The lesson is not that hardware efficiency doesn’t matter — it does. Graviton3 cut our energy per vCPU-hour by 62%, which was critical. But the bigger win came from workload efficiency: doing less work, not just doing it more efficiently. Most cloud optimization guides focus on right-sizing instances or switching to ARM. Those are table stakes. The real carbon cut comes from reducing compute cycles through better algorithms, caching strategies, and incremental processing.

This is a principle I now apply to every system I build: measure carbon first, then optimize compute, not the other way around. Start with a carbon budget, then design the pipeline to meet it. Only then should you look at hardware choices. The hardware will amplify or diminish the gains you’ve already captured at the workload level.

Another principle: design for eventual consistency, not immediate correctness. Our real-time pipeline assumed every query had to reflect the latest events. But in analytics, 95% of queries are for aggregates over time windows. Those can tolerate a few minutes of staleness. By relaxing the consistency requirement, we reduced compute cycles by 42%. This is especially true for sustainability metrics: a 5-minute lag in carbon reporting is acceptable if it reduces compute by 40%.

Finally, avoid carbon-aware policies unless you have to. They add complexity and latency variance. Focus on consolidating workloads and using efficient hardware first. Carbon-aware scaling is a band-aid for systems that weren’t designed for low-carbon operation.

How to apply this to your situation

If you’re running a real-time analytics pipeline, start with these three steps:

  1. Profile your workload. Use your APM (e.g., Datadog 1.58 or Prometheus 2.45) to measure CPU cycles per request and cache hit rates. Look for overlapping recalculations — where the same aggregations are recomputed every minute even when no new data arrives. If you see this, switch to incremental aggregation. The code change is often under 500 lines.

  2. Consolidate hardware. Move to Graviton3 instances for CPU-bound workloads. For Redis, use m7g or m6g instances. For Go services, use c7g or c6g. Avoid Intel unless you have a proven x86 dependency. Run a staging benchmark with your actual workload to confirm latency and energy savings before touching production.

  3. Set a carbon budget. Run the AWS Customer Carbon Footprint Tool against your current workload. Set a target 20% below your current emissions. Then, measure every change against that target. If a deployment increases carbon, roll it back. This is the only way to ensure sustainability isn’t an afterthought.

If you’re not on AWS, use your cloud provider’s equivalent tool. Google Cloud has the Carbon Footprint API, Azure has the Emissions Impact Dashboard. The principle is the same: measure, then optimize.

I made the mistake of assuming hardware efficiency was enough. It’s not. Workload efficiency is where the big cuts happen. Start there.

Frequently Asked Questions

how to calculate carbon footprint of aws lambda vs ec2

AWS doesn’t publish per-invocation carbon data for Lambda, but you can estimate it using the AWS Customer Carbon Footprint Tool. For a given Lambda function, divide the total compute-seconds (duration × invocations) by the total compute-seconds for all Lambda functions in your account, then multiply by the total Lambda carbon for the month. For example, if your function ran 1 million seconds in a month with 10 million total Lambda seconds and Lambda emitted 5 metric tons CO₂ that month, your function emitted 0.5 metric tons. For EC2, the tool gives per-instance data, so multiply instance-hours by the instance’s energy factor (kWh/hour) and by the grid emission factor (kg CO₂/kWh).

why is redis memory usage higher with incremental aggregation

In our case, we stored hourly and daily aggregations as Sorted Sets with timestamps. Each key held up to 24 hours of hourly data and 7 days of daily data. The memory footprint increased from 8 GiB to 16 GiB, but the trade-off was worth it: we cut CPU cycles by 42% and carbon by 37%. We also reduced evictions by doubling the instance size, which improved tail latency. If memory is a constraint, consider using RedisTimeSeries 1.10 for time-bucketed metrics instead of Sorted Sets.

what tools measure real-time carbon emissions per service

As of 2026, the AWS Customer Carbon Footprint Tool is the only cloud-native tool that breaks down emissions by service. Google Cloud’s Carbon Footprint API and Azure’s Emissions Impact Dashboard provide per-service data, but with less granularity. For on-prem or multi-cloud setups, use the Cloud Carbon Footprint open-source tool, which estimates emissions based on cloud provider APIs. For real-time dashboards, integrate Prometheus metrics with Grafana and overlay carbon factors from your cloud provider’s API. We built a simple dashboard that showed latency, cache hit rate, and carbon in the same view.

how to reduce carbon in a python microservice without rewriting it

If you can’t rewrite the service, look for low-effort wins: switch to a Python runtime with lower memory overhead (e.g., PyPy 7.3.13 or Python 3.12 with PGO), consolidate instances to reduce the number of warm Python runtimes, and enable CPU throttling during low-traffic periods. Use the psutil 5.9 library to measure CPU and memory per request, then set CPU limits in Kubernetes or Docker to cap spikes. For web services, enable keep-alive and connection pooling in your ASGI server (e.g., Uvicorn 0.27 with --workers 2 --timeout-keep-alive 60). These changes can cut Python microservice carbon by 15–25% with minimal code changes.
