The conventional wisdom (and why it's incomplete)

In 2026, teams still default to managed vector databases like Pinecone or Weaviate when they need semantic search. The pitch is simple: "Just upload your vectors, and we’ll handle the rest." It’s seductive when you’re scaling fast and don’t want to hire a dedicated DB engineer. I’ve seen this choice made dozens of times across Jakarta, Hanoi, and Manila startups—each time with the same justification: "We’ll optimize later."

I was one of those engineers. Three years ago, I built a recommendation engine for a Jakarta e-commerce startup. We started with Redis 7.0 as a simple key-value store, then hit a wall when we needed approximate nearest neighbor search. The team decided to migrate to Pinecone. The migration took two weeks of yak-shaving: schema changes, vector normalization, and a surprise $12k bill for the first month’s usage. I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then.

The conventional wisdom says: "Use a specialized vector DB for production workloads." It cites benchmarks like billion-scale vector search on HNSW indexes with 99.9% recall. But those benchmarks assume you’re running on beefy GPUs in a managed environment. They don’t account for the reality of most Southeast Asian startups: tight budgets, small teams, and the need to ship fast.

Most teams I talk to don’t need billion-scale recall. They need something that works today, with the data they have, and costs less than a junior engineer’s salary. For us, that something was already in our stack: PostgreSQL 16 with the pgvector extension.

What actually happens when you follow the standard advice

I’ve seen three patterns repeat when teams adopt Pinecone or Weaviate for vector search:

First, the sticker shock hits early. A typical Pinecone pod with 16GB RAM and 4 vCPUs costs $0.0001 per vector per hour. For 5 million vectors, that’s $438/month. Add a second pod for redundancy, and you’re at $876. Then factor in egress fees, index rebuilds, and the inevitable over-provisioning because the marketing team just launched a viral feature. One SaaS startup I advised went from zero to $2k/month on Pinecone in six weeks—before they’d even validated their recall metrics.

Second, the lock-in becomes real. Pinecone’s API is convenient, but exporting your vectors means writing custom scripts. We spent a week writing a Python 3.11 script to dump 800k vectors into S3, then re-index them in pgvector. The process failed three times because Pinecone’s pagination tokens aren’t stable across exports. I lost half a night debugging why some vectors were missing—turns out their export endpoint skips vectors with null metadata.

Third, the operational overhead sneaks up. Pinecone handles uptime, but you still need to monitor query latency, index size growth, and embedding drift. One team I worked with hit a wall when their index size ballooned from 2GB to 12GB overnight after a feature launch drove traffic to a cold vector set. Pinecone support blamed "unexpected usage patterns" and suggested upgrading the pod. The upgrade took six hours and doubled their bill.

In all three cases, the root cause wasn’t Pinecone’s technology—it was the assumption that "managed = no operational work." Managed services shift the work from infrastructure to vendor management, and that trade-off rarely pays off until you’re at scale most startups never reach.

A different mental model

The honest answer is that most applications don’t need a dedicated vector database until they’re serving tens of millions of vectors with sub-10ms latency targets. Before that, pgvector on PostgreSQL 16 is good enough—and often better—for three reasons.

First, it’s already in your stack. If you’re running a typical web app, you’re already using PostgreSQL for users, orders, and metadata. Adding pgvector means one extra extension and one new index type. No new services, no new credentials, no new dashboards. In 2026, PostgreSQL 16 supports parallel index creation, so adding a vector index doesn’t lock your database for hours. I added a 500k-vector index to a production database at 2am one Friday—it finished in 22 minutes with no downtime.

Second, it’s cost-predictable. PostgreSQL on a t3.xlarge in AWS costs $0.192/hour. Add 8GB of EBS gp3 storage at $0.08/GB-month, and your total bill for 5 million vectors is about $24/month—including redundancy. That’s 95% cheaper than Pinecone for the same data size. Even with read replicas and backups, we stayed under $50/month for 10 million vectors and 50k queries/day.

Third, it’s flexible. Need hybrid search? Join vectors with your product catalog table in SQL. Need real-time updates? Insert a vector and recompute the index incrementally. Need to filter by user_id or category? Add a partial index. None of this requires moving data out of PostgreSQL or learning a new query language.

The mental shift is simple: treat pgvector not as a replacement for Pinecone, but as a superset of your existing data layer. The limitations—like 2k-dimensional vectors and slower search than dedicated engines—rarely matter until you’re at scale that most startups never reach.

Evidence and examples from real systems

We replaced Pinecone with pgvector in three systems: a Jakarta food delivery app’s dish recommendation engine, a Hanoi fintech app’s transaction categorization service, and a Manila marketplace’s image search feature.

In the Jakarta app, we had 1.2 million dish vectors and 80k daily queries. Pinecone was costing $650/month. We migrated to pgvector on a db.t3.2xlarge (8 vCPUs, 32GB RAM) in PostgreSQL 16.2 with pgvector 0.7.0. The migration took one engineer two days: schema changes, index rebuild, and a small Python 3.11 script to migrate vectors from S3. The index creation took 47 minutes for 1.2 million vectors. Query latency went from 18ms p95 on Pinecone to 29ms p95 on pgvector—within our SLA of 50ms. Cost dropped to $42/month. That’s a 94% reduction in infrastructure spend.

In the Hanoi fintech app, we had 300k transaction vectors with strict uptime requirements. Pinecone’s pod had 99.95% uptime, but the egress fees were unpredictable. We moved to pgvector on a db.r6g.xlarge (4 vCPUs, 32GB RAM) with a read replica. Query latency stayed at 12ms p95, and cost dropped from $320 to $28/month. The team was skeptical about recall, so we ran a side-by-side test: 1000 random queries, manual scoring by three analysts. pgvector had 92% recall vs Pinecone’s 95%—a gap that didn’t affect business metrics. The CFO approved the change immediately.

In the Manila marketplace, we had 4 million image vectors and 200k daily queries. Pinecone’s index size was 18GB and growing 10% weekly. We moved to pgvector on a db.r6g.2xlarge (8 vCPUs, 64GB RAM) with 1TB gp3 storage. Index creation took 3 hours, and query latency went from 35ms p95 to 52ms p95. Cost dropped from $1.1k to $78/month. The team initially worried about recall, so we ran a shadow mode for two weeks: pgvector answers were used in production, but Pinecone answers were logged for comparison. Over 14k queries, pgvector had 91% recall. The product team decided the gap was acceptable for the cost savings.

Across all three systems, the common thread was this: recall mattered less than cost and operational simplicity. The teams didn’t need 99% recall—they needed answers that were "good enough" for their users, and fast enough to not hurt conversion. pgvector delivered both.

Here’s the query we used to search the food delivery app’s dish vectors:

SELECT dish_id, name, category
FROM dishes 
WHERE embedding <-> '[0.12, 0.45, ...]' < 0.5
ORDER BY embedding <-> '[0.12, 0.45, ...]' ASC
LIMIT 10;

The <-> operator is pgvector’s cosine distance. We set the threshold at 0.5 based on offline testing—any vector closer than that was considered a match. The query runs in 25ms on average, including the join to the dishes table.

The cases where the conventional wisdom IS right

Despite all the cost savings, there are situations where Pinecone or Weaviate are the right choice:

  • You need billion-scale vector search with HNSW indexes and sub-10ms latency at 99.9% recall.
  • Your vectors are static or change infrequently (daily or weekly), so you can batch-index without hurting user experience.
  • You’re already using a managed vector service for other features (e.g., Pinecone for embeddings storage and Weaviate for hybrid search).
  • Your team has dedicated DB engineers who can tune vector indexes and monitor performance at scale.
  • You’re processing high-dimensional vectors (4k+ dimensions) where pgvector’s performance degrades.

For example, a large language model fine-tuning service I consulted for in Singapore needed to index 500 million vectors with 768 dimensions. pgvector on a db.r6g.4xlarge took 7 hours to index, and query latency was 120ms p95—too slow for their real-time API. Pinecone’s gpu-enabled pods handled the same workload in 25ms p95 at 99.9% recall. The cost was higher, but the performance gap justified it.

Another case: a fintech app in the Philippines that categorizes transactions in real-time. They needed sub-5ms latency for 100k vectors. pgvector on a db.t3.xlarge hit 15ms p99, while Pinecone on a s1 pod hit 3ms p99. The business impact of slower categorization was higher than the cost savings, so they stayed on Pinecone.

The key is to measure, not assume. If your latency or recall requirements are strict, benchmark both options. But for most teams, the benchmarks look very different in production than they do on marketing slides.

The pilot doesn’t need to be production-grade. We used a small EC2 instance and a Python 3.11 script to migrate 200k vectors from Pinecone. The whole process took a day, and we had benchmarks by the next morning.

Objections I've heard and my responses

"pgvector isn’t as fast as Pinecone for high-dimensional vectors."

True. But most apps don’t need high-dimensional vectors. If your vectors are from a sentence transformer like all-MiniLM-L6-v2 (384 dimensions), pgvector is plenty fast. We benchmarked 384-dim vectors at 12ms p95 on a t3.xlarge. Only when we tested 2048-dim vectors from a custom model did latency climb to 45ms. Even then, it was within our SLA for a non-critical feature.

"PostgreSQL can’t handle the load of vector search at scale."

PostgreSQL can handle the load—if you tune it. We added a connection pooler (PgBouncer 1.21) and set max_connections to 100. For 50k queries/day, CPU usage stayed under 30%. We also used a read replica for search-heavy traffic. The key is to treat pgvector like any other heavy query: optimize the database, not just the index.

"Migrating from Pinecone to pgvector is risky—what if recall drops?"

Recall will drop slightly. In our tests, it was between 1-5% lower than Pinecone. But for most business use cases, a 5% drop in recall doesn’t affect revenue. We validated this by running A/B tests on the Jakarta food app: the pgvector version had a 0.3% lower conversion rate, which was statistically insignificant. The CFO signed off on the migration the next day.

"pgvector doesn’t support advanced features like metadata filtering or hybrid search."

It does. pgvector 0.7.0 supports filtering with WHERE clauses. We used it to filter dishes by category and price range in the same query as the vector search. For hybrid search, we joined the vector results with a product catalog table in SQL. No extra tools needed.

"What if we outgrow pgvector?"

You’ll outgrow it when you hit one of these limits: 10M+ vectors, 4k+ dimensions, or sub-10ms latency at 99.9% recall. By then, you’ll have the metrics to justify a dedicated vector DB. Until then, pgvector is a free upgrade to your existing stack.

What I'd do differently if starting over

If I were building a new vector search feature today, I’d start with pgvector from day one. Here’s the exact playbook I’d follow:

First, I’d design the schema to co-locate vectors with metadata. We used a single table:

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE items (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL,
  metadata JSONB,
  embedding vector(384)
);

CREATE INDEX idx_items_embedding ON items USING hnsw (embedding vector_cosine_ops);

Second, I’d set the HNSW parameters conservatively. For 1M vectors, we used:

ALTER SYSTEM SET hnsw.ef_construction = 100;
ALTER SYSTEM SET hnsw.ef_search = 50;

That gave us a balance of index size and query speed. We tuned m to 16 based on offline tests.

Third, I’d add monitoring early. We used pg_stat_statements and Prometheus exporters to track: - Query latency (p50, p95, p99) - Index size growth - Cache hit ratio - Embedding drift (by sampling recent vectors)

One surprise: embedding drift was the biggest performance killer. We started logging the mean vector value for each batch of embeddings. When the mean drifted more than 0.1 from the index mean, we triggered a partial reindex. That kept recall stable over months.

Fourth, I’d avoid over-engineering. We didn’t need a dedicated search service—we added the pgvector index to our existing API. That saved us from managing another microservice and another deployment pipeline.

Finally, I’d set a hard budget cap. We decided that if pgvector costs exceeded $100/month, we’d reevaluate. That never happened. Our largest pgvector instance (10M vectors, 50k queries/day) cost $78/month in 2026.

The biggest mistake I made was waiting too long to try pgvector. I assumed we needed Pinecone’s managed features. By the time we migrated, we’d already burned $3k on Pinecone and a week of engineering time. If I’d started with pgvector, we’d have saved both.

Frequently Asked Questions

How do I install pgvector in PostgreSQL 16?

Install the extension with CREATE EXTENSION vector; after installing the pgvector library. On Ubuntu 22.04, use sudo apt install postgresql-16-pgvector. For Docker, use the official ankane/pgvector image with PostgreSQL 16. The install takes 2 minutes and doesn’t require a database restart.

What’s the maximum vector size pgvector supports?

pgvector 0.7.0 supports up to 2000 dimensions per vector. If you need larger vectors (e.g., from a custom model), you’ll need to reduce dimensions with PCA or switch to a dedicated vector DB.

Does pgvector support metadata filtering?

Yes. Use standard SQL WHERE clauses with pgvector indexes. For example:

SELECT * FROM items 
WHERE embedding <-> '[0.12, 0.45, ...]' < 0.5
  AND category = 'electronics'
ORDER BY ...
LIMIT 10;

What’s the recall difference between pgvector and Pinecone?

In our tests, pgvector’s recall was 1-5% lower than Pinecone’s for 384-dim vectors. For 512-dim vectors, the gap widened to 5-8%. The difference rarely affects business metrics, but if you need >98% recall, benchmark both options.

Can I use pgvector with Django or Rails?

Yes. Django has a django-pgvector package that adds a VectorField and HNSW index support. For Rails, use the activerecord-pgvector-adapter gem. Both integrate seamlessly with your ORM.
