How Database Sharding Actually Works Under the Hood

Database sharding works by dividing data into smaller, more manageable pieces, called shards. Each shard contains a portion of the overall data and is typically stored on a separate server. When a query is executed, the database must determine which shard(s) to access to retrieve the required data. This is typically done using a shard key, which is used to map data to a specific shard. For instance, when using PostgreSQL 13.4, developers can use the pg_shard extension to manage sharding. The shard key can be based on a specific column or set of columns, such as a user ID or location. A well-designed shard key can reduce the number of shards that need to be accessed, resulting in a 30% decrease in query latency.
