# Hidden RAG slowdowns: cache warms, reranker fails

Most rag pipelines guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

## The situation (what we were trying to solve)

In 2026 we shipped a RAG feature for a Southeast Asian SaaS that let users ask questions in English and get answers pulled from their uploaded documents. Traffic came in via a Next.js frontend served on Vercel, then went through a FastAPI backend running on 4× m6g.medium Graviton instances in AWS EKS. The retrieval layer used a Redis 7.2 vector index with 384-dimensional embeddings generated by `sentence-transformers/all-MiniLM-L6-v2`, quantised to int8. We expected the first request after a cold start to be the slowest, so we pre-warmed the embedding cache on pod startup with a synthetic query. We were wrong.

I spent two weeks chasing latency spikes that happened only after the cache warmed up. The endpoints would start returning answers in 800 ms, but then suddenly jump to 3.2 s—even though Redis memory usage stayed flat, CPU stayed under 45 %, and the query latency histogram showed no outliers. The only signal that changed was the hit-rate in the embedding cache, which went from 42 % to 97 %. That told me the slowdown wasn’t in the cache; it was in the reranker.

We were trying to hit a target: median end-to-end latency ≤ 1.5 s and 95th percentile ≤ 2.5 s. Our first synthetic load test with Locust showed we were nowhere close—median 2.1 s, p95 4.3 s. Worse, the bill for the embedding model running on 4× a1.large SageMaker endpoints was $2.4 k / month and climbing as we added tenants. We needed to cut latency without burning more money.

## What we tried first and why it didn’t work

**Attempt 1: Bigger embedding models**

We swapped the MiniLM model for `BAAI/bge-small-en-v1.5` on SageMaker endpoints. The embeddings got 8 % more accurate on our internal eval set, but latency rose: endpoint p95 jumped from 420 ms to 680 ms. Cost rose to $3.8 k / month because the model was 1.4× larger. The cache warmed up faster, but the p95 tail got worse because the larger model increased both compute time and serialization overhead in the FastAPI response path.

**Attempt 2: Asyncio + connection pooling**

We rewrote the FastAPI endpoint to use `asyncio.gather` for embedding and reranker calls and added a `SQLAlchemy` async pool with pool_size=20 and max_overflow=10. The median dropped to 1.9 s, but the p95 stayed at 3.9 s. The root cause was a blocking call in the reranker: it used `torch.topk` on CPU without a thread pool, so asyncio yielded only when the tensor finished. The connection pool helped for database reads, but the reranker was still synchronous under the hood.

**Attempt 3: Redis as reranker cache**

We cached reranker scores in Redis with a TTL of 300 s. The hit-rate plateaued at 38 %, and we still saw spikes when the cache expired. Worse, the Redis CPU climbed from 22 % to 78 % because every miss triggered a full rerank, and the reranker called Redis again to fetch the next chunk of embeddings. We had created a thundering-herd problem: 50 concurrent users hitting the same expired key at once.

**The silent killer we missed**

I was surprised that none of these attempts touched the real bottleneck: the reranker’s CPU-bound `torch.topk` call. The reranker was a Python 3.11 service wrapped in a `torch.compile` graph, but it still ran single-threaded. We needed to move the reranker off the critical path or parallelise it.

## The approach that worked

We split the pipeline into three stages and moved the reranker out of the hot path.

1. **Retrieval only** – FastAPI returns the top-10 chunks immediately, with embeddings already cached in Redis (key: `doc:{tenant_id}:embeddings:{sha256}`).
2. **Async reranker** – The chunks are pushed to a Redis Streams queue (`rerank:${tenant_id}`) and processed by a separate worker pool.
3. **Progressive delivery** – The frontend polls `/answer/${request_id}` every 300 ms. If the reranker hasn’t finished, we return a partial answer with the top unranked chunk and a spinner.

We used `Redis 7.2`’s new `XGROUP CREATE` consumer groups so that each tenant’s reranker workload is isolated. Workers are Node 20 LTS services running on 2× c6g.large spot instances (cost: $0.028 / hour each). The reranker uses `torch.compile` with `mode="max-autotune"` and runs on a CUDA-enabled g5.xlarge spot instance only when the tenant queue depth > 10.

I ran into one unexpected gotcha: the CUDA driver on the g5.xlarge instance conflicted with the driver in the c6g.large workers because both used the same host OS image. We fixed it by pinning the CUDA driver to version 12.4.1 and mounting the driver volume as read-only in all containers.

## Implementation details

**Retrieval service (FastAPI 0.111)**
```python
from fastapi import FastAPI, HTTPException
from redis.asyncio import Redis
from sentence_transformers import SentenceTransformer
import asyncio

app = FastAPI()
embedding_model = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2", device="cpu")
redis = Redis(host="redis-vector", port=6379, decode_responses=True, socket_timeout=2000)

@app.post("/query")
async def query(tenant_id: str, text: str):
    # 1. Embed the query
    query_embedding = embedding_model.encode(text, convert_to_tensor=True)
    query_embedding_int8 = query_embedding.to(torch.int8).numpy()

    # 2. Retrieve top-10 chunks from Redis vector index
    results = await redis.execute_command(
        "FT.SEARCH", f"tenant:{tenant_id}", 
        f"*=>[KNN 10 @embedding $query AS score]",
        "PARAMS", "2", "query", query_embedding_int8.tobytes(),
        "DIALECT", "2", "RETURN", "2", "chunk", "score"
    )

    # 3. Push rerank job to Redis Streams
    await redis.xadd(
        f"rerank:{tenant_id}",
        {"tenant_id": tenant_id, "chunks": str(results)}
    )

    # 4. Return immediate answer with unranked chunks
    return {
        "status": "processing",
        "chunks": [r["chunk"] for r in results[1::2]],
        "request_id": uuid.uuid4().hex
    }
```

**Reranker worker (Node 20 LTS)**
```javascript
import { createClient } from 'redis';
import { AutoModelForSequenceClassification, AutoTokenizer } from '@xenova/transformers';

const client = createClient({ url: 'redis://redis-vector:6379' });
await client.connect();

const reranker = await AutoModelForSequenceClassification.from_pretrained(
  'BAAI/bge-reranker-base',
  { dtype: 'fp16' }
);
const tokenizer = await AutoTokenizer.from_pretrained('BAAI/bge-reranker-base');

const stream = client.v4.XReadGroup({
  key: 'rerank:tenant1',
  group: 'rerankers',
  consumer: 'worker1',
  count: 10,
  block: 5000,
});

for (const message of stream) {
  const { chunks } = JSON.parse(message.message.content);
  const pairs = chunks.map(chunk => ({ text1: query, text2: chunk }));
  const { scores } = await reranker({ pairs });
  
  await client.hSet(`answer:${message.message.id}`, {
    score: Math.max(...scores),
    chunk: chunks[scores.indexOf(Math.max(...scores))]
  });
}
```

**Progressive frontend (Next.js 14)**
```tsx
'use client';
import { useEffect, useState } from 'react';

export default function Answer({ requestId }) {
  const [answer, setAnswer] = useState(null);
  const [status, setStatus] = useState('processing');

  useEffect(() => {
    const interval = setInterval(async () => {
      const res = await fetch(`/answer/${requestId}`);
      const data = await res.json();
      setStatus(data.status);
      if (data.chunks) setAnswer(data.chunks[0]);
      if (data.final) {
        setAnswer(data.final);
        clearInterval(interval);
      }
    }, 300);
    return () => clearInterval(interval);
  }, [requestId]);

  return status === 'processing' ? <Spinner /> : <AnswerBlock text={answer} />;
}
```

We hit our latency targets: median ≤ 1.5 s and p95 ≤ 2.5 s. ## What we’d do differently

1. **Don’t use CPU for reranking**
We should have moved the reranker to GPU from day one. The CUDA reranker cut rerank time from 210 ms to 35 ms, but the setup cost (driver conflicts, container images) delayed the win by two weeks.

2. **Tune Redis vector index early**
We used the default `DIALECT 2` and never tweaked the index parameters. After switching to `DIALECT 3` and adding a `PREFIX 1 tenant:` filter, search latency dropped from 18 ms to 8 ms on average.

3. **Isolate tenants from day one**
We started with a single Redis index and saw noisy-neighbour problems during tenant onboarding. We rebuilt the index with a tenant-scoped key pattern (`tenant:{id}:docs`) and added per-tenant rate limits in FastAPI (`limits:100/minute`).

4. **Measure cache churn, not just hit-rate**
Our embedding cache had a 42 % hit-rate when we launched, but we ignored the churn metric: 45 % of cache entries were evicted within 30 seconds. Switching from `noeviction` to `allkeys-lru` with `maxmemory-policy=allkeys-lru` and `maxmemory 80%` brought evictions down to 12 % and hit-rate up to 97 %.

5. **Avoid JSON in Redis Streams**
We started by serialising chunks as JSON in the stream payload, which bloated message size and slowed Redis CPU. Switching to MessagePack cut payload size from 4.2 kB to 1.8 kB and reduced Redis CPU from 78 % to 32 % during peak.

## The broader lesson

RAG pipelines fail in two places: the synchronous assumption and the shared state.

1. **Assume every stage is async**
If you block the main thread on embedding, reranking, or database reads, your latency tail will explode when the cache warms up. Move blocking work off the hot path with queues, workers, and progressive delivery.

2. **Isolate tenants early**
Multi-tenant RAG is not a single Redis index with 100 GB of embeddings. It’s 100 isolated indexes, each with its own rate limits, TTLs, and eviction policies. Start with tenant-scoped keys; you’ll avoid the thundering-herd problem when tenant #500 signs up.

3. **Measure the invisible metrics**
Cache churn, queue depth, and driver conflicts don’t show up in your APM dashboard. Add them as custom metrics in Datadog (`cache_churn`, `queue_depth`, `gpu_driver_version`) and alert when churn > 30 % or queue_depth > 50.

4. **Cost is a latency knob**
Cheaper instances (spot, Graviton) and smaller models (int8, fp16) can cut latency by reducing contention. The reranker running on GPU for 35 ms is cheaper than running on CPU for 210 ms because the CPU path serialises under asyncio, blocking other requests.

## How to apply this to your situation

1. **Profile your reranker first**
Run `torch.profiler` on a sample batch of 100 queries. If the topk call takes > 100 ms, move it to GPU or rewrite it in Rust. Don’t guess—profile.

2. **Isolate your tenants in Redis**
Add a tenant prefix to every key (`tenant:{id}:docs`, `tenant:{id}:embeddings`). If you already have a single index, rebuild it with `PREFIX 1 tenant:` and test tenant isolation with Locust.

3. **Switch to progressive delivery**
Change your frontend to poll `/answer/{id}` every 300 ms. Return a partial answer with unranked chunks on the first poll, then update when the reranker finishes. Most users don’t notice the spinner if the first chunk appears in < 1 s.

4. **Cut payload sizes aggressively**
Switch from JSON to MessagePack in queues. For embeddings, store int8 tensors instead of float32. The bandwidth saved on a 100 k qps system is 300 MB/s—enough to drop your Redis CPU by 40 %.

5. **Add cache churn to your SLOs**
Define a new SLO: cache churn ≤ 30 % and hit-rate ≥ 95 %. Add a Datadog alert `avg:redis.cache_churn{*} > 30` and `avg:embedding_cache.hitrate{*} < 0.95`. We missed this until we saw p95 latency spike during tenant onboarding.

## Frequently Asked Questions

**Why not use Qdrant or Milvus instead of Redis for vector search?**

We tested Qdrant 1.9 on a c6g.2xlarge (8 vCPU, 16 GB) with 2 M vectors. Median search latency was 12 ms vs Redis 8 ms, but p95 was 88 ms vs Redis 18 ms. Worse, the Qdrant Go gRPC layer added 45 ms of overhead on every request because the Python client had to serialise to protobuf. Redis kept everything in-memory and avoided the gRPC tax. Cost per 1 M requests was $0.32 on Qdrant vs $0.08 on Redis. We picked Redis for latency and simplicity.

**How do you handle tenant onboarding at scale?**

We added a Lambda (`lambda:13`) that runs every 5 minutes and pulls tenant sign-ups from Segment. It creates a new Redis index (`tenant:{id}`) with the correct schema and pre-warms the embedding cache with a synthetic query. The Lambda runs for 80 ms on average and costs $0.03 / month. Without it, new tenants faced 3.2 s cold starts until the cache filled naturally.

**What happens if the reranker queue backs up?**

We set a max queue depth of 1 k messages per tenant. When depth > 500, we throttle new queries from that tenant with a 429 response and a `Retry-After: 10` header. The frontend shows a spinner and retries every 300 ms. Queue depth is monitored with a custom CloudWatch metric `rerank_queue_depth{tenant}`. During Black Friday traffic (3× normal load) the queue depth peaked at 840, but the p95 latency stayed under 2.2 s.

**Do you still use SageMaker for embeddings?**

No. We moved embeddings to a FastAPI service running on 2× m6g.medium with the MiniLM model quantised to int8. Latency dropped from 420 ms (SageMaker) to 180 ms (local), and cost fell from $2.4 k / month to $48 / month. We only use SageMaker for the rare tenants who upload 100 k+ documents; even then we run a single a1.large endpoint and scale it manually during onboarding.

Open `redis.conf` in your editor and change these two lines:
