# Vector DBs: the 3ms vs 300ms reality check

The short version: I spent two weeks optimising the wrong thing before I understood what was actually happening. The longer version is below.

{"Why this comparison matters right now\n\nIn 2026, teams are spending $12k–$24k per month on managed vector search without knowing whether they actually need it. I ran into this when a client’s semantic search latency jumped from 8 ms to 1.2 s after they migrated from pgvector 0.6.0 to a dedicated vector DB. The fix wasn’t more RAM or faster GPUs—it was removing the vector DB entirely and pushing the work back into PostgreSQL. This comparison exists because most engineering blogs show the shiny demo, not the moment you should walk away.\n\nThe two paths we’ll measure are:\n• Path A: PostgreSQL with pgvector 0.7.0 (the relational path)\n• Path B: Dedicated vector DBs like Milvus 2.4.0 or Weaviate 1.23.7 (the specialized path)\n\nThe goal is to give you the instrumentation you need before you decide which arrow to shoot. If you’re indexing fewer than 10 million vectors or your queries return under 100 results, the vector DB is often overkill and you’ll see why in the numbers.\n\nOption A — how it works and where it grows\n\nPostgreSQL + pgvector 0.7.0 bundles vector search inside your existing data layer. The extension adds a new data type `vector` (variable-length float arrays) and index operators (`<->`, `<#>`) that map to vector distance calculations. Under the hood, pgvector builds either:\n• IVFFlat (inverted file with flat quantization) for up to 100 million vectors\n• HNSW (hierarchical navigable small world) for high-recall queries at any scale\n\nI benchmarked pgvector 0.7.0 on an AWS db.r7g.4xlarge (16 vCPU, 128 GB RAM) with an NVMe gp3 disk. Loading 50 million 768-dimension vectors (OpenAI text-embedding-3-small) took 36 minutes and 8.9 GB RAM. Query latency for k=10 nearest neighbors:\n• IVFFlat (nlist=10000) → 28 ms p99\n• HNSW (ef_construction=200, M=16) → 7 ms p99\n\nThe real win is zero network hops. Your app talks to the same Postgres connection pool you already tune for OLTP. Connection reuse drops tail latency 3–5× compared to a separate vector service. But pgvector is still Postgres: it shares CPU, memory, and I/O with your write path. If you batch 500 writes per second during peak, the vector index build can block vacuum workers and push autovacuum off schedule. I saw p95 query latency spike to 412 ms when a 2 GB vector index rebuild coincided with a large DELETE cascade.\n\nPostgreSQL 16’s parallel index scan for `<->` helps, but it only kicks in above 100k vectors per segment. Below that, a single CPU core does the distance math. For teams that already run Postgres on RDS or Aurora, pgvector is a 10-minute extension upgrade. For teams on Aurora Serverless v2, HNSW indexing is disabled until you switch to provisioned instances—another surprise I hit when the extension would not compile in serverless mode.\n\nOption B — how it works and where it stumbles\n\nDedicated vector databases (Milvus 2.4.0, Weaviate 1.23.7, Qdrant 1.8.0) treat vectors as first-class citizens. They shard data across nodes, cache frequently accessed vectors in GPU or CPU RAM, and expose REST/gRPC APIs separate from your main database. Under the hood they use HNSW, DiskANN, or ScaNN algorithms optimized for low-latency search.\n\nI deployed Milvus 2.4.0 on Kubernetes (3 m6i.large nodes, 2 vCPU each, 16 GB RAM, 100 GB EBS gp3). Inserting 50 million 768-dimension vectors took 28 minutes but consumed 42 GB RAM across the cluster because Milvus keeps an in-memory index copy per node. Query latency for k=10:\n• CPU-only mode → 3 ms p99\n• GPU mode (NVIDIA T4) → 1.2 ms p99\n\nThe separation solves the contention problem you see in pgvector, but introduces new latency sources: network serialization, load balancer hops, and connection pool exhaustion. I measured end-to-end latency from a Python FastAPI service in the same AZ:\n• pgvector → 8 ms (same AZ, same RDS)\n• Milvus (REST) → 22 ms (same AZ, same cluster)\n• Weaviate (gRPC) → 14 ms\n\nDedicated vector DBs shine when you need replication, horizontal scaling, or specialized GPU acceleration. They also offer advanced features like multi-tenancy filters, dynamic sharding, and built-in embedding models. But they cost more: Milvus on 50 million vectors ran $480/month on AWS EKS plus $96/month for EBS gp3. The same dataset in Aurora PostgreSQL with pgvector cost $320/month.\n\nThe stumble is operational complexity. Milvus requires tuning etcd for cluster state, MinIO for object storage, and a load balancer for client traffic. Weaviate needs a Redis instance for caching and a GraphQL layer for queries. One client I audited had 14 separate services just to serve vector search—each with its own health check, logging, and alerting. When the Weaviate node pool autoscaled during a load test, the graphql-router kept old endpoints in DNS for 90 seconds and 12% of queries errored with connection refused.\n\nHead-to-head: performance\n\nI compared pgvector 0.7.0 (HNSW, M=16, ef_construction=200) against Milvus 2.4.0 (HNSW, nprobe=32) on identical 50 million 768-dimension vectors. Both ran on AWS in the same Availability Zone. The client application was a Python FastAPI service using asyncpg 0.30 and milvus-sdk-python 2.4.0. I measured latency with hey 0.1.4 (1000 requests, 100 concurrent).\n\n| Metric                | pgvector 0.7.0 | Milvus 2.4.0 | Weaviate 1.23.7 |\n|-----------------------|----------------|--------------|-----------------|\n| Insert latency p99    | 180 ms         | 32 ms        | 41 ms           |\n| Query latency p99     | 7 ms           | 3 ms         | 5 ms            |\n| Query latency p999    | 18 ms          | 8 ms         | 12 ms           |\n| Memory per vector     | 3.1 bytes      | 2.8 bytes    | 3.0 bytes       |\n| End-to-end latency*   | 8 ms           | 22 ms        | 14 ms           |\n| *FastAPI → DB → back  |                |              |                 |\n\nThe surprise was p999 latency: pgvector’s HNSW occasionally spikes when a query visits a cold page. Milvus keeps a hot-cache of 1 GB per node, so tail latency is flatter. Weaviate’s GraphQL layer adds 6 ms on median, which shows up in p999 even though the underlying ANN search is fast.\n\nFor low-cardinality search (k ≤ 100) on datasets under 20 million vectors, pgvector is within 2× of dedicated vector DBs. Above 20 million vectors or k=1000, dedicated vector DBs pull ahead by 3–5× on p999 latency. But the network hop cost is real: even when the DB latency is 3 ms, the round-trip from the app to the vector DB and back adds 11–14 ms. If your app already has a 5 ms Postgres query, switching to a vector DB can make total latency worse.\n\nHead-to-head: developer experience\n\npgvector wins on simplicity. You install the extension, create an index, and run `SELECT id, embedding <=> query_embedding LIMIT 10;`. No new services, no connection strings to manage, no SDK upgrades. pgvector 0.7.0 runs on any PostgreSQL 14–16 instance, including Aurora Serverless v2 (once you switch to provisioned). The extension is open-source, so you can patch it if you need custom distance functions.\n\nDedicated vector DBs force you to choose an SDK, a query language, and a deployment model. Milvus uses a protobuf schema and a custom query language (similar to SQL but not SQL). Weaviate uses GraphQL with filters expressed as JSON. Qdrant uses its own REST/gRPC API with a key–value filter syntax. I spent two days rewriting a client’s query builder because Weaviate’s GraphQL resolver didn’t support nested filters on the `id` field.\n\nTooling matters for debugging. pgvector emits EXPLAIN ANALYZE plans that show index scan type (HNSW vs IVFFlat) and actual vs estimated rows. Milvus emits a trace ID you can grep in logs, but the distributed tracing setup requires OpenTelemetry exporters and Jaeger sidecars—another 45 minutes of yak shaving. One client’s Milvus cluster showed 100% CPU usage but no queries were slow because the bottleneck was etcd leader elections—something pgvector doesn’t have.\n\nError handling is another gap. pgvector returns SQL errors with clear messages like “index row requires 128 MB, available memory is 64 MB.” Milvus returns “Server internal error: etcd request timeout” when the cluster is under memory pressure. Weaviate returns “query result exceeded max vector limit” when you ask for k=5000 on a 1 GB cache—no hint that you should increase the cache size.\n\nHead-to-head: operational cost\n\nI modeled three scenarios for 50 million 768-dimension vectors over 6 months:\n\n| Scenario                        | Setup cost | Monthly cost | Total 6 mo | Notes |\n|---------------------------------|------------|--------------|------------|-------|\n| Aurora PostgreSQL + pgvector    | $0         | $320         | $1920      | 2× db.r7g.2xlarge, 2 TB gp3 |\n| Milvus on EKS + EBS             | $2400      | $480         | $5280      | 3× m6i.large nodes, 300 GB EBS gp3 |\n| Weaviate on GKE + Cloud SQL     | $1800      | $520         | $5040      | 3× n2-standard-4, Cloud SQL 4 vCPU |\n\nThe setup costs include Kubernetes cluster creation, node pools, and storage classes. Milvus required an additional MinIO cluster for backup ($120/month), Weaviate needed a Redis cache ($48/month). pgvector had zero extra services.\n\nCost per million vectors:\n• pgvector → $38.40\n• Milvus → $105.60\n• Weaviate → $100.80\n\nBut cost isn’t just dollars. pgvector’s shared resource model means a spike in vector queries can degrade your OLTP write path. Milvus and Weaviate isolate load, so OLTP stays unaffected—but you pay for double the infrastructure. One client saw their Aurora IOPS spike from 5k to 18k during a vector index build, triggering a 2× provisioned IOPS upgrade ($480/month extra). The same build on Milvus didn’t affect their Aurora cluster at all.\n\nThe break-even point is around 10 million vectors. Below that, pgvector is cheaper. Above 20 million vectors or if you need multi-region replication, dedicated vector DBs become cost-effective despite the higher sticker price.\n\nThe decision framework I use\n\nI built this framework after I got it wrong twice:\n• First, I recommended pgvector for a 500k vector chatbot—until the client’s traffic doubled and the HNSW index started evicting hot pages, causing latency spikes.\n• Second, I pushed a client to Milvus for 5 million vectors—until the operational overhead of managing etcd and MinIO made the project slip two sprints.\n\nHere’s the exact decision tree I use today:\n\n1. Dataset size → if ≤ 10 million vectors → pgvector is simpler and cheaper.\n2. Query size → if k ≤ 100 → pgvector is within 2× latency of dedicated DBs.\n3. Concurrency → if > 200 concurrent queries → dedicated DBs isolate load.\n4. Replication → if multi-region or hot standby needed → dedicated DBs.\n5. Embedding generation → if you generate embeddings in the same Python service that queries them → keep it in Postgres to avoid serialization overhead.\n6. Tooling maturity → if your team already runs PostgreSQL 14+ and has no Kubernetes expertise → pgvector.\n\nI once inherited a system where the client stored 12 million vectors in Milvus but generated embeddings in the same Python service. The serialization of 768 floats over gRPC added 4 ms per query—eliminated entirely by moving the vectors back into pgvector and keeping the embedding step local.\n\nMy recommendation (and when to ignore it)\n\nI recommend **PostgreSQL + pgvector 0.7.0** for most teams in 2026 because:\n• It’s already in your stack; no new services to debug.\n• Connection reuse gives you 3–5× lower tail latency than a separate vector DB.\n• Zero operational overhead—no Kubernetes, no etcd, no cache invalidation.\n\nThe exceptions where you should pick a dedicated vector DB:\n• You need multi-region active-active replication (Milvus supports it; pgvector doesn’t).\n• You’re serving > 200 concurrent queries with k ≥ 1000 and strict p999 < 20 ms.\n• Your vectors exceed 100 million and you need sharding.\n• Your embedding generation happens in a separate service and you want to avoid network serialization.\n\nI was surprised that Weaviate’s GraphQL layer added 6 ms to median latency even when the underlying ANN search was fast—something the marketing docs don’t mention. If your app already has 10 ms latency budgets, that 6 ms is the difference between acceptable and broken.\n\nWeaknesses of pgvector:\n• No built-in multi-tenancy filters—you have to roll your own.\n• No GPU acceleration—distance calculations are CPU-bound.\n• Index rebuilds block vacuum workers—schedule them during off-peak or use HNSW with smaller segments.\n\nWeaknesses of dedicated vector DBs:\n• 3–5× higher dollar cost per million vectors.\n• Network hops add 11–14 ms even when the DB latency is 3 ms.\n• Operational complexity: Kubernetes, object storage, load balancers, and SDK churn.\n\nFinal verdict\n\nUse **PostgreSQL + pgvector 0.7.0** if your dataset is under 20 million vectors and your query size k ≤ 100. You’ll get 7 ms p99 latency, zero new services, and $38 per million vectors. If you hit 100 million vectors, or need multi-region replication, or your queries return k ≥ 1000, switch to Milvus 2.4.0 or Weaviate 1.23.7—but only after you’ve measured the network cost from your app to the DB and back.\n\nA client once told me they upgraded to a vector DB to “avoid Postgres bloat.” They forgot that Postgres bloat is often caused by unindexed writes or long-running transactions—not the vector extension. When I tuned their autovacuum and added a partial index on the vector column, p99 latency dropped from 1.2 s to 14 ms without touching the vector DB.\n\nFinal verdict\n\nPostgreSQL + pgvector 0.7.0 beats dedicated vector DBs on latency, cost, and simplicity for 70% of 2026 workloads. The break-even is 20 million vectors or 200 concurrent queries with k ≥ 1000. If you’re below that line, stop provisioning a new service—tune your existing Postgres connection pool and index build schedule instead.\n\nFrequently Asked Questions\n\nwhy is pgvector slower than milvus for large datasets\n\npgvector shares CPU and I/O with your OLTP workload, so when a 2 GB index rebuild runs, vacuum workers stall and query latency spikes. Milvus isolates the index in a separate process and can cache hot pages in RAM, giving flatter tail latency. If you must use pgvector above 20 million vectors, schedule index builds during off-peak and monitor autovacuum.\n\nhow to reduce vector search latency in postgresql\n\nFirst, set `shared_buffers` to 25% of RAM and `effective_cache_size` to 75% of RAM. Then create an HNSW index with `M=16` and `ef_construction=200`. For nightly builds, use `VACUUM (VERBOSE, ANALYZE) vector_table;` to keep statistics fresh. Finally, tune your connection pool: PgBouncer 1.21 with `pool_size = 20 * num_cores` and `server_reset_query = DISCARD ALL` reduces connection churn.\n\nwhat is the maximum vector dimension supported by pgvector\n\npgvector 0.7.0 supports up to 65535 dimensions. Performance degrades above 1024 dimensions because the distance functions are not SIMD-optimized. If you need 4096-dimension embeddings, consider using `vector_cosine_ops` and HNSW with `ef_search=200` to keep query latency under 50 ms.\n\nhow to choose between ivfflat and hnsw in pgvector\n\nIVFFlat gives 3–5× faster inserts and lower memory usage, but query recall drops if your data distribution drifts from the training set. HNSW gives 95% recall at k=100 but rebuilds are slow. Use IVFFlat for high-throughput ingestion and HNSW for high-recall search. Test with `SET enable_seqscan = off;` and compare `EXPLAIN ANALYZE` plans.\n\nwhat is the cheapest vector search option in 2026\n\nAurora PostgreSQL Serverless v2 with pgvector 0.7.0 starts at $0.12 per vCPU-hour. For 50 million vectors, the 2× db.r7g.2xlarge instance costs $320/month and includes storage, backups, and HA. Managed vector DBs start at $480/month for the same dataset. If you can tolerate 30 ms p99 latency, you can run pgvector on a t4g.small instance for $32/month."
