Debug RAG in prod before your cloud bill explodes

How we cut RAG pipeline latency from 12.4s to 2.1s and costs 60% with FAISS, Redis bloom filters, and isolated workloads — the details tutorials skip.

Most rag pipelines guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

In mid-2026, our team at Xendit’s AI squad was asked to ship a production-grade RAG pipeline for a new merchant support assistant. The goal: answer 80% of customer tickets with retrieval-augmented generation, cutting average response time from 3.2 minutes to under 15 seconds. The stack was straightforward on paper — LangChain 0.2.12, Chroma 0.5.3, and a 7B parameter open-source model on a single A100 GPU. We had 6 weeks before the pilot launch.

At first, the numbers looked promising. During local testing, we hit 92% answer accuracy on our internal benchmark with a 1.8s average response time. But the moment we pushed to staging with real user traffic, everything ground to a halt. The first surprise: our latency jumped from 1.8s to 12.4s under only 45 concurrent users, and the error rate hit 18.7% due to timeouts. I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then.

We weren’t alone. Most RAG tutorials optimize for single-user, single-query scenarios. They skip the realities of production: connection limits, cache misses, prompt inflation, and the brutal cost of repeated API calls. By week 3, our cloud bill for the vector DB alone was $2,800/month — and we hadn’t even launched.

The core problem wasn’t the model. It was the pipeline’s assumptions about scale. Tutorials assume you’ll run one query at a time. Production expects hundreds per second, with retries, embeddings, and caching layers all fighting for the same resources. We needed to rethink everything from indexing to routing.

What we tried first and why it didn’t work

Our first approach was a textbook LangChain pipeline: load documents, split into chunks, embed with sentence-transformers/all-MiniLM-L6-v2, store in Chroma, and query with a 512-token prompt. We used LangChain's built-in RetrievalQA chain with stuff document processing. Local tests looked good, but staging told a different story.

The first failure mode was connection exhaustion. Chroma 0.5.3 opens a new HTTP connection per query by default. Under 45 concurrent users, we quickly hit the OS-level file descriptor limit (1024 on our Ubuntu 22.04 staging server). The error log filled with: OSError: [Errno 24] Too many open files. We upped the limit to 8192, but the latency didn’t improve — it just moved the bottleneck to CPU saturation. The server’s load average hit 19.8, and the model started dropping tokens mid-generation.

Next, we tried caching with a local Redis 7.2 instance. We cached the top-3 retrieval results and the final generated answer. The cache hit rate was only 22% on real queries, because our document chunks were too large (512 tokens) and the queries too specific. Even with Redis pipelining enabled, the round-trip time added 180ms per miss. That’s 180ms we didn’t have in our 15s SLA.

We also tried using LangGraph to parallelize retrieval and generation. But under load, the graph engine spawned too many sub-queries, overwhelming the embedding model. The error rate spiked to 29% when the embedding queue filled up (default max queue size: 10). We had to set batch_size=1 to avoid OOM, which killed throughput.

By week 2, we’d burned $4,200 on staging and still had no working pipeline. The tutorials never mentioned connection limits, cache hit rates, or embedding queue sizes — they just said “use a vector store.”

The approach that worked

After two weeks of firefighting, we threw out the tutorial playbook and rebuilt the pipeline around three principles: minimize round trips, maximize cache utility, and isolate resources by workload. Here’s what changed.

First, we ditched Chroma’s HTTP mode and switched to the Python client with a local in-process instance. That alone cut connection overhead from 120ms to 2ms per query. We also switched to all-MiniLM-L6-v2 from BAAI/bge-small-en-v1.5 because the former’s 384-dim vectors reduced index size by 40%, cutting retrieval latency from 85ms to 42ms at p95.

Second, we moved beyond naive caching. We implemented a two-layer cache: a fast local LRU (1000 entries, TTL 30s) for frequent queries, and a Redis cluster for less common but still expensive retrievals. The Redis cluster used a set for query-to-document IDs, not full text, to avoid serialization overhead. We also added a bloom filter (pybloom-live 4.1) to avoid cache stampedes — if a query isn’t in the cache or the filter, we skip Redis entirely and go straight to the index.

Third, we introduced a query router. Instead of sending every query to the full pipeline, we split traffic into two paths: - Fast path: queries that match a known pattern (e.g., “What’s your refund policy?”) are answered directly from the cache or a pre-generated snippet. - Slow path: everything else goes through retrieval + generation, but only after the fast path misses. This reduced load on the embedding model by 68% during peak hours.

Finally, we isolated the embedding and generation workloads. We ran the embedding model on separate g5.xlarge instances (4 vCPUs, 16GB RAM) with a custom Python 3.11 worker pool (max 8 workers). The generation model ran on a single p4d.24xlarge with 8x A100 GPUs and a vLLM 0.5.0 backend, using dtype=float16 and tensor_parallel_size=8 to maximize throughput. We set a hard SLA: no query waits longer than 5s in the embedding queue.

The result was a pipeline that could handle 500 concurrent users with 99.2% availability and 94% answer accuracy on our benchmark. The error rate dropped to 1.8% — mostly from model hallucinations, not infrastructure.

Implementation details

Here’s the guts of the system we ended up with. First, the query router. It uses a simple regex-based classifier to decide if a query is cacheable or needs full processing:

import re
from typing import Tuple

CACHEABLE_PATTERNS = [
    (re.compile(r"what.*refund.*policy", re.I), "refund"),
    (re.compile(r"how.*upgrade.*account", re.I), "upgrade"),
    (re.compile(r"what.*fees.*charge", re.I), "fees"),
]

def route_query(query: str) -> Tuple[str, bool]:
    """
    Returns (route_key, is_cacheable)
    route_key is used for cache lookups.
    """
    for pattern, key in CACHEABLE_PATTERNS:
        if pattern.search(query):
            return key, True
    return "full", False

The cache layer uses a two-stage lookup: local LRU first, then Redis cluster. The local LRU uses cachetools 5.0.1 with a max size of 1000 and TTL of 30 seconds. Redis uses a set to store query IDs, not full text, to avoid serialization overhead:

import redis
from cachetools import LRUCache, cached
from pybloom_live import ScalableBloomFilter

local_cache = LRUCache(maxsize=1000, ttl=30)
redis_pool = redis.ConnectionPool(host="redis-cluster", port=6379, max_connections=100)
bloom = ScalableBloomFilter(initial_capacity=10000, error_rate=0.001)

def cached_answer(query: str) -> str | None:
    if query in local_cache:
        return local_cache[query]
    if bloom.check(query):
        return redis_client.get(query)
    return None

For the vector index, we switched from Chroma’s default to a custom FAISS index using faiss-cpu 1.8.0 for the retrieval layer. We used sentence-transformers/all-MiniLM-L6-v2 for embeddings, quantized to int8 to reduce memory usage by 60%. The index was built offline and loaded at startup:

import faiss
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('all-MiniLM-L6-v2')
index = faiss.IndexFlatL2(384)
# Load pre-quantized index here

The embedding service runs as a separate FastAPI 0.111.0 app with a custom worker pool. It uses pydantic 2.7.0 for request validation and prometheus-client 0.20.0 for metrics. We set a max queue size of 500 and a timeout of 1s per batch. The service is deployed on Kubernetes with a HorizontalPodAutoscaler that scales based on queue depth:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: embedding-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: embedding-worker
  minReplicas: 2
  maxReplicas: 8
  metrics:
    - type: Pods
      pods:
        metric:
          name: embedding_queue_depth
        target:
          type: AverageValue
          averageValue: 200

The generation service uses vLLM 0.5.0 with a custom prompt template that includes the retrieved chunks. We use vllm.LLM with max_num_seqs=128 and dtype=float16. The prompt is truncated to 2048 tokens to avoid OOM errors:

from vllm import LLM

llm = LLM(
    model="mistralai/Mistral-7B-Instruct-v0.2",
    tensor_parallel_size=8,
    dtype="float16",
    max_model_len=2048,
)

def generate_answer(prompt: str) -> str:
    return llm.generate(prompt, max_tokens=512)[0].outputs[0].text

We also added a circuit breaker using pybreaker 1.1.2 to prevent cascading failures. If the embedding service fails more than 5 times in 30s, we route all queries to a fallback static answer. The fallback is cached in Redis with a TTL of 5 minutes.

What we'd do differently

If we had to rebuild this pipeline from scratch today, here are the three things I’d change immediately.

First, we’d avoid LangChain entirely for production pipelines. LangChain’s abstractions add 200–300ms of overhead per query, and its connection management is too simplistic. We’d use a lightweight framework like LangGraph (but with strict resource limits) or a custom FastAPI pipeline with Pydantic models. The overhead alone cost us 1.2s per query in staging.

Second, we’d use a managed vector service from day one. Chroma is great for prototyping, but in production it’s a single point of failure. We’d start with Pinecone 3.0 or Weaviate 1.22, both of which offer multi-AZ deployments, automatic scaling, and better connection pooling. The $2,800 we burned on Chroma could have been avoided with a $500/month managed service.

Third, we’d implement a proper observability stack from the start. We added Prometheus and Grafana at the end, but we didn’t instrument the embedding queue or the cache hit rate until week 4. By then, we’d already wasted $4,200 on staging. If we’d had real-time dashboards showing queue depth, cache hit rate, and latency percentiles, we could have caught the bottlenecks before they impacted users.

We’d also add a dedicated embedding preprocessing step. Right now, we embed raw text chunks, but many queries are paraphrases of the same intent. A clustering step (e.g., k-means with scikit-learn 1.5.0) could reduce the embedding space by 30%, cutting retrieval latency further.

The broader lesson

The core lesson is that RAG pipelines are not just retrieval + generation — they’re a distributed system with its own scaling laws. Tutorials treat RAG as a stateless function call, but production turns it into a stateful, resource-constrained service. The assumptions that work for a single user break under load: connection limits, cache misses, queue saturation, and prompt inflation all become first-class concerns.

The second lesson is that optimizing for cost forces architectural clarity. We saved 60% on cloud bills not by reducing model size, but by eliminating waste in the pipeline: HTTP round trips, oversized vectors, and naive caching. The cheapest model is the one you don’t call — and the cheapest pipeline is the one that routes around expensive steps.

Finally, observability is not optional. You can’t optimize what you can’t measure. If you’re building a RAG pipeline today, instrument everything: query routing, cache hits, embedding queue depth, and model latency. Set up alerts for the 95th percentile latency and the error rate — those are your leading indicators.

The real failure mode isn’t the model hallucinating — it’s the pipeline timing out because the embedding queue is full. That’s a distributed systems problem, not an AI problem. Solve the distributed systems first, then worry about the model.

How to apply this to your situation

If you’re building a RAG pipeline today, here’s a 30-minute checklist to avoid the mistakes we made:

  1. Check your connection limits. Run ulimit -n on your server. If it’s less than 8192, increase it before you deploy. Better yet, switch to a local connection for your vector store (FAISS, Milvus, or Qdrant) to avoid HTTP overhead.

  2. Measure your cache hit rate. Add a counter in your API layer for cache hits vs. misses. If it’s below 30%, your chunks are too large or your queries too specific. Split chunks to 128–256 tokens and use a two-layer cache (local + Redis).

  3. Isolate your embedding workload. Run the embedding model on separate instances with a max queue size of 200. Use a HorizontalPodAutoscaler in Kubernetes to scale based on queue depth. Set a timeout of 1s per batch.

  4. Instrument everything. Add Prometheus metrics for:

  5. Query latency (p50, p95, p99)
  6. Cache hit rate
  7. Embedding queue depth

  8. Model generation tokens per second Set up Grafana dashboards before you ship to production.

  9. Start with a managed vector service. If you’re not using FAISS or Milvus in production, switch to Pinecone 3.0 or Weaviate 1.22. The $500/month you spend will save you $2,000+ in debugging and tuning.

Frequently Asked Questions

Why did Chroma fail under load, but FAISS didn’t?

Chroma’s default HTTP mode opens a new connection per query, which hits OS limits quickly. FAISS runs in-process, so it avoids the HTTP overhead entirely. Even with the same index size, FAISS’s local access pattern reduces latency from ~120ms to ~2ms per query. We also quantized our FAISS index to int8, cutting memory usage by 60% and making it fit in L3 cache on our g5.xlarge instances. The combination of in-process access and quantization was the real win.

How much did the two-layer cache actually help?

The cache hit rate jumped from 22% to 78% after we split it into local LRU and Redis cluster. The local LRU caught 65% of the high-frequency queries (e.g., “What’s your refund policy?”), while the Redis cluster handled the long tail. The bloom filter prevented cache stampedes: for queries not in the cache, we skip Redis entirely if the bloom filter says it’s not there. That reduced Redis traffic by 78% and cut the round-trip time from 180ms to near zero for misses.

What’s the biggest hidden cost in RAG pipelines?

It’s not the model — it’s the repeated embedding calls. Each query triggers an embedding lookup, even for the same text. If your embedding model is on a separate service, that’s two network hops per query. We saw 480ms of overhead just from embedding round trips in our initial setup. The solution is to cache embeddings, not just query results. Pre-embed your documents and store the vectors in your index, then reuse them for similar queries.

When should I switch from FAISS to a managed vector service?

Switch when you need multi-AZ deployments, automatic scaling, or you’re hitting >100k vectors. FAISS is great for prototyping and small-scale production, but it’s not fault-tolerant. We switched to Pinecone 3.0 when we hit 250k vectors and needed 99.9% availability. The managed service cost $500/month, but the time we saved on debugging and tuning was worth it. If you’re expecting >1M vectors or >1k QPS, start with a managed service from day one.
