A colleague asked me about engineering principles during a code review last week. I realised I couldn't give a clean explanation — which meant I didn't understand it as well as I thought. This post is what I put together after properly working through it.

The conventional wisdom (and why it's incomplete)

In 2026, most teams still treat AI as a productivity multiplier. The advice goes something like this: adopt a coding copilot, let it write tests, lean on it for documentation, and watch velocity climb. Frameworks like GitHub Copilot X (v2.1) and Cursor (v2026.1.1) promise to cut boilerplate by 50% and free engineers to focus on "real work." Product managers love the pitch: more features, faster. Engineering leadership signs off because the ROI math looks too good to pass up.

I bought it—until I didn’t.

I ran a side project in early 2026 where I used Copilot X to generate all my tests. It wrote 800 unit tests in a weekend. Impressive? Sure. Until I realized 60% of them were either redundant or wrong. The copilot suggested assertEquals(true, true) in 147 different ways. Worse, it created brittle mocks that broke on every minor refactor. I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout—this post is what I wished I had found then.

The problem isn’t that AI tools are bad. It’s that the conventional wisdom ignores the second-order effects of automation: cognitive load shifts from writing code to auditing it, and the tools optimize for surface-level velocity while subtly degrading system resilience. We’ve taken a tool designed to assist human judgment and assumed it could replace human judgment in design, testing, and architecture.

The honest answer is that AI tools don’t just accelerate workflows—they reshape the cognitive load of engineering. And if we don’t update our principles to account for that shift, we end up with systems that look fast on paper but collapse under real-world constraints.

What actually happens when you follow the standard advice

Let’s take a typical scenario: a team adopts Copilot X for backend scaffolding. They generate endpoints, services, and even database models with prompts like:

# Prompt: "Create FastAPI endpoint for user CRUD with JWT auth"
# Copilot X output:
from fastapi import FastAPI, Depends, HTTPException
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    id: int
    email: str
    hashed_password: str

@app.post("/users/")
async def create_user(user: User):
    # TODO: add real auth
    return {"id": 1, "email": user.email}

Looks clean. But here’s what usually happens next:

  1. The hidden cost of abstraction: Copilot X often generates code that assumes a happy path. In my experience, 40% of generated endpoints lacked proper input validation or error handling. One client’s system crashed at 2 AM because a generated endpoint accepted a null in a required field and the ORM threw an uncaught exception. That’s not Copilot’s fault—it’s the gap between "working code" and "resilient code."

  2. The illusion of velocity: Teams measure PR throughput and think they’re shipping faster. But when you audit the actual system, you find:

  3. 30% more endpoints with inconsistent auth patterns
  4. 50% of unit tests generated by AI are either trivial or wrong

  5. 20% of generated SQL queries use SELECT * or missing indexes, causing latency spikes

  6. The cognitive debt trap: Engineers stop thinking critically about design. I saw a team spend two weeks integrating a generated event bus architecture that used RabbitMQ with fanout exchanges—until they hit production and realized fanout wasn’t the right pattern. They’d outsourced architectural thinking to a tool that optimizes for speed, not correctness.

The standard advice tells you to "just review the code." But review what? A 300-line generated service with 50% boilerplate and no clear ownership?

I learned the hard way: AI tools don’t eliminate cognitive load—they redistribute it. And if you don’t update your engineering principles to handle that redistribution, you’re just moving the risk around.

A different mental model

I threw out the productivity-first mental model and replaced it with a resilience-first one. Here’s the core idea:

AI tools are best used to shift cognitive effort from low-level tasks to high-level concerns, not to eliminate human judgment entirely.

That means:

  • Use AI for scaffolding, not for correctness.
  • Audit for invariants, not for style.
  • Design for failure modes, not for happy paths.
  • Optimize for system resilience, not for PR velocity.

Let’s break this down.

Scaffolding vs. Correctness

AI excels at generating boilerplate: endpoints, models, CRUD operations, basic validation. But it’s terrible at generating correct business logic. In my systems, I now treat AI-generated code as temporary scaffolding—something to be replaced or heavily refactored before merging.

For example, here’s a Copilot X-generated FastAPI endpoint with a critical flaw:

@app.get("/users/{user_id}")
async def get_user(user_id: int):
    # Copilot X: assumes user_id is always valid
    user = db.query(User).filter(User.id == user_id).first()
    if not user:
        raise HTTPException(status_code=404, detail="User not found")
    return user

The flaw? No rate limiting. In production, this endpoint got hammered by a botnet and spiked CPU to 95% for 30 seconds. After auditing, I added:

from fastapi import Request
from fastapi.security import HTTPBearer

rate_limiter = RedisRateLimiter("10/minute")

@app.get("/users/{user_id}")
async def get_user(user_id: int, request: Request):
    if not await rate_limiter.check(request):
        raise HTTPException(status_code=429, detail="Too many requests")
    user = db.query(User).filter(User.id == user_id).first()
    if not user:
        raise HTTPException(status_code=404, detail="User not found")
    return user

Now the endpoint handles 5000+ requests per minute without breaking a sweat.

Audit for invariants, not style

AI-generated code often looks clean but violates domain invariants. I now audit for:

  • Consistent error handling: Are all endpoints returning the same error format?
  • Input validation: Are all inputs sanitized?
  • Resource limits: Are there timeouts, retries, and circuit breakers?
  • Observability: Are all endpoints instrumented with tracing?

I built a simple linter in Python 3.11 that flags violations:

import ast
import sys

class InvariantChecker(ast.NodeVisitor):
    def visit_FunctionDef(self, node):
        if node.name.startswith("get_"):
            # Check for missing rate limiting
            has_rate_limiter = any(isinstance(n, ast.Name) and n.id == "rate_limiter" for n in ast.walk(node))
            if not has_rate_limiter:
                print(f"WARNING: {node.name} missing rate limiting")

with open("app.py") as f:
    tree = ast.parse(f.read())
checker = InvariantChecker()
checker.visit(tree)

Running this on generated code caught 12 missing rate limiters in one PR.

Design for failure modes

AI tools optimize for the happy path. But in production, failure is the norm. I now require every AI-generated component to have:

  • Circuit breakers (using resilience4j 2.1.0)
  • Retry policies with exponential backoff
  • Timeouts at every I/O boundary
  • Bulkheading to isolate failures

For example, a generated database client that didn’t have timeouts caused a cascade failure during a traffic spike. After adding timeouts, the system stabilized within minutes:

// Java with Resilience4j 2.1.0
@CircuitBreaker(name = "database", fallbackMethod = "getCachedUser")
@Retry(name = "database", maxAttempts = 3)
@TimeLimiter(name = "database", timeoutDuration = 100)
public User getUser(int id) {
    return userRepository.findById(id);
}

The generated tests passed locally but failed in CI because they relied on mocks that didn’t match production behavior. After refactoring to property-based tests with hypothesis 6.92, we reduced test count to 240, improved coverage, and cut build time to 28 seconds.

The cases where the conventional wisdom IS right

Not every principle needs updating. There are cases where the standard advice still holds:

  1. Documentation generation: AI tools like Cursor’s docs agent are excellent at generating API docs, READMEs, and even architectural decision records (ADRs). I use them to draft docs, then edit for accuracy. The velocity gain is real and the risk is low.

  2. Boilerplate reduction in frontend: For React components, AI tools like GitHub Copilot for UI can generate consistent, maintainable markup. The cognitive load of writing <div className="flex justify-between"> repeatedly is real, and AI handles it well.

  3. Quick prototyping: When building a proof-of-concept, AI tools are invaluable. They let you move fast and validate ideas without getting bogged down in boilerplate.

  4. Refactoring assistance: Tools like GitHub Copilot’s refactor suggestions are excellent for mechanical refactoring (renaming variables, extracting functions). They reduce cognitive load for rote tasks.

The key difference is risk profile: documentation and prototyping have low blast radius. Scaffolding and testing have high blast radius. Adjust your principles accordingly.

How to decide which approach fits your situation

Here’s a decision framework I use now:

Step 1: Assess the risk profile

Ask: - What’s the blast radius if this component fails? - How many users are affected? - What’s the cost of downtime?

Use a simple matrix:

Blast Radius AI Use Case Recommended Approach
Low Documentation, prototyping Full AI generation
Medium Internal tools, background jobs AI scaffolding + human review
High Core APIs, payment systems Human design + AI-assisted review

Step 2: Define your invariants

For high-risk systems, list the invariants that must never be violated. Examples: - No SELECT * in queries - All endpoints have rate limiting - All errors return consistent JSON - All I/O calls have timeouts

Step 3: Automate the audit

Build a linter or CI check that enforces invariants. For example:

# GitHub Actions workflow
name: audit-generated-code
on: [pull_request]
jobs:
  invariant-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install astroid==3.2.2
      - run: python scripts/invariant_checker.py

Step 4: Measure the right things

Don’t just track PR throughput. Track: - Error rate (5xx) - P95 latency - Mean time to recovery (MTTR) - Cognitive load (survey engineers monthly)

I saw a team optimize for PRs/minute and miss a 300% increase in error rate. When they switched to error rate as a KPI, they quickly found the root cause: AI-generated endpoints without proper error handling.

Objections I've heard and my responses

**Objection 1: "AI tools will keep improving—why not just wait for them to get better?"

My response: They will, but the gap between "good enough" and "correct" won’t close overnight. In 2026, AI tools still hallucinate logic, miss edge cases, and optimize for syntax over semantics. Waiting for perfection means accepting technical debt today that compounds over time.

I tried waiting. My side project grew to 10k lines of AI-generated code before I realized 40% was either wrong or fragile. Refactoring it took six weeks. The lesson: don’t let AI write code you’re not prepared to maintain.

**Objection 2: "Manual review is enough—why change principles?"

Manual review catches style issues, but it’s terrible at catching invariants. A human reviewer might miss that an AI-generated endpoint lacks rate limiting or has a missing timeout. Principles like "audit for invariants" force you to codify what matters, not just what looks clean.

I audited a PR that passed review but failed in production because the generated SQL used LIMIT 1000 in a paginated endpoint. The reviewer didn’t catch it because the code looked fine. The invariant linter did.

**Objection 3: "This slows us down—we need velocity."

Velocity without resilience is a house of cards. I’ve seen teams ship features fast only to spend weeks in firefighting mode. The real question is: What’s the cost of velocity today vs. the cost of resilience tomorrow?

In one case, a client’s AI-accelerated API shipped in 2 weeks. It took 6 weeks to stabilize. If they’d built resilience in from day one, they’d have saved 4 weeks and avoided customer churn.

**Objection 4: "We don’t have time to refactor AI-generated code."

You don’t have time not to. The technical debt from AI-generated code compounds exponentially. I once inherited a system with 5k lines of AI-generated code. Refactoring it took 3 months. If it had been designed correctly from the start, it would have taken 2 weeks.

What I'd do differently if starting over

If I were building a new system in 2026 with AI tools, here’s what I’d do:

  1. Start with a resilience-first architecture: Design your system for failure from day one. Use circuit breakers, timeouts, and rate limiting as primitives, not afterthoughts.

  2. Use AI for scaffolding, not for correctness: Treat AI-generated code as temporary. Plan to refactor or replace it.

  3. Automate invariant enforcement: Build linters, CI checks, and even pre-commit hooks that catch violations early.

  4. Measure the right things: Track error rate, latency, and MTTR—not just PR throughput.

  5. Educate the team: Run a workshop on AI-generated code risks. Show examples of what can go wrong and how to catch it.

  6. Use AI for documentation and prototyping: Let it handle low-risk tasks while humans focus on high-risk ones.

Here’s a concrete example of what I’d do differently: I’d use Copilot X to generate a basic FastAPI scaffold, then immediately replace the database layer with a well-tested ORM and add timeouts, circuit breakers, and rate limiting. I’d also add a property-based testing suite to catch edge cases the AI missed.

Summary

AI tools haven’t replaced engineering principles—they’ve exposed their gaps. The conventional wisdom of "adopt AI, measure velocity" is incomplete because it ignores the second-order effects of automation: cognitive load shifts, resilience degrades, and technical debt compounds.

Here’s what I’ve updated:

  • From velocity-first to resilience-first
  • From "just review the code" to "audit for invariants"
  • From "AI writes everything" to "AI scaffolds, humans design"
  • From measuring PR throughput to measuring error rate and MTTR

The tools aren’t the problem. The mental model is.

Frequently Asked Questions

How do I catch AI-generated code that misses edge cases?

Start with property-based testing. Tools like hypothesis (Python) or fast-check (JavaScript) generate random inputs and check invariants. For example, test that a generated user endpoint never returns null for a valid user ID. Combine this with fuzz testing for I/O boundaries (e.g., timeouts, network failures). In one system, this caught 14 edge cases the AI missed.

What’s the minimum set of invariants every system should enforce?

For any API or service: 1. All endpoints have rate limiting 2. All I/O calls have timeouts 3. All errors return consistent JSON 4. All database queries use parameterized queries (no string formatting) 5. All secrets are injected via environment variables (never hardcoded)

I’ve seen systems violate these invariants in AI-generated code and cause outages within hours.

Is it worth using AI for testing?

Only if you treat it as a starting point, not a final solution. AI can generate tests quickly, but 60-70% are often redundant or wrong. Use it to bootstrap, then refactor with property-based or mutation testing. In my experience, the effort saved in writing boilerplate is offset by the time spent fixing AI-generated tests.

How do I convince my team to slow down and audit AI-generated code?

Frame it as risk reduction, not velocity reduction. Show them data: a system with 1.8% error rate due to AI-generated code costs more to stabilize than one built with invariants from day one. Use examples from production: outages, customer churn, and MTTR spikes. If they still resist, propose a 2-week experiment: audit one AI-generated PR with invariants, measure the error rate, and compare to previous PRs.
