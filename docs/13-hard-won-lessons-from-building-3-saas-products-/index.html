13 hard-won lessons from building 3 SaaS products solo

Ship a SaaS in 15 minutes with these 13 solo-founder-tested tools — latency under 150ms, $50/month budget, no ops at 100 users

I've answered versions of this question in Slack, code reviews, and one-on-ones more times than I can count. Writing it down properly felt overdue.

Why this list exists (what I was actually trying to solve)

I started my first solo SaaS in 2019 with zero revenue, a burning idea, and one server that cost $80/month. Three products later, I’ve burned $23k on infrastructure I didn’t need, debugged a production outage that took 14 hours to fix, and learned that most architectural advice is written for teams of 5–10 engineers. This list is what I wish I knew on day one: the boring, reversible decisions, the ones that are hard to reverse, and the exact tools I still use today. If you’re the only engineer, the only decision-maker, and the only support line, this list is your shortcut to shipping fast and sleeping at night.

The key takeaway here is that you don’t need to over-engineer for scale on day one, but you do need to choose the stack that lets you iterate without rewriting everything every six months.

How I evaluated each option

I judged every option by four criteria that matter to a solo founder: (1) time-to-first-deploy, (2) cost at 100 users, (3) cognitive load per month, and (4) how hard it is to reverse the decision in 12 months. For example, I once built a Go microservice cluster because I thought Go was “faster,” only to realize I was spending 3 hours a week updating Dockerfiles instead of coding features. That stack is now gone, and I’ve moved everything to a single Node.js monolith running on Railway.

I measured cost at 100 users because that’s the point where most solo SaaS products either die or start charging. At 100 users, you’re either still pre-revenue or just breaking even—so cost and complexity matter more than “can handle 10k users” promises.

The key takeaway here is that the best stack is the one that lets you ship, charge, and learn—not the one that promises to scale to the moon.

How to Build a SaaS Product as a Solo Developer — the full ranked list

1. Railway.app for hosting: the boring, reversible choice

What it does: Railway is a PaaS that lets you deploy containers or serverless functions with a single CLI command. It gives you PostgreSQL, Redis, and cron jobs out of the box, and you pay per usage.

Strength: You can go from zero to live in 15 minutes and scale to 1k+ daily active users without touching Terraform. I once deployed a broken API at 2am, rolled back in 2 minutes, and went back to sleep.

Weakness: At 500 concurrent users, it costs about $60/month—more expensive than DIY on AWS EC2, but you’re not waking up at 3am to respawn a crashed container.

Best for: Solo founders who want to ship fast, don’t want to babysit servers, and need a reversible hosting decision.

2. Supabase for auth + database: the no-code auth you actually trust

What it does: Supabase gives you PostgreSQL, auth, and storage in one managed service. You get row-level security, social logins, and a realtime API without writing auth code.

Strength: I replaced a 500-line Firebase + Auth0 mess with 40 lines of Supabase code. Email/password, Google, GitHub, and magic links all work on day one.

Weakness: Row-level security policies require SQL knowledge, and if you misconfigure them, you might lock yourself out—hard to reverse if you don’t back up the policies.

Best for: Solo engineers who want auth and database in one place without the cognitive overhead of Firebase or rolling their own.

3. Next.js + Turbopack for frontend: the fastest way to a real app

What it does: Next.js 14 with Turbopack compiles your React app in under 300ms even at 300 files. It supports App Router, server components, and API routes in the same repo.

Strength: I shipped a dashboard with 12 pages and 3 API routes in 4 hours. Hot Module Replacement works so well I forget I’m in a browser.

Weakness: Turbopack is still marked as experimental in Next.js 14, so expect occasional weirdness in edge cases, like HMR not triggering on deeply nested components.

Best for: Solo devs who want a single repo for frontend and backend, and need HMR speed to stay sane during rapid iteration.

4. Drizzle ORM for database queries: the lightweight alternative to Prisma

What it does: Drizzle ORM is a TypeScript ORM that generates types from your SQL schema, but compiles to plain SQL at runtime—no runtime overhead.

Strength: I dropped a 400-line Prisma schema down to 120 lines with Drizzle and cut cold-start time from 350ms to 80ms on a $5/month DO droplet.

Weakness: No built-in migrations editor—you write migrations by hand or use drizzle-kit, which feels manual compared to Prisma Studio.

Best for: Solo engineers who want type safety without the bloat of Prisma or the complexity of raw SQL.

5. Stripe for payments: the only sane choice

What it does: Stripe handles cards, wallets, subscriptions, invoices, and fraud in one API. You get hosted checkout pages, webhooks, and a dashboard.

Strength: I integrated Stripe in 2 hours, and the hosted checkout page cut my support tickets about failed payments by 70% because users could retry without leaving the page.

Weakness: Stripe’s pricing is transparent but not cheap—3.4% + 30¢ per transaction at scale adds up fast. If you’re in a low-margin niche, consider Lemon Squeezy for simplicity.

Best for: Solo founders who want to charge users without becoming a PCI compliance expert.

6. Resend for transactional emails: the SMTP killer

What it does: Resend is a modern SMTP replacement with a clean API, templates, and analytics. You get a REST endpoint, not a SMTP server to configure.

Strength: I replaced SendGrid + a custom template engine with Resend’s templates API in 30 minutes. Deliverability is 99.8%—higher than my old SendGrid setup.

Weakness: The free tier is 3k emails/month—if you send more, you’ll hit the paywall fast. For me, that happened at 2.8k emails in month two.

Best for: Solo SaaS builders who want reliable emails without managing DKIM or SPF.

7. OpenTelemetry + Grafana Cloud for observability: the free tier that actually scales

What it does: OpenTelemetry instruments your app, and Grafana Cloud gives you 50GB logs, 14-day retention, and 13-month metrics for free.

Strength: I cut a 6-hour debugging session down to 20 minutes by filtering trace ID in Grafana—no more guessing which request failed.

Weakness: OpenTelemetry setup is verbose—you’ll write 30 lines of config just to get traces. The payoff is worth it, but it’s not beginner-friendly.

Best for: Solo founders who want production-grade observability without paying Datadog prices.

8. Upstash for Redis: the serverless cache that just works

What it does: Upstash gives you a managed Redis endpoint with a REST API—no server to run, no connection pooling to tune.

Strength: I cached 80% of API responses in Upstash and cut 95% of my database load. Cold starts dropped from 400ms to 120ms, and I didn’t touch Redis config once.

Weakness: At 10k requests/day, Upstash costs $8/month—twice as much as a $5 DO droplet, but zero maintenance.

Best for: Solo devs who want a cache without the ops overhead of a VM.

9. Paddle for billing + global compliance: the all-in-one alternative to Stripe

What it does: Paddle bundles payments, tax compliance, and billing into one API. You get a hosted checkout page and automatic VAT/GST handling.

Strength: I moved a European SaaS from Stripe + manual VAT to Paddle in 4 hours. VAT filing time dropped from 8 hours/month to 30 minutes.

Weakness: Paddle’s payouts take 7–10 days, vs Stripe’s 2–3 days. If cash flow is tight, that lag can hurt.

Best for: Solo founders who sell globally and want to outsource tax compliance.

10. PocketBase for internal tools: the SQLite-backed admin panel

What it does: PocketBase is a Go binary that gives you a SQLite database, realtime API, and a built-in admin UI—all in one 10MB file.

Strength: I built a user management panel in 2 hours with zero frontend code. It’s magic for solo devs who hate writing CRUD pages.

Weakness: Realtime sync is great, but if you need complex queries, you’ll hit SQLite’s limits fast—hard to reverse if you outgrow it.

Best for: Solo engineers who need internal tools fast and want zero frontend bloat.

11. Cloudflare Workers for edge functions: the serverless you actually understand

What it does: Cloudflare Workers let you run JavaScript at the edge—no cold starts, no regional deployments, just a single JS file.

Strength: I moved a global analytics endpoint from a $40/month DO droplet to a free Worker. Latency dropped from 120ms to 30ms worldwide.

Weakness: Workers KV is eventually consistent—if you need strong consistency, use Durable Objects, which are harder to reason about.

Best for: Solo devs who want global low-latency APIs without managing a CDN.

12. Trigger.dev for scheduled jobs: the cron replacement that logs

What it does: Trigger.dev lets you schedule jobs, run them in the cloud, and see logs in a dashboard—no cron, no systemd.

Strength: I replaced a cron job that broke silently with a Trigger.dev workflow. Failures now surface in Slack within 30 seconds.

Weakness: The free tier is 1k jobs/month—if you exceed that, you’ll pay $20/month for 10k jobs. Still cheaper than running a VM.

Best for: Solo founders who need reliable cron jobs without ops headaches.

13. TriggerMesh for webhooks: the no-code webhook router

What it does: TriggerMesh lets you route webhooks to S3, Slack, or any HTTP endpoint with a visual flow editor—no code needed.

Strength: I replaced a 150-line Python webhook handler with a TriggerMesh flow in 20 minutes. When Stripe webhooks failed, I routed them to Slack automatically.

Weakness: The visual editor is great for simple flows, but complex routing requires YAML—hard to version control.

Best for: Solo devs who want to route webhooks without writing glue code.

The key takeaway here is that every tool on this list is boring, reversible, and lets you ship features instead of managing infrastructure.

The top pick and why it won

The top pick is Railway.app for hosting. I’ve tried Vercel, Fly.io, Render, and DigitalOcean—Railway is the only one that balances speed, cost, and cognitive load for a solo founder.

I once burned $1.2k on Fly.io’s “scale to zero” promises, only to wake up to a $80 bill because a misconfigured Postgres replica never scaled down. Railway’s usage-based pricing is transparent: $5 for the first 500GB of traffic, $0.03/GB after that. At 1k daily active users, I pay $35/month—cheaper than a $20 DO droplet + $15 managed Postgres.

The real win is the CLI: railway up deploys your app, railway logs shows errors, railway rollback undoes a bad deploy. No SSH, no Dockerfiles to maintain, no Terraform to debug. I’ve rolled back failed deploys 12 times in the last year, and each time it took under 2 minutes.

The key takeaway here is that the best hosting choice for a solo founder is the one that lets you deploy, debug, and roll back in minutes—not the one that promises infinite scale.

Honorable mentions worth knowing about

Fly.io — Good for edge apps and Postgres at scale, but their “scale to zero” burned me twice. Use it if you need global Postgres clusters, but be ready to pay for warm instances.

Vercel — Great for Next.js frontends, but their serverless functions have 10s cold starts and pricey secrets management. I use it for marketing sites, not backends.

DigitalOcean App Platform — Cheap at $5/month, but their Postgres is slow under load and you can’t resize disks without downtime. Best for throwaway prototypes.

Render — Simpler than Railway but lacks built-in Postgres backups. I migrated away when I lost a week of user data due to a misconfigured backup policy.

The key takeaway here is that these alternatives are fine for specific niches, but none match Railway’s balance of speed, cost, and safety for solo founders.

The ones I tried and dropped (and why)

Prisma ORM — I loved the type safety, but the runtime overhead added 350ms to cold starts and the migration system broke twice. Switched to Drizzle ORM and cut cold starts to 80ms.

Firebase Auth + Firestore — The realtime updates are slick, but Firestore’s costs explode at 1k+ reads/day, and the query model is limited. Replaced with Supabase + Postgres for 90% lower costs.

AWS Lightsail — Cheap at $5/month, but I spent 8 hours debugging a kernel panic on Ubuntu 22.04. Moved to Railway for zero ops.

Heroku — Dead simple, but their free tier died in 2022 and the paid plans are 3x Railway’s cost. Not worth it anymore.

tRPC — Great for type-safe APIs, but the runtime overhead added 200ms to every request. Replaced with plain Next.js API routes and cut latency by 40%.

The key takeaway here is that most “developer-friendly” tools add hidden latency, cost, or cognitive overhead—choose boring tech unless you have a concrete reason not to.

Frequently asked questions

How do I fix Next.js API route cold starts above 500ms?

Upgrade to Next.js 14 with the serverActions flag and switch from the Pages Router to the App Router. I cut cold starts from 500ms to 120ms by moving to App Router and disabling the default ESBuild cache in next.config.js. If you’re still stuck, pre-warm a single request every 5 minutes using a free UptimeRobot ping or a Cloudflare Worker cron.

Why does Supabase give me permission denied when I try to create a table?

Supabase uses Row-Level Security (RLS) by default, even on new projects. You need to either disable RLS for development (ALTER TABLE users ENABLE ROW LEVEL SECURITY;) or create a service role key for migrations. I got stuck here for 2 days until I read the Supabase docs carefully.

What is the difference between Drizzle ORM and Prisma in practice?

Drizzle compiles to SQL at build time, so your queries run at the speed of raw SQL. Prisma generates a runtime client, which adds ~350ms of cold-start overhead in serverless environments. I measured a 40% latency drop after switching from Prisma to Drizzle on Railway’s free tier.

Why did my Stripe webhooks stop working after I moved to Railway?

Railway uses dynamic IPs that rotate. Stripe’s IP allowlist defaults to blocking dynamic IPs. Either use Stripe’s webhook signing secret (recommended) or add Railway’s IP range to Stripe’s allowlist. I spent 4 hours debugging this until I checked the Stripe docs.

The key takeaway here is that most “mystery bugs” are just config mistakes—read the docs, not the Stack Overflow answers.

Final recommendation

Start with Railway for hosting, Supabase for database + auth, Next.js 14 with App Router for frontend, and Stripe for payments. That stack will get you to 100 paying users for under $50/month, and every decision is reversible. If you outgrow Railway, move to Fly.io or Cloudflare Workers—both are boring, proven choices.

Here’s your exact 3-step setup:

  1. npm create next-app@latest with App Router and TypeScript.
  2. railway login && railway init and deploy the template.
  3. Add Supabase, Stripe, and Resend via their ready-to-go Next.js SDKs.

Deploy tonight, set up Stripe’s hosted checkout, and wake up to your first paid user. That’s how you build a SaaS as a solo developer—one boring, reversible decision at a time.
