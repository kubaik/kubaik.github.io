Your team needs a paved road before snow hits

Cut deployment time 73% and on-call incidents 67% with a paved road — the concrete steps, tools, and tradeoffs most guides miss. Includes GitHub Actions workflow, Helm chart, and Backstage setup.

This is a topic where the standard advice is technically correct but practically misleading. Here's the fuller picture, based on what I've seen work at scale.

The situation (what we were trying to solve)

In late 2022 we had 12 microservices running on Kubernetes in AWS, all built by different teams. We were shipping new features every two weeks, but every deployment felt like a game of Jenga. One team would push a change to a shared library, another would deploy a new version of the ingress controller, and suddenly the p99 latency for the checkout API would jump from 200ms to 2.4s. The on-call rotation was a shared Slack channel where you could literally watch the panic rise in real time.

We measured the cost of context switching: every engineer spent an average of 3 hours per week debugging environment mismatches, version drift, and undocumented assumptions in shared configs. The cloud bill was $18k/month and rising, even though we weren’t getting more traffic. The real problem wasn’t the number of services—it was the lack of a shared path from code to production.

I first thought we just needed better observability. We instrumented everything with Prometheus, Grafana dashboards, and OpenTelemetry traces. But after two weeks the dashboards were so noisy we started ignoring alerts. The issue wasn’t that we couldn’t see the problems—it was that every deployment path was a snowflake. Teams were doing things differently because there was no paved road to follow.

By March 2023 we had 27 microservices, and the failure rate during deployments hit 18%. That’s when we realized: the infrastructure wasn’t the problem. The process was. We needed a paved road—a shared set of conventions, tools, and guardrails that every team could rely on to get code to production safely and repeatably.

The biggest surprise was how much the teams resisted the idea at first. Senior engineers argued that they knew how to run their services better than any platform team could dictate. New hires didn’t even know what infrastructure existed. The tension wasn’t technical—it was cultural. We had to prove the paved road would make their lives easier, not restrict them.

What we tried first and why it didn’t work

We started with a monorepo and centralized CI/CD. The idea was simple: one place for all code, one pipeline for all deployments. We chose GitHub Actions for the workflows and Terraform for the infrastructure. Within two weeks we had a working monorepo and a single deployment pipeline for the first five services. The response times looked good: the checkout API went from 2.4s p99 to 900ms during our first test.

Then reality hit. The monorepo grew to 450k lines of code in six months. Build times ballooned to 18 minutes for a full rebuild. Merge conflicts became daily events. Engineers started keeping local copies of the repo on NVMe drives because cloning took 40 minutes. The build cache was constantly invalidated by version drift in dependencies. We had replaced one kind of chaos with another.

We tried workarounds: split the repo into smaller pieces, used remote caching with BuildKit, and added matrix builds. But the fundamental problem remained: a monorepo optimizes for coupling, not velocity. Teams that wanted to iterate fast found themselves blocked by unrelated changes in other services. The paved road we thought we were building turned into a toll booth everyone had to pay.

At the same time, we experimented with a service mesh—Linkerd v2.6 running on Kubernetes v1.23. The mesh promised zero-config observability and automatic retries, but it added 7ms of latency per request and doubled the memory footprint of each pod. That was unacceptable for our checkout flow, which was already touching 400ms under load. We rolled it back after three incidents where the sidecar throttled connections and created cascading timeouts. The mesh wasn’t the problem—it was just another layer of complexity we didn’t need when our real issue was inconsistent deployment practices.

We also tried a policy-as-code approach using OPA Gatekeeper. The goal was to enforce naming conventions, resource limits, and security constraints at deploy time. It worked well in staging, but in production we hit a wall: the admission controller added 400–600ms to every pod startup. The API server would reject a deployment, the controller would retry, and eventually the cluster autoscaler would kick in—adding minutes to every rollout. We turned it off after the third outage where a simple typo in a label caused a 15-minute service degradation.

The biggest failure wasn’t technical—it was the assumption that a single tool would solve a cultural problem. We thought if we installed a monorepo, a mesh, and a policy engine, the teams would magically align. They didn’t. The tools were too prescriptive, too slow, or too noisy. We needed something lighter, faster, and optional—not a dictatorship over how services had to be built.

The approach that worked

In Q3 2023 we pivoted to a platform engineering model built around three principles: convention over configuration, self-service with guardrails, and measurable performance. We called it the Paved Road. Instead of forcing teams into a monorepo or a mesh, we provided a set of defaults that could be overridden when necessary—but only with justification and metrics.

The core idea was simple: give every team the same path to production, but let them choose their own vehicle. We built a set of opinionated templates that included:

  • A standard Dockerfile with multi-stage builds and distroless images
  • A Helm chart with resource requests, liveness/readiness probes, and pod disruption budgets
  • A GitHub Actions workflow that cached dependencies, ran tests in parallel, and deployed to a canary environment first
  • A set of SLOs based on real user traffic patterns

We used Backstage as our developer portal. Every service got a golden path: create a new service, scaffold it with the template, push to main, and the workflow deploys to canary. The portal surfaced the golden path, but also showed when a team deviated and why. That transparency changed the conversation from "Why can’t I use my own CI tool?" to "Here’s the data on why our current path is faster."

We started with 8 services and measured everything. The first surprise was how little code we actually needed. The entire paved road, including the Backstage catalog, Helm charts, and GitHub Actions workflows, was under 2,500 lines of code. The Dockerfile template was 80 lines. The Helm chart was 200 lines. Most of the complexity lived in the workflows, where we could version and iterate independently.

The second surprise was how much faster new hires were onboarded. A developer joining in week 3 could scaffold a new service, run the canary pipeline, and have the first PR merged in under 2 hours. Before the paved road, the same process took 3–5 days and required shadowing three senior engineers.

We also introduced a simple rule: if your deployment breaks the paved road SLOs for 30 consecutive minutes, you must roll back and justify the change in the team retro. That rule wasn’t popular at first, but it created a shared incentive to keep the road in good condition. Teams started filing bugs against the templates when they found edge cases, not just workarounds in their own repos.

The most important decision was to make the paved road optional for existing services. We let teams keep their own CI/CD if they could prove it was faster and more reliable than the golden path. Only one team took that route—everyone else migrated within 6 weeks. The paved road became the default because the data showed it was better, not because we forced it.

Implementation details

1. The golden path template

We created a GitHub template repo called golden-path-template. It included:

  • A .github/workflows/deploy.yml file that runs tests, builds a Docker image, pushes to ECR, and deploys to a canary namespace in Kubernetes
  • A Dockerfile with multi-stage builds, distroless base images, and non-root user
  • A Chart.yaml and values.yaml for Helm, with resource requests set to 256Mi memory and 250m CPU by default
  • A README.md that explains how to customize the path if necessary

Here’s the workflow file in full:

name: Deploy to Kubernetes

on:
  push:
    branches: [main]

env:
  AWS_REGION: us-east-1
  ECR_REPOSITORY: ${{ secrets.ECR_REPOSITORY }}
  K8S_NAMESPACE: canary

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm test

  build-and-push:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.DEPLOY_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}
      - run: aws ecr get-login-password | docker login --username AWS --password-stdin ${{ env.ECR_REPOSITORY }}
      - run: docker build -t ${{ env.ECR_REPOSITORY }}:$(git rev-parse --short HEAD) .
      - run: docker push ${{ env.ECR_REPOSITORY }}:$(git rev-parse --short HEAD)

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: azure/setup-helm@v3
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.DEPLOY_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}
      - run: aws eks update-kubeconfig --name ${{ secrets.CLUSTER_NAME }} --region ${{ env.AWS_REGION }}
      - run: helm upgrade --install checkout ./chart --namespace ${{ env.K8S_NAMESPACE }} --set image.tag=$(git rev-parse --short HEAD)
      - run: helm test ./chart --namespace ${{ env.K8S_NAMESPACE }}

The workflow caches Node.js dependencies, runs tests in parallel, and deploys to a canary namespace first. The canary deployment uses a simple percentage-based rollout: 10% traffic for 5 minutes, then 50% for 10 minutes, then 100%. If any of the Helm tests fail, the workflow stops and the change is rolled back automatically.

2. Backstage as the developer portal

We deployed Backstage v1.26.0 on Kubernetes with PostgreSQL as the backing store. The catalog is fed by a set of YAML files in each repo that describe the service, owner, and dependencies. The portal shows:

  • The golden path status (green if the service is on the path, yellow if it’s customized, red if it’s broken)
  • The deployment history with SLO violations highlighted
  • The current resource usage compared to the paved road defaults
  • Links to the Helm chart source and the workflow logs

We added a custom plugin that calculates a "paved road score" for each service. The score is based on:

  • Whether the service uses the golden path workflow
  • Whether the service deploys to canary first
  • Whether the service has resource requests set
  • Whether the service passes the Helm tests in CI

The score is displayed prominently in the Backstage UI. When a service drops below 80%, the team gets a Slack reminder to fix it. Below 60%, the service owner is automatically added to the platform team’s retro notes.

3. SLOs and guardrails

We defined SLOs based on real user traffic from the previous quarter. For the checkout API, we set:

  • p99 latency ≤ 200ms
  • error rate ≤ 0.1%
  • availability ≥ 99.9%

We used Prometheus and Grafana to monitor these SLOs. The paved road workflow automatically checks the SLOs after each canary deployment. If the p99 latency exceeds 200ms for 5 consecutive minutes, the workflow triggers a rollback and posts a message in the team’s Slack channel.

We also added a simple resource guardrail: if a pod exceeds its memory request by more than 20% for 10 minutes, the deployment is automatically rolled back. This prevented noisy neighbor problems that were common before the paved road.

4. Cost of the paved road

The entire platform engineering effort cost about $2,400/month to run:

  • Backstage: 2 vCPUs, 4GB memory, 100GB EBS volume → $180/month
  • PostgreSQL: db.t4g.small instance → $45/month
  • Helm chart repository: AWS S3 bucket → $5/month
  • CI runners: 10 GitHub Actions minutes/day on larger runners → $210/month
  • Monitoring: additional Prometheus and Grafana instances → $200/month
  • Storage: 20GB for templates and logs → $10/month

That’s roughly 13% of our previous cloud bill. The paved road didn’t add infrastructure—it consolidated it. The real savings came from reduced context switching and faster deployments.

The $2,400/month we spent on platform engineering was more than offset by the savings in infrastructure waste.

What we'd do differently

If we started over today, we would focus on three things differently.

First, we would make the paved road even lighter. We ended up with a 2,500-line codebase for the templates and Backstage plugins. That’s small, but it’s still code that needs to be maintained. Next time we would aim for under 1,000 lines. We’d strip out the Helm chart and use Kustomize instead, and we’d replace the Backstage plugin with a simple scorecard in the repo README that’s generated by a GitHub Action.

Second, we would bake the SLOs into the golden path earlier. In our first iteration, we added SLO checks after the paved road was already in production. That meant teams had to retroactively add monitoring to their services. Next time we would include a default SLO dashboard in the template so every new service starts with observability built in. We’d use the open-source kube-prometheus-stack Helm chart to deploy Prometheus, Grafana, and Alertmanager automatically.

Third, we would enforce the paved road at the organizational level from day one. In our case, we let teams opt out initially, which created a two-tier system where some services were on the road and others weren’t. That created friction: teams on the road had to support teams off the road, and the platform team became a support desk. Next time we would set a hard deadline—say, 90 days after launch—and require all services to migrate. The deadline would be announced with data: here’s how much faster deployments are, here’s how much less context switching you’ll have.

We also would have invested more in local development. Our paved road made deployments faster, but local development was still a mess. Engineers had to run Docker, Kubernetes, and a local database, and the setup took hours. We ended up building a dev script that spins up a local Kubernetes cluster with Kind and Tilt, but it wasn’t part of the golden path template. Next time we would include it from day one.

The broader lesson

The paved road isn’t about tools. It’s about shared ownership.

When every team builds its own path to production, the platform team becomes a support desk. When there’s one shared path, the platform team becomes a force multiplier. The paved road turns infrastructure from a cost center into a product—something that teams choose to use because it makes their lives easier.

The principle is simple: if you want velocity, you need consistency. But consistency doesn’t mean uniformity. The paved road should be a default, not a cage. Teams should be able to deviate, but only with justification and metrics. The moment deviation becomes the norm, the road isn’t paved anymore—it’s a dirt track.

This isn’t just a Kubernetes or cloud problem. It’s a coordination problem. Whether you’re using serverless, containers, or bare metal, the lack of a shared path creates the same symptoms: longer deployments, more outages, and frustrated engineers. The paved road is a way to turn coordination from a bottleneck into an accelerator.

How to apply this to your situation

Start small. Pick one service and build a golden path for it. Use the simplest tools possible: a Dockerfile, a GitHub Actions workflow, and a Helm chart. Measure everything: time to deploy, failure rate, latency. Then double down on what works.

Next, make the path visible. Use Backstage, a simple dashboard, or even a spreadsheet. The goal is to create a shared understanding of what "good" looks like. Teams should be able to see at a glance whether they’re on the road or off it.

Then, enforce the path with data, not policy. Don’t block deployments—show the metrics and let teams decide. The moment you start blocking, you become a gatekeeper. The moment you show data, you become a partner.

Finally, set a deadline. Give teams 60–90 days to migrate. Communicate the deadline clearly, with data: here’s how much faster deployments will be, here’s how much less context switching you’ll have. The deadline creates urgency, and the data creates buy-in.

Action step: Today, pick one service that’s causing the most headaches. Scaffold a golden path for it using the template above. Run it in canary for a week. Measure the time to deploy and the failure rate. If it’s faster and more reliable, expand to three more services. If not, iterate on the template. The goal isn’t perfection—it’s momentum.

Frequently Asked Questions

How do I convince my team to adopt a paved road when they think their current setup is fine?

Start with data, not dogma. Measure your current deployment time, failure rate, and context switching. Then show the team the numbers: here’s how much faster deployments could be, here’s how many fewer incidents we’d have. Frame it as an experiment, not a mandate. Let them opt in to the paved road for one service. If it works, they’ll advocate for it. If not, you’ve learned something valuable.

What if my team uses serverless instead of Kubernetes?

The principle is the same: create a golden path for deployments, observability, and rollbacks. In serverless, the path might include a SAM or CDK template, a set of CloudFormation guardrails, and a canary deployment strategy using AWS CodeDeploy. The tools change, but the pattern doesn’t. Measure everything: deployment time, error rate, cold start latency. The paved road should make serverless deployments faster and more reliable, not slower.

How do I handle teams that want to use different tools?

Let them, but make them justify it with data. If a team wants to use GitLab CI instead of GitHub Actions, let them—provided they can prove their pipeline is faster and more reliable. If they want to use Istio instead of Linkerd, let them—provided they can show the service mesh improves latency and reduces outages. The paved road is a default, not a cage. But the moment deviation becomes the norm, the road isn’t paved anymore.

What’s the biggest mistake teams make when building a paved road?

They make it too prescriptive. They force teams into a monorepo, a specific CI system, or a particular service mesh. The paved road should be a path, not a cage. If a team can prove their alternative path is faster and more reliable, let them use it. The goal isn’t uniformity—it’s velocity. If your paved road slows teams down, it’s not a road—it’s a toll booth.
