The primary knob is prompt engineering, which is cheap and fast to iterate.\n\nBut there are hidden costs. The LangChain 0.1.x release we used leaked exceptions on every third retry, causing 2–3% of requests to return 500 errors. We patched it to LangChain 0.2.13, but the churn cost us two sprints. You write a new prompt, test locally with a notebook, push to staging, and hope the managed service doesn’t change under you. The LangChain 0.2.x release cycle introduced breaking changes twice in six months, costing us two days of debugging per change. The developer loop feels productive until you hit the wall of managed service quirks.\n\nVector-search infra is slower to iterate but more predictable. You benchmark different index types offline, deploy a new shard, run a canary, and promote once SLOs are green. The Rust-based stack (qdrant, milvus) compiles to native code, so performance deltas are repeatable. The downside is the cognitive load: you need to understand SIMD, NVMe geometry, and network partitions.\n\nTooling parity is uneven. For prompt-tuning, the ecosystem is mature: LangChain 0.2.13, LlamaIndex 0.10, TruLens 0.15, and LiteLLM for proxying.
