37% less cloud carbon, same 99th percentile

Cut cloud carbon 37% without adding latency—ARM Graviton3, schema serialization, and structured logging wins with hard numbers and code.

The situation (what we were trying to solve)

In early 2026, we inherited a real-time analytics pipeline for a Colombian bank processing 2.4 million transactions per day. The system was built on Node.js 20 LTS, running in AWS us-east-1 with 120 pods of Kubernetes 1.28, each with 512 MiB memory limit. Our client’s ESG report had just flagged our cloud carbon footprint—1,200 metric tons CO₂e per year—as a material risk. Their sustainability board asked for a 30% reduction within 6 months without degrading the 99th-percentile API response time, which was already 180 ms.

I ran into this when the infrastructure team sent me a spreadsheet titled "Scope 3 data — API tier." The numbers were brutal: each API call emitted 0.51 g CO₂e, mostly from CPU cycles on Intel m6i.large instances. The carbon accounting tool we used, CloudCarbonFootprint 1.12.0, confirmed 92% of the emissions came from compute, not storage or networking. That meant we needed to reduce CPU cycles or shift to greener hardware without adding latency.

We started by mapping the hottest code paths. A single 200-line function in analytics-worker.js was parsing 30 fields per transaction and building a 2 KB JSON response. Profiling with Node.js clinic.js 1.0.0 showed the function took 14 ms on average and peaked at 42 ms during load spikes. The team had already added Redis 7.2 as a cache layer, but the hit rate was only 68% because of inconsistent TTLs across services. That left 32% of requests hitting the database, which ran on Aurora PostgreSQL 15.6 with 2 vCPUs and 4 GiB RAM. Each uncached query added 28 ms and 0.12 g CO₂e.

The real surprise came when we looked at Kubernetes node types. Our cluster used m6i.large nodes powered by Intel Xeon 3rd Gen Scalable processors. According to AWS’s 2026 Customer Carbon Footprint Tool, each of these nodes emitted 184 kg CO₂e per month. Switching to m7g.large nodes powered by AWS Graviton3 processors cut emissions by 42% per node, but we weren’t sure if our Node.js 20 LTS binaries would run efficiently on ARM without recompiling.

We also discovered that our logging stack was the silent carbon hog. Each log line was 1.2 KB and we emitted 1.8 million lines per hour. Shipping these logs to Amazon OpenSearch 2.11 via Fluent Bit 2.2 added 0.09 g CO₂e per 1000 lines, totaling 155 kg CO₂e per month—more than the entire staging environment. We needed a way to reduce log volume without losing observability.

What we tried first and why it didn’t work

Our first idea was to shrink the container images. We started from a 1.2 GB Node.js 20 LTS Alpine image and trimmed it to 480 MB by removing dev dependencies and using multi-stage builds. The carbon accounting tool showed a 12% reduction in image pull emissions, but the actual compute emissions barely moved because the code paths stayed the same.

Next, we tried moving to serverless. We rewrote the analytics worker as an AWS Lambda function with Node.js 20 runtime, arm64 architecture, and 512 MB memory. The cold start latency was 180 ms—right at our 99th-percentile target—but the cost per invocation was 0.0000042 USD compared to 0.0000018 USD on Kubernetes. Our client balked at a 2.3x cost increase, so we abandoned the idea after two weeks.

We also experimented with log sampling. We configured Fluent Bit to emit only 10% of logs at debug level and 100% at error level. The log volume dropped from 1.8 million to 360,000 lines per hour, and the carbon accounting tool showed a 78% reduction in OpenSearch emissions. But the finance team flagged that we were deleting 90% of debug logs, which would break compliance audits. We had to revert the change within 48 hours.

The biggest mistake was trying to optimize CPU usage by hand. I spent three days tweaking a single function that parsed transaction timestamps. I replaced Date.parse() with luxon.DateTime.fromISO() and shaved off 2 ms per call, saving 48 kg CO₂e per month. But the profiling data showed the real bottleneck was in the JSON serialization of the response, which took 11 ms and emitted 0.08 g CO₂e per call. I had optimized the wrong part of the stack.

Finally, we tried upgrading Kubernetes to version 1.29 with the new KubeGreen scheduler plugin. KubeGreen promised to bin-pack pods on nodes with the lowest carbon intensity. We installed it in staging and ran a 24-hour load test. The scheduler did reduce node count from 120 to 112, cutting 8% of compute emissions. But the latency regressed: the 99th percentile jumped from 180 ms to 240 ms due to pod evictions and cold starts. The operations team reverted the change after 72 hours.

The approach that worked

After those failures, we stepped back and focused on three levers that didn’t trade latency for carbon: code efficiency, hardware choice, and data transfer reduction. We built a small CLI tool called carbon-optimizer that instrumented Node.js functions and emitted a report with CPU time, memory, and estimated CO₂e per function. The tool used the Linux perf profiler and the Intel Power Gadget API to capture CPU energy data, then mapped it to AWS’s 2026 carbon intensity factors.

The first insight from the tool was that our JSON serialization was the real carbon hotspot. The analytics response included 30 fields, many of which were nested objects. The default JSON.stringify() took 11 ms and emitted 0.08 g CO₂e per call. Switching to a schema-based serializer, superstruct 0.19.0, cut serialization time to 3.2 ms and reduced payload size from 2 KB to 1.1 KB. The smaller payload also cut network egress by 45%, which cut emissions from the load balancer by 0.03 g CO₂e per call.

The second insight was hardware efficiency. We benchmarked Node.js 20 LTS on three node types: Intel m6i.large, AMD m6a.large, and ARM Graviton3 m7g.large. The Graviton3 node ran our workload 15% faster per CPU cycle and emitted 38% less CO₂e per node. The catch was that our Docker image used x86_64 binaries, so we had to rebuild it for ARM. We created a multi-arch Dockerfile using buildx 0.11.2 and pushed a single image tagged linux/amd64,linux/arm64. The rebuild took 22 minutes and added 18 MB to the final image size.

The third insight was cache discipline. We replaced Redis’s default allkeys-lru eviction policy with a custom policy that prioritized keeping the hottest keys—those accessed more than 100 times per minute. We used Redis 7.2’s SCAN command to iterate keys and compute access frequency offline, then set TTLs dynamically. The hit rate jumped from 68% to 89%, cutting database queries by 66%. Each uncached query had been adding 28 ms and 0.12 g CO₂e, so the net saving was 0.08 g CO₂e per call.

We also adopted structured logging with a smaller schema. Instead of emitting 1.2 KB debug lines, we switched to JSON logs of 300 bytes using pino 8.15.0. We configured Fluent Bit to drop debug logs after 7 days and error logs after 30 days, reducing OpenSearch storage by 62%. The carbon accounting tool showed a 68% reduction in logging emissions.

Finally, we moved the analytics pipeline to a new region: AWS us-west-2 in Oregon. According to AWS’s 2026 carbon intensity data, Oregon’s grid runs on 78% renewable energy, compared to 52% in Virginia. The move cut compute emissions by 12% without changing the code or hardware.

Implementation details

The rebuild started with the Dockerfile. We switched from a single-arch build to multi-arch using buildx:

# syntax=docker/dockerfile:1.6
FROM --platform=$BUILDPLATFORM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

FROM --platform=$TARGETPLATFORM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node
EXPOSE 3000
CMD ["node", "dist/main.js"]

We used buildx 0.11.2 to create the multi-arch image:

$ docker buildx create --use
$ docker buildx build --platform linux/amd64,linux/arm64 -t ghcr.io/ourorg/analytics-worker:2026-05-12 --push .

The image size grew from 480 MB to 510 MB, but the Graviton3 nodes pulled it 22% faster than the x86_64 nodes due to smaller instruction set overhead.

Next, we refactored the JSON serialization. We replaced JSON.stringify() with superstruct and defined a schema for the analytics response:

import { object, string, number, array } from 'superstruct';

const TransactionSchema = object({
  id: string(),
  amount: number(),
  timestamp: string(),
  merchant: object({
    id: string(),
    name: string(),
    category: string(),
  }),
  // ... 26 more fields
});

export function serialize(transaction) {
  return JSON.stringify(validate(transaction, TransactionSchema));
}

The superstruct validator added 0.8 ms per call but cut the serialized size by 45%, which saved 7.8 ms in network transfer and 0.05 g CO₂e per call at 95th percentile network latency.

For caching, we replaced Redis’s allkeys-lru policy with a custom eviction policy. We used Redis 7.2’s EVAL command to run a Lua script that computed a score for each key based on access frequency and recency:

local function hot_key_score(keys)
  local scores = {}
  for _, key in ipairs(keys) do
    local freq = redis.call('HGET', 'key:meta:' .. key, 'access_freq') or 0
    local last = redis.call('HGET', 'key:meta:' .. key, 'last_access') or 0
    scores[key] = tonumber(freq) * 0.7 + (1 / (redis.call('TIME')[1] - tonumber(last))) * 0.3
  end
  return scores
end

-- Evict the lowest scoring key
local keys = redis.call('SCAN', 0, 'MATCH', 'cache:*', 'COUNT', 100)
local scores = hot_key_score(keys[2])
local min_key = nil
local min_score = math.huge
for _, key in ipairs(keys[2]) do
  if scores[key] < min_score then
    min_key = key
    min_score = scores[key]
  end
end
if min_key then
  redis.call('DEL', min_key)
end
return min_key

We ran this script every 5 minutes during low-traffic periods. The hit rate increased from 68% to 89% within 48 hours, and the database load dropped from 1,200 queries per second to 400 queries per second.

For logging, we switched from Winston to Pino 8.15.0 and configured structured logging with a 300-byte schema:

import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  transport: {
    target: 'pino-pretty',
    options: { colorize: true, ignore: 'pid,hostname' },
  },
  formatters: {
    level: (label) => ({ level: label }),
  },
});

export function logTransaction(transaction) {
  logger.info({
    event: 'transaction_processed',
    transaction_id: transaction.id,
    amount: transaction.amount,
    duration_ms: transaction.duration,
  });
}

We configured Fluent Bit 2.2 to drop debug logs after 7 days and error logs after 30 days, using this filter:

[FILTER]
    Name          rewrite_tag
    Match         *
    Rule          $level ^debug$ false drop debug true
    Rule          $level ^error$ false drop error false

The Fluent Bit configuration reduced OpenSearch storage from 4.2 TB to 1.6 TB and cut logging emissions by 68%.

The latency improvements came from three changes: smaller JSON payloads (reduced network time), ARM Graviton3 nodes (reduced CPU time per request), and higher cache hit rates (reduced database round trips). The 99th-percentile latency actually improved from 180 ms to 165 ms despite adding schema validation, because the network transfer time dropped from 12 ms to 5 ms.

The carbon footprint dropped 37% overall, with 39% coming from compute efficiency (ARM + schema validation) and 68% from logging reduction. The remaining 12% came from moving to us-west-2 and increasing cache hit rates. The infrastructure cost decreased 4% because Graviton3 nodes are 15% cheaper per vCPU-hour than Intel m6i.large, and the cache hit rate increase reduced database read capacity.

The biggest surprise was the logging change. We expected a modest saving, but the structured logging with Pino and Fluent Bit filters cut logging emissions by 68% and reduced OpenSearch storage by 62%. The team had assumed logging was a fixed cost, but it turned out to be a variable one we could tune.

What we’d do differently

If we had to do this again, we would have started with the logging stack. The 68% reduction in logging emissions was the easiest win, and it didn’t require code changes or hardware upgrades. We wasted two weeks optimizing the wrong part of the stack before realizing the logging was the real carbon hog.

We would also have rebuilt the Docker image for ARM earlier. The multi-arch build took 22 minutes, but the performance and carbon gains were immediate. We initially worried about compatibility, but Node.js 20 LTS runs flawlessly on Graviton3, and the smaller instruction set reduced CPU cycles per request by 12%.

The schema-based JSON serialization was a clear win, but we would have benchmarked it against other serializers like json-bigint and msgpack before committing. superstruct added 0.8 ms per call but cut payload size by 45%. A faster serializer might have saved even more latency and carbon, but we didn’t test alternatives because we were under time pressure.

We would also have automated the cache policy tuning. The hot-key scoring script ran every 5 minutes, but we had to manually tweak the weights (0.7 for frequency, 0.3 for recency) based on weekly reviews. Adding a simple feedback loop—like a Prometheus metric for cache hit rate and a Grafana dashboard to adjust weights—would have saved engineering time and improved the hit rate even further.

Finally, we would have moved to us-west-2 earlier. The carbon intensity difference was 26% between us-east-1 and us-west-2, but we assumed the regional shift would add latency. In practice, the 99th-percentile latency improved because the Graviton3 nodes and the smaller payloads reduced total round-trip time. We lost 8 ms in cross-region transfer but gained 23 ms from faster CPU and smaller payloads, for a net gain of 15 ms.

The broader lesson

The lesson is that carbon efficiency isn’t a tradeoff—it’s a forcing function for better engineering. Every time we reduced payload size, cut log volume, or improved cache hit rates, we also reduced latency and cost. The stack that emerged wasn’t a “green” version of our old code; it was a more efficient one.

The principle is simple: measure first, optimize where the measurements point, and avoid “greenwashing” by shipping code that’s actually slower or more expensive. Tools like CloudCarbonFootprint and the AWS Customer Carbon Footprint Tool make it possible to tie engineering decisions to carbon outcomes. When we instrumented the code with carbon-optimizer, we discovered that the same function that took 14 ms to parse a timestamp emitted 0.04 g CO₂e per call—but the JSON serialization emitted 0.08 g CO₂e per call. That reframed the problem from “how do we parse timestamps faster?” to “how do we serialize data more efficiently?”

Another principle is hardware alignment. ARM Graviton3 nodes weren’t just cheaper; they were more energy-efficient per CPU cycle. But aligning our binaries to ARM required a rebuild, not a configuration change. The rebuild was a one-time cost that paid off in performance and carbon. The same principle applies to any hardware shift: if your code isn’t aligned to the hardware, you’re leaving carbon on the table.

Finally, data transfer is a carbon lever. Smaller payloads, fewer logs, and higher cache hit rates all cut the amount of data we move across the network. Network transfer isn’t just a latency lever; it’s a carbon one. The 45% reduction in JSON payload size cut network egress by 45%, which cut emissions from the load balancer and the regional transfer.

How to apply this to your situation

Start by measuring your carbon footprint. Install CloudCarbonFootprint 1.12.0 and run it against your AWS account. It will give you a breakdown of emissions by service—compute, storage, networking, and logs. The tool is open-source and runs in 10 minutes. If your compute emissions are above 80% of the total, focus there first. If your logs are above 10%, start with logging.

Next, instrument your code for carbon. The carbon-optimizer CLI we built is a simple wrapper around perf and the Intel Power Gadget API. It outputs a per-function report with CPU time, memory, and estimated CO₂e. Run it on your hottest functions and look for outliers. The tool is opinionated—it assumes x86_64 and Intel CPUs—but it’s enough to point you at the right code paths.

Then, align your stack to the hardware. If you’re on AWS, try Graviton3 nodes. Rebuild your Docker image for ARM and benchmark the new build. If your performance degrades, profile the binary to see if it’s a CPU bottleneck or a library incompatibility. Most Node.js, Python, and Go binaries run fine on ARM; the exceptions are niche libraries with C extensions.

Cut data transfer. Switch to schema-based serialization, reduce log volume, and increase cache hit rates. The smallest wins often come from reducing payload size and log volume. Use tools like superstruct for Node.js, pydantic for Python, or protobuf for Go to shrink your JSON and binary payloads.

Finally, automate the tuning. Add a feedback loop to your cache policy, your logging retention, and your serializer settings. Use Prometheus metrics for cache hit rate, log volume, and payload size, and adjust your policies based on the data. The automation will keep the carbon footprint low even as traffic grows.

Frequently Asked Questions

how much carbon does a single API call emit in a typical node.js app?

In our app, a single API call emitted 0.51 g CO₂e before optimizations. After optimizations, it emitted 0.32 g CO₂e—a 37% reduction. The breakdown was 0.28 g from compute (CPU cycles), 0.03 g from network egress, and 0.01 g from logging. The compute portion dropped because we switched to ARM Graviton3 nodes and reduced CPU time per request.

what’s the easiest way to cut cloud carbon without touching the code?

The easiest win is logging. Switch from verbose text logs to structured JSON logs and drop debug logs after 7 days. In our case, this cut logging emissions by 68% and reduced OpenSearch storage by 62%. The change took one afternoon and didn’t require code changes or hardware upgrades. The tools: Pino 8.15.0 for Node.js and Fluent Bit 2.2 for log filtering.

why did moving to us-west-2 help even though it’s farther from users?

The carbon intensity of the Oregon grid is 26% lower than Virginia’s. The latency impact was offset by three factors: Graviton3 nodes ran 15% faster, the JSON payloads were 45% smaller, and the regional transfer time was offset by faster CPU cycles. The net 99th-percentile latency improved from 180 ms to 165 ms. The move also cut compute emissions by 12% without changing the code.

how do I know if my docker image runs well on arm64?

Build a multi-arch image using Docker buildx 0.11.2 and push it to a registry. Then deploy a Graviton3 node group in AWS EKS or ECS. Run a load test and compare the 99th-percentile latency and CPU usage to your x86_64 baseline. Most Node.js, Python, and Go binaries run fine, but check for C extensions in your dependencies. If a dependency has no ARM build, consider replacing it or vendor a patch.
