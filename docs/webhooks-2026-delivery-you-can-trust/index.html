Webhooks 2026: delivery you can trust

12 min read  ·  2451 words  ·  Updated 2026-06-12

Most building webhook guides assume a clean environment and a patient timeline. Production gives you neither. Here's what I learned building this under real constraints.

The situation (what we were trying to solve)

In late 2026 we launched a new feature that let customers push events from their SaaS apps into ours. Within two weeks we averaged 2 300 webhook calls per second at peak, with bursts up to 7 800. The existing endpoint lived inside a Node 18 LTS Express service and accepted events without any delivery guarantees. After the first major outage—where we lost 14 % of the events during a 30-second AWS us-east-1 outage—I realised we were storing receipts for customers but not the payloads themselves. That meant the only proof an event ever existed was an internal event_id we generated after the downstream service returned 200 OK.

I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then.

By the numbers: - 14 % events vanished during a 30-second outage - 7 800 calls/sec peak burst - 2 300 calls/sec sustained average

We needed a system that could: 1. Survive 30-second regional outages without losing events. 2. Avoid duplicates when retries kick in. 3. Fail gracefully so customers saw 5xx only when we were truly down.

Everything I read in 2026 tutorials told me to keep it simple: HTTP 200, retry with exponential back-off, store the payload in DynamoDB, done. But reality taught me that simplicity hides two traps: idempotency keys that collide and retry storms that turn one outage into a cascade of throttled downstream services.

What we tried first and why it didn’t work

Our first cut was a Node 20 LTS Lambda function behind an Application Load Balancer. Each request got an idempotency key built from customer_id + event_type + client_generated_event_id. We stored the key in DynamoDB with a TTL of 7 days and returned 200 only after the write succeeded. Retries used exponential back-off starting at 100 ms and capped at 5 s.

The good news: we didn’t lose events during the next us-east-1 hiccup.

The bad news: - Retry storms saturated our downstream API, causing 429 responses and back-pressure that took 8 minutes to clear. 20 % of our Lambda invocations timed out at 15 s and left the connection in CLOSE_WAIT, eventually exhausting the ALB’s ephemeral port pool after 30k concurrent connections. - Idempotency keys collided when two Lambda functions raced to insert the same key; one succeeded, the other threw ConditionalCheckFailedException, and we treated it as a duplicate and dropped the event.

I had read ‘use idempotency keys’ in every blog post, but none mentioned the race between concurrent writers. By February 2026 we had 3 200 duplicate events slipping through because of that single race condition.

Here is the code we shipped first:

// v1 – naive idempotency
exports.handler = async (event) => {
  const { body } = event;
  const { idempotencyKey } = JSON.parse(body);

  // Insert or fail
  await dynamodb.put({
    TableName: 'WebhookEvents',
    Item: {
      idempotencyKey,
      payload: body,
      status: 'pending'
    },
    ConditionExpression: 'attribute_not_exists(idempotencyKey)'
  }).promise();

  // Send downstream
  await axios.post(process.env.DOWNSTREAM_URL, body);

  return { statusCode: 200 };
};

The retry policy lived in our customer SDK, so every SDK retry created another Lambda invocation. We learned the hard way that AWS Lambda has a default concurrency limit of 1 000 per region, and we blew past it during Black-Friday traffic.

The approach that worked

We threw away the Lambda-per-event model and rebuilt around three principles:

  1. Store before you accept – put the payload in durable storage first, then acknowledge.
  2. Idempotency at the write – use a transactional outbox so only one writer can claim an idempotency key.
  3. Back-pressure via visibility timeout – let the queue itself rate-limit retries instead of hammering downstream.

The new stack: - Ingest: Node 20 LTS HTTP server on ECS Fargate (arm64) with 4 vCPU and 8 GB memory. - Queue: Amazon SQS FIFO with 300 ms visibility timeout and maxReceiveCount = 3. - Idempotency store: DynamoDB table with a transactional write that checks both the idempotency key and a sequence number. - Worker: Python 3.12 service on ECS Fargate consuming from the queue, calling downstream with a circuit breaker. - Dead-letter: SQS back to a Lambda that emails the customer with the event_id for manual replay.

Key changes: - Idempotency key = SHA256(customer_id + event_type + client_event_id). Collisions drop to one in 2^256, so we can rely on ConditionalCheckFailedException as a genuine duplicate. - Sequence number is a timestamp in microseconds; if two workers race, only the one with the higher sequence wins, the other fails the condition and becomes a duplicate. - Visibility timeout is 300 ms, not the default 30 s; this prevents a single slow downstream from holding the message hostage for minutes.

Here is the idempotent writer in Python 3.12:

# v2 – transactional outbox with sequence
import boto3, hashlib, time

ddb = boto3.client('dynamodb', region_name='us-east-1')
sqs = boto3.client('sqs', region_name='us-east-1')

IDEMPOTENCY_TABLE = 'WebhookIdempotency2026'
QUEUE_URL = 'https://sqs.us-east-1.amazonaws.com/123456789012/WebhookQueue2026.fifo'

def store_and_enqueue(payload: dict, customer_id: str, event_type: str) -> str:
    client_event_id = payload.get('eventId')
    key = hashlib.sha256(f"{customer_id}:{event_type}:{client_event_id}".encode()).hexdigest()
    seq = int(time.time() * 1_000_000)

    # Transactional write guarantees only one succeeds
    ddb.transact_write_items(
        TransactItems=[
            {
                'Put': {
                    'TableName': IDEMPOTENCY_TABLE,
                    'Item': {
                        'idempotencyKey': {'S': key},
                        'sequence': {'N': str(seq)},
                        'status': {'S': 'pending'},
                        'payload': {'S': json.dumps(payload)}
                    },
                    'ConditionExpression': 'attribute_not_exists(idempotencyKey)'
                }
            }
        ]
    )

    # Enqueue with deduplication ID = key
    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(payload),
        MessageDeduplicationId=key,
        MessageGroupId=customer_id
    )

    return key

The worker consumes the message, calls the downstream API with a circuit breaker (pybreaker 3.0.0), and on success updates the status to ‘delivered’. If the breaker is open or the downstream returns 5xx, the message becomes visible again after the 300 ms timeout and the next worker retries.

Implementation details

We migrated in three stages without downtime:

Stage 1 – Dual-write (30 minutes) - Old Lambda kept running. - New ECS service read from a SQS queue fed by the same ALB. - We set the old Lambda’s concurrency to 50 to avoid port exhaustion while we tested.

Stage 2 – Flip traffic (15 minutes) - Changed ALB target group to the ECS service. - Old Lambda still received events but returned 503 immediately, draining old connections.

Stage 3 – Cleanup (7 days) - Old Lambda retired. - DynamoDB stream on the idempotency table backfilled any events that had slipped through the cracks.

We used these versions: - Node 20 LTS (ingest server) - Python 3.12 (worker) - AWS SDK for JavaScript v3.450.0 / boto3 1.34.0 - SQS FIFO – 1 200 messages/sec limit per queue - DynamoDB on-demand capacity with 5 ms single-digit millisecond writes - ECS Fargate (arm64) with 4 vCPU / 8 GB per task

The ECS cost includes 4 tasks running 24×7 plus 20 % burst capacity. Even with the higher fixed cost, we saved 51 % because Lambda’s per-invocation pricing was eating us alive during traffic spikes.

What we’d do differently

  1. Deduplication ID length – We capped SQS MessageDeduplicationId at 128 bytes. SHA256 produces 64 bytes, so we’re safe, but if we ever need longer customer IDs we’ll switch to a truncated SHA3-512 instead of truncating the key.

  2. Visibility timeout tuning – 300 ms was aggressive for some customers whose downstream APIs occasionally took 400 ms. We’ll move to a tiered timeout: 300 ms for 95 % of traffic, 1.5 s for the slow 5 %.

  3. Payload storage – Storing full payloads in DynamoDB cost $110/month. In hindsight we should have used S3 for payloads > 1 KB and DynamoDB only for metadata; that would cut DynamoDB costs by ~40 % and still keep us under 5 ms latency for the metadata lookup.

  4. Circuit breaker thresholds – Our pybreaker configuration was too aggressive. We set failure_threshold=5 within 10 s, but during the Black-Friday sale we saw 30 consecutive 5xx from one customer; the breaker tripped and we lost 1 200 events before we noticed. We’ll widen the window to 30 s and require 10 failures.

  5. Observability – We added a CloudWatch metric Webhook.DuplicateRaceCount but forgot to alarm on it. During the migration we had one race that produced 28 duplicates before we caught it. Add an alarm at >0 duplicates per 5 minutes.

The broader lesson

The trap we fell into isn’t new: we treated the webhook endpoint as just another REST controller. REST controllers are allowed to fail and the client will retry. Webhooks are a reverse-REST system: the client fires once, so the server must guarantee delivery.

Delivery guarantee ≠ idempotency key in the request path. Delivery guarantee = durable storage + transactional write + back-pressure boundary.

Most 2026 tutorials still teach the naive pattern:

Client → Webhook → Store key → Call downstream

The working pattern in 2026 is:

Client → Webhook → Store payload → Enqueue → Worker → Call downstream → Update status

The difference is the queue as a back-pressure boundary. A queue with a short visibility timeout lets the downstream API dictate its own pace without cascading failures. A durable outbox lets you replay lost messages even if the entire region goes dark.

In other words: stop accepting events until you’re sure you can keep them.

How to apply this to your situation

If you’re running webhooks today, do this in the next 30 minutes:

  1. Check your idempotency key collision rate. Run: bash aws dynamodb query \ --table-name WebhookEvents \ --key-condition-expression 'idempotencyKey = :k' \ --expression-attribute-values '{ ":k": {"S":"YOUR_COLLIDING_KEY"} }' \ --region us-east-1 Any result > 1 row is a collision you need to fix.

  2. Measure your retry storm impact. Look at CloudWatch metrics for 5xx and 429 during the last 7 days. If 429 > 5 % of total invocations, you need a queue with visibility timeout.

  3. Pick one customer with the highest event volume and enable the new pattern behind a feature flag for 24 hours. Use the same idempotency key logic but route through SQS FIFO. Compare duplicate rates and latency.

  4. If your current cost per event > $0.30, budget a 50 % cut by switching from Lambda to ECS Fargate for the ingest tier.

The minimum viable improvement is: store the payload before you return 200 OK.

Frequently Asked Questions

how to handle idempotency keys in a distributed system with eventual consistency

Use a transactional outbox so only one writer can commit the key. In practice this means a DynamoDB transaction or a Postgres advisory lock. Do not rely on the caller’s key alone; always append a server-side sequence number so you can break ties when races happen.

what visibility timeout should i set for sqs when retrying webhooks

Start with 300 ms. If your downstream P95 latency is > 300 ms, increase in 100 ms increments up to 1 500 ms. Monitor ApproximateReceiveCount CloudWatch metric; if it exceeds 3 consistently, you need to widen the timeout or scale your downstream capacity.

why did my lambda cost explode after adding retries

Lambda charges per invocation and per GB-second. Adding retries multiplies invocations. If your downstream fails 20 % of the time and you retry 3 times, you jump from 1 invocation to 1.6 invocations on average. Move to a queue-based architecture so only successful deliveries count toward compute cost.

how to test webhook delivery guarantees in staging

Use a chaos tool like AWS Fault Injection Simulator to kill the downstream service for 30 seconds while sending 1 000 events. Verify that no events are lost and that duplicates are < 0.01 %. Then repeat the test with a regional outage using AWS Global Accelerator to shift traffic to us-west-2.
