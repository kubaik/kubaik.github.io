Stop polishing your GitHub profile

Reduce portfolio noise by 80% with one rule: publish latency graphs, cost data, and incident logs—examples from real AWS and DigitalOcean systems that landed $95k–$135k remote offers.

The conventional wisdom (and why it's incomplete)

The standard advice for getting a remote job from Africa goes like this: build a GitHub with polished READMEs, contribute to open-source, write Medium posts, and rack up 6+ LeetCode problems a day. The gospel is that exposure is everything. I’ve seen this advice work—sometimes. A friend cleared 15 interviews last year using this playbook. But the honest answer is that most candidates who follow it end up with 200 GitHub stars and no offers because the signal they’re sending is noise.

I built my first public repo in 2015—an Express + React starter. I added a snazzy README, a demo link, and a GIF of the UI. I waited. Six months later, zero messages. I doubled down: added TypeScript, a Dockerfile, GitHub Actions CI, and a blog post explaining the architecture. Still nothing. I finally got a recruiter’s note in 2018—not for the repo, but because I’d fixed a real bug in a popular open-source library and documented the fix in a GitHub issue with performance numbers.

The conventional wisdom misses one hard truth: most hiring pipelines outside Silicon Valley care about problem-solving over polish. Your GitHub stars won’t override a resume that screams “I wrote a todo-app with 500 npm packages.”

Summary: The standard playbook optimizes for visibility, not relevance. Exposure without proof of impact is noise.

What actually happens when you follow the standard advice

Most developers who chase the polished profile hit a wall at step two: the interview loop. They’ve spent months tweaking repos that never see production traffic, and when they get to technical screens, they can’t explain why their system behaved the way it did under load. One candidate I interviewed claimed their Flask API handled 500 RPS because they tested it locally with ab -n 1000 -c 10. I asked what happened at 400 concurrent connections. They froze. I probed further and learned their “production” was a $5 DigitalOcean droplet with no auto-scaling, no monitoring, and a single gunicorn worker. They’d never measured latency above 200ms.

Another common failure: writing Medium posts about “best practices” that contradict their own code. A dev published a post titled “Why Redis is overkill for session storage” but used Redis in their portfolio project because it looked good in the README. The inconsistency screams inexperience.

I once spent three weeks helping a Nairobi fintech engineer rewrite his portfolio from a dozen micro-repos into a single app with a README that showed real AWS cost data, latency percentiles, and incident logs. Before the rewrite, he averaged one recruiter message per month. After, he had three offers in eight weeks. The signal changed from “I can write code” to “I can run systems.”

Summary: Polishing artifacts doesn’t prove skill. Measured outcomes do, and most candidates skip the measurement step.

A different mental model

Think of your portfolio as a product: it must solve a real user problem. Your user is the hiring manager who needs to ship features tomorrow. Your product is “your brain on call.” Therefore, your portfolio must demonstrate three things:

  1. You can design systems under constraints.
  2. You can measure and explain behavior.
  3. You can ship fixes under pressure.

That means: stop writing “e-commerce clones.” Start building something that breaks in production and show how you fixed it. Use real infra (even if small), real logs, real cost data. I built a micro-lending simulator in 2020: a Node.js service with PostgreSQL, Redis for rate limiting, a React dashboard, and a CI pipeline on GitHub Actions. I simulated 10,000 concurrent loan applications using k6, graphed latency percentiles in Grafana, and published the dashboard publicly. One recruiter DM’d me after seeing the latency spike at 8,000 RPS and the 503 errors. She hired me within two weeks.

Another shift: stop chasing open-source. Unless you’re fixing a dependency that your employer uses, most OSS contributions are noise to recruiters. I once contributed to a Go library with 50 stars. The maintainer thanked me, but no recruiter ever mentioned it. Contrast that with a bug I fixed in a production Node.js service at my fintech job: I reduced 95th percentile latency from 1.2s to 280ms by adding connection pooling to the MySQL driver. That fix landed me interviews at three remote-first companies.

Summary: Treat your portfolio as proof of on-call readiness, not as a showcase of syntax mastery.

The pattern is clear: recruiters click on projects that publish hard numbers. They ignore READMEs that say “built with love.”

I’ve seen candidates ship a “URL shortener” with Redis and claim 10k RPS. When I dug deeper, their Redis instance ran on a t3.micro with no persistence and a single client thread. The honest truth: it would die at 3k RPS. The moment I asked for the Redis INFO stats output and the load test CSV, the facade crumbled.

Another candidate built a “real-time chat” with Socket.io and a single Node.js instance on Railway. They added a Grafana dashboard showing “connections: 100.” When I asked how many messages per second the system sustained without dropping, they said they hadn’t measured. I walked them through a quick wrk2 test: the instance collapsed at 400 messages/sec. They never got past the first screen.

Summary: Recruiters want evidence of resilience. Numbers without context are noise; context without numbers is fiction.

The cases where the conventional wisdom IS right

Despite the critique, there are two times the standard advice works:

  1. Early-career candidates with no production experience. A polished GitHub can stand in for a resume when you lack battle scars. Example: a Nairobi intern built a React + Firebase TODO app with a clean README. They landed a junior role at a fintech because they could explain the React hooks and Firebase security rules clearly.

  2. Open-source maintainers for infra libraries. If you’re the primary maintainer of a package with 5k+ downloads weekly (e.g., a Python async HTTP client), recruiters will notice. I once interviewed a candidate who maintained a Go HTTP client used by 200+ repositories. The hiring manager skipped the LeetCode round because the candidate’s GitHub activity proved real-world impact.

Even in these cases, the signal must map to a real need. A React tutorial repo with 1k stars won’t help if the job is a backend role at a payments company.

Summary: The conventional playbook helps only when it substitutes for missing proof. Otherwise, it’s filler.

How to decide which approach fits your situation

Use this decision table:

Your profile Portfolio strategy Tooling focus Proof to publish
Junior / no prod experience Clean README, small app, basic CI Vercel, Railway, Firebase README screenshots, basic load test
Mid-level / infra curiosity System with real infra, logs, cost AWS (ECS Fargate, RDS, ElastiCache), k6, Grafana Cloud Latency/throughput graphs, infra cost, incident log
Senior / staff / infra maintainer OSS library or internal tool GitHub Actions, Terraform, Prometheus, Sentry Dependency usage stats, bug fix PRs, performance diffs

I made a mistake in 2021: I assumed every remote job required AWS expertise. I spent three months learning EKS, ALB, and WAF, only to realize most African fintech companies run on DigitalOcean or bare-metal in Mombasa data centers. I pivoted to showing how I optimized a DigitalOcean droplet fleet for a 30% cost cut and 60% latency drop. That project got me interviews at three Kenyan startups and one US remote-first company.

Summary: Choose infra that matches the job market you’re targeting, not the one you fantasize about.

Objections I've heard and my responses

Objection 1: “I don’t have time to build a system with real infra.”

I’ve heard this from candidates who work full-time jobs. My response: you don’t need a production system—you need a production-like sandbox. Use AWS Free Tier credits ($100/month for 12 months) or DigitalOcean’s $200 credit. Spin up a micro service with a single endpoint, add a load test, and publish the Grafana dashboard. I built a payment retry engine in two weeks using only AWS Lambda, DynamoDB, and CloudWatch Synthetics. The total bill was $3.47. The project got me a senior offer.

Objection 2: “Recruiters only care about LeetCode.”

I’ve placed engineers at US and EU remote-first companies. The honest answer: LeetCode helps only if the company’s interview loop is LeetCode. Most African fintech companies now use system design rounds and take-home assignments. I placed two engineers at a Nairobi payments company using only a take-home system design that included latency budgets, cost constraints, and incident response plans. No LeetCode required.

Objection 3: “My open-source contributions will never match big repos.”

You don’t need to match React or Kubernetes. Focus on fixing a dependency that your target employer uses. Example: if you’re targeting a Python fintech, fix a bug in httpx or asyncpg and document the fix with a before/after latency chart. One candidate fixed a memory leak in asyncpg that affected a Nairobi payments startup. The CTO noticed the PR and hired the candidate within two weeks.

Summary: Time and scale objections ignore cheap sandboxes and niche impact. Find the smallest slice that proves skill.

What I'd do differently if starting over

If I rebuilt my portfolio today, I would:

  1. Start with a real pain point. I’d pick a slow API in my city’s public transport system (matatu) and build a caching layer. I’d measure the API’s 95th percentile latency at 2.1s, cache it with Redis, and show the drop to 340ms. Then I’d publish the Grafana dashboard and GitHub repo with the load test CSV. That project would get me interviews faster than any “full-stack” clone.

  2. Use real infra from day one. I’d spin up AWS ECS Fargate with a single service, add RDS, ElastiCache, and CloudWatch. I’d publish the cost per 1k requests and the latency percentiles. No local dev server screenshots allowed.

  3. Simulate an incident. I’d write a postmortem for a 5-minute outage I caused by a misconfigured Redis eviction policy. I’d include the timeline, the fix, and the remediation steps. Recruiters eat this up.

  4. Avoid frameworks I won’t use again. I wasted months on NestJS because it looked good on a tutorial. When I finally used FastAPI at work, I realized the framework mattered less than the system design. Today, I’d use FastAPI for Python or Go’s standard library for HTTP servers.

  5. Measure everything. I’d add Prometheus metrics, Grafana dashboards, and load tests to every repo. No README without a screenshot of the dashboard.

Summary: Today I’d build a system that breaks, measure the breakage, and show the fix—not a system that pretends to work.

Summary

Your portfolio must prove you can run systems, not write code. Recruiters care about latency graphs, incident logs, and cost per request—not stars, not Medium posts, not LeetCode scores. Build a project that fails in production, measure the failure, and publish the remediation. That’s the signal that gets you hired remotely from Africa.

Next step: Pick a public API you use daily, measure its latency, build a caching layer, and publish the before/after graphs with the repo link. Ship it in one week. No excuses.

Frequently Asked Questions

How much does it cost to run a portfolio project on AWS?

Most portfolio projects can run under $10/month on AWS Free Tier or DigitalOcean. Example: a FastAPI service with RDS micro, ElastiCache micro, and CloudWatch logs costs about $7/month at low traffic. I ran a payment retry engine for $3.47/month using AWS Lambda, DynamoDB, and CloudWatch Synthetics. The key is to use serverless or micro instances and delete them after the interview season.

Do I need Kubernetes for my portfolio?

No. Unless you’re targeting a staff engineer role at a hyperscaler, Kubernetes is overkill. Most African fintech companies run on DigitalOcean, AWS ECS Fargate, or bare-metal. I once interviewed a candidate who bragged about their EKS setup. The hiring manager asked how they’d handle a node failure. The candidate froze. Stick to what you’ll actually use in the job.

How do I add load testing to my portfolio?

Use k6 or Artillery. Add a GitHub Action that runs a 1000-RPS test on every push and uploads the CSV to an S3 bucket. Then embed a Grafana dashboard screenshot in your README. Example workflow:

# .github/workflows/load-test.yml
name: Load test
on: [push]
jobs:
  load-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: grafana/k6-action@v0.2.0
        with:
          filename: tests/api_test.js
      - uses: actions/upload-artifact@v3
        with:
          name: k6-results
          path: output.csv

Include a link to the dashboard in your README. Recruiters click on numbers.

What if my target companies don’t use AWS?

Use the cloud your target uses. In Kenya, many fintechs run on DigitalOcean or Linode. In Nigeria, it’s AWS or Azure. I built a portfolio on DigitalOcean for a Nigerian payments company. I used a $20/month droplet, PostgreSQL, and a $5/month Redis instance. The project got me an interview because the infra matched the company’s stack.

Postscript: One metric to rule them all

There’s a single metric that predicts whether your portfolio will convert recruiters: the time it takes a hiring manager to find proof of impact after opening your repo. If they have to read 10 files to understand what your system does, you’ve lost them. If they see a Grafana link in the README and a load test CSV attached, you’ve won the first 30 seconds. Make impact obvious.

That’s the secret.
