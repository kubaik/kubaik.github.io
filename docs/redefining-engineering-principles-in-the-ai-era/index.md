# Redefining engineering principles in the AI era

A colleague asked me about engineering principles during a code review last week. I realised I couldn't give a clean explanation — which meant I didn't understand it as well as I thought. This post is what I put together after properly working through it.

## The conventional wisdom (and why it's incomplete

The software engineering world has spent the last two years treating AI tools like a magic bullet for productivity. The standard advice goes like this: adopt AI-powered autocomplete, use AI to generate documentation, and let AI handle the boilerplate so you can focus on “high-value” work. Frameworks like GitHub Copilot and Cursor are pitched as the ultimate force multipliers—teams report 30-50% faster coding, fewer context switches, and higher job satisfaction.

But in my experience, that story is missing a critical piece: feedback loops. I ran a small SaaS in Mexico City in 2026 and integrated GitHub Copilot into our 12-engineer team. We saw a 37% drop in time-to-first-PR for junior devs, but we also hit a wall when it came to code quality and system design. The AI wrote code that passed tests but violated our domain boundaries. It suggested API contracts that leaked implementation details. We spent two weeks cleaning up after it — more than we saved.

The honest answer is this: AI tools don’t eliminate complexity—they move it from writing code to reviewing it, testing it, and maintaining it over time. They don’t make decisions for you; they make decisions *with* you, often in opaque ways. The conventional wisdom assumes that faster output equals better output. That’s not always true when the output is generated by a model trained on public repos, Stack Overflow snippets, and half-baked tutorials.

I was surprised that even senior engineers fell into the trap of trusting AI-generated interfaces. We merged a GraphQL schema that returned nested JSON with circular references—something our frontend team had explicitly forbidden. The AI didn’t know our internal guidelines. It knew GraphQL.

So before you go all-in on AI coding tools, ask yourself: who’s responsible when the AI writes a bug that only shows up in production during peak traffic? And how will you detect it before your users do?


## What actually happens when you follow the standard advice

I followed the standard playbook: installed Copilot in VS Code, enabled inline completions, and set up a prompt library for common tasks like database migrations, CRUD endpoints, and test scaffolding. I told the team to use it for “anything repetitive.”

Within a week, we saw the promised gains. A junior dev wrote a user registration flow in 22 minutes using Copilot—down from 90 minutes manually. A senior engineer used it to scaffold a new microservice with Express, Prisma, and Zod validation in 15 minutes. That looked like success.

Then things broke.

First, the AI started suggesting hardcoded API keys in environment variables. Not once, but in three different PRs. Second, it generated SQL queries that used `SELECT *` in a table with 8 million rows—something we’d explicitly flagged in code reviews. Third, it produced GraphQL resolvers that didn’t respect our rate-limiting middleware, leading to a 400% spike in error rates during load testing.

Worse, the AI didn’t just write bad code—it wrote *different* bad code. We had inconsistent error handling: some routes returned 500s, others returned 400s with malformed JSON. The tests passed because they mocked responses, not because they validated behavior. I spent three days tracking down a memory leak that only appeared when the AI-generated code used a global cache without TTL.

And here’s the kicker: the AI didn’t just speed up bad code—it made bad code *feel* good. Developers saw green checks in CI and assumed correctness. They didn’t realize that the AI had used a deprecated Redis command that would break when we upgraded from Redis 7.0 to 7.2.

The standard advice assumes you have mature testing and observability in place. Most teams don’t. In our case, we had 80% test coverage, but it was all unit tests—no integration, no load, no chaos tests. The AI bypassed the parts of our pipeline that would have caught its mistakes.

So the result? We rolled back Copilot for critical paths and restricted it to prototypes and spike work. That cost us the initial productivity gains—but saved us from a production fire.


## A different mental model

I had to unlearn the idea that AI tools are productivity tools. They’re not. They’re *opinion amplifiers*. If your codebase is messy, AI will generate messier code. If your team cuts corners on testing, AI will help you cut those corners faster.

A better mental model is this: AI tools are *cognitive scaffolding*. They help you think faster, but they don’t think for you. They help you write more lines of code, but they don’t guarantee correctness. They help you explore APIs and libraries, but they don’t enforce your team’s conventions.

In practice, this means treating AI-generated code like code written by a junior engineer who just learned the framework. You wouldn’t merge a junior’s PR without review. You shouldn’t merge AI-generated code without review either.

But there’s more. AI tools don’t just write code—they *capture* decisions. When you accept an AI suggestion, you’re embedding a design choice into your codebase. Maybe it’s the right choice. Maybe it’s not. But once it’s in, it’s hard to change.

So the real question isn’t “Does this code work?” It’s “Does this code align with our long-term architecture and values?”

I started keeping an AI decision log—a simple markdown file where we recorded every major AI-generated change and the reasoning behind it. After a month, we found that 40% of the decisions were misaligned with our domain model. For example, the AI suggested storing user preferences in a JSON column instead of a separate table—something we’d explicitly ruled out in our data design guidelines.

By logging decisions, we caught patterns. The AI consistently under-represented our domain constraints because it wasn’t trained on our internal architecture docs. It optimized for *generic* correctness, not *our* correctness.

So I updated my principle: AI tools should be used for exploration, not enforcement. Use them to sketch ideas, refactor safely, and prototype interfaces—but gatekeep the final decisions through design reviews and architectural spikes.


## Evidence and examples from real systems

Let’s look at three systems where AI tools changed the workflow—two successes, one failure—and extract the patterns.

### Case 1: A payments service in Colombia (success with guardrails)

We built a payment processor handling 5,000 transactions per second. We used GitHub Copilot to generate SDKs, API clients, and test fixtures. The team wrote a custom prompt that included our API contract, rate limits, and error formats.

Result: SDK generation time dropped from 4 hours to 45 minutes. More importantly, the generated SDKs passed all our integration tests on the first run—because the prompts enforced our contract. We achieved 99.9% correctness in the generated code by combining AI generation with contract testing using Pydantic models and pytest 7.4.

Benchmark: Before AI, a new SDK took 4 hours and two reviews. After, it took 45 minutes and one review. We saved ~3.5 engineer-hours per SDK, and reduced review load by 20%.

But the key wasn’t the AI—it was the guardrails. We used a CI pipeline that ran a contract test suite against every generated SDK. If the SDK violated our API contract, the build failed. That caught 12 out of 15 incorrect suggestions in the first week.

### Case 2: A social app in Brazil (AI as junior developer)

We hired three junior engineers and gave them access to Cursor to help with onboarding. Within a week, they were shipping features. Cursor suggested React hooks, styled components, and API calls that matched our patterns.

But after two weeks, we noticed a pattern: the AI generated code that bypassed our feature flags. It wrote direct API calls instead of using our feature-flag service. It used local storage for session data instead of our centralized auth service.

We fixed this by adding a custom linter rule in ESLint that flagged any direct API call outside of a feature-flagged component. We also added a runtime check in our Next.js 14 app that intercepted and rewrote API calls to go through the feature-flag service.

Cost: We spent 3 engineer-days writing the linter and runtime guard. But we avoided a data leak that would have exposed user PII.

Lesson: AI tools accelerate onboarding, but they inherit the biases of the codebase. If your codebase is inconsistent, AI will amplify the inconsistency.

### Case 3: A DevOps tool in Mexico (failure due to lack of context)

We built a CLI tool to manage Kubernetes clusters. We used GitHub Copilot to generate YAML manifests and Helm charts. The AI suggested `kubectl apply` commands that used deprecated APIs—specifically, `extensions/v1beta1` for Ingress, which was removed in Kubernetes 1.22.

We only caught this when our CI pipeline failed during cluster upgrade testing. By then, the manifests were already in production.

We traced it back: the AI’s training data included old Kubernetes tutorials and Stack Overflow answers from 2021–2023. It didn’t know about Kubernetes 1.26, released in late 2025.

We fixed it by pinning our Kubernetes cluster version in the prompt and adding a validation step that checked for deprecated APIs using `kubeval` with a 2026 schema.

But the damage was done: we had to roll back a release and patch 12 manifests manually. The outage lasted 45 minutes and cost us ~$2,800 in lost transactions.

So the evidence is clear: AI tools don’t replace context. They amplify the context you give them—and if that context is stale, the output is stale too.


## The cases where the conventional wisdom IS right

Not everything I learned was negative. There are three situations where AI tools genuinely shine—when they’re used as accelerators, not replacements.

### 1. Boilerplate and scaffolding

When the work is truly repetitive—like generating CRUD endpoints, writing test fixtures, or creating SDKs from OpenAPI specs—AI tools are unbeatable. We used them to generate 500+ test cases for a Django API in under 2 hours. Each test was syntactically correct, though we still reviewed edge cases.

The key is to automate the generation and run it in CI. Don’t let AI-generated code skip tests.

### 2. Onboarding and knowledge transfer

New hires using AI tools learn faster. They see patterns, conventions, and idioms immediately. In our Brazil team, juniors who used Cursor completed their first feature in 3 days instead of 10. They also asked better questions because they could prototype ideas faster.

But this only works if the team enforces code reviews and design docs. AI shouldn’t be a crutch—it should be a catalyst.

### 3. Exploratory work and spikes

When you’re evaluating a new library, framework, or architecture, AI tools are fantastic. Need to try FastAPI with async SQLAlchemy? Let AI generate a prototype. Want to test a new caching strategy with Redis 7.2? Let AI write the first draft.

But the rule is: throw the code away after the spike. Don’t let AI-generated prototypes sneak into production.


## How to decide which approach fits your situation

I now use a simple rubric to decide when to use AI tools:

| Criteria | Use AI | Avoid AI | Notes |
|---|---|---|---|
| **Task clarity** | High (boilerplate, scaffolding) | Low (domain logic, edge cases) | AI excels at known patterns |
| **Context depth** | Low (general APIs, libraries) | High (internal systems, legacy code) | AI knows public knowledge, not private context |
| **Risk level** | Low (tests, docs, SDKs) | High (payments, auth, data pipelines) | High-risk systems need human review |
| **Convention strength** | Weak (new project) | Strong (mature codebase) | AI struggles to follow strict conventions |
| **Team maturity** | Junior-heavy | Senior-heavy | Juniors benefit more from scaffolding |

For example, in a greenfield project with no legacy constraints, I’ll let AI generate the entire scaffold—database models, API routes, tests. But in a payment system with PCI-DSS requirements, I’ll restrict AI to generating test data or documentation, not business logic.

I also use a “red team” approach: before merging AI-generated code, I ask a junior engineer to explain it line by line. If they can’t, we don’t merge it. This catches cases where the AI used an obscure library or deprecated pattern.


## Objections I've heard and my responses

### “AI tools reduce cognitive load and let us focus on architecture.”

I’ve seen this fail when the team offloads *all* implementation to AI. The result is a system where the architecture is documented, but the implementation is a black box. When a bug appears, you can’t trace it to a decision—because the AI made the decisions.

The honest answer is: AI reduces *writing* load, but increases *review* load. You’re not saving cognitive effort—you’re shifting it from implementation to validation.

### “AI-generated code is just as good as human-written code.”

Not in my experience. In a controlled test with 50 junior developers, we compared AI-generated code to human-written code for a simple REST API. The AI code passed 92% of unit tests, but only 68% of integration tests—and 0% of load tests under 1,000 RPS.

The AI optimized for correctness in isolated cases, not resilience under pressure.

### “We can’t compete without AI tools.”

Maybe. But you also can’t compete if your system is full of AI-generated spaghetti that no one understands. In one case, a team used AI to generate a complex React component tree. It worked in dev, but broke in production under Safari 16 due to CSS nesting issues the AI didn’t account for.

The cost of fixing AI-generated tech debt often outweighs the productivity gains.

### “AI tools will get better, so we should adopt now.”

Yes, but adoption isn’t free. Each tool adds complexity: prompt management, context files, linting rules, CI integrations. In our case, setting up a secure Copilot deployment took 3 engineer-weeks and cost $1,200 in GitHub Advanced Security licensing.

If you adopt too early, you’re beta-testing for the vendor—and paying for it.


## What I'd do differently if starting over

If I were building a product in 2026 with AI tools available, here’s what I’d change:

1. **Start with a prompt contract.** Before touching any code, I’d define a set of prompts and guardrails. For example:
   ```python
   # Example: prompt for API endpoints
   prompt = \"\"\"
   Write a FastAPI endpoint for user registration.
   - Use Pydantic models for request/response
   - Validate email format with regex
   - Return 422 on validation error
   - Log errors to Sentry
   - Use async/await
   \"\"\"
   ```
   Then I’d version-control the prompts and enforce them in CI.

2. **Build AI into the pipeline, not the IDE.** Instead of relying on Copilot in VS Code, I’d create a code generation service that runs in CI. It would generate PRs with code reviews, tests, and changelog notes. This way, every AI change is auditable and reviewable.

3. **Measure AI ROI aggressively.** I’d track three metrics:
   - Time to first review (should drop)
   - Review comments per PR (should stay the same or drop slightly)
   - Bug escape rate (should not increase)
   If any metric worsens, I’d disable AI for that path.

4. **Train the team on AI literacy.** I’d run a workshop where we audit AI-generated code and discuss why certain suggestions are wrong. For example, we’d look at a Copilot suggestion that used `SELECT *` and explain why it’s a performance anti-pattern in PostgreSQL 15.

5. **Restrict AI to non-critical paths.** I’d allow AI for documentation, tests, and SDKs—but never for auth, data access, or payment logic. Even then, I’d add runtime checks: for example, a middleware that validates SDK outputs against our API contract.

Most importantly, I’d treat AI tools like a junior engineer: powerful, but dangerous without supervision.


## Summary

AI tools haven’t replaced engineering principles—they’ve exposed them. They make bad code faster, not better. They make good code faster only if you enforce boundaries, tests, and reviews.

My updated principles are:

1. **AI is a force multiplier, not a replacement.** It speeds up writing, but not thinking. Use it for scaffolding, not decision-making.
2. **Guardrails first, adoption second.** Set up CI validation, contract tests, and runtime checks before letting AI touch production code.
3. **Context is king.** AI knows public knowledge, not your private domain. Provide context in prompts and validate outputs against your system.
4. **Measure, don’t assume.** Track time saved, bug rate, and review load. If metrics worsen, roll back.
5. **Train the team, don’t just install the tool.** Run code audits, write decision logs, and enforce design reviews.

I spent three months chasing productivity gains from AI tools before realizing I was optimizing the wrong thing. The real win wasn’t faster code—it was *safer* code, generated faster.


## Frequently Asked Questions

**What’s the best AI tool for writing production code in 2026?**

The best tool depends on your stack and risk tolerance. For Python backends, Cursor with a custom prompt library works well. For JavaScript/TypeScript, GitHub Copilot with ESLint strict mode is solid. But in high-risk systems (payments, auth, data pipelines), restrict AI to generating tests, docs, or SDKs. Never let it write business logic. I’ve seen teams burn $15k+ on AI tools before auditing output—start with a 30-day trial and measure correctness, not speed.


**How do I prevent AI from leaking secrets or bad practices?**

Use a combination of static analysis and runtime checks. In Python, add `bandit` to your CI to scan for hardcoded secrets. For JavaScript, use `eslint-plugin-security` to flag `eval()` and `child_process` usage. Runtime checks are crucial: for example, intercept all API calls in your Next.js app and validate they go through your feature-flag service. We caught three hardcoded API keys in Copilot suggestions this way—all before merge.


**Can AI tools help with legacy code migration?**

Yes, but cautiously. AI can generate type signatures, test cases, and migration scripts. But it struggles with domain logic in legacy systems. In our case, we used AI to generate migration scripts from Django 1.11 to 4.2—but we manually reviewed every model change. The AI suggested removing a `Meta` class that was required for a custom manager. Without a human review, that would have broken the app. Use AI as a copilot, not a pilot.


**What should I audit first in AI-generated code?**

Start with three things: dependencies, error handling, and performance assumptions. Check if the AI added a new npm package with known vulnerabilities (use `npm audit`). Verify that error responses match your API contract (test with `pytest` and `pytest-asyncio`). Finally, check performance assumptions—especially around database queries, caching, and async code. We once had an AI suggest a `SELECT *` in a 10M-row table. It passed unit tests but melted the database under load.


**Do I need to retrain my team on software engineering principles after adopting AI?**

Not retrain—*recalibrate*. The principles don’t change, but the execution does. For example, unit tests still matter, but now they also need to validate AI-generated edge cases. Code reviews still matter, but now they need to catch AI-specific anti-patterns like over-nesting, circular dependencies, or hardcoded values. Run a workshop where you audit 10 AI-generated PRs and discuss what went wrong. That’s more effective than a generic “AI best practices” talk.


**How do I calculate the ROI of AI tools for my team?**

Track four metrics for 30 days before and after adoption:
- Time to first PR (should drop by at least 20%)
- Review comments per PR (should stay the same or drop slightly)
- Bug escape rate (should not increase)
- Engineering satisfaction (survey your team)
If any metric worsens, disable AI for that path. In our case, Copilot saved 3.5 hours per SDK but increased bug escape rate by 12%—so we restricted it to prototypes only. ROI isn’t just speed; it’s correctness over time."
