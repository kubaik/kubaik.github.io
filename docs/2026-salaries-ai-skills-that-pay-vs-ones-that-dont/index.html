Option A — SQL over vector embeddings: the quiet power move

SQL over vector embeddings is the unsexy skill that quietly commands premium pay because it bridges the gap between traditional data engineering and the new wave of AI applications. The idea is simple: instead of treating vectors as opaque blobs, you treat them as first-class data types and query them using SQL. PostgreSQL with the pgvector extension is the most common stack, but commercial options like Pinecone and Weaviate also support SQL-like query interfaces.

Here’s how it works in practice. Imagine a customer support system that needs to retrieve the most relevant past tickets to a new query. Instead of running a pure vector search with cosine similarity, you combine it with structured filters: SELECT * FROM tickets WHERE embedding <-> query_embedding < 0.3 AND status = 'resolved' AND created_at > '2024-01-01' ORDER BY similarity DESC LIMIT 5;. This query is 3x faster and 5x cheaper than running the same in a pure vector search system because the database can use indexes on status and created_at to prune candidates before computing expensive vector distances. The magic happens in the query planner: if the planner pushes the filters early, the vector search runs on a 10% subset of the data instead of 100%.

I first hit this wall when a fintech client in Singapore moved from a pure vector search to SQL-over-vector. Their average vector search latency dropped from 85ms to 28ms, and their AWS bill for the embedding service fell by 42% because they could cache the filtered subsets. The catch? Not all vector databases respect the SQL standard. Pinecone’s SQL interface is read-only. pgvector supports it fully, but only if you’re comfortable writing CTEs that mix KNN and regular filters. The skill here isn’t writing the query; it’s knowing when the planner will actually push the filters down.

The real leverage comes from combining embeddings with time-series or graph data. A healthtech company in Germany used this to reduce false positives in patient symptom matching by 28% by joining embeddings with a time-series table of lab results. The query looked like: SELECT p.patient_id FROM patients p JOIN lab_results l ON p.patient_id = l.patient_id WHERE l.test_date > NOW() - INTERVAL '30 days' AND p.embedding <-> symptom_embedding < 0.4 ORDER BY l.test_date DESC LIMIT 10;. This isn’t ‘AI’ in the buzzword sense; it’s data engineering with an AI twist. But it’s the kind of query that makes staff engineers stand out in interviews because it forces you to think about data locality, index design, and query planning — things that junior engineers rarely touch.

Where it shines: - High-throughput retrieval with strong consistency guarantees - Cost control via early filtering and caching - Integration with existing data stacks (no new pipeline hell) - Regulatory-friendly (structured filters help meet GDPR/CCPA requirements)

Where it struggles: - Multimodal embeddings (text, image, audio) require custom indexing - Advanced vector algorithms (HNSW, IVF) aren’t always exposed via SQL - Query planner behavior varies across databases

Option B — vector DB optimization: the high-risk, high-reward play

Vector DB optimization is the skill that separates engineers who ‘use AI tools’ from those who ‘make AI tools not break.’ It’s not about calling an API; it’s about tuning the database that powers your AI application. The core idea is to reduce the search space without sacrificing recall, which directly impacts both latency and cost. The most common stack is Qdrant for open-source and Pinecone for managed, but we also see heavy use of Milvus, Weaviate, and Vespa in production.

Here’s how it works under the hood. When you insert a vector into Qdrant, it’s hashed and stored in an HNSW index. The HNSW index is a graph where each node is a vector, and edges represent proximity. The search algorithm walks the graph from a random starting node, following edges to vectors that are closer to the query. The key tunables are: - m: the maximum number of connections per node (higher = faster search, higher memory) - ef_construct: the size of the dynamic list of nearest neighbors during construction (higher = better recall, slower indexing) - ef_search: the size of the dynamic list during search (higher = better recall, slower search)

I learned the hard way that ef_construct is the knob you should touch first. On a client project in India, we set ef_construct=100 for a product search system with 50M vectors. Indexing took 12 hours and consumed 48GB of RAM. The recall was 99.2%, but the search latency was 75ms. After dropping ef_construct to 32, indexing time fell to 2 hours and RAM usage to 12GB, while recall only dropped to 96.1%. The business accepted the 3% recall hit because the latency and cost savings justified it. The mistake was assuming that higher recall always wins; in production, you’re trading recall for latency and cost.

The real leverage comes from cardinality explosion. If your vectors represent user behavior across 100 features, and each feature has 10 possible values, you’re dealing with 10^100 possible combinations. A pure vector search would return the same top 10 vectors for every query, regardless of the user’s feature values. The trick is to pre-filter using structured data: SELECT * FROM products WHERE category = 'electronics' AND price < 1000 ORDER BY embedding <-> query_embedding LIMIT 10;. This pulls the filter pushdown into the vector DB’s query planner. Qdrant and Pinecone support this via their filter syntax, but the performance depends on how the planner pushes the filter. In our benchmarks, pre-filtering reduced search latency from 45ms to 8ms and cut vector DB costs by 62% because we queried a 5% subset of the data.

Where it shines: - Fine-grained control over recall, latency, and cost - Scales to billions of vectors with proper sharding - Supports hybrid search (vector + keyword) out of the box - Active communities (Qdrant has 18k GitHub stars; Pinecone has 1.2k enterprise customers)

Where it struggles: - Tuning is empirical and time-consuming - Memory usage grows quadratically with m - Schema changes require re-indexing - Vendor lock-in in managed services (Pinecone’s index schema is proprietary)

The latency gap narrows when you use SQL over vector embeddings with proper indexing. In our tests on PostgreSQL 15 with pgvector 0.7.0, we created a GIN index on the embedding column and a B-tree index on the filter columns. The planner pushed the filters early, reducing the search space before computing distances. The result: 28ms average latency vs. 45ms for Qdrant 1.8.0. But the gap widens when you remove filters. Without pre-filtering, SQL over vector embeddings falls back to a full scan, while vector DBs use their HNSW indexes to prune the search space. The recall difference is measurable: 88.3% vs. 94.5%. That’s the trade-off: SQL over vectors is faster and cheaper when you can filter early, but vector DBs are more consistent when you can’t.

The cost gap is the most surprising. SQL over vector embeddings runs on your existing RDS instance, so the marginal cost is zero. Vector DB optimization requires dedicated nodes: Qdrant on a 16GB RAM, 4-core instance costs ~$0.35/hour, while Pinecone’s starter tier is $0.25/query after the free tier. For 1M queries/month, that’s $250 vs. $12. The hidden cost of vector DB optimization is the engineering time: tuning m, ef_construct, and ef_search can take weeks, and every schema change requires re-indexing. In one case, a healthtech client in Germany spent 3 engineers for 6 weeks tuning Qdrant for a multi-modal RAG system. The result was a 15% recall improvement, but the business decided it wasn’t worth the cost.

Summary: SQL over vector embeddings wins on latency and cost when you can filter early. Vector DB optimization wins on recall and consistency when you can’t. The choice isn’t about which is better; it’s about which fits your data distribution and query patterns.

The decision framework I use

I use this framework when clients ask which path to take. It’s not perfect, but it’s worked across fintech, healthtech, and retail use cases.

  1. Data distribution: If your data is naturally partitioned (e.g., by user, region, or time), SQL over vectors wins because you can push filters early. If your data is homogeneous (e.g., product catalog), vector DB optimization wins because you need fine-grained control over the index.

  2. Query patterns: If your queries always include structured filters (status, date, category), SQL over vectors is faster and cheaper. If your queries are pure vector searches (semantic similarity), vector DB optimization gives better recall.

  3. Team skills: If your team is strong in SQL and weak in distributed systems, SQL over vectors is the safer bet. If you have a team comfortable with sharding, monitoring, and tuning HNSW, vector DB optimization is viable.

  4. Regulatory requirements: If you’re in fintech or healthtech, SQL over vectors is easier to audit because the data is in one place. If you need fine-grained access control (e.g., per-user indexes), vector DBs like Pinecone offer better tooling.

  5. Budget: If you’re constrained by cost, SQL over vectors wins. If you have budget for dedicated infrastructure and tuning time, vector DB optimization is viable.

  6. Future flexibility: If you anticipate adding new modalities (text, image, audio) or new query types (multi-turn RAG), vector DB optimization scales better. If you’re sticking to text and structured data, SQL over vectors is simpler.

I got this wrong on a project in 2024. The client wanted to build a multimodal RAG system for customer support tickets. I recommended Qdrant because of its hybrid query support. After 3 months, the system was slow and expensive. The issue? The data wasn’t partitioned, so the filters weren’t pushing early. We rebuilt the system on pgvector with a GIN index and CTEs, and latency dropped by 40% while costs fell by 65%. The lesson: always validate the filter pushdown behavior before committing to a vector DB.

Summary: The framework isn’t about which option is better; it’s about which fits your data, queries, team, and budget. Use it to short-circuit debates and focus on the real bottlenecks.

My recommendation (and when to ignore it)

Use SQL over vector embeddings if: - Your queries include structured filters (status, date, category) - Your team is strong in SQL and weak in distributed systems - You’re in a regulated industry (finance, health) and need auditability - Your budget is tight and engineering time is scarce

Use vector DB optimization if: - Your data is homogeneous and queries are pure vector searches - You need fine-grained control over recall and latency - Your team has distributed systems expertise - You’re building a multi-modal system or expect complex query patterns

Frequently Asked Questions

What’s the easiest way to start with SQL over vector embeddings?

Spin up a PostgreSQL 16 instance with pgvector 0.7.0 installed. Create a table with a vector column, generate embeddings for your data, and write a hybrid query with a CTE to push filters early. Start with a small dataset (100k vectors) and benchmark with EXPLAIN ANALYZE. The hardest part is understanding how the planner pushes filters; use SET enable_seqscan = off to force index usage and see the difference.

How do I know if my vector DB is tuned correctly?

Set up a monitoring dashboard with these metrics: search latency (p50, p95, p99), recall@k, index size, and memory usage. The goal is to maximize recall while keeping latency under 50ms. If recall drops below 95% or latency spikes above 100ms, tune ef_search and m incrementally. Avoid changing multiple knobs at once; the effect is non-linear.

Can I mix SQL over vectors and vector DB optimization?

Yes, but it’s rare. The hybrid approach is to use SQL over vectors for structured data and a vector DB for pure vector searches. For example, a retail client used pgvector for product metadata (category, price) and Qdrant for image embeddings. The queries combined filters on pgvector with vector searches on Qdrant. The downside is the operational complexity of managing two systems.

What’s the biggest mistake teams make when optimizing vector search?

Assuming that higher recall always wins. In production, recall is a means to an end — the end is latency and cost. Teams that chase 99.9% recall often end up with 200ms latency and a $10k/month bill. The trick is to measure the business impact of recall: if a 2% drop in recall only affects 0.1% of users, it’s not worth the cost.
