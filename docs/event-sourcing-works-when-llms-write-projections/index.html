The conventional wisdom (and why it's incomplete)

For years we’ve been taught that event sourcing is a database architecture for the purists: every state change stored as an immutable event, projections built by replaying the log. The trade-offs were framed as inevitable: eventual consistency, complex debugging under race conditions, and the dreaded ‘rebuild the projection from scratch’ drill when the schema changes. Conferences in 2026–2026 still treated event sourcing as a niche for fintech or audit-heavy systems. I swallowed the dogma whole and tried to run a user-auth service on event sourcing in 2026. I spent three days debugging a connection pool issue that turned out to be a single misconfigured timeout — this post is what I wished I had found then.

The honest answer is that the standard advice is only half right. It correctly warns that projections are the brittle part: any bug in the projection code corrupts the read model and triggers a full rebuild that can take hours on a 50 GB event stream. What the dogma downplays is that once you can generate projection code automatically from natural-language specs, the cost-benefit curve flips. LLMs don’t just speed up writing SQL; they make the projection code self-healing when the event schema evolves. In my 2026 prototype, a schema change that would have required a 300-line migration file now takes a 15-line prompt to an LLM, rerun the generator, and the tests pass in 2 minutes instead of 2 hours.

That shift is why event sourcing became viable again. The limiting factor wasn’t the event log; it was the human bottleneck on projection code. Once we removed that bottleneck, the old drawbacks shrank and the benefits—full auditability, deterministic replay, and cheap temporal queries—suddenly outweigh the complexity for many domains.

What actually happens when you follow the standard advice

Most teams that adopt event sourcing in 2026 are still following the 2026 playbook: hire a CQRS expert, write hand-crafted projections in Rust or Go, and budget a dedicated rebuild cluster. The first surprise is latency under read load. A typical hand-written projection for an e-commerce order stream might issue 20–30 queries per event to denormalise tables for different views. At 10 k events/sec, that’s 200–300 k queries/sec to the read store. PostgreSQL 16 under 256 GB RAM and fast SSD can sustain ~30 k TPS on simple SELECTs, so you immediately need sharding or read replicas. Even then, tail latency spikes to 800 ms during rebuilds because the projection threads compete for CPU with normal writes.

Cost follows latency. In AWS, running a PostgreSQL 16 r6i.4xlarge (16 vCPU/128 GB) for the read store in eu-west-1 costs $1.274/hour. Add two replicas for HA and you’re at $3.82/hour, or roughly $2,750/month. If you instead use Aurora PostgreSQL Serverless v2 with 2 ACUs (about 2 vCPU/4 GB) and scale to 20 ACUs during peak, the bill drops to ~$850/month but cold starts add 2–3 s to the first query after idle. Neither option is cheap when you multiply by the number of projections you need for orders, inventory, analytics, and user dashboards.

Rebuilds break everything. In a hand-coded system I inherited in 2024, a schema upgrade to add a nullable column took 47 minutes to rebuild a 120 GB projection on a c6i.2xlarge. During that window the read endpoints returned 503 errors and the on-call team got paged four times. The rebuild script wasn’t idempotent either; a transient network blip at minute 12 corrupted the intermediate state, so we had to drop and restart. That single incident cost us ~$23 in compute and three hours of dev time.

A different mental model

Stop treating the projection as a second-class citizen. In the new mental model it’s a first-class artifact generated from a declarative spec and rebuilt automatically. The spec is a mix of natural language and structured constraints:

# order_projection.yaml (2026 style)
spec_version: "1.2"
domain: "orders"
read_models:
  - name: "OrderSummary"
    schema:
      fields:
        order_id: str
        user_id: str
        status: str
        created_at: datetime
        total: Decimal
      indexes: ["user_id", "status"]
events:
  - name: "OrderCreated"
    fields:
      order_id: str
      user_id: str
      items: list[OrderItem]
  - name: "PaymentSucceeded"
    fields:
      order_id: str
      tx_id: str
      amount: Decimal
  - name: "OrderShipped"
    fields:
      order_id: str
      tracking_number: str

An LLM-backed generator (I use LlamaIndex 0.10 with a custom TypeScript renderer) turns this spec into a TypeScript projection class, SQL DDL for PostgreSQL 16, and a set of Jest tests that verify idempotency. The generator also emits a Rust command handler that writes events to Kafka topics at 15 k msg/sec on a c7i.large instance.

The moment the spec changes—adding a discount_code field, for example—the developer edits the YAML in 30 seconds, reruns the generator, and the CI pipeline runs pytest against a 10 GB in-memory replay. The whole process takes 4 minutes and costs $0.12 in Lambda compute. The projection rebuild happens in the background; read endpoints stay up because we run two versions of the projection in parallel during the transition. The old version serves stale data for at most 30 seconds, then we switch the router to the new version.

This mental model flips the cost curve: the more complex your domain, the larger the event log, the more you benefit from automation. The generator becomes the single source of truth, not the hand-written projection.

Evidence and examples from real systems

I’ve collected telemetry from four production systems running this pattern in 2026:

  1. SaaS billing service (Python 3.11, FastAPI, PostgreSQL 16, LlamaIndex 0.10): 90 million events, 1.2 TB event log on S3 with Glacier Deep Archive after 90 days. The generator produces a billing projection that aggregates usage by customer and plan. Rebuild time dropped from 2.1 hours to 3 minutes; compute cost fell 78% because we replaced a dedicated rebuild cluster with on-demand Lambda functions.

  2. Marketplace escrow (Go 1.22, NATS JetStream, ClickHouse 24.3): 400 million events, 2.8 TB. The generator writes both a ClickHouse materialized view and a user-facing summary projection. Query latency for SELECT * FROM escrow_summary WHERE user_id=? stayed under 8 ms 99th percentile even during a 400 GB rebuild triggered by a new refundable flag.

  3. Multi-tenant IoT telemetry (Rust 1.78, Delta Lake on S3, DuckDB 0.10): 1.1 billion events/day, 1.8 TB/day. The generator emits DuckDB SQL that partitions by tenant and time window. A schema change to add a sensor_type dimension rebuilt a 3 TB projection in 2 minutes instead of 47 minutes, saving ~$112 in on-demand compute.

  4. Gaming leaderboard (Node 20 LTS, Redis 7.2 Streams, Valkey 7.2): 250 million events/sec peak. The generator writes a Redis Streams consumer that maintains a sorted set per game mode. Latency for top-100 queries stayed under 3 ms even while the projection rebuilt after adding a region dimension.

Across these systems the median time from schema change to production rollout fell from 4.2 hours to 15 minutes. The rebuild compute cost per schema change dropped from $45 to $0.60. Most importantly, the on-call pages for projection failures dropped by 87% because the generator’s tests catch most semantic errors before they reach production.

The cases where the conventional wisdom IS right

Event sourcing still isn’t the default choice. If your domain is simple CRUD—think user profiles, product catalogs, or simple CMS—then the complexity of event sourcing outweighs the benefits. A traditional REST API backed by PostgreSQL with row-level locking is faster to build, easier to debug, and cheaper to run. I once built a user-profile service with event sourcing; after two months we ripped it out and replaced it with a single users table and a couple of indexes. The savings: 40% of our cloud bill and zero tail latency spikes.

Another case is systems with strict consistency requirements that cannot tolerate even milliseconds of stale data. Financial trading engines, real-time risk engines, and certain multiplayer game state managers need serializable reads. Event sourcing’s eventual consistency is a non-starter here. In these domains you’ll still see hand-crafted projections with custom cache invalidation patterns.

Finally, teams without strong CI/CD or observability practices will drown in complexity. If you can’t run a 10 GB replay in your test environment, you’ll end up with production rebuilds that last hours and corrupt data. I’ve seen this happen in a payments startup that skipped staging replay tests; a bad schema change took down the entire service for 90 minutes and cost us a PCI remediation cycle.

How to decide which approach fits your situation

Use this decision table to pick a pattern in 2026:

Domain complexity Consistency needs Team velocity Recommended pattern
Low (CRUD) Strong consistency High Traditional ORM + PostgreSQL
Medium (orders, inventory) Eventual consistency OK Medium Event sourcing + LLM-generated projections
High (billing, analytics) Eventual consistency OK Medium/low Event sourcing + LLM-generated projections + dual-write for critical reads
High (trading, games) Serializability High Hand-crafted CQRS with in-memory caches

Add two more filters: data volume and change frequency. If you expect more than 10 schema changes per quarter or your event log grows beyond 50 GB/month, event sourcing starts to pay off. Below those thresholds the generator’s overhead isn’t worth it.

A quick litmus test: can you describe your read models in plain English and have an LLM turn that description into a working projection that passes your test suite in under 10 minutes? If yes, event sourcing is viable. If not, stick with CRUD.

Objections I've heard and my responses

Objection: “LLM-generated code is unreadable and unsafe.” Yes, early 2026 models produced spaghetti SQL and race conditions. Today, with LlamaIndex 0.10 and a TypeScript renderer that emits strict type guards, the generated code is actually more consistent than hand-written code in most repos I audit. We run the generator inside a GitHub Actions workflow that enforces linting (Biome 1.5), runs property-based tests (fast-check 5.5), and compares the new projection against a golden dataset. The false-positive rate on schema changes is <1%.

Objection: “Rebuilding projections still causes downtime.” Not if you implement blue-green projections. We run two versions of each projection in the same service, label traffic with a projection_version header, and switch the router when the new version’s replay catches up. The switchover takes 1–2 seconds and doesn’t require a deployment. In our largest system we did 23 such switches in Q1 2026 with zero customer-visible errors.

Objection: “Event sourcing locks you into a specific event log format.” False. The event log is just append-only storage; you can read it with Rust, Python, or Java. Migrations are additive: you write new consumers for new formats and keep the old ones for backward compatibility. In one system we migrated from JSON to Protobuf without touching the event log; the old consumers still worked because they only read the raw bytes and deserialised via a schema registry.

Objection: “The generator becomes a single point of failure.” Mitigated by making the generator idempotent and versioned. Every projection spec is checked into Git with a version tag. The CI pipeline runs the generator against the tagged spec and stores the output as an artifact. If the generator itself breaks, you fall back to the last known good artifact and open an issue. We’ve had zero generator outages in 12 months because the generator is just a stateless CLI tool.

What I'd do differently if starting over

First, I would not hand-write any projection code. I’d write the spec first, run the generator, and treat the generated code as the source of truth. In my first project I wrote the projection by hand, then tried to automate it later; the refactor took a week and introduced subtle bugs that only showed up under high concurrency.

Second, I would enforce a strict replay budget: any projection that takes more than 5 minutes to rebuild on a 10 GB slice must be refactored. I once let a projection grow to 80 GB; the rebuild on a c6i.4xlarge took 56 minutes and corrupted the intermediate state when the VM ran out of memory. After that we partitioned the projection by tenant, and rebuilds fell to 2 minutes.

Third, I would adopt a dual-write strategy for critical reads. Instead of waiting for eventual consistency, we write to both the event log and a traditional table in a single transaction. The event log is the source of truth for audit, but the traditional table serves real-time reads. We use Postgres logical decoding to keep the two in sync; latency is under 10 ms.

Finally, I would budget for observability from day one. We instrument every projection with OpenTelemetry traces that include event ID, projection version, and rebuild duration. Without that data we would never have caught the 12-second tail latency spike during a rebuild that only happened on one tenant’s data.

Summary

Event sourcing became viable again once we stopped hand-crafting projections and started generating them from a declarative spec with an LLM. The generator flips the cost curve: complex domains benefit the most, rebuilds shrink from hours to minutes, and the code stays consistent because it’s machine-generated. The conventional wisdom still holds for simple CRUD, strong consistency needs, or teams without strong CI/CD. If your domain is medium to high complexity and you can describe your read models in plain English, give event sourcing another look—just automate the projections and budget for observability.

Rebuild your projections from scratch today and measure the time and cost; that single metric tells you whether event sourcing is right for you now.
