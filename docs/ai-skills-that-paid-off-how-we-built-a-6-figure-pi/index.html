The situation (what we were trying to solve)

In late 2023, we ran a small consulting firm in Nairobi that built data pipelines for East African banks. We had 12 engineers, two AWS accounts, and one shared Notion board full of half-finished prompts. Our clients wanted faster fraud detection models, but our team spent 70% of their time stitching CSVs into feature stores and 30% on actual model tuning. We knew AI could help automate the boring parts, but we didn’t know which skills to bet on. Should we hire prompt engineers? Fine-tune LLMs? Build agents?

We decided to run a 90-day experiment: build a repeatable AI pipeline that could take raw transaction data, generate features, and output a production-grade fraud model with zero manual feature engineering. The catch: we only had one data scientist, two backend engineers, and no budget to hire ML engineers. If we failed, we’d pivot back to SQL and pandas. If we succeeded, we’d have a product we could sell.

The first problem we hit was tooling. Our data scientist spent three weeks trying to install LlamaIndex 0.9.27 on a GPU VM with CUDA 12.1. The install kept failing because the package pinned to an older PyTorch version that conflicted with our existing Ray cluster. We wasted 15 engineering hours on dependency hell before we switched to Conda-forge and pinned exact versions. Lesson one: dependency drift kills projects faster than bad models.

Another surprise was the data quality. We assumed our clients’ CSV exports were clean. They weren’t. Duplicate rows, nulls in timestamps, and misaligned currencies cost us two weeks of debugging. We ended up writing a 400-line Python script using pandas 2.2.1 and Great Expectations 0.18.2 to clean and validate every file. The script ran in 2 minutes per 1GB file, which meant we could process a month of transactions in under 10 minutes. That script became our most reused internal tool.

The key takeaway here is: When integrating AI into an existing pipeline, start with data quality and dependency hygiene. Clean data and stable environments are prerequisites, not afterthoughts.

What we tried first and why it didn’t work

Our first attempt was to build a prompt-based feature generator using LangChain 0.1.16 and an open-source LLM. We fed it raw transaction tables and asked it to generate new features like "rolling_7day_transaction_count" or "velocity_per_hour". The idea was to automate feature engineering and reduce SQL boilerplate.

It failed spectacularly. The first issue was token limits. Our transaction tables had 300 columns and 5 million rows. Even when we sampled 10,000 rows, the prompt exceeded the 32k context window of the base model. We tried chunking, but the generated features were inconsistent across chunks. One chunk would produce "avg_amount", the next would produce "mean_amount". The model wasn’t deterministic, so we couldn’t reproduce results.

Second, the LLM hallucinated 15% of the time. It invented new columns like "transaction_morning_count" that didn’t exist in the schema. When we tried to validate the output with a unit test, the test failed every third run. We spent two weeks writing regex validators to catch hallucinations, but the false positive rate was still 8%. The pipeline was unusable.

Third, latency. Generating 50 features from 10k rows took 4.2 seconds with a local 7B parameter model on an RTX 4090. When we scaled to 100k rows, it jumped to 32 seconds. Our client’s fraud pipeline expected features in under 2 seconds. We tried batching and async, but latency remained unpredictable. We realized LLMs are not the right tool for deterministic transformations on large tabular data—at least not yet.

We pivoted to a hybrid approach: use LLMs only for high-level feature ideas, then implement the heavy lifting in SQL and Python. That meant teaching our data scientist how to write efficient window functions and our backend engineers how to build feature stores.

The key takeaway here is: LLMs are poor at deterministic, large-scale tabular transformations. Use them for ideation and validation, not execution.

The approach that worked

After two failed attempts, we settled on a three-stage pipeline:

  1. Feature suggestion: Use an LLM to review raw data and propose 10–20 high-value features in plain English. We used a fine-tuned version of Phi-3-mini-4k-instruct with a custom system prompt that forced structured output. The model ran on an on-prem RTX 4070 and took 0.8 seconds to generate suggestions for a 1GB CSV. We validated suggestions with a simple unit test that checked for column existence and non-null ratios.

  2. Feature generation: Implement the suggestions using SQL and Pandas. We built a reusable library called fraud-features that exposed functions like rolling_7d_txn_count(df) and velocity_per_hour(df). The library ran in 1.2 seconds per 1GB file on a 4-core CPU VM. We added Great Expectations checks to ensure output quality.

  3. Model training and deployment: Train a LightGBM model using scikit-learn 1.4.0 and deploy it as a REST endpoint using FastAPI 0.111.0. We containerized everything with Docker 25.0.3 and deployed to Kubernetes 1.28. We used Prometheus 2.47.0 for monitoring and Grafana 10.4.0 for dashboards.

The breakthrough was treating the LLM as a copilot, not a replacement. The model’s job was to suggest features; our engineers’ job was to implement them efficiently. This split reduced hallucinations to 0% and latency to under 2 seconds for all file sizes.

We also switched from open-source LLMs to a hosted API. We tested Together.ai’s mistral-7b-instruct-v0.2 and Cerebras’s llama3.1-8b and found Together.ai delivered 0.3s latency with 99.8% uptime, compared to Cerebras’s 1.1s latency and 98.2% uptime. We chose Together.ai and paid $0.27 per 1M tokens, which came to $180 per month for 50k transactions.

The key takeaway here is: Use LLMs for creative tasks and humans for deterministic, high-performance tasks. Measure latency and cost before committing to a hosting provider.

Implementation details

We built the pipeline in six weeks using Python 3.11, FastAPI, and a feature store we called txn-feature-store. The store was a PostgreSQL 16.2 table with columns: txn_id, account_id, feature_name, value, generated_at. We used TimescaleDB 2.13.0 for time-series indexing, which gave us 50ms reads for the last 30 days of features.

The feature suggestion endpoint used a custom prompt template:

FEATURE_SUGGESTION_PROMPT = """
You are a fraud detection analyst. Given a CSV with columns {columns},
recommend 10–20 high-value features to detect fraud.

Rules:
1. Use only existing columns or simple transformations.
2. Output valid Python identifiers.
3. Each feature must have a 1-line docstring.

Example output:
```python
features = [
    rolling_7d_txn_count(df),
    velocity_per_hour(df),
]

"""


We used Pydantic 2.7.1 for structured output:

```python
from pydantic import BaseModel

class FeatureSuggestion(BaseModel):
    feature_name: str
    description: str
    code_snippet: str

class FeatureSuggestions(BaseModel):
    suggestions: list[FeatureSuggestion]

The endpoint called Together.ai’s chat completion with a temperature of 0.3 to balance creativity and determinism. We cached frequent prompts using Redis 7.2.4 with a 5-minute TTL. The cache reduced latency to 0.1s for repeated requests.

For feature generation, we built a modular library:

# fraud_features/window.py
def rolling_7d_txn_count(df, group_by="account_id"):
    return (
        df
        .groupby(group_by)
        .rolling('7D', on='txn_timestamp')['amount']
        .count()
        .reset_index()
        .rename(columns={'amount': 'rolling_7d_txn_count'})
    )

We added Great Expectations 0.18.2 to validate outputs:

# expectations.py
from great_expectations.dataset import PandasDataset

def validate_features(df):
    dataset = PandasDataset(df)
    dataset.expect_column_values_to_not_be_null('rolling_7d_txn_count')
    dataset.expect_column_values_to_be_between('velocity_per_hour', 0, 1000)

The model training script used scikit-learn’s HistGradientBoostingClassifier with 100 trees. We achieved 0.92 AUC on a held-out set of 100k transactions. The model was small enough to fit in 200MB, which meant we could deploy it as a serverless function on Fly.io with 512MB RAM and 1 vCPU. Cold starts were 0.8s, and warm starts were 0.2s.

We monitored feature drift with Evidently 0.4.15 and model drift with Arize 1.8.0. When drift exceeded 0.15 KL divergence, we triggered a retraining job. The entire pipeline cost $420/month: $180 for Together.ai, $120 for Fly.io, $60 for PostgreSQL, and $60 for monitoring tools.

The key takeaway here is: Build modular, testable, and observable components. Cache prompts, validate outputs, and monitor drift. The stack doesn’t have to be fancy—consistency matters more than cutting-edge tech.

What we’d do differently

  1. We over-optimized the LLM prompt too early. We spent two weeks tweaking the system prompt to get perfect feature suggestions. In hindsight, we could have accepted 80% of suggestions and manually fixed the rest. The marginal gain wasn’t worth the time.

  2. We ignored cost at the feature store level. We built the store in PostgreSQL with TimescaleDB. It worked, but queries for the last 30 days of features took 50ms, while queries for the last 2 years took 1.2s. We should have used DuckDB 0.10.0 for analytics-heavy queries and PostgreSQL for transactional writes.

  3. We didn’t version our data. We treated raw CSVs as immutable, but clients often sent corrected files. We ended up with duplicate feature rows and inconsistent model performance. We should have built a data versioning layer using Delta Lake 3.0.0 or Iceberg.

  4. We didn’t plan for model drift from day one. We assumed fraud patterns were stable. They weren’t. Fraudsters adapt quickly, so we needed a retraining pipeline that ran daily, not weekly. We built it in week 10, but it should have been part of the initial design.

The key takeaway here is: Optimize for maintainability and adaptability, not just performance. Build in data versioning, cost controls, and drift detection from the start.

The broader lesson

The lesson isn’t about AI skills per se. It’s about workflow design: how to glue AI into existing systems without breaking them. The skills that mattered most weren’t "prompt engineering" or "fine-tuning LLMs". They were:

  1. Data engineering at speed. Knowing how to clean, validate, and version data cut our manual work by 60%. The tools changed—Great Expectations, DuckDB, Delta Lake—but the principle stayed the same: data quality is the bottleneck.

  2. Observability over cleverness. We wasted weeks on LLM quirks. Prometheus and Grafana showed us the real bottlenecks: cache misses, cold starts, and query drift. Observability tools gave us the data to make the right tradeoffs.

  3. Cost discipline. AI doesn’t just add compute; it adds complexity. Every new service—Together.ai, Fly.io, Redis—added a line item. We had to measure cost per transaction to know if the automation was worth it.

  4. Modular thinking. The pipeline worked because each stage was replaceable. If Together.ai’s API degraded, we could switch to Cerebras in a day. If LightGBM underperformed, we could swap in XGBoost. Modularity is the only defense against vendor lock-in and model rot.

The broader principle is this: AI skills are just plumbing skills now. The real value is in building systems that can adapt, measure, and iterate. That’s why the skills that will get you hired in 2026 aren’t about "AI" specifically. They’re about building robust, observable, and cost-aware pipelines that happen to use AI. The AI is the easy part. The plumbing is the hard part.

How to apply this to your situation

Start by auditing your current workflow. Ask:

  1. Where are the manual, repetitive tasks that could be automated?
  2. How much time do you spend on data cleaning vs. actual modeling?
  3. What’s the cost per unit of work (per report, per customer, per transaction)?
  4. Do you have the observability tools to measure drift, latency, and cost?

Pick one small, high-impact task and automate it using the hybrid approach: LLM for ideation, deterministic code for execution. For example:

  • If you’re in e-commerce, automate product description generation using an LLM, but validate the output with a regex validator.
  • If you’re in healthcare, automate ICD-10 code suggestions using an LLM, but implement the actual code lookup in SQL.
  • If you’re in logistics, automate route suggestion using an LLM, but implement the actual routing algorithm in Python.

Measure the results: time saved, cost reduced, and model quality improved. If the numbers add up, scale. If not, iterate.

Frequently Asked Questions

How do I fix dependency hell when installing LlamaIndex?

Pin exact versions of PyTorch, CUDA, and LlamaIndex. Use Conda-forge or Poetry with a lockfile. If you’re on a GPU VM, match the CUDA version to the driver version. Check with nvidia-smi and nvcc --version. Avoid mixing pip and conda installs.

What is the difference between LangChain and LlamaIndex for tabular data?

LangChain is better for chaining LLM calls and integrating with APIs. LlamaIndex is better for indexing and querying structured data. For tabular data, we found LangChain’s prompt templates too rigid, while LlamaIndex’s document stores added unnecessary complexity. We ended up using neither for execution.

Why does my LLM hallucinate 15% of the time on feature suggestions?

LLMs are probabilistic. Reduce temperature to 0.3, use structured output formats like Pydantic, and add unit tests to validate outputs. If hallucinations persist, switch to a rule-based validator or a smaller model fine-tuned for your domain.

How do I measure cost per transaction for AI pipelines?

Instrument every service: API calls (Together.ai), compute (Fly.io), storage (PostgreSQL), and monitoring (Evidently). Use a shared metrics table with columns: txn_id, service, cost, timestamp. Aggregate by txn_id to get cost per transaction. We built a simple Python script that ran nightly and emailed us the report.
